# Claude Code Ultimate Guide - Контекст проекту

## Призначення

Цей репозиторій є **вичерпною документацією для Claude Code** (CLI-інструменту від Anthropic). Він вчить користувачів ефективно використовувати Claude Code за допомогою посібників, прикладів та шаблонів.

**Мета-примітка**: Оскільки цей репо документує Claude Code, його власна конфігурація має бути зразковою.

## Ключові файли

| Файл | Призначення |
|------|---------|
| `VERSION` | Єдине джерело істини для версії (наразі 3.40.0) |
| `guide/ultimate-guide.md` | Головний довідник (шукайте тут в першу чергу) |
| `guide/cheatsheet.md` | Швидка довідка для щоденного використання |
| `machine-readable/reference.yaml` | Оптимізований для LLM індекс із номерами рядків |
| `CHANGELOG.md` | Усі зміни з детальними описами |

## Команди

### Управління версіями
```bash
# Перевірка узгодженості версій у всіх документах
./scripts/sync-version.sh --check

# Виправлення невідповідностей версій (оновлення з файлу VERSION)
```

### Слеш-команди (Обслуговування)

Кастомні слеш-команди, доступні в цьому проекті:

| Команда | Опис |
|---------|-------------|
| `/release <bump-type>` | Реліз версії посібника (CHANGELOG + VERSION + sync + commit + push) |
| `/update-infos-release [bump-type]` | Оновлення трекінгу релізів Claude Code + опціональне підняття версії посібника |
| `/version` | Відображення поточної версії посібника та Claude Code зі статистикою |
| `/changelog [count]` | Перегляд останніх записів CHANGELOG (за замовчуванням: 5) |
| `/sync` | Перевірка статусу синхронізації посібника та лендингу |
| `/audit-agents-skills [path]` | Аудит якості агентів, скілів та команд у конфігу .claude/ |
| `/security-check` | Швидка перевірка конфігу за базою даних відомих загроз (~30с) |
| `/security-audit` | Повний 6-фазний аудит безпеки з оцінкою /100 (2-5хв) |
| `/update-threat-db` | Дослідження та оновлення бази даних розвідки загроз |

**Приклади:**
```
/release patch                 # Bump patch + release (3.20.4 → 3.20.5)
/release minor                 # Bump minor + release (3.20.4 → 3.21.0)
/update-infos-release          # Тільки оновлення CC релізів
/update-infos-release patch    # Оновлення CC + bump посібника (3.9.11 → 3.9.12)
/update-infos-release minor    # Оновлення CC + bump посібника (3.9.11 → 3.10.0)
/version                       # Показати версії та статистику контенту
/changelog 10                  # Останні 10 записів CHANGELOG
/sync                          # Перевірка статусу синхронізації
/audit-agents-skills           # Аудит поточного проекту
/audit-agents-skills --fix     # Аудит + пропозиції виправлень
/audit-agents-skills ~/other   # Аудит іншого проекту
/security-check                # Швидке сканування на відомі загрози
/security-audit                # Повний аудит з оцінкою безпеки /100
/update-threat-db              # Дослідження + оновлення threat-db.yaml
```

Ці команди визначені в `.claude/commands/` та автоматизують:
- Відстеження релізів Claude Code (YAML + Markdown + бейдж на лендингу)
- Управління версією посібника (файл VERSION + синхронізація у всіх доках)
- Оновлення CHANGELOG
- Перевірка синхронізації лендингу
- Git commit та push в обидва репозиторії

### Конвенції найменування команд

Неявні префікси, що використовуються в `.claude/commands/`:

| Префікс | Патерн | Приклади |
|--------|---------|---------|
| `audit-*` | Перевірки якості з бальною оцінкою | `audit-agents-skills`, `audit-deps` |
| `update-*` | Синхронізація або оновлення даних із зовнішнього джерела | `update-infos-release`, `update-threat-db` |
| `security-*` | Сканування безпеки, зростаюча глибина | `security-check` (швидко), `security-audit` (повністю) |
| *(без префіксу)* | Основні команди воркфлоу посібника | `release`, `sync`, `version`, `changelog` |

При додаванні нової команди обирайте префікс, що відповідає типу дії. Уникайте створення нових категорій префіксів, якщо жоден із чотирьох існуючих не підходить.

## Поведінкові правила

Ці правила виведені з виявлених патернів тертя під час реальних сесій у цьому репо.

### Завжди оновлюйте CHANGELOG.md
Після будь-якої модифікації файлу або впровадження функції, оновлюйте `CHANGELOG.md` у розділі `[Unreleased]`. Ніколи не пропускайте цей крок, якщо не отримано іншої вказівки. Це найпоширеніший крок, який забувають.

### Будьте вичерпними при першому проході
Коли вас просять проаналізувати, провести аудит або рев'ю будь-чого — прочитайте кожен відповідний файл. Не робіть поверхневого сканування. Якщо не впевнені в обсязі, краще запитайте, ніж надавайте неглибокі результати. Це стосується оцінки ресурсів, аудитів документів та оглядів кодобази.

### Використовуйте абсолютні шляхи
При посиланні на файли в документації, звітах або оцінках ресурсів завжди використовуйте повні абсолютні шляхи. Ніколи не використовуйте відносні шляхи.

### Фінальний чек-лист
Після виконання всіх запитаних завдань завжди підтверджуйте самостійно:
1. Змінені файли (перелічіть їх)
2. CHANGELOG.md оновлено
3. Закомічено та запушено (якщо застосовно) — вкажіть хеш коміту

### Схильність до дії
Не витрачайте надто багато часу на цикли дослідження або планування. Створюйте файли та конкретні результати на ранніх етапах, потім ітеруйте. Якщо ви застрягли на будь-якому кроці більше ніж на 2 спроби, поясніть причину замість зациклювання.

## Модель конфігурації

**Рекомендований режим**: `/model opusplan`

**Обґрунтування**: Цей репозиторій документації отримує вигоду від гібридного інтелекту:
- **Фаза планування** (Opus + thinking): архітектурні рішення, синтез досліджень, багатофайловий аналіз.
- **Фаза виконання** (Sonnet): оновлення документів, синхронізація версій, редагування шаблонів, форматування.

**Воркфлоу OpusPlan**:
1. `/model opusplan` → встановлення гібридного режиму.
2. `/plan` або `Shift+Tab × 2` → планування з Opus (thinking увімкнено).
3. `Shift+Tab` → виконання з Sonnet (швидше, дешевше).

**Типовий розподіл завдань**:
| Тип завдання | Модель | Обґрунтування |
|-----------|-------|---------------|
| Редагування доків, виправлення помилок | Sonnet | Прямолінійно, без глибоких міркувань |
| Синхронізація версій, форматування | Sonnet | Механічне зіставлення патернів |
| Реструктуризація посібника | Opus (plan) → Sonnet (execute) | Потребує архітектурного мислення |
| Синтез досліджень | Opus (plan) → Sonnet (write) | Складний аналіз, потім чіткий текст |
| Перевірка цілісності декількох файлів | Opus (plan) → Sonnet (fix) | Аналіз залежностей, потім редагування |

**Оптимізація витрат**: OpusPlan оплачує Opus лише за планування (зазвичай 10-20% токенів), Sonnet виконує 80-90% роботи.

## Воркфлоу оцінки ресурсів

Зовнішні ресурси (статті, відео, обговорення) оцінюються перед інтеграцією в посібник.

### Процес

1. **Дослідження**: Початковий пошук Perplexity → Збереження промпту + результатів у `claudedocs/resource-evaluations/` (приватно).
2. **Оцінка**: Систематичне нарахування балів (1-5) → Створення файлу оцінки в `docs/resource-evaluations/` (відстежується).
3. **Виклик (Challenge)**: Технічне рев'ю агентом для забезпечення об'єктивності.
4. **Рішення**: Інтеграція (бал 3+), згадка (бал 2) або відхилення (бал 1).

### Організація файлів

| Розміщення | Контент | Відстеження |
|----------|---------|----------|
| `docs/resource-evaluations/` | Фінальні оцінки (151 файл) | ✅ Git tracked (публічно) |
| `claudedocs/resource-evaluations/` | Робочі документи, промпти, приватні аудити | ❌ Gitignored (приватно) |

### Гістограма оцінювання

| Бали | Дія |
|-------|--------|
| 5 | Критично - Інтегрувати негайно (<24г) |
| 4 | Висока цінність - Інтегрувати протягом 1 тижня |
| 3 | Помірно - Інтегрувати при наявності часу |
| 2 | Гранично - Мінімальна згадка або пропуск |
| 1 | Низько - Відхилити |

Дивіться повну методологію: [`docs/resource-evaluations/README.md`](docs/resource-evaluations/README.md)

## Швидкий пошук

Для надання відповідей на запитання про Claude Code:
0. **Офіційна дока Anthropic (оптимізовано для LLM)**: `https://code.claude.com/docs/llms.txt` (індекс ~65 сторінок) або `https://code.claude.com/docs/llms-full.txt` (повна дока ~98KB) для офіційних фактів.
1. Спочатку шукайте в `machine-readable/reference.yaml` (має номери рядків для повного посібника).
2. Використовуйте ці номери рядків, щоб прочитати відповідні розділи з `guide/ultimate-guide.md`.
3. Перевірте `examples/` на наявність готових шаблонів.
4. Перевірте `guide/core/claude-code-releases.md` на предмет недавніх функцій/змін.
