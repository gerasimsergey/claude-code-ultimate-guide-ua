# Claude Code — Для техлідів та Engineering Managers

> Ви напевно чули, як ваші розробники обговорюють Claude Code. Можливо, дехто вже використовує його. Ця сторінка для вас — людини, відповідальної за те, щоб це впровадження було послідовним, безпечним та масштабованим на рівні всієї команди.

---

## Основна проблема

Якщо залишити все на самоплив, кожен розробник будує власну конфігурацію Claude Code. Різні файли `CLAUDE.md`, різні налаштування дозволів, відсутність спільних хуків та прозорості. Приріст продуктивності є реальним, але хаотичним, а поверхня ризиків безпеки зростає безконтрольно.

Команда зі спільною конфігурацією у 3-5 разів ефективніша за ту саму групу розробників, які працюють з індивідуальними налаштуваннями.

## 3 речі, які варто зробити цього тижня

**1. Додайте CLAUDE.md у свій репозиторій**
Створіть `CLAUDE.md` у корені вашого основного репозиторію. Він автоматично застосовуватиметься до всіх членів команди. Почніть із конвенцій кодування, архітектурних рішень та правил "ніколи не робити X".

**2. Визначте одного "чемпіона"**
Знайдіть інженера, який вже ефективно працює з Claude Code. Дайте йому час задокументувати свою конфігурацію та провести 1-годинну сесію для команди.

**3. Додайте один хук безпеки**
Мінімум: pre-tool хук, який блокує запис у файли `.env` та шляхи `**/secrets/**`. Налаштування займає 10 хвилин, але закриває реальний вектор загрози.

```bash
# Приклад: hooks/block-sensitive-files.sh
if [[ "$TOOL_INPUT_PATH" =~ \.env$|secrets/ ]]; then
  echo "BLOCKED: sensitive file path"
  exit 2
fi
```
Дивіться [Розділ 7.4 — Хуки безпеки](../guide/ultimate-guide.md#74-security-hooks) для повного набору.

## Огляд стану безпеки

Цей посібник містить **єдину публічну базу загроз для Claude Code**: закаталоговано 15 вразливостей та 655 шкідливих скілів. Ключові ризики для команд:

- **Промпт-ін'єкції** через неперевірений вміст файлів або MCP сервери.
- **Надмірно вільні налаштування** — наприклад, `allowedTools: ["*"]` у продакшені.
- **Неперевірені MCP сервери** — ставтеся до них як до npm-пакетів (ризик ланцюжка поставок).
- **Відсутність аудиту** — хто, що і коли робив.

## Навчання вашої команди

Якщо вам потрібен структурований онбординг замість самостійного навчання:

- **Brown Bag Lunch або семінар (1-3 год, безкоштовно)** — вступна сесія, демо, Q&A.
- **Аудит конфігурації** — огляд поточної системи на відповідність стандартам безпеки та продуктивності.
- **Формування команди (1-2 дні)** — практична робота на вашій кодобазі та воркфлоу.
