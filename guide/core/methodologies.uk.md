## Карта методологій

Де кожна методологія знаходиться на двох осях: **Спочатку Специфікація проти Спочатку Код** (Y) та **Легкий/Solo проти Корпоративний/Керований** (X).

```
                     СПОЧАТКУ СПЕЦИФІКАЦІЯ / ПЛАНУВАННЯ
                                 ▲
   ── легка · специф. ──         │         ── керована · специф. ──
                                 │
   [Doc-Driven]  [SDD]           │    [BDD]  [ATDD]   [Req-Driven]
   [GSD]  [Plan-First]           │ [CDD] [ADR-Driven]  [DDD]  [BMAD]
                                 │
   LEAN / SOLO ──────────────────┼───────────────────────────────► ENTERPRISE
                                 │
   ── легкий · код ──            │         ── керований · код ──
                                 │
   [Context Eng.]   [TDD]        │       [Multi-Agent]
   [Prompt Eng.]  [Iterative]    │       [Eval-Driven]       [FDD]
   [Ralph Loop]                  │           [JiTTesting]
                                 │
                            КОД / ЕМЕРДЖЕНТ
```

**Як це читати:**

-   **Вгорі ліворуч** — Легкі підходи зі специфікацією: `SDD`, `Doc-Driven`, `Plan-First`. Природна точка входу для соло-розробників та малих команд, які хочуть відійти від підходу "спочатку код".
-   **Вгорі праворуч** — Керовані підходи зі специфікацією: `BMAD`, `Req-Driven`, `ATDD`, `DDD`. Справжнє управління, але дорого в налаштуванні. ROI залежить від складності проекту та стабільності вимог, а не лише від кількості людей.
-   **Внизу ліворуч** — Легкі підходи через код: природна територія Claude Code. `TDD` + `Ralph Loop` + `Iterative` = ядро соло-роботи.
-   **Внизу праворуч** — Код-орієнтовані підходи у масштабі: `Multi-Agent`, `Eval-Driven`, `JiTTesting` (Meta, 100M+ рядків коду). Нові патерни для команд з великим обсягом роботи.
-   **На осях** — `Plan-First`, `CDD`, `ADR-Driven`, `GSD`: гібридні підходи, що адаптуються до будь-якого контексту.

## 15 методологій

Організовані у 6-рівневу піраміду від стратегічної оркестрації до технік оптимізації.

### Рівень 1: Стратегічна оркестрація

| Назва | Що це | Найкраще для | Сумісність з Claude |
|------|------|----------|------------|
| **BMAD** | Управління багатьма агентами через "Конституцію" | Складні проекти зі стабільними вимогами | ⭐⭐ Нішево, але потужно |
| **GSD** | 6-фазний робочий процес з чистим контекстом для кожного завдання | Соло-розробники, Claude Code CLI | ⭐⭐⭐ Схоже на патерни в гайді |

**BMAD (Breakthrough Method for Agile AI-Driven Development)** перевертає традиційну парадигму: документація стає джерелом істини, а не код. Використовує спеціалізованих агентів (Аналітик, ПМ, Архітектор, Розробник, QA), оркестрованих за суворими правилами.

-   **Ключова концепція**: `Constitution.md` як стратегічний обмежувач.
-   **Коли використовувати**: Складні корпоративні проекти, що потребують нагляду.
-   **Коли уникати**: MVP, швидке прототипування — BMAD стає крихким, коли вимоги часто змінюються.

**GSD (Get Shit Done)** бореться з "гниттям контексту" через систематичний 6-фазний процес (Ініціалізація → Обговорення → План → Виконання → Перевірка → Завершення) з новими сесіями по 200 тис. токенів на кожне завдання.

> **Нове**: [Ralph Inferno](https://github.com/sandstream/ralph-inferno) реалізує автономні робочі процеси з багатьма ролями (Аналітик→ПМ→UX→Архітектор→Бізнес) з виконанням у VM та циклами самокорекції.

### Фундаментальна дисципліна: Спочатку План (Plan-First)

> **"Коли план хороший — код хороший."**
> — Борис Черний, творець Claude Code

**Це не просто функція (команда `/plan`) — це систематична дисципліна.**

**Ментальна модель**:
Планування не є опціональним для складних завдань. Це різниця між:
- ❌ 8 ітераціями "спробував → виправив → ще раз виправив"
- ✅ 1 ітерацією "спланував → перевірив → чисто виконав"

**Коли планувати спочатку**:

| Складність завдання | Планувати? | Чому |
|----------------|-------------|-----|
| Змінено >3 файлів | ✅ Так | Залежності між файлами потребують архітектури |
| Змінено >50 рядків | ✅ Так | Достатньо складно, щоб помилитися |
| Архітектурні зміни | ✅ Так | Потрібен аналіз впливу |
| Незнайомий код | ✅ Так | Потрібне дослідження перед дією |
| Очевидна помилка | ❌ Ні | Витрати на план > час на завдання |

**Як працює Plan-First**:

1.  **Фаза дослідження** (Режим планування через `Shift+Tab` або `/plan`):
    - Claude читає файли, вивчає архітектуру.
    - Редагування заборонено → це змушує думати перед дією.
    - Пропонує підхід з описом компромісів.
2.  **Фаза валідації** (ви перевіряєте):
    - План викриває припущення та прогалини.
    - Легше змінити напрямок зараз, ніж після 100 написаних рядків.
3.  **Фаза виконання** (перехід у Normal Mode):
    - План стає контрактом для виконання.
    - Менше сюрпризів, чистіша реалізація.

### Рівень 2: Специфікація та Архітектура

| Назва | Що це | Найкраще для | Сумісність |
|------|------|----------|------------|
| **SDD** | Специфікації перед кодом | API, контракти | ⭐⭐⭐ Ядро системи |
| **Doc-Driven** | Документація = джерело істини | Узгодження команд | ⭐⭐⭐ Нативно для CLAUDE.md |
| **Req-Driven** | Багатий контекст (20+ артефактів) | Складні вимоги | ⭐⭐ Важке налаштування |
| **DDD** | Спочатку доменна мова | Бізнес-логіка | ⭐⭐ Етап дизайну |

**SDD (Spec-Driven Development)** — Специфікації ПЕРЕД кодом. Одна добре структурована ітерація дорівнює восьми неструктурованим. Файл `CLAUDE.md` — це ваша специфікація.

**DDD (Domain-Driven Design)** — Узгоджує ПЗ з мовою бізнесу через "Єдину мову" (Ubiquitous Language) та Обмежені контексти (Bounded Contexts).

---

### Рівень 3: Поведінка та Прийняття

| Назва | Що це | Найкраще для | Сумісність |
|------|------|----------|------------|
| **BDD** | Сценарії "Дано-Коли-Тоді" | Співпраця з бізнесом | ⭐⭐⭐ Тести та специ |
| **ATDD** | Критерії прийняття спочатку | Регульовані галузі | ⭐⭐ Багато процесів |
| **CDD** | API-контракти як інтерфейс | Мікросервіси | ⭐⭐⭐ OpenAPI |

**BDD (Behavior-Driven Development)** — Сценарії у форматі:
```gherkin
Scenario: Неможливо купити без наявності
  Given товар з 0 залишку
  When клієнт намагається купити
  Then система видає помилку
```

**JiTTesting (Just-in-Time Testing)** — Тести, що генеруються "на льоту" під час створення PR, перевіряють зміни і видаляються після мержу. Немає витрат на підтримку, немає роздування тестової бази.

---

### Рівень 4: Поставка функцій

| Назва | Що це | Найкраще для | Сумісність |
|------|------|----------|------------|
| **FDD** | Поставка по одній функції | Паралельні команди | ⭐⭐ Структура |
| **Context Eng.** | Контекст як першокласний об'єкт | Довгі сесії | ⭐⭐⭐ Фундаментально |

**Контекстна інженерія (Context Engineering)** — Ставтеся до контексту як до елемента дизайну: поступове розкриття інформації, керування пам'яттю, динамічне оновлення списку TODO.

### Рівень 5: Реалізація

| Назва | Що це | Найкраще для | Сумісність |
|------|------|----------|------------|
| **TDD** | Red-Green-Refactor | Якісний код | ⭐⭐⭐ Ядро воркфлоу |
| **Eval-Driven** | Евали для виходів ШІ | Продукти з ШІ | ⭐⭐⭐ Агенти |
| **Multi-Agent** | Оркестрація під-агентів | Складні завдання | ⭐⭐⭐ Інструмент Task |

**TDD (Test-Driven Development)** — Класичний цикл:
1. **Red**: Написати тест, що падає.
2. **Green**: Мінімальний код, щоб тест пройшов.
3. **Refactor**: Очистити код, тести залишаються зеленими.

**Цикли верифікації (Verification Loops)**: Дайте Claude механізм перевірки власного результату (браузер, тести, лінтери). Агент, який "бачить", що він зробив, працює в рази краще.

### ADR-Driven Development

**Патерн**: Пишіть ADR (Architecture Decision Records) англійською → Передайте скілу `implement-adr` → Виконуйте нативно.

**Переваги**:
- ✅ **Керованість документацією**: Архітектура та код синхронізовані.
- ✅ **Прозорість**: Чіткий слід від рішення до реалізації.

---

### Рівень 6: Оптимізація

| Назва | Що це | Найкраще для | Сумісність |
|------|------|----------|------------|
| **Iterative Loops** | Автономне вдосконалення | Оптимізація | ⭐⭐⭐ Ядро |
| **Fresh Context** | Скидання контексту на кожне завдання | Довгі сесії | ⭐⭐⭐ Power users |
| **Prompt Eng.** | Фундамент технік | Все | ⭐⭐⭐ База |

**Fresh Context Pattern (Ralph Loop)** — Вирішує проблему "гниття контексту" шляхом створення нових інстансів агента для кожного завдання. Стан зберігається у файлах та git, а не в історії чату.

## Написання ефективних специфікацій

На основі аналізу 2500+ конфігурацій агентів (джерело: Адді Османі):

1.  **Команди**: Точні рядки запуску (`npm test -- --coverage`).
2.  **Тестування**: Фреймворк, очікуване покриття, шлях до тестів.
3.  **Структура**: Явні директорії для коду та логів.
4.  **Стиль коду**: Один приклад кращий за десять абзаців опису.
5.  **Git воркфлоу**: Назви гілок та формат комітів.

**Прокляття інструкцій**: Дослідження показують, що **більше інструкцій = гірше їх виконання**. Рішення: подавайте Claude лише релевантні частини специфікації для конкретного завдання.

## Патерни комбінування

| Ситуація | Рекомендований стек | Примітки |
|-----------|-------------------|-------|
| Solo MVP | SDD + TDD | Мінімум накладних витрат, фокус на якості |
| Команда 5-10 осіб | Spec Kit + TDD + BDD | Управління + якість + співпраця |
| Мікросервіси | CDD + Specmatic | Спочатку контракт, паралельна розробка |
| Складні/Regulated проекти | BMAD + Spec Kit | Повний контроль та звітність |
