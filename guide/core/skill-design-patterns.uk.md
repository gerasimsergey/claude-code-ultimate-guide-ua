# Патерни проектування скілів

> **Пов'язано**: [Методології розробки](./methodologies.uk.md) | [Багатоагентна координація](../ultimate-guide.uk.md#920-agent-teams-multi-agent-coordination)

Практичні патерни для скілів, які виходять за межі простого промпту для одного агента. Ці патерни вирішують специфічні проблеми: субагенти, що повторно відкривають ті самі факти; правила, що застосовуються не до тих файлів; скіли, що змішують виявлення проблем із їх виправленням.

---

## Ін'єкція спільного джерела істини (Shared Ground Truth Injection)

**Проблема**: Коли ви запускаєте N паралельних субагентів для аудиту або аналізу набору артефактів, кожен агент незалежно виявляє ті самі базові факти (список файлів, структура навігації, команди CLI, схема). Це N надлишкових читань, N незалежних фактів, яким треба довіряти, і N шансів для одного з агентів побачити застарілий стан файлу.

**Патерн**: Оркестратор один раз обчислює спільну фактичну базу, а потім вставляє цей блок дослівно в промпт кожного субагента.

```
Оркестратор (Фаза 1)
  ├── Читання структури навігації → вилучення груп
  ├── Glob файлів → отримання поточного списку
  ├── Список команд CLI → отримання поточних команд
  └── Компіляція в рядок "Джерело істини" (Ground Truth)

Оркестратор (Фаза 2)
  ├── Промпт Агента 1: "## Ground Truth\n{спільний блок}\n\n## Ваша область\nСекція A"
  ├── Промпт Агента 2: "## Ground Truth\n{спільний блок}\n\n## Ваша область\nСекція B"
  └── Промпт Агента 3: "## Ground Truth\n{спільний блок}\n\n## Ваша область\nСекція C"
```

**Що входить у спільний блок**:
- Навігація або ієрархія файлів (щоб кожен агент знав, що існує)
- Поточні команди CLI або кінцеві точки API (щоб виявити посилання на видалені команди)
- Список доменних сутностей (пакети, модулі, сервіси)
- Поточна дата (щоб виявити посилання на версії, які вже застаріли)

**Чому це працює**: Кожен агент починає з одного і того ж знімка. Агент не може галюцинувати команду CLI, яка існує, якщо список джерела істини показує протилежне. Оркестратор є єдиним джерелом істини для структури; субагенти є єдиним джерелом істини для вмісту призначених їм секцій.

**Витрати токенів**: Спільний блок додає токени до промпту кожного субагента. Для паралельного аудиту з 5 агентами блок джерела істини на 500 токенів коштує 2500 токенів авансом. Усунення повторного виявлення фактів кожним агентом компенсує ці витрати з лишком. Обмежуйте розмір блоку тим, що дійсно необхідно.

**Примітка щодо реалізації**: Блок джерела істини має бути рядком, а не посиланням на файл. Якщо ви вкажете агентам "прочитати `docs.json`", кожен агент прочитає його незалежно. Скомпілюйте його один раз в оркестраторі та вставте текстом.

## Попередньо відфільтровані посилання через Frontmatter-шляхи

**Проблема**: У вас є набір файлів правил (стандарти кодування, політики безпеки, гайди по стилю). Кожне з них застосовується до специфічної підмножини файлів у коді. Якщо ви передасте всі правила агенту рев'ю, він застосовуватиме правила до файлів, для яких вони не були написані, створюючи помилкові спрацьовування (false positives) та витрачаючи контекст.

**Патерн**: Додайте поле `paths:` у frontmatter кожного файлу правил із шаблоном glob. Перед запуском агентів рев'ю оркестратор читає цей frontmatter, зіставляє шаблони зі списком змінених файлів і передає кожному агенту лише ті правила, які до нього застосовуються.

```yaml
# .claude/rules/standard-testing-good-practices.md
---
name: Найкращі практики тестування
paths: "**/*.spec.ts,**/*.test.ts"
alwaysApply: false
description: Конвенції тестування для юніт- та інтеграційних тестів на TypeScript
---

## Правила
- Використовуйте блоки `describe` для групування пов'язаних тестів
...
```

Логіка оркестратора (прості кроки):

```
1. Glob .claude/rules/**/*.md → список усіх файлів правил
2. Для кожного файлу правил:
   a. Читання YAML frontmatter → вилучення paths (шаблон glob)
   b. Якщо alwaysApply: true → включити безумовно
   c. Якщо alwaysApply: false → порівняти paths зі списком змінених файлів
   d. Включити, якщо хоча б один змінений файл відповідає glob-шаблону
3. Передати відфільтрований набір правил кожному агенту рев'ю
```

**Ключові переваги**:
- Тестовий файл не перевіряється на відповідність правилам безпеки бекенда.
- Файл міграції не перевіряється на відповідність гайдам стилю фронтенда.
- Контекст кожного агента містить лише дієві правила для того, що він перевіряє.

**Масштабованість**: При наявності 50 файлів правил типовий PR, що стосується 8 файлів специфікацій TypeScript, може відповідати 3 правилам. Без фільтрації агент читає 50 правил. З фільтрацією — лише 3. Співвідношення сигнал/шум для агента покращується пропорційно.

**Коли додавати `paths:` у frontmatter**: Будь-яке правило, яке має чітку область застосування (тести, міграції, фронтенд-компоненти, маршрути API). Правила, які є справді глобальними, використовують `alwaysApply: true` і пропускають крок зіставлення.

> Спостерігається у [скілі qa-review від Packmind](https://github.com/packmind/packmind) (.claude/skills/qa-review/SKILL.md, Фаза 5). Див. [Credits](./credits.md).

## Межа видимості: лише виявлення (Detection-Only Scope Boundary)

**Проблема**: Скіл, який одночасно виявляє проблеми та виправляє їх, має два типи збоїв: помилкові спрацьовування (виявив і виправив те, що не було проблемою) та неповні виправлення (виявив правильно, але виправив невірно). Змішування цих функцій погіршує обидві та позбавляє користувача точки перевірки (checkpoint).

**Патерн**: Чітко обмежте область дії скілів лише виявленням. Скіл створює звіт. Виправлення (remediation) — це окремий крок, який ініціюється користувачем після ознайомлення зі звітом.

Вкажіть це на початку скіла:

```markdown
**Цей скіл лише виявляє проблеми. Він не виправляє їх.**
```

Потім забезпечте виконання: ніяких записів у файли, редагувань чи комітів git всередині скіла. Результатом завжди є звіт у форматі markdown або знахідки в консолі.

**Коли порушувати це правило**: Винятком є скіли, спеціально розроблені для автоматизованого виправлення (скіли типу codemod, оновлення залежностей). Вони мають чітко вказувати, що вносять зміни, і обов'язково включати режим попереднього перегляду (dry-run).

**Практична цінність**: Скіли, орієнтовані лише на виявлення, безпечно запускати в main, у CI та проти продуктових конфігурацій. Користувач може викликати такий скіл, не побоюючись ненавмисних змін, що робить їх використання частішим.

> Повторюваний патерн у скілах qa-review, playbook-audit та doc-audit у [репозиторії Packmind](https://github.com/packmind/packmind). Див. [Credits](./credits.md).

## Диспетчеризація обробників вводу (Input-Handler Dispatch)

**Проблема**: Скіл, який обробляє два або більше гетерогенних типи вводу (зображення проти тексту, GitHub issues проти дизайн-макетів), або обростає складною логікою розгалуження в основному файлі, або стає занадто жорстким для різних точок входу.

**Патерн**: Зберігайте один файл обробника вводу на кожен тип вводу в піддиректорії `inputs/`. Основний SKILL.md запитує користувача, який тип застосовується, а потім завантажує відповідний файл обробника.

```
.claude/skills/my-skill/
├── SKILL.md                 # Оркестратор: запитує користувача, направляє до обробника
└── inputs/
    ├── github-issue.md      # Обробник для вводу з GitHub issue
    └── visual-mockup.md     # Обробник для скріншотів/зображень
```

SKILL.md (скорочено):

```markdown
## Крок 1: Визначення типу вводу

Запитайте користувача: "Ви надаєте GitHub issue чи візуальний макет?"

- Якщо GitHub issue → дотримуйтесь інструкцій у `inputs/github-issue.md`
- Якщо візуальний макет → дотримуйтесь інструкцій у `inputs/visual-mockup.md`
```

Кожен файл обробника містить повні інструкції з парсингу для цього типу вводу, включаючи вилучення полів, нормалізацію формату та валідацію. Основний файл залишається чистим.

**Коли використовувати**: Коли типи вводу вимагають суттєво різної логіки парсингу (а не просто різних назв полів). Якщо два типи вводу відрізняються менш ніж на 5 кроків, обробляйте їх інлайново. Патерн стає вигідним, коли кожен обробник перевищує 100 рядків.

> Спостерігається у [скілі create-em-spec від Packmind](https://github.com/packmind/packmind) (.claude/skills/create-em-spec/inputs/). Див. [Credits](./credits.md).

## Версіоновані піддиректорії для прив'язки до версії інструменту

**Проблема**: Скіл є обгорткою для CLI-інструменту, поведінка якого змінюється між версіями. Скілу потрібно визначити, яка версія у користувача, і відповідно адаптувати свої інструкції.

**Патерн**: Створіть по одній піддиректорії на кожну версію інструменту, кожна з яких містить специфічні для версії інструкції:

```
.claude/skills/update-playbook/
├── SKILL.md                         # Визначає версію CLI, виконує диспетчеризацію
└── tool-versions/
    ├── v1.21/
    │   └── apply-changes.md         # Інструкції для v1.21
    ├── v1.23/
    │   └── apply-changes.md         # Інструкції для v1.23 (інший API)
    └── v1.24/
        └── apply-changes.md         # Інструкції для v1.24 (критична зміна)
```

SKILL.md визначає версію під час виконання:

```markdown
## Крок 1: Визначення версії інструменту

Запустіть: `my-cli --version`

- Якщо вивід починається з "1.21" → прочитайте `tool-versions/v1.21/apply-changes.md`
- Якщо вивід починається з "1.23" → прочитайте `tool-versions/v1.23/apply-changes.md`
- Якщо вивід починається з "1.24" → прочитайте `tool-versions/v1.24/apply-changes.md`
- Якщо версію не розпізнано → зупиніться і повідомте користувачеві, які версії підтримуються
```

**Антипатерн, якого слід уникати**: Не створюйте директорію `my-skill-v2/` поруч із оригінальною `my-skill/`. Версіонуйте *вміст всередині* скіла, а не сам скіл. Два майже ідентичні скіли важче підтримувати, і вони плутають користувача при виклику.

**Коли використовувати**: Коли CLI-інструмент, який ви обгортаєте, має критичні зміни між версіями, і вам потрібно підтримувати кілька версій одночасно. Якщо вам потрібно підтримувати лише останню версію, просто оновіть скіл на місці.

> Спостерігається у [скілі update-playbook від Packmind](https://github.com/packmind/packmind) (.claude/skills/packmind-update-playbook/). Див. [Credits](./credits.md).

## Дворівневі стандарти (Two-Tier Standards)

**Проблема**: Повний стандарт кодування може бути дуже об'ємним (1,000–5,000 слів). Завантаження всього стандарту в контекст при кожному рев'ю файлу роздуває витрати токенів і розпорошує увагу ШІ.

**Патерн**: Зберігайте коротке резюме (100–300 слів, із glob-шаблоном у frontmatter `paths:`) у директорії `.claude/rules/`. Повний канонічний стандарт зберігайте в окремому місці, де Claude зможе прочитати його за запитом.

```
.claude/rules/standard-testing.md          # Резюме + glob шляхів (200 слів)
.standards/standard-testing-full.md        # Повний канонічний стандарт (2,000 слів)
```

Резюме в `.claude/rules/` включає:
- 3-5 обов'язкових правил (в наказовому способі: "Використовуйте X", "Не робіть Y")
- frontmatter `paths:`, щоб правило завантажувалося лише для відповідних файлів
- Посилання на повний стандарт для випадків, коли потрібна деталізація

Повний канонічний стандарт знаходиться поза межами `.claude/rules/`, тому він не завантажується автоматично.

**Коли використовувати**: Командам, у яких більше 10 файлів правил, або файли правил яких у середньому перевищують 500 слів. Для менших конфігурацій достатньо одного рівня.

**Підтримка**: При оновленні канонічного стандарту оновлюйте також і резюме. Розподіл на два рівні створює ризик дублювання. Мінімізуйте його, залишаючи в резюме лише пункти правил (без розлогого тексту), щоб воно змінювалося рідко.

> Спостерігається у [конфігурації правил Packmind](https://github.com/packmind/packmind) (.claude/rules/packmind/). Див. [Credits](./credits.md).

## Плани та специфікації як зафіксовані артефакти

**Проблема**: Плани в межах сесії (режим `/plan`, тимчасові файли) зникають після завершення сесії. Наступна сесія не має доступного для пошуку запису про те, чому було прийнято певне дизайнерське рішення, які варіанти розглядалися або який крок впровадження був завершений останнім.

**Патерн**: Фіксуйте (commit) плани та пов'язані з ними специфікації дизайну безпосередньо в директорію `.claude/` у вигляді датованих пар markdown-файлів:

```
.claude/
├── plans/
│   └── 2026-03-15-auth-refactor.md        # План: фази, завдання, повідомлення комітів
└── specs/
    └── 2026-03-15-auth-refactor-design.md  # Специфікація: контекст, альтернативи, обґрунтування
```

Файл плану містить деталізацію впровадження з чіткими повідомленнями комітів для кожного завдання:

```markdown
# План рефакторингу авторизації

## Завдання 1: Вилучення middleware валідації токенів
- Перенести логіку валідації з контролера в `middleware/auth.ts`
- git commit -m "refactor(auth): extract token validation into middleware"

## Завдання 2: Додавання ротації refresh-токенів
- Реалізувати логіку ротації з терміном дії 7 днів
- git commit -m "feat(auth): add refresh token rotation with 7-day expiry"
```

Специфікація фіксує те, на основі чого було складено план: обмеження, відхилені альтернативи та міркування, які не будуть очевидними з коду через півроку:

```markdown
# Дизайн рефакторингу авторизації

## Контекст
Зберігання токенів сесій позначено юридичним відділом як проблему (невідповідність ISO 27001).
Ротація обов'язкова згідно з новою внутрішньою політикою безпеки v2.3.

## Відхилені підходи
- Підхід з HttpOnly cookie: вимагає змін у крос-доменній конфігурації (відкладено)
- Сховище сесій у Redis: команда експлуатації не готова підтримувати ще один stateful сервіс
```

Обидва файли комітяться разом і залишаються в репозиторії назавжди.

**Чому це краще за плани лише на одну сесію**: Зафіксовані плани можна шукати (grep), порівнювати (diff) та продовжувати. Нова сесія може прочитати `.claude/plans/`, щоб зрозуміти поточну роботу, не реконструюючи її з логів git. Специфікація зберігає логіку, яка стирається з пам'яті, але стає важливою при перегляді рішення через шість місяців.

**Конвенція назв**: `РРРР-ММ-ДД-<slug>.md` для плану, `РРРР-ММ-ДД-<slug>-design.md` для специфікації. Це дозволяє візуально асоціювати пари в списку директорії.

**Коли використовувати**: Завдання впровадження на кілька сесій, де важлива безперервність. Не варто витрачати час на це для односесійного завдання, яке поміщається в один коміт. Порогове значення: якщо робота триває більше одного дня або більше однієї сесії Claude — фіксуйте план.

> Спостерігається у конвенції [.claude/plans/ від Packmind](https://github.com/packmind/packmind). Див. [Credits](./credits.md).

## Логування промптів під час виконання (Runtime Prompt Logging)

**Проблема**: Коли виклик провайдера ШІ завершується тайм-аутом або помилкою, точний промпт, який був надісланий, втрачається. Якщо ви створюєте скіл або конвеєр оцінки з кількома агентами, що працюють паралельно, ви втрачаєте можливість діагностувати, що саме було сказано кожному агенту. Прапорець `--debug` допомагає лише тоді, коли ви не забули його додати.

**Патерн**: Записуйте промпт на диск як блокуючу операцію ПЕРЕД викликом провайдера. Використовуйте постійну директорію з файлом, назва якого містить часову мітку. Ніколи не дозволяйте логуванню зупиняти основний процес.

```typescript
// Запускайте ПЕРЕД provider.invoke(), а не після
await writeFile(
  `prompts/debug/${evaluatorName}-${timestamp}.md`,
  fullPrompt,
  "utf-8",
);
// Лише тепер викликайте провайдера
const response = await provider.invoke(fullPrompt);
```

Три правила успішного використання цього патерна:

1. **Блокуючий запис** (`await`, а не "вистрілив і забув"): файл існує на диску до початку виклику провайдера. Помилка під час виклику залишає промпт цілим для аналізу.
2. **Завжди ввімкнено** (не залежить від прапорця налагодження): накладні витрати — це один запис файлу на агента. Перевага в тому, що будь-який несподіваний результат має постійний запис.
3. **Ніяких помилок**: збої логування не повинні переривати основну роботу. Огорніть запис у try/catch, виведіть попередження у разі помилки та продовжуйте.

**Конвенція директорій**: Використовуйте шлях відносно кореня проекту (`prompts/debug/` або `claudedocs/debug/`), а не тимчасову директорію. Тимчасові директорії очищуються; ви ж хочете, щоб ці записи зберігалися.

**Вартість токенів**: Нуль (дискове вводу-виводу відбувається поза викликом ШІ). Промпт уже в пам'яті, ви просто фіксуєте його.

**Коли використовувати**: Будь-який скіл, що викликає провайдера ШІ та де промпт збирається динамічно (не просто статичний рядок). Цінність пропорційна складності промпту: якщо ваш промпт вставляє "джерело істини", інструкції оцінювача та вміст файлів, один запис перед викликом — це дешева страховка.

> Спостерігається в [PackmindHub/context-evaluator](https://github.com/PackmindHub/context-evaluator) (`src/shared/evaluation/runtime-prompt-logger.ts`). Див. [Credits](./credits.md).

## Адаптивний уніфікований/паралельний режим

**Проблема**: У вас є N файлів для оцінки, і потрібно вирішити: надіслати всі файли одному агенту (краще для виявлення проблем між файлами, але вища вартість контексту) чи надіслати кожен файл власному паралельному агенту (дешевше, швидше, але не бачить суперечностей між файлами)?

**Патерн**: Оцініть загальну кількість токенів перед вибором стратегії. Якщо загальна кількість вписується в поріг, використовуйте уніфікований режим (один агент бачить усе). Якщо перевищує — перейдіть до незалежних паралельних агентів для кожного файлу.

```typescript
const DEFAULT_MAX_UNIFIED_TOKENS = 100_000;

function canUseUnifiedMode(context: MultiFileContext, maxTokens = DEFAULT_MAX_UNIFIED_TOKENS) {
  if (context.totalTokenEstimate > maxTokens) {
    return {
      viable: false,
      reason: `Загальний вміст (~${context.totalTokenEstimate} токенів) перевищує ліміт (${maxTokens})`,
    };
  }
  return { viable: true };
}
```

Викликайте це перед створенням промптів агентів. Рішення визначає решту конвеєра:

```
Оцінка токенів (усі файли)
  ↓
< 100K → runUnifiedEvaluation(allFiles)   // 1 агент, виявлення зв'язків між файлами
> 100K → runAllEvaluators(file)           // N агентів, паралельно для кожного файлу
```

**Чому поріг важливий**: В уніфікованому режимі агент може побачити суперечності між кореневим `CLAUDE.md` та `CLAUDE.md` у піддиректорії. У паралельному режимі кожен агент бачить лише один файл і не може виявити ці суперечності. Поріг зберігає інтелект для невеликих репозиторіїв, залишаючись у межах практичних лімітів контексту для великих.

**Калібрування порогу**: 100K токенів залишає місце для промпту оцінювача (зазвичай 2–5K) та буфера відповіді моделі. Для вашого випадку встановіть поріг як `model_context_window - evaluator_prompt_tokens - response_buffer`. Початкове значення `100_000` є консервативним стандартом для більшості сучасних моделей.

**Оцінка токенів**: Вам не потрібна точна кількість. Грубої оцінки (`символи / 4` для більшості англійських текстів) достатньо для прийняття рішення. Вартість помилки на 10% в оцінці набагато нижча за вартість надмірної точності.

> Спостерігається в [PackmindHub/context-evaluator](https://github.com/PackmindHub/context-evaluator) (`src/shared/evaluation/runner.ts`). Див. [Credits](./credits.md).
