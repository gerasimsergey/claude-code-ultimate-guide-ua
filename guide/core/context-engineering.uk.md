## 1. Що таке інженерія контексту

### Визначення

Андрій Карпати ввів цей термін: **"Інженерія контексту — це мистецтво заповнення вікна контексту правильною інформацією у правильний час."**

Це речення містить три не очевидні вимоги:

-   **Заповнення**: вікно контексту має наповнюватися свідомо, а не випадково. Напівпорожнє вікно марнує потужність моделі; хаотично заповнене — марнує ваші токени та погіршує якість результату.
-   **Правильна інформація**: не вся інформація однаково цінна. Рішення про архітектуру важливіші за вподобання щодо лінтера. Негативні обмеження ("ніколи не повертай клієнту сирі помилки SQL") дієвіші за загальні цілі ("пиши чистий код").
-   **Правильний час**: правила для бекенд-коду не мають цінності, коли ви редагуєте фронтенд-компонент. Завантаження всього і завжди — це лінивий підхід, який погіршує дотримання інструкцій.

### Промпт-інженерія проти Інженерії контексту

Ці терміни часто плутають, але різниця суттєва:

| Вимір | Промпт-інженерія | Інженерія контексту |
|-----------|-------------------|---------------------|
| Область | Один запит | Вся сесія або система |
| Тривалість | Одна взаємодія | Зберігається між взаємодіями |
| Зусилля | Створення запиту щоразу | Початкове проектування системи |
| Масштаб | Індивідуальний | Командний або організаційний |
| Результат | Рядок промпту | Система конфігурації |

**Промпт-інженерія** — це створення правильного запитання для одного завдання. **Інженерія контексту** — це система, яка гарантує, що Claude має правильні фонові знання *перед* початком будь-якого завдання.

## 2. Бюджет контексту

### Математика токенів

Базові показники для проекту середнього розміру:

| Джерело | Типовий діапазон токенів |
|--------|---------------------|
| Глобальний `CLAUDE.md` | 1,000 – 3,000 токенів |
| Проектний `CLAUDE.md` (root) | 2,000 – 8,000 токенів |
| Модулі з обмеженням за шляхом | 1,000 – 5,000 токенів |
| Імпортовані скіли / команди | 500 – 3,000 токенів |
| **Всього постійного контексту** | **~5,000 – 20,000 токенів** |

Практичне правило: **постійно завантажений контекст має займати менше 5% вікна контексту.** У Claude Sonnet 4.6 (200K токенів) це близько 10,000 токенів. Все, що більше, починає витісняти реальний вміст завдання (код, історію).

### Межа в 150 інструкцій
Спостереження показують: якщо кількість окремих правил перевищує ~150, модель починає вибірково ігнорувати деякі з них. Якість правил важливіша за їх кількість. 20 конкретних, дієвих правил працюють краще, ніж 200 загальних побажань.

## 3. Ієрархія конфігурацій

### Трирівневий стек

```
┌──────────────────────────────────────────────┐
│  Глобальний (~/.claude/CLAUDE.md)            │
│  Ідентичність, тон, універсальні правила     │
├──────────────────────────────────────────────┤
│  Проектний (./CLAUDE.md + модулі)            │
│  Архітектура, стек, правила команди          │
├──────────────────────────────────────────────┤
│  Сесійний (інструкції в чаті, прапорці)      │
│  Тимчасові виправлення, параметри завдання   │
└──────────────────────────────────────────────┘
```

Нижні шари мають пріоритет над верхніми: сесійна інструкція може перевизначити проектне правило, а проектне — глобальне.

## 4. Модульна архітектура

### Проблема монолітного конфігу
600-рядковий `CLAUDE.md` без структури — найпоширеніша помилка. Модель втрачає фокус, а розробники не можуть знайти потрібне правило.

### Патерн Path-Scoping (Обмеження за шляхом)
Використовуйте імпорт модулів `@path/to/file.md` у головному `CLAUDE.md`. Правила з такого модуля додаються в контекст *лише тоді*, коли Claude працює з файлами у вказаній директорії.

Приклад структури:
- `src/api/CLAUDE-api.md` (правила API)
- `src/components/CLAUDE-ui.md` (правила фронтенду)
- `prisma/CLAUDE-db.md` (правила БД)

Це економить 40-50% бюджету контексту, не втрачаючи глибини інструкцій.

## 5. Командна збірка

Для великих команд підтримка сотень файлів `CLAUDE.md` вручну неможлива. Рішення — **декларативні профілі розробників**.

```yaml
# profiles/alice.yaml
profile:
  name: "Alice"
  role: "frontend"
modules:
  include:
    - shared/core-rules.md
    - frontend/react-patterns.md
    - frontend/tailwind-conventions.md
overrides:
  - "Використовуй лише іменовані експорти"
```

Команда запускає скрипт збірки, який об'єднує спільні модулі та персональні налаштування у фінальний `CLAUDE.md`.

## 6. Життєвий цикл контексту

### Борг інструкцій (Instruction Debt)
Правила накопичуються, але рідко видаляються. Застарілі правила — це "шум", який плутає модель. Проводьте **квартальний аудит контексту**:
- Чи актуальна ця бібліотека?
- Чи немає суперечливих правил?
- Чи не занадто розмите це правило?

### Конвеєр ACE (Assemble → Check → Execute)
Для автоматизованих робочих процесів:
1.  **Assemble**: Збірка контексту з профілю та модулів.
2.  **Check**: Запуск "тестових промптів" (Canary Checks) для перевірки, чи розуміє Claude ключові правила.
3.  **Execute**: Виконання реального завдання.

---

## 7. Вимірювання якості

### Canary Checks (Перевірочні промпти)
Прості тестові завдання для перевірки дотримання правил. Наприклад: "Створи React-компонент". Якщо Claude використав `any` (а у вас правило `no-any`), значить конфіг потребує виправлення.

---

## 8. Методи скорочення контексту

1.  **Стиснення правил**: Замість абзаців тексту використовуйте короткі тези.
2.  **Негативні обмеження**: "Ніколи не роби X" працює на 15-25% краще, ніж "Роби Y".
3.  **Видалення дублікатів**: Одне чітке правило краще за три схожих у різних файлах.
4.  **Архівування**: Переміщуйте застарілі правила в окремий архівний файл для історії, але прибирайте з активного вікна контексту.

---

## 9. Оцінка зрілості

| Рівень | Назва | Стан |
|-------|------|-------------|
| **0** | Без конфігу | Немає `CLAUDE.md`, загальні відповіді |
| **1** | Плоский конфіг | Один файл, правила накопичуються хаотично |
| **2** | Структурований | Розділи, відокремлення глобальних правил |
| **3** | Модульний | Використання Path-scoping та скілів |
| **4** | Вимірюваний | Тестові промпти (Canaries), аудит токенів |
| **5** | Інженерна система | Декларативні профілі, CI-контроль за дрейфом контексту |

## 10. Робочий процес аудиту токенів

Якщо Claude починає "тупити" або ігнорувати правила — перевірте обсяг завантажених даних:
1.  Порахуйте токени в усіх `CLAUDE.md` (символи / 4).
2.  Перевірте вивід хуків (Hooks): якщо хук видає 1000 рядків тексту на кожну команду, він швидко "з'їсть" весь контекст.
3.  Приберіть з автозавантаження правила, які потрібні лише в 10% випадків.
