## 1. Зони керування контекстом

Як діяти залежно від використання вікна контексту (перевіряйте через `/status`):

```
Використання контексту
0%          50%         70%         90%       100%
├───────────┼───────────┼───────────┼──────────┤
│  ЗЕЛЕНА   │   ЖОВТА   │ ПОМАРАНЧЕВА │ ЧЕРВОНА  │
│ працюйте  │ вибірково │ /compact  │  /clear  │
│  вільно   │ з увагою  │   ЗАРАЗ   │ обов'язк.│
└───────────┴───────────┴───────────┴──────────┘
              ▲                       ▲
              │                       │
         Будьте вибагливі        Ризик: забування
         щодо читання та         інструкцій,
         інструментів            галюцинації
```

**Дії за зонами:**
- **Зелена (0-50%)** — Повна швидкість. Читайте файли, досліджуйте вільно.
- **Жовта (50-70%)** — Будьте вибірковими. Уникайте непотрібного читання файлів.
- **Помаранчева (70-90%)** — Негайно запустіть `/compact`. Якість контексту падає.
- **Червона (90%+)** — Запустіть `/clear` і почніть спочатку. Відповіді стають ненадійними.

→ Джерело: [ultimate-guide.uk.md:1335](./ultimate-guide.uk.md)

## 2. Цикл режимів дозволів

Перемикайте режими за допомогою `Shift+Tab`:

```
                 Shift+Tab              Shift+Tab
   ┌──────────┐ ────────────→ ┌───────────────┐ ────────────→ ┌───────────┐
   │ ЗА ЗАМОВЧ.│               │ АВТО-ПРИЙНЯТТЯ │               │ РЕЖИМ ПЛАНУ│
   │           │               │                │               │           │
   │ ред=запит │               │ ред=авто       │               │ ред=ні    │
   │ вик=запит │               │ вик=запит      │               │ вик=ні    │
   └──────────┘ ←──────────── └───────────────┘ ←──────────── └───────────┘
                 Shift+Tab              Shift+Tab
```

**Коли використовувати кожен режим:**

| Режим | Коли використовувати... | Рівень ризику |
|------|-------------|------------|
| **За замовчуванням** | Звичайна розробка — перевірка кожної зміни | Низький |
| **Авто-прийняття** | Довірені завдання (форматування, рефакторинг) | Середній |
| **Режим плану** | Складні/ризиковані операції — спочатку безпечне дослідження | Відсутній |

**Гарячі клавіші:**
- `Shift+Tab` — Перехід до наступного режиму
- `Shift+Tab × 2` — Стрибок до режиму плану із замовчуваного
- `/plan` — Прямий вхід у режим плану
- `/execute` — Вихід з режиму плану

→ Джерело: [ultimate-guide.uk.md:760](./ultimate-guide.uk.md)

## 3. Конвеєр робочого процесу (9 кроків)

Рекомендований робочий процес для кожного завдання:

```
  ┌─────────┐    ┌──────────┐    ┌────────────┐    ┌─────────────┐
  │ 1.СТАРТ │───→│ 2./status│───→│ 3. план?   │───→│ 4. опис     │
  │ claude  │    │ перев.кон│    │ Shift+Tab×2│    │ ЩО/ДЕ       │
  └─────────┘    └──────────┘    │(якщо ризик)│    │ ЯК/ПЕРЕВІРКА│
                                 └────────────┘    └──────┬──────┘
                                                          │
      ┌───────────────────────────────────────────────────┘
      │
      ▼
  ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
  │ 5.рев'ю  │───→│ 6. так/ні│───→│ 7. тест  │───→│ 8.коміт  │───→│9./compact│
  │   diff   │    │ прийняти?│    │ запуск   │    │ по заверш│    │ при >70% │
  └──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
```

**Ключові принципи:**
- **Крок 2**: Завжди перевіряйте контекст перед початком. Якщо >70%, спочатку `/compact`.
- **Крок 3**: Використовуйте режим плану для всього ризикованого або складного.
- **Крок 4**: Будьте конкретними — розмиті промпти дають розмиті результати.
- **Крок 5**: Читайте кожен diff. Ніколи не приймайте наосліп.
- **Крок 9**: Стискайте контекст після кожного завдання, щоб залишатися в зеленій зоні.

## 8. Атака MCP "Rug Pull"

Як шкідливий MCP-сервер може використати модель одноразового схвалення:

```
┌─────────────────────────────────────────────────────────────┐
│  1. Атакувальник публікує корисний MCP "code-formatter"     │
│                         ↓                                    │
│  2. Користувач додає в ~/.claude.json, схвалює один раз     │
│                         ↓                                    │
│  3. MCP працює нормально 2 тижні (будує довіру)             │
│                         ↓                                    │
│  4. Атакувальник випускає шкідливе оновлення (без запиту!)  │
│                         ↓                                    │
│  5. MCP викрадає ~/.ssh/*, .env, облікові дані              │
└─────────────────────────────────────────────────────────────┘
ЗАХИСТ: Фіксація версій + перевірка хешу + моніторинг
```

## 10. Вибір інструменту пошуку

3-рівневе дерево рішень для вибору правильного інструменту пошуку:

**Рівень 1: Що ви знаєте?**

```
Ви знаєте ТОЧНИЙ текст/патерн?
│
├─ ТАК → Використовуйте rg (ripgrep)
│  ├─ Відома назва функції: rg "createSession"
│  ├─ Відомий імпорт: rg "import.*React"
│  └─ Відомий патерн: rg "async function"
│
└─ НІ → Переходьте до Рівня 2
```

**Рівень 2: Що ви шукаєте?**

```
Яка ваша мета пошуку?
│
├─ "Знайти за ЗМІСТОМ/КОНЦЕПЦІЄЮ"
│  → Використовуйте grepai
│  └─ Приклад: grepai search "логіка валідації оплати"
│
├─ "Знайти визначення ФУНКЦІЇ/КЛАСУ"
│  → Використовуйте Serena
│  └─ Приклад: serena find_symbol --name "UserController"
│
├─ "Знайти за СТРУКТУРОЮ КОДУ"
│  → Використовуйте ast-grep
│  └─ Приклад: async без обробки помилок
│
└─ "Зрозуміти ЗАЛЕЖНОСТІ"
   → Використовуйте grepai trace
   └─ Приклад: grepai trace callers "validatePayment"
```

**Рівень 3: Оптимізація**

```
Знайдено забагато результатів?
│
├─ rg → Додайте фільтр --type або звузьте шлях
├─ grepai → Додайте фільтр --path або використовуйте trace
├─ Serena → Фільтруйте за типом символу (function/class)
└─ ast-grep → Додайте обмеження до патерну
```

→ Джерело: [search-tools-mastery.md:75](./workflows/search-tools-mastery.md)

## 17. Цикл TDD Red-Green-Refactor

Ітеративний цикл в основі розробки через тестування:

```
                    ┌──────────────────────────┐
                    │                          │
                    ▼                          │
            ┌──────────────┐                   │
            │   🔴 RED      │                   │
            │              │                   │
            │  Напишіть    │                   │
            │  тест, що    │                   │
            │  не проходить│                   │
            └──────┬───────┘                   │
                   │                           │
                   │ Тести НЕ проходять        │
                   │ (очікувано)               │
                   ▼                           │
            ┌──────────────┐                   │
            │   🟢 GREEN   │                   │
            │              │                   │
            │  Напишіть    │                   │
            │  мінімальний │                   │
            │  код для     │                   │
            │  успіху      │                   │
            └──────┬───────┘                   │
                   │                           │
                   │ Тести ПРОХОДЯТ            │
                   │ (мінімально)              │
                   ▼                           │
            ┌──────────────┐                   │
            │   🔵 REFACTOR│                   │
            │              │                   │
            │  Очистіть код│                   │
            │  поки тести  │                   │
            │  зелені      │                   │
            └──────┬───────┘                   │
                   │                           │
                   │ Наст. функція             │
                   └───────────────────────────┘

Ключові правила:
  RED      → Тест МАЄ провалитися перед написанням коду
  GREEN    → Пишіть ТІЛЬКИ достатньо коду для тесту
  REFACTOR → Покращуйте структуру, тести мають бути зелені
  REPEAT   → Одна функція за раз, завжди в такому порядку
```

> Джерело: [workflows/tdd-with-claude.md:78](./workflows/tdd-with-claude.md)
