## Чому варто моніторити сесії

Використання Claude Code може швидко накопичуватися, особливо під час активної розробки. Моніторинг допомагає:

- **Розуміти витрати**: Оцінювати витрати на API до того, як надійдуть рахунки.
- **Виявляти патерни**: Бачити, які інструменти ви використовуєте найчастіше та які файли редагуються повторно.
- **Оптимізувати воркфлоу**: Знаходити неефективність (наприклад, повторне читання того самого великого файлу).
- **Відстежувати проекти**: Порівнювати використання в різних кодових базах.
- **Видимість для команди**: Агрегувати дані для бюджетування команди (при об'єднанні логів).

## Пошук та відновлення сесій

Після тижнів використання Claude Code пошук минулих розмов стає складним завданням. Цей розділ охоплює нативні опції та інструменти спільноти.

### Нативні команди

| Команда | Варіант використання |
|---------|----------|
| `claude -c` / `claude --continue` | Відновити останню сесію |
| `claude -r <id>` / `claude --resume <id>` | Відновити конкретну сесію за ID |
| `claude --resume` | Інтерактивний вибір сесії |

Сесії зберігаються локально в `~/.claude/projects/<project>/` як JSONL-файли.

### Порівняння інструментів спільноти

| Інструмент | Встановлення | Швидкість списку | Швидкість пошуку | Залежності | Відновлення |
|------|---------|------------|--------------|--------------|----------------|
| **session-search.sh** (цей репо) | Копіювати скрипт | **10мс** | **400мс** | Немає (bash) | ✅ Відображається |
| claude-conversation-extractor | `pip install` | 230мс | 1.7с | Python | ❌ |
| claude-code-transcripts | `uvx` | Н/Д | Н/Д | Python | ❌ |
| ran CLI | `npm -g` | Н/Д | Швидко | Node.js | ❌ (тільки команди) |

### Рекомендовано: session-search.sh

Bash-скрипт без залежностей, оптимізований для швидкості, з готовими командами для відновлення.

**Встановлення:**
```bash
cp examples/scripts/session-search.sh ~/.claude/scripts/cs
chmod +x ~/.claude/scripts/cs
echo "alias cs='~/.claude/scripts/cs'" >> ~/.zshrc
source ~/.zshrc
```

**Використання:**
```bash
cs                          # Список 10 останніх сесій (~15мс)
cs "authentication"         # Пошук за ключовим словом (~400мс)
cs "Prisma migration"       # Пошук за кількома словами (AND-логіка)
cs -n 20                    # Показати 20 результатів
cs -p myproject "bug"       # Фільтр за назвою проекту
cs --since 7d               # Сесії за останні 7 днів
cs --json "api" | jq .      # JSON-вивід для скриптів
```

## Обмеження відновлення сесій та міграція між папками

**Коротко**: Нативний `--resume` обмежений поточним робочим каталогом. Для міграції між папками використовуйте ручні операції з файловою системою або інструменти автоматизації.

#### Чому відновлення прив'язане до директорії

Claude Code зберігає сесії за шляхом `~/.claude/projects/<encoded-path>/`, де `<encoded-path>` походить від абсолютного шляху до вашого проекту.

**Ризики та застереження міграції:**

| Ризик | Вплив | Як запобігти |
|------|--------|------------|
| **Жорстко задані секрети** | Облікові дані в новому контексті | Перевірте `.jsonl` файли перед міграцією |
| **Абсолютні шляхи** | Посилання на файли зламаються | Перевірте існування шляхів у цілі |
| **Конфіги MCP-серверів** | Відсутність серверів у цілі | Встановіть потрібні MCP перед відновленням |

## Налаштування логування сесій

### 1. Встановіть Logger Hook

Скопіюйте логер сесій у вашу директорію хуків:

```bash
mkdir -p ~/.claude/hooks
cp examples/hooks/bash/session-logger.sh ~/.claude/hooks/
chmod +x ~/.claude/hooks/session-logger.sh
```

### 2. Зареєструйте в налаштуваннях

Додайте до `~/.claude/settings.json`:

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "type": "command",
        "command": "~/.claude/hooks/session-logger.sh"
      }
    ]
  }
}
```

## Аналіз даних сесій

### Використання session-stats.sh

```bash
session-stats.sh             # Підсумок за сьогодні
session-stats.sh --range week # За останній тиждень
session-stats.sh --json      # Машинозчитуваний вивід
```

### Аналіз якості, а не тільки кількості

Три патерни, що сигналізують про необхідність оновлення скілів або `CLAUDE.md`:

1.  **Повторне читання того самого файлу**: Якщо Claude читає файл 3+ рази за сесію, потрібний контекст варто винести в скіл.
2.  **Збої інструментів на одній команді**: Сигналізує про застарілі шляхи або зміни в стеку.
3.  **Висока частота редагування одного файлу**: Означає відсутність чітких конвенцій або контексту навколо цього файлу.

## Відстеження витрат

### Метод оцінки токенів

Логер оцінює токени за евристикою: **~4 символи на токен**. Це приблизне значення, яке зазвичай трохи завищує витрати.

### Бюджетні сповіщення (ручний патерн)

Додайте до профілю оболонки (shell profile) для щоденних попереджень:

```bash
claude_budget_check() {
  local cost=$(session-stats.sh --json 2>/dev/null | jq -r '.summary.estimated_cost.total // 0')
  local threshold=5.00  # Денний бюджет $5

  if (( $(echo "$cost > $threshold" | bc -l) )); then
    echo "⚠️  Витрати Claude Code за сьогодні: \$$cost (поріг: \$$threshold)"
  fi
}
```

## Моніторинг активності

Моніторинг активності показує, *що саме зробив Claude Code*: які файли читав, які команди запускав, які URL відвідував. Це рівень аудиту.

### Що розкривають виклики інструментів

| Інструмент | Що він показує |
|------|----------------|
| `Read` | Доступ до файлів (шлях, діапазон рядків) |
| `Write` / `Edit` | Модифікації файлів (шлях, дельта вмісту) |
| `Bash` | Виконані команди (повний рядок команди) |
| `WebFetch` | Відвідані URL (може включати дані в POST) |
| `Task` | Запуск субагентів (промпт для субмоделі) |

## Зовнішні інструменти моніторингу

| Інструмент | Тип | Що робить |
|------|------|-------------|
| **ccusage** | CLI / TUI | Відстеження витрат з JSONL — стандарт де-факто. |
| **claude-code-otel** | OpenTelemetry | Передає дані в Prometheus / Grafana. |
| **Akto** | SaaS | Запобіжники безпеки API + аудиторський слід. |
| **MLflow Tracing** | SDK | Точний підрахунок токенів та оцінка якості. |

## Трасування MLflow

**Коли використовувати**: Командам, які вже в екосистемі MLflow, або для точного підрахунку токенів та оцінки якості "ШІ-як-суддя" (LLM-as-judge).

### Регресійне тестування агентів

Ключова можливість MLflow — вимірювання того, чи покращилася якість після зміни інструкцій агента:

```python
from mlflow.genai.scorers import ConversationCompleteness, RelevanceToQuery

# Оцінка записаних трас
results = mlflow.genai.evaluate(
    data=traces,
    scorers=[
        ConversationCompleteness(),
        RelevanceToQuery(),
    ]
)
```

## Проксіювання Claude Code

Як побачити, що Claude Code надсилає до Anthropic?

### mitmproxy (рекомендовано)

Найчистіше open-source рішення. Забезпечує скриптований HTTPS-проксі з веб-інтерфейсом.

```bash
# В одному терміналі
mitmproxy --listen-port 8080

# В іншому терміналі для Claude Code
export NODE_EXTRA_CA_CERTS="$(python3 -c 'import mitmproxy.certs; print(mitmproxy.certs.Cert.default_ca_path())')"
export HTTPS_PROXY="http://localhost:8080"
claude
```

Веб-інтерфейс `mitmweb` покаже повні тіла запитів/відповідей, включаючи системні промпти та визначення інструментів.

## Контрольний список аудиту для менеджера

Для лідів команд, яким потрібно перевірити належне використання Claude Code.

### Щотижнева вибіркова перевірка (5 хвилин)

1.  **Файли поза межами проекту**: Перевірка, чи не читав агент файли, до яких не повинен мати доступу.
2.  **Деструктивні команди**: Пошук `rm -rf`, `drop table`, `git push --force`.
3.  **Зовнішні запити**: Перевірка `WebFetch` на невідомі URL.
