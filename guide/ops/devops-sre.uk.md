## Фреймворк FIRE

Кожна діагностика інфраструктури з Claude слідує цьому патерну:

```
F - First Response (Перша відповідь) → Надайте Claude симптом + контекст
I - Investigate (Дослідження)        → Claude аналізує логи, метрики, конфігурацію
R - Remediate (Усунення)             → Claude пропонує виправлення (з вашим схваленням)
E - Evaluate (Оцінка)                → Постмортем, документація, запобігання
```

### Чому FIRE?

| Фаза | Роль людини | Роль Claude |
|-------|------------|-------------|
| **F**irst Response | Описати симптом, надати контекст | Тріаж, пріоритезація перевірок |
| **I**nvestigate | Запускати команди, вставляти вивід | Аналізувати, корелювати, будувати гіпотези |
| **R**emediate | **Схвалити або відхилити** | Пропонувати виправлення, пояснювати вплив |
| **E**valuate | Переглянути, поділитися знаннями | Генерувати постмортем, документацію |

**Важливо**: Claude пропонує, ви схвалюєте. Це особливо критично для:
- Деструктивних операцій (видалення, масштабування вниз, перезапуск)
- Продуктових середовищ (Production)
- Змін, чутливих до безпеки

## Ваша перша діагностика інфраструктури

### Приклад: Pod у стані CrashLoopBackOff

```bash
# Крок 1: FIRST RESPONSE - Надайте контекст
claude "У мене под у стані CrashLoopBackOff у просторі імен payment-service.
Середовище: production
Кластер: gke-prod-us-east1
Почалося: 10 хвилин тому
Які перші 3 речі мені варто перевірити?"

# Крок 2: INVESTIGATE - Запустіть запропоновані команди та поділіться виводом
kubectl describe pod payment-api-7d4b8c6f5-x2j9k -n payment-service
# Скопіюйте вивід та вставте в Claude

# Крок 3: REMEDIATE - Claude пропонує, ви схвалюєте
claude "На основі цього виводу describe, яке виправлення?
ОБМЕЖЕННЯ: Покажи мені точну команду, але нічого не виконуй."

# Крок 4: EVALUATE - Задокументуйте інцидент
claude "Створи коротку нотатку про інцидент для нашої wiki:
- Що сталося
- Коренева причина (Root cause)
- Застосоване виправлення
- Рекомендація щодо запобігання"
```

### Що повертає Claude (Приклад першої відповіді)

```markdown
## Тріаж CrashLoopBackOff (у порядку пріоритетності)

1. **Перевірте код виходу та останні логи**
   ```bash
   kubectl logs payment-api-7d4b8c6f5-x2j9k -n payment-service --previous
   ```
   Найчастіші причини: OOM (код 137), помилки конфігурації (код 1), збої залежностей.

2. **Перевірте події пода та ліміти ресурсів**
   ```bash
   kubectl describe pod payment-api-7d4b8c6f5-x2j9k -n payment-service
   ```
   Шукайте: OOMKilled, ImagePullBackOff, збої liveness probe.

3. **Перевірте останні зміни у розгортанні**
   ```bash
   kubectl rollout history deployment/payment-api -n payment-service
   ```
   Якщо зміни були нещодавно: розгляньте відкат (rollback) під час розслідування.
```

# Патерн: Діагностика інфраструктури

**Мета**: Системне усунення несправностей для поширених проблем інфраструктури.

## Kubernetes Troubleshooting

### Налаштування MCP-сервера K8s

Для постійного доступу до контексту K8s встановіть MCP-сервер:

```json
// ~/.claude.json (або .mcp.json)
{
  "mcpServers": {
    "kubernetes": {
      "command": "npx",
      "args": ["-y", "@anthropic/mcp-kubernetes"]
    }
  }
}
```

**Переваги**: Claude може запитувати стан кластера безпосередньо, що зменшує кількість циклів копіювання-вставки.

**Без MCP**: Ви будете вручну передавати вивід kubectl до Claude (це все ще ефективно).

### Промпти за симптомами

Скопіюйте ці промпти, замінивши значення у `<дужках>`.

#### CrashLoopBackOff

```bash
kubectl describe pod <pod> -n <ns> | claude "Проаналізуй цей CrashLoopBackOff:
1. Який код виходу і що він означає?
2. Перевір патерн останніх 5 перезапусків (час, стабільний чи прискорюється?)
3. Запропонуй 3 найбільш ймовірні кореневі причини на основі подій
4. Дай мені точні команди для перевірки кожної гіпотези"
```

**Типові причини, які виявить Claude**:
- Код 137: OOMKilled (перевищено ліміт пам'яті)
- Код 1: Помилка застосунку (неправильний конфіг, відсутня залежність)
- Код 143: SIGTERM (тайм-аут коректного завершення)

#### OOMKilled

```bash
kubectl top pods -n <ns> && kubectl describe pod <pod> -n <ns> | claude "Цей под був вбитий за OOMKilled:
1. Порівняй запити (requests) проти лімітів (limits) та реального використання
2. Це витік пам'яті чи недостатнє виділення ресурсів?
3. Якщо витік: які патерни в контейнері вказують на шляхи розслідування?
4. Якщо недостатньо ресурсів: запропонуй оптимальні налаштування на основі цих даних"
```

#### ImagePullBackOff

```bash
kubectl describe pod <pod> -n <ns> | claude "Діагностика ImagePullBackOff:
1. Це проблема авторизації, мережі чи неправильна назва образу?
2. Про що саме говорить нам повідомлення про помилку?
3. Дай мені команди для перевірки існування образу та роботи облікових даних"
```

### Кейс: Коренева причина збою в продакшені

**Ситуація**: Платформа електронної комерції, виклик о 3 ранку, сервіс оформлення замовлень повертає 503 помилки.

**FIRE в дії**:

```bash
# F - First Response
claude "ІНЦИДЕНТ: checkout-service повертає 503, почалося 10 хв тому.
Вплив: 100% спроб оплати невдалі.
Середовище: AWS EKS production, us-east-1.
Нещодавні зміни: деплой 2 години тому (нова логіка feature flags).
Який найшвидший шлях діагностики?"

# I - Investigate (Claude запропонував перевірити поди)
kubectl get pods -n checkout -l app=checkout-service
# Вивід: 3/5 подів у CrashLoopBackOff

kubectl logs checkout-service-xxx --previous | tail -50 | claude "Проаналізуй логи краху"
# Claude виявляє: panic: nil pointer dereference у коді feature flag

# R - Remediate
claude "Кореневу причину виявлено: nil pointer у логіці feature flag після деплою.
Варіанти:
А) Відкат (Rollback) до попередньої версії
Б) Хотфікс перевірки на nil
Що швидше та безпечніше о 3 ранку?"
# Claude рекомендує: Rollback (швидше, перевірений стан, виправлення завтра)

kubectl rollout undo deployment/checkout-service -n checkout
# Сервіс відновлено за 2 хвилини

# E - Evaluate (наступного дня)
claude "Створи постмортем на основі цього інциденту:
Хронологія: 3:02 alert, 3:15 знайшли причину, 3:17 rollback, 3:19 вирішено
Причина: Feature flag nil pointer з коміту abc123
Вплив: 15 хвилин простою checkout
Формат: Без звинувачень (Blameless), фокус на запобіганні"
```

## Що Claude НЕ МОЖЕ (Обмеження)

Розуміння обмежень запобігає розчаруванню та небезпечній довірі.

| Обмеження | Вплив | Спосіб вирішення |
|------------|--------|------------|
| **Немає стану кластера в реальному часі** | Не бачить поточний статус подів | Використовуйте K8s MCP або вставляйте вивід kubectl |
| **Немає прямого доступу до API хмари** | Не може викликати API AWS/GCP | Використовуйте MCP-сервери або діліться виводом CLI |
| **Ліміти вікна контексту** | Макс. ~100K токенів | Фокусуйтеся на релевантних логах, а не на повних дампах |
| **Немає постійної пам'яті** | Забуває між сесіями | Використовуйте CLAUDE.md для контексту проекту |
| **Ризик галюцинацій** | Може запропонувати неіснуючі прапорці | Завжди перевіряйте команди перед запуском |
| **Немає секретів** | Не може читати vault/secrets | Чудово! Ніколи не діліться секретами з ШІ |

# Патерн: Реагування на інциденти

**Мета**: Структуровані робочі процеси для управління інцидентами.

## Соло-інцидент (Нічний виклик)

**Реальність**: О 3 ранку ви самі. Цей воркфлоу розроблений для однієї людини.

### FIRE в дії: Соло-інцидент

#### F - First Response (30 секунд)

```bash
claude "ІНЦИДЕНТ: [симптом - будьте конкретними]
Контекст: [сервіс], [середовище], [час початку]
Нещодавні зміни: [деплої, зміни інфраструктури, сплески трафіку]
Поточний вплив: [% постраждалих користувачів, вплив на дохід]
Які 3 найбільш критичні речі перевірити першими?"
```

#### R - Remediate (зі схваленням)

```bash
claude "На основі розслідування:
- Причина: [ваше розуміння]
- Докази: [ключові знахідки]

Запропонуй варіанти усунення:
1. Швидка мінімізація (відновлення сервісу)
2. Правильне виправлення (усунення причини)

ОБМЕЖЕННЯ: Мені потрібно схвалити дію. Покажи точні команди."
```

## Комунікація під час інцидентів

### Генератор оновлень для стейкхолдерів

```bash
claude "Згенеруй оновлення щодо інциденту для стейкхолдерів:
Інцидент: Деградація сервісу Checkout
Поточний статус: Мінімізовано, моніторимо
Вплив: 15 хвилин 30% відмов при оплаті
Орієнтовний час повного вирішення: 2 години
Аудиторія: Нетехнічні керівники
Тон: Професійний, заспокійливий, фактичний
Довжина: макс. 3 речення"
```

**Приклад виводу**:
> Ми зафіксували 15-хвилинний збій у сервісі оформлення замовлень, що вплинув на приблизно 30% транзакцій. Проблема вже усунута шляхом відкату нещодавнього оновлення програмного забезпечення. Ми розгорнемо остаточне виправлення під час наступного планового технічного вікна без очікуваного впливу на клієнтів.

```bash

# Запобіжники та впровадження

**Мета**: Безпечне впровадження Claude Code та отримання підтримки команди.

## Обізнаність щодо вартості

| Модель | Вхід (1M токенів) | Вихід (1M токенів) |
|-------|-------------------|-------------------|
| Sonnet 4 | $3 | $15 |
| Opus 4 | $15 | $75 |

**Типова DevOps сесія**: 20K-50K токенів = $0.10-$0.50.

## Межі безпеки

### Ніколи не діліться з Claude

| Тип даних | Чому ні | Альтернатива |
|-----------|---------|-------------|
| API ключі, токени | Можуть бути закешовані | Використовуйте `<API_KEY>` |
| Прод-секрети | Ризик безпеки | Опишіть тип секрету, а не значення |
| Дані клієнтів (PII) | Приватність/комплаєнс | Анонімізовані приклади |
| Власні алгоритми | Захист інтелект. власн. | Опишіть поведінку, а не код |
