## Чому трасування важливе саме зараз

Поява ШІ-помічників для кодування створила новий виклик: **розуміння того, який код надійшов від ШІ, а який — від людини**.

### Період напіврозпаду ШІ-коду

Дослідження репозиторіїв, що відстежуються за допомогою git-ai, виявило вражаючу метрику: **Період напіврозпаду ШІ-коду** становить приблизно **3.33 роки** (медіана). Це означає, що половина згенерованого ШІ коду замінюється протягом 3.33 років — швидше, ніж звичайний код.

Чому так відбувається? ШІ-код часто:
- Не має глибокого розуміння архітектури проекту.
- Використовує загальні патерни, що не підходять для конкретного контексту.
- Вимагає переробки при зміні вимог.
- Замінюється, коли розробники краще розуміють проблему.

### Чотири драйвери трасування

| Драйвер | Проблема | Стейкхолдер |
|--------|---------|-------------|
| **Аудит та комплаєнс** | Регульовані галузі потребують підтвердження походження | Юристи, Безпека |
| **Ефективність рев'ю** | ШІ-код часто потребує ретельнішої перевірки | Мейнтейнери |
| **Юридичні аспекти** | Походження тренувальних даних, неоднозначність ліцензій | Юристи |
| **Налагодження** | Розуміння "чому" за вибором ШІ | Розробники |

**Claude Code** за замовчуванням додає мітку `Co-Authored-By: Claude`, але це лише одна точка на ширшому спектрі.

## Спектр розкриття використання

Не всі проекти потребують однакового рівня атрибуції. Вибирайте залежно від контексту:

| Рівень | Метод | Коли використовувати | Приклад |
|-------|--------|-------------|---------|
| **Нульовий** | Без розкриття | Особисті проекти, експерименти | Pet-проект |
| **Мінімальний** | Мітка `Co-Authored-By` | Невеликі команди, прості OSS | Утиліта |
| **Стандартний** | `Assisted-by` + опис у PR | Командні проекти, активні OSS | Фреймворки |
| **Повний** | git-ai + збереження промптів | Enterprise, комплаєнс, медицина | Фінтех, GovTech |

## Методи атрибуції

### 3.1 Co-Authored-By (За замовчуванням у Claude Code)

Найпростіший метод. Claude Code автоматично додає це до комітів:

```
feat: implement user authentication

Implemented JWT-based auth with refresh tokens.

Co-Authored-By: Claude <noreply@anthropic.com>
```

**Плюси:** Автоматично, розпізнається GitHub/GitLab, відображається в графіках контриб'юторів.
**Мінуси:** Не показує обсяг участі ШІ, не зберігає контекст промпту.

### 3.2 Assisted-by Trailer (Стандарт LLVM)

Політика LLVM від січня 2026 року ввела точнішу мітку:

```
commit abc123
Author: Jane Developer <jane@example.com>

Implement RISC-V vector extension support

Assisted-by: Claude (Anthropic)
```

**Різниця між Co-Authored-By та Assisted-by:**
- `Co-Authored-By`: ШІ як співавтор.
- `Assisted-by`: Людина — автор, ШІ — лише допомагав. Чітка підзвітність людини.

### 3.3 Розкриття в PR/MR (Патерн Ghostty)

Ghostty (емулятор термінала) вимагає розкриття на рівні Pull Request, а не окремих комітів:

```markdown
## AI Assistance

Цей PR був розроблений за допомогою Claude (Anthropic).
Зокрема:
- Початкова структура алгоритму
- Генерація тестових випадків
- Чернетка документації

Увесь код був переглянутий та зрозумілий автором.
```

## Інструменти та автоматизація

### 5.1 Entire CLI

**Засновано**: Лютий 2026 Томасом Домке (колишній CEO GitHub).

**Що він робить**:
- Фіксує сесії ШІ-агентів як версіоновані **Чекпоінти** в Git.
- Зберігає промпти, міркування, використання інструментів та зміни файлів.
- Створює запис того, ЯК був написаний код, доступний для аудиту.
- Дозволяє "перемотувати" сесії та передавати контекст між агентами (напр., Claude → Gemini).

**Воркфлоу з Claude Code**:

```bash
# 1. Почати фіксацію сесії Entire
entire capture --agent="claude-code" --task="auth-refactor"

# 2. Працюйте з Claude як зазвичай
claude

# 3. Створити іменований чекпоінт
entire checkpoint --name="jwt-implemented"
```

**Архітектура**:
Entire зберігає чекпоінти в **автономній гілці (orphan branch)** `entire/checkpoints/v1`. Вона не має спільного предка з `main`, тому не створює конфліктів при злитті та не засмічує історію основного коду.

## Безпекові наслідки

### 6.1 Вразливість PromptPwnd

Клас атак, де шкідливі промпти в репозиторії експлуатують ШІ-помічників.

**Вектор атаки**:
1. Зловмисник додає приховані інструкції у файл (напр., у коментар або README).
2. Розробник використовує ШІ-помічника, який читає цей файл.
3. ШІ виконує шкідливі команди (напр., викрадає секрети з `.env` або впроваджує бекдори).
4. Розробник, не помітивши підступу, комітить скомпрометований код.

**Захист**:
- Використовуйте режим **пісочниці** в Claude Code.
- Завжди перевіряйте diff, згенерований ШІ.
- Обмежуйте доступ ШІ до чутливих шляхів через `.claudeignore`.

## Шаблони

### Коміт-повідомлення з Assisted-by

```
feat: implement rate limiting middleware

Add token bucket algorithm for API rate limiting.
Configurable per-endpoint limits with Redis backing.

Assisted-by: Claude (Anthropic)
```

### Розділ для CONTRIBUTING.md

```markdown
## Розкриття використання ШІ

Якщо ви використовували інструменти ШІ для підготовки вашого внеску, будь ласка, вкажіть це в описі Pull Request.

### Що потрібно декларувати:
- Код, згенерований ШІ
- Дослідження кодової бази за допомогою ШІ
- Підходи та алгоритми, запропоновані ШІ

### Що НЕ потребує декларування:
- Просте автодоповнення слів
- Перевірка граматики та орфографії
```
