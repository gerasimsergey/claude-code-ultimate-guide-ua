# TDD з Claude Code

> **Впевненість**: Рівень 1 — На основі офіційних рекомендацій Anthropic та валідації спільноти.

Розробка через тестування (TDD) з Claude потребує явних вказівок. За своєю природою Claude спочатку пише реалізацію, а потім тести. TDD вимагає зворотного.

---

## 1. Проблема

Без чітких інструкцій Claude буде:
1. Писати код реалізації.
2. Писати тести, які проходять проти цієї реалізації.

Це нівелює мету TDD: тести повинні визначати дизайн, а не просто підтверджувати існуючий код.

---

## 2. Налаштування CLAUDE.md

Додайте до вашого `CLAUDE.md`, щоб Claude знав правила гри:

```markdown
## Конвенції тестування

### Воркфлоу TDD
- Завжди пиши тести, що НЕ проходять, ДО реалізації.
- Використовуй патерн AAA: Arrange-Act-Assert (Підготовка-Дія-Перевірка).
- Назви тестів повинні описувати поведінку: "should_return_empty_when_no_items".

### Правила Test-First
- Коли я прошу нову функцію, спочатку пиши тести.
- Тести повинні ПАДАТИ на початку (реалізації ще немає).
- Лише після написання тестів пиши мінімальний код для їх проходження.
```

---

## 3. Цикл Red-Green-Refactor

### Фаза 1: Red (Червона — тест, що не проходить)
**Промпт**:
> "Напиши тест, що не проходить (failing test) для [опис функції]. Поки що НЕ пиши реалізацію. Тест повинен впасти, бо функція ще не існує."

### Фаза 2: Green (Зелена — мінімальна реалізація)
**Промпт**:
> "Тепер напиши мінімально необхідний код, щоб ці тести пройшли. Лише стільки коду, скільки потрібно для успішного тесту, нічого зайвого."

### Фаза 3: Refactor (Рефакторинг — очищення)
**Промпт**:
> "Проведи рефакторинг реалізації для покращення якості коду. Тести повинні залишатися зеленими. Зосередься на: [читабельності / швидкодії / видаленні дублювання]."

---

## 4. Анти-патерни: Прірва верифікації (Verification Gap)

**Прірва верифікації** — це стан, коли агент повідомляє про виконання завдання до того, як набір тестів підтвердить його. Це найпоширеніша помилка надійності в роботі ШІ-агентів.

**Як це виглядає:**
- Агент пише "Готово", хоча жодна команда тестування ще не запускалася.
- Тести запускаються, але агент ігнорує повідомлення про помилки в `stderr`.
- Проходять лише юніт-тести, хоча критерієм прийняття була поведінка всієї системи (E2E).

**Рішення**: Трьохшарова верифікація (Lint -> Unit -> E2E) та принцип "Незалежного оцінювача" (використання окремих хуків або агентів для перевірки результату).

---

## 5. Чого НЕ варто робити

| Анти-патерн | Чому це погано | Правильний підхід |
|--------------|----------------|------------------|
| "Додай тести та реалізацію" | Втрачається перевага TDD | Розділяйте на два окремі промпти |
| "Напиши тести для цього коду" | Claude підлаштує тести під баги | "Напиши тести для поведінки, наче коду ще немає" |
| Пропуск фази рефакторингу | Накопичується технічний борг | Завжди очищуйте код після проходження тестів |
