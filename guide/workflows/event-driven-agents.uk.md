# Подієво-орієнтована автоматизація агентів

> **Впевненість**: Рівень 3 — Новий патерн, перші користувачі звітують про позитивні результати, але інструментарій все ще вдосконалюється.

Замість того, щоб щоразу викликати Claude Code вручну, дозвольте зовнішнім подіям керувати роботою. Картка переміщується в "In Progress" у Linear — і Claude автоматично бере її в роботу. Ішью в GitHub отримує лейбл `claude-fix` — і агент розпочинає впровадження змін за лічені секунди.

Це перехід від пуш-моделі ("Клод, зроби це") до пуш-моделі на основі подій ("події запускають агентів").

---

## 1. Основна концепція

Традиційне використання Claude Code є інтерактивним. Подієво-орієнтована автоматизація усуває людину з етапу тригера. Людина все ще перевіряє результат (PR, зміни в коді), але ініціація відбувається через ваш існуючий воркфлоу управління проектом.

Якість ваших тикетів безпосередньо визначає якість автоматизації: хороші картки з чіткими критеріями прийняття (Acceptance Criteria) дають якісний код.

---

## 2. Патерн "Подія-Агент"

Пайплайн складається з 5 компонентів:
1. **Джерело події**: Linear, GitHub Issues, Slack, Jira або моніторингові сповіщення.
2. **Фільтр подій**: Визначає, чи є подія дієвою (наприклад, тільки для карток з лейблом `claude-auto`).
3. **Вилучення контексту**: Перетворення даних з події (опис тикета) у промпт для Claude Code.
4. **Вибір агента**: Підбір конфігурації агента (моделі, інструментів) залежно від типу завдання.
5. **Маршрутизація результатів**: Відкриття PR, коментар до картки або Slack-сповіщення про завершення.

---

## 3. Захисні механізми (Guardrails)

Оскільки такі агенти працюють з меншим наглядом людини, захисні механізми стають критичними:
- **Ідемпотентність**: Агент має перевіряти, чи робота вже не була виконана раніше (наприклад, чи не існує вже гілка для цього тикета).
- **Обмеження швидкості (Rate Limiting)**: Не дозволяйте черзі з 50 тикетів запустити 50 агентів одночасно. Встановіть ліміт (напр. 3-5 паралельних агентів).
- **Запобіжник (Circuit Breaker)**: Якщо агенти постійно зазнають невдач у певному типі завдань, система має автоматично зупинити спроби та сповістити людину.

## 4. Типові помилки (Anti-Patterns)

| ❌ Помилка | Проблема | ✅ Як правильно |
|-----------|----------|----------------|
| Агресивне опитування API | Марнування ресурсів, ризик блокування | Використовуйте вебхуки (webhooks), де це можливо |
| Відсутність запобіжника | Безперервне витрачання токенів на помилкові завдання | Зупиняйте роботу після 3 невдалих спроб |
| Розмиті описи в тикетах | Claude згенерує "сміттєвий" код | Вимагайте чітких критеріїв прийняття для автоматизації |
