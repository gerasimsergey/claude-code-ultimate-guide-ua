# Вибір підходу до впровадження

> **Застереження**: Claude Code — молодий інструмент (~1 рік). Остаточних відповідей ще немає — включаючи цей посібник. Це лише відправні точки, засновані на спостереженнях за патернами використання. Адаптуйте їх під свій контекст.

---

## Чого ми ще не знаємо

Перш ніж зануритися, ось моменти, які залишаються невизначеними:
- **Оптимальний розмір CLAUDE.md** — одні команди процвітають з 10 рядками, інші — зі 100.
- **Патерни впровадження** — чи перемагає стандартизація "зверху вниз" органічне впровадження, ще не доведено.
- **Показник ROI просунутих функцій** — не завжди зрозуміло, коли витрати на налаштування MCP-серверів та хуків окуповуються.

---

## Що ми знаємо (Емпіричні дані)

Деякі закономірності вже підтверджені практикою:

| Знахідка | Дані | Висновок |
|---------|------|-------------|
| **Важливість обсягу** | 1-3 файли: ~85% успіху, 8+ файлів: ~40% | Починайте з малого, розширюйте поступово |
| **"Золота середина" CLAUDE.md** | 4-8 КБ оптимально, >16 КБ втрата когерентності | Лаконічність краща за повноту |
| **Межі сесії** | 15-25 кроків до появи помилок | Скидайте сесію для нових завдань |
| **ROI написання скриптів** | 70-90% економії часу | Найкращий перший кейс використання |

---

## Відправні точки (Не інструкції)

| Ваш контекст | Один з підходів для спроби |
|--------------|---------------------|
| Обмежений час на налаштування | **Готове рішення (Turnkey)** — мін. конфіг, ітерації за потребою |
| Соло-розробник | **Автономний шлях** — спочатку концепції, потім конфігурація |
| Мала команда (4-10 осіб) | **Гібридний** — спільні основи + простір для особистих налаштувань |
| Велика команда (10+) | **Готове рішення + документація** — стабільність важливіша за гнучкість |

## Готове рішення (Turnkey Quickstart)

### Крок 1: Мінімальний конфіг
Створіть `.claude/CLAUDE.md`:
```markdown
# Проект: [назва]
## Стек
- Runtime: Node 22 / Python 3.12
- Framework: Next.js / FastAPI
## Команди
- Тести: `npm test`
- Лінтер: `npm run lint`
```

### Крок 2: Перевірка
Запустіть `claude` та запитайте: *"Яка команда тестування в цьому проекті?"*. Якщо відповідь правильна — конфіг завантажено.

## Корпоративне розгортання (50+ розробників)

У такому масштабі індивідуальних налаштувань замало. Потрібен поетапний план:

**Фаза 1 — Фундамент (Тиждень 1–2)**:
- Створення репозиторію зі спільними шаблонами `.claude/`.
- Публікація **Хартії використання ШІ**.
- Встановлення глобальних хуків безпеки на машини розробників.

**Фаза 2 — Впровадження (Тиждень 3–6)**:
- Класифікація проектів за рівнями ризику (Starter / Standard / Strict / Regulated).
- Додавання Claude Code у чекліст онбордингу нових інженерів.
- Перший аудит комплаєнсу.

**Фаза 3 — Оптимізація (Місяць 2–3)**:
- Тюнінг хуків (зменшення хибних спрацювань).
- Додавання перевірок у CI/CD для запобігання "дрейфу" конфігурацій.

---

## Типові помилки (Pitfalls)

- **Копіювання величезних конфігів**: Claude починає ігнорувати правила. Краще додавати по одному правилу при виникненні реальних проблем.
- **Надмірне проектування (Over-engineering)**: Витрати часу на налаштування перевищують час на власне кодування.
- **Відсутність спільних конвенцій**: У великих командах кожен починає працювати у своєму стилі, що ускладнює рев'ю коду.
