## Навіщо оцінювати агентів?

Коли ви створюєте кастомних агентів у `.claude/agents/`, ви кодуєте спеціалізовану експертизу в багаторазові воркфлоу. Але як дізнатися, чи вони дійсно ефективні?

**Без оцінки ви будуєте наосліп:**
- ❌ Немає способу дізнатися, чи стають відповіді кращими чи гіршими з часом.
- ❌ Неможливо об'єктивно порівняти різні конфігурації.
- ❌ Важко зрозуміти, яка частина інструкцій потребує доопрацювання.

**З оцінкою ви ітеруєте впевнено:**
- ✅ Кількісно оцінюєте якість (час відповіді, точність, використання інструментів).
- ✅ Проводите A/B тести з вимірюваними результатами.
- ✅ Будуєте цикли зворотного зв'язку для постійного вдосконалення.

## Метрики для відстеження

### 1. Якість відповіді
- **Показник виконання завдань (Task completion rate)**: Чи виконав агент поставлену мету?
- **Правильність**: Чи є вихідні дані фактично точними?
- **Рівень галюцинацій**: Як часто агент вигадує інформацію?

### 2. Використання інструментів
- **Успішність викликів інструментів**: % викликів без помилок.
- **Точність вибору**: Чи вибрав агент правильний інструмент для завдання?
- **Ефективність**: Мінімальна кількість викликів для досягнення мети.

### 3. Продуктивність
- **Час відповіді**: Загальний час від запиту до відповіді.
- **Ефективність токенів**: Кількість вхідних/вихідних токенів на завдання.
- **Вартість завдання**: Витрати на API для повної взаємодії.

## Патерни впровадження

### Паттерн 1: Система хуків логування
Автоматичне відстеження всіх взаємодій через `.claude/hooks/post-tool-use.sh`.
**Переваги**: Нульові ручні витрати, повне покриття даних.

### Паттерн 2: Юніт-тести для агентів
Регресійне тестування для перевірки, що покращення агента не ламають існуючі можливості.
```bash
# Тест: Аналіз архітектури
RESULT=$(claude agent architect "Проаналізуй src/auth.ts")
if echo "$RESULT" | grep -q "domain layer"; then
  echo "✅ PASS"
else
  echo "❌ FAIL"
fi
```

## Приклад: Агент з оцінкою (SQL-аналітик)

Шаблон `analytics-agent.md` включає хук `post_response`, який автоматично:
1. Витягує згенерований SQL-запит.
2. Перевіряє його на безпеку (відсутність `DROP`, `DELETE`).
3. Вимірює час виконання запиту в тестовій БД.
4. Логує результат у `analytics-metrics.jsonl`.

## Найкращі практики

- **Починайте з простого**: Тиждень 1 — логування викликів інструментів, Тиждень 2 — збір оцінок від користувачів.
- **Фокусуйтеся на дієвих метриках**: Не відстежуйте те, на що ви не будете реагувати. Пріоритет — показник виконання завдань.
- **Автоматизуйте все, що можливо**: Ручна оцінка не масштабується. Використовуйте CI/CD для тестів агентів.
