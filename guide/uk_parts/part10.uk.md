**Коли це працює**: Тільки для інженерів, на поглядах яких був навчений Claude і чиї думки складаються у стабільний, впізнаваний стиль. Хорошими кандидатами є DHH (Rails), Кент Бек (TDD, простота), Мартін Фаулер (рефакторинг, патерни). Випадкові імена — ні.

**Приклад** (з плагіна compound-engineering від Every.to):

```markdown
---
name: dhh-reviewer
description: Рев'ю коду з перспективи DHH. Пріоритет конвенціям Rails, "товстим" моделям, "тонким" контролерам, прагматичному REST та скептицизму щодо зайвої абстракції.
allowed-tools: Read, Grep
---
```

Цінність агента полягає у висвітленні цілісної перспективи, яка може не збігатися з вашим підходом за замовчуванням, а не в імітації людини.

**Застереження**: Агенти іменованих перспектив можуть дрейфувати в міру розвитку навчання Claude. Сприймайте ім'я як зручне скорочення, а не гарантію того, що агент буде точно слідувати поточним думкам реальної людини.

*Джерело: Every.to compound-engineering plugin (2026)*

### Патерн агента, що самостійно розвивається (Self-Evolving Agent)

Агент, який оновлює власні скіли після кожного виконання. Замість ручної підтримки документації агент читає поточний стан своєї галузі та переписує знання, вприснуті в нього самого.

**Коли використовувати**: Довгоживучі агенти, чия галузь розвивається — редактори презентацій, клієнти API, що відстежують зміни схем, агенти, що керують "живими" документами.

**Основний механізм** (у системному промпті агента):

```markdown
### Крок N: Саморозвиток (після кожного виконання)

Після завершення основного завдання онови свої попередньо завантажені скіли, щоб залишатися в курсі:

1. Прочитай поточний стан [домену, який ти змінив]
2. Онови `.claude/skills/<your-skill>/SKILL.md`, щоб відобразити реальність
3. Запиши, що змінилося і чому, у розділ "## Learnings" цього файлу агента

Це запобігає розриву між тим, що ти знаєш, і тим, що є насправді.
```

**Повний приклад** — агент-куратор презентацій, який підтримує знання про структуру/вагу слайдів актуальними:

```yaml
---
name: presentation-curator
description: PROACTIVELY використовуй при оновленні слайдів, структури або ваги
tools: Read, Write, Edit, Grep, Glob
model: sonnet
color: magenta
skills:
  - presentation/slide-structure
  - presentation/styling
---

## Крок 5: Саморозвиток (після кожного виконання)

Прочитай presentation/index.html та онови свої скіли:
- slide-structure: онови діапазони розділів, таблицю ваги, кількість слайдів
- styling: онови патерни CSS, якщо були впроваджені нові
- Додай нові знахідки у розділ "## Learnings" нижче

## Learnings
_Кожен запуск додає знахідки сюди. Майбутні виклики починаються вже поінформованими._
- Бейджі слайдів вприскуються через JS — ніколи не хардкодь їх в HTML.
```

**Чому це працює**: Поле `skills:` у frontmatter вприскує вміст скілів при старті агента. Записуючи дані назад у ці файли після кожного запуску, наступний виклик агента починається з актуальними знаннями. Ручна підтримка з боку людини не потрібна.

**Ключові обмеження**:
- Оновлюйте область вузько — тільки те, що справді змінилося
- Ведіть лог `## Learnings`, щоб агент накопичував знання протягом сесій
- Використовуйте разом із `memory: project` для збереження ширшого контексту між сесіями

## 5.0 Два види скілів

> **Нове у березні 2026 року**: Оновлення Anthropic Skill Creator формалізує таксономію, яка змінює підхід до проектування, тестування та, зрештою, виведення скілів з експлуатації. Джерела: ainews.com, mexc.co, claudecode.jp — ще не відображено в офіційному `llms-full.txt`.

Не всі скіли старіють однаково. Тип, який ви створюєте, визначає, як його писати, тестувати та коли від нього відмовлятися.

| | Capability Uplift (Розширення можливостей) | Encoded Preference (Закодовані вподобання) |
|---|---|---|
| **Що робить** | Заповнює прогалину, з якою базова модель не справляється стабільно | Вибудовує послідовність існуючих можливостей у специфічний для вашої команди спосіб |
| **Приклади** | Точне розміщення тексту в PDF, кастомні патерни коду | Чек-лист перевірки NDA, воркфлоу щотижневого звіту |
| **Довговічність** | Зникає в міру вдосконалення моделі | Залишається актуальним, поки воркфлоу є релевантним |
| **Сигнал до виведення** | Модель проходить eval без скіла | Воркфлоу змінюється або стає нерелевантним |
| **Підхід до оцінювання** | A/B тест: зі скілом та без нього | Перевірка точності: чи дотримується послідовність? |

**Capability Uplift** вчить Claude тому, що він справді не може робити добре сам по собі — поки що. Висока цінність сьогодні, але несе за собою борг обслуговування: в міру покращення Claude ці скіли можуть стати зайвими. Evals підкажуть вам, коли це станеться, раніше за користувача.

**Encoded Preference** кодує специфічний спосіб виконання вашою командою того, що Claude вже вміє робити. Перевірка NDA слідує критеріям вашого юридичного відділу, а не загальному чек-листу. Ці скіли не конкурують із вдосконаленням моделі — вони фіксують ваші робочі рішення, які залишаються актуальними, поки діє ваш процес.

> **Практичний наслідок**: При створенні скіла Capability Uplift виділяйте час на evals. При створенні Encoded Preference — на підтримку точності опису воркфлоу в міру еволюції вашого процесу.

## 5.1 Розуміння скілів

Скіли — це пакети знань, які агенти можуть успадковувати.

### Скіли проти Агентів проти Команд

| Концепція | Мета | Виклик |
|---------|---------|------------|
| **Агент** | Інструмент ізоляції контексту | Делегування завдань інструментам |
| **Скіл** | Модуль знань | `/skill-name` або автозавантаження |
| **Команда** | Робочий процес (воркфлоу) | Slash-команда |

#### Детальне порівняння

| Аспект | Команди | Скіли | Агенти |
|--------|----------|--------|--------|
| **Що це** | Шаблон промпту | Модуль знань | Інструмент ізоляції контексту |
| **Місце** | `.claude/commands/` | `.claude/skills/` | `.claude/agents/` |
| **Виклик** | `/command-name` | `/skill-name` або автозавантаження | Делегування завдань інструментам |
| **Виконання** | В основній розмові | Завантажується в контекст | Окремий підпроцес |
| **Контекст** | Спільний основний контекст | Додається до контексту агента | Ізольований контекст |
| **Найкраще для** | Повторюваних воркфлоу | Багаторазових знань | Обмеженого аналізу |
| **Вартість токенів** | Низька (тільки шаблон) | Середня (знання завантажені) | Висока (повний агент) |
| **Приклади** | `/commit`, `/pr`, `/ship` | TDD, security-guardian | security-audit, perf-audit |

#### Дерево рішень: Що використовувати?

```
Чи є це повторюваним воркфлоу з кроками?
├─ Так → Використовуйте КОМАНДУ (COMMAND)
│        Приклад: /commit, /release-notes, /ship
│
└─ Ні → Чи є це специфічними знаннями, які потрібні багатьом агентам?
        ├─ Так → Використовуйте СКІЛ (SKILL)
        │        Приклад: методологія TDD, чек-лист безпеки
        │
        └─ Ні → Чи потребує це ізольованого контексту або паралельної роботи?
                ├─ Так → Використовуйте АГЕНТА (AGENT)
                │        Приклад: code-reviewer, performance-auditor
                │
                └─ Ні → Просто запишіть це в CLAUDE.md як інструкції
```

> **Правило 20%**: якщо інструкція стосується понад 20% ваших розмов, помістіть її в `CLAUDE.md` (завжди завантажений). Якщо менше ніж 20% — зробіть її скілом (завантаження за запитом). Ріниця важлива для економії токенів: системний промпт скіла вприскується тільки тоді, коли Claude його викликає, тоді як вміст CLAUDE.md враховується в контексті кожного запиту.

> **Див. також**: [§2.7 Посібник з прийняття рішень щодо конфігурації](#27-configuration-decision-guide) для ширшого дерева рішень, що охоплює всі сім механізмів (включаючи хуки, MCP та правила). Для автоматичного виявлення того, що до якої категорії належить, використовуйте [`cc-sessions discover`](#session-pattern-discovery) — ця команда застосовує поріг 20% до вашої реальної історії сесій.

#### Поширені патерни

| Потреба | Рішення | Приклад |
|------|----------|---------|
| Запуск тестів перед комітом | Команда | `/commit` з кроком тестування |
| Знання для рев'ю безпеки | Скіл + Агент | security-guardian skill → security-audit agent |
| Паралельне рев'ю коду | Кілька фокусованих агентів | Запуск 3 агентів рев'ю з ізольованими областями |
| Швидкий Git воркфлоу | Команда | `/pr`, `/ship` |
| Знання архітектури | Скіл | architecture-patterns skill |
| Складне налагодження | Агент | debugging-specialist agent |

### Навіщо потрібні скіли?

Без скілів:
```
Агент A: Має знання про безпеку (дубльовано)
Агент B: Має знання про безпеку (дубльовано)
Агент C: Має знання про безпеку (дубльовано)
```

Зі скілами:
```
security-guardian skill: Єдине джерело знань про безпеку
Агент A: успадковує security-guardian
Агент B: успадковує security-guardian
Агент C: успадковує security-guardian
```

### Що робить скіл хорошим?

| Хороший скіл | Поганий скіл | Очікувана тривалість життя |
|------------|-----------|-------------------|
| Багаторазовий для різних агентів | Специфічний для одного агента | — |
| Фокусований на домені | Занадто широкий | — |
| Містить довідкові матеріали | Тільки інструкції | — |
| Включає чек-листи | Відсутня верифікація | — |
| Має визначені evals | Валідація "начебто працює" | Capability Uplift: регулярний моніторинг; Encoded Preference: стабільний |
| Чіткі критерії виведення | Немає плану життєвого циклу | Capability Uplift: короткий-середній; Encoded Preference: довгий |

## 5.2 Створення скілів

Скіли живуть у директоріях `.claude/skills/{skill-name}/`.

### Структура папки скіла

```
skill-name/
├── SKILL.md          # Обов'язково - Основні інструкції
├── reference.md      # Опціонально - Детальна документація
├── checklists/       # Опціонально - Списки верифікації
│   ├── security.md
│   └── performance.md
├── examples/         # Опціонально - Патерни коду
│   ├── good-example.ts
│   └── bad-example.ts
└── scripts/          # Опціонально - Допоміжні скрипти
    └── audit.sh
```

### Frontmatter у SKILL.md

```yaml
---
name: skill-name
description: Короткий опис для активації (макс. 1024 симв.)
allowed-tools: Read Grep Bash
---
```

| Поле | Специфікація | Опис |
|-------|------|-------------|
| `name` | [agentskills.io](https://agentskills.io) | Нижнім регістром, 1-64 симв., тільки дефіси, без `--`, має збігатися з назвою папки |
| `description` | [agentskills.io](https://agentskills.io) | Що робить скіл і коли його використовувати (макс. 1024 симв.) |
| `allowed-tools` | [agentskills.io](https://agentskills.io) | Розділений пробілами список схвалених інструментів. Підтримує маски: `Bash(npm run *)`, `Bash(agent-browser:*)`, `Edit(/docs/**)` |
| `license` | [agentskills.io](https://agentskills.io) | Назва ліцензії або посилання на вкладений файл |
| `compatibility` | [agentskills.io](https://agentskills.io) | Вимоги до середовища (макс. 500 симв.) |
| `metadata` | [agentskills.io](https://agentskills.io) | Довільні пари ключ-значення (автор, версія тощо) |
| `effort` | **Тільки CC** (v2.1.80+) | `low\|medium\|high` — перевизначає рівень зусиль сесії при виклику цього скіла. Ставте `low` для механічних завдань (коміт, форматування), `high` — для аналізу чи архітектурних роздумів. |
| `argument-hint` | **Тільки CC** | Підказка в меню slash-команд, коли скіл приймає `$ARGUMENTS`. Формат: `"[--flag] [positional_arg]"`. |
| `disable-model-invocation` | **Тільки CC** | `true`, щоб зробити скіл доступним тільки вручну (воркфлоу з побічними ефектами) |

**`effort` для кожного скіла** (v2.1.80+) — перевизначає рівень зусиль сесії для конкретного виклику скіла. Незалежно від `effortLevel` у settings.json: значення скіла має пріоритет тільки під час його роботи, потім рівень повертається до попереднього.

```yaml
---
name: security-audit
description: Глибокий аналіз безпеки з моделюванням загроз
effort: high      # Завжди високі зусилля, незалежно від налаштувань сесії
allowed-tools: Read Grep Glob Bash
---
```

```yaml
---
name: commit
description: Індексація та коміт змін у конвенційному форматі
effort: low       # Механічно — бюджет на міркування не потрібен
allowed-tools: Bash
---
```

**Чому це важливо**: Зусилля (effort) контролюють глибину мислення, деталізацію викликів інструментів та глибину аналізу — а не тільки токени. Скіл з рівнем `low` працює швидше та дешевше. Скіл з `high` міркує глибше без необхідності вручну змінювати налаштування сесії. Це дозволяє автоматично розподіляти когнітивний бюджет за типами завдань: платіть за мислення тільки там, де воно додає цінності.

**Обмеження через маски в `allowed-tools`** — обмежте скіл конкретними просторами команд замість надання повного доступу до Bash:

```yaml
# Доступ тільки до конкретного інструменту CLI — жодних інших команд Bash
allowed-tools: Bash(agent-browser:*)

# Доступ тільки до скриптів npm
allowed-tools: Bash(npm run *)

# Тільки читання + обмежений запис
allowed-tools: Read Grep Glob Edit(/docs/**)
```

Це набагато безпечніше, ніж надання широкого доступу до `Bash`: скіл зможе запускати тільки команди, що відповідають патерну. Ідеально для скілів, що обгортають конкретний CLI-інструмент.

> **Відкритий стандарт**: Agent Skills слідують [специфікації agentskills.io](https://agentskills.io), створеній Anthropic та підтримуваній 30+ платформами (Cursor, VS Code, GitHub, Codex, Gemini CLI, Goose, Roo Code тощо). Скіли, які ви створюєте для Claude Code, є портативними. Поле `disable-model-invocation` є розширенням Claude Code.

### "Ворота якості" (Quality Gates) для скілів

**Чек-лист (критерії Every.to compound-engineering, адаптовані)**:

- [ ] **Frontmatter заповнений**: `name`, `description`, `allowed-tools` присутні та точні
- [ ] **Розділ "Коли застосовувати"**: явно вказано тригери та антитригери (коли НЕ використовувати)
- [ ] **Методологія структурована**: пронумеровані кроки або чітка послідовність рішень, а не вільні абзаци
- [ ] **Жодних TODO або плейсхолдерів**: кожен розділ завершений та придатний до виконання
- [ ] **allowed-tools мінімізовані**: якщо скіл тільки читає файли, не давайте Bash; якщо шукає — не давайте Edit
- [ ] **Формат виводу задокументований**: що саме створює Claude? Додайте приклад або шаблон
- [ ] **Жодних AskUserQuestion для кросплатформних скілів**: скіли, що викликаються іншими агентами, не повинні блокуватися на інтерактивних запитах
- [ ] **Єдина відповідальність**: один скіл — один домен знань, а не "звалище" всього підряд
- [ ] **Опис є реченням-тригером**: поле `description` має повідомляти Claude, коли активувати цей скіл, а не що він робить внутрішньо

## 5.X Життєвий цикл скілів та їх виведення з експлуатації

Скіли мають життєвий цикл. Сприйняття їх як постійних артефактів призводить до "гниття": мертвий код у `.claude/skills/` споживає токени і не приносить користі.

Два патерни визначають, коли пора діяти:

```
ВІДСТЕЖЕННЯ РЕГРЕСІЙ               ВИЯВЛЕННЯ ПЕРЕРОСТАННЯ
────────────────────               ──────────────────────
Модель еволюціонує                 Модель вдосконалюється
       ↓                                    ↓
Скіл дрейфує                       Скіл проходить тест сам
       ↓                               (без допомоги)
Сповіщення Eval                             ↓
(ранній сигнал)                      Скіл виведено
       ↓                           (більше не потрібен)
Виправити або вивести
```

**Відстеження регресій**: Ваш скіл працював минулого місяця. Модель оновилася. Тепер він поводиться інакше. Без evals ви дізнаєтесь про це тільки з репорту користувача. З evals ви зловите це до того, як помилка до когось дійде.

**Виявлення переростання**: Ви створили скіл Capability Uplift, щоб закрити прогалину. Через півроку Claude справляється з цим нативно. Запустіть eval без скіла — якщо він проходить, скіл більше не потрібен. Видаліть його, щоб зменшити навантаження на контекст та витрати на обслуговування.

### Чек-лист рішення про виведення скіла

- [ ] **Запустіть eval без скіла**: чи проходить Claude тест самостійно?
- [ ] **Перевірте дату останньої активації**: коли цей скіл востаннє викликався на практиці?
- [ ] **Перевірте точність воркфлоу**: для скілів Encoded Preference — чи не змінився базовий процес?
- [ ] **Архівуйте перед видаленням**: перемістіть у `.claude/skills/archive/` з нотаткою про дату та причину виведення

> **Див. також**: [§5.Y Оцінювання скілів (Skill Evals)](#5y-skill-evals) — як запускати тести для прийняття рішень про виведення.

## 5.Y Оцінювання скілів (Skill Evals)

Evals переводять якість з "здається, працює" у "точно працює". Це рівень тестування, що робить скіли придатними для продакшену.

> **Доступно через**: плагін Skill Creator (Anthropic GitHub). Станом на березень 2026 року працює на Claude.ai та Cowork. Джерела: ainews.com, mexc.co — ще немає в офіційному `llms-full.txt`.

### Як це працює

```
Скіл → Тестові промпти + Файли
               ↓
    Очікуваний результат (еталон)
               ↓
        Запуск Evals
               ↓
      Пройшов ✓ / Провалив ✗
               ↓
     Покращити скіл → Перезапуск
```

Ви визначаєте три речі: тестові промпти (реалістичні вхідні дані), очікувані результати (опис того, як виглядає "добре" — не точне порівняння рядків) та поріг проходження. Claude виконує скіл проти кожного кейсу та оцінює результат.

Звіт містить: відсоток проходження, витрачений час, використання токенів на кожен кейс.

### Три інструменти оцінювання

**Режим Benchmark** — відстежує відсоток проходження, час та токени після оновлень моделі. Запускає тести паралельно в чистих, ізольованих контекстах. Використовуйте для автоматичного виявлення регресій при оновленні Claude.

**A/B тестування (Comparator Agents)** — сліпе порівняння "лоб у лоб" двох версій скіла. Версія А проти Версії Б, оцінювання без знання, де яка. Виключає упередженість розробника при покращенні скіла.

**Trigger Tuning (Оптимізатор описів)** — аналізує поле `description` вашого скіла та пропонує покращення для зменшення хибнопозитивних (скіл спрацьовує, коли не треба) та хибнонегативних (не спрацьовує, коли треба) результатів. Внутрішній тест Anthropic: 5 із 6 скілів створення документів показали кращу точність активації після оптимізації.

### Два сценарії використання Evals

| Сценарій | Коли | Дія |
|----------|------|--------|
| **Відстеження регресій** | Після оновлень моделі | Запуск benchmark → сповіщення, якщо відсоток впав |
| **Виявлення переростання** | Періодично для Capability Uplift | Запуск eval *без* скіла → якщо проходить, вивести |

### Практична структура Eval

```
.claude/skills/my-skill/
├── SKILL.md
└── tests/                      ← Директорія для Eval
    ├── test-01-basic.md        # Промпт + опис очікуваного результату
    ├── test-02-edge-case.md    # Покриття крайових випадків
    └── benchmark-config.md     # Поріг проходження, бюджет токенів
```

### Принципи проектування Eval

- **Одна поведінка на один тест**: не змішуйте кілька перевірок — причини падіння стануть неоднозначними.
- **Включайте крайові випадки**: тестуйте вхідні дані, які з самого початку зробили цей скіл необхідним.
- **Визначайте "добре" точно**: розпливчасті очікування роблять оцінки ненадійними.
- **Встановіть поріг проходження**: 80% — гарний старт; коригуйте залежно від критичності.

### Приклад 2: TDD Skill

```markdown
---
name: tdd
description: Методологія та патерни розробки через тестування (Test-Driven Development)
allowed-tools: Read Write Bash
---

# TDD (Test-Driven Development)

## Цикл TDD

┌─────────────────────────────────────────────────────────┐
│                    RED → GREEN → REFACTOR               │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   1. RED     ──→  Напишіть тест, що падає               │
│        │                                                │
│        ▼                                                │
│   2. GREEN   ──→  Напишіть мін. код для проходження     │
│        │                                                │
│        ▼                                                │
│   3. REFACTOR ──→  Покращіть код, зберігши тести        │
│        │                                                │
│        └────────────→  Повторити                        │
│                                                         │
└─────────────────────────────────────────────────────────┘


## Методологія

### Крок 1: RED (Тест, що падає)

Напишіть тест для бажаної поведінки ПЕРЕД написанням коду.

```typescript
// user.test.ts
describe('User', () => {
  it('should validate email format', () => {
    expect(isValidEmail('test@example.com')).toBe(true);
    expect(isValidEmail('invalid')).toBe(false);
  });
});
```

Запуск: `pnpm test` → має ВПАСТИ (функція не існує)

### Крок 2: GREEN (Мінімальна реалізація)

Напишіть МІНІМУМ коду, щоб тест пройшов.

```typescript
// user.ts
export const isValidEmail = (email: string): boolean => {
  return email.includes('@');
};
```

Запуск: `pnpm test` → має ПРОЙТИ

### Крок 3: REFACTOR (Покращення)

Тепер покращуйте реалізацію, зберігаючи тести зеленими.

```typescript
// user.ts (покращено)
export const isValidEmail = (email: string): boolean => {
  const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
  return emailRegex.test(email);
};
```

Запуск: `pnpm test` → все ще має ПРОЙТИ

## Структура тесту: патерн AAA

```typescript
it('should calculate order total', () => {
  // Arrange - підготовка даних
  const items = [
    { price: 10, quantity: 2 },
    { price: 5, quantity: 3 }
  ];

  // Act - виконання коду
  const total = calculateTotal(items);

  // Assert - верифікація результату
  expect(total).toBe(35);
});
```
