### Приклад 4: Tally Form Builder Skill

**Мета**: Створення та модифікація форм Tally через MCP — без браузера, без інтерфейсу, тільки `/tally-form-builder` та опис.

**Місце**: `~/.claude/skills/tally-form-builder/`

**Що демонструє цей патерн**: Обертання MCP з відкладеним завантаженням інструментів. Інструменти Tally MCP не доступні за замовчуванням — їхні схеми мають бути отримані через `ToolSearch` перед будь-яким викликом. Цей скіл обробляє це автоматично та документує всі підводні камені, що спричиняють помилки при "сліпому" виклику API.

**Ключові особливості**:
- Управління потоком OAuth (авторизація → браузер → callback URL → завершення)
- Ланцюжки блоків через `insertAfterBlockUuid` для збереження порядку
- Підтримка HTML (у блоках TEXT — так, у мітках опцій — ні)
- Пакетне оновлення тексту одним викликом
- Файл відомих проблем із 7 задокументованими обмеженнями та рішеннями

**Структура**:
```
tally-form-builder/
├── SKILL.md                     # Повний воркфлоу + правила + антипатерни
└── references/
    ├── block-types.md           # Усі типи блоків із корисним навантаженням та прикладами
    └── known-issues.md          # 7 обмежень із рішеннями
```

**Основна концепція: Відкладені інструменти (Deferred Tools)**

Інструменти Tally MCP є відкладеними — їх виклик без попереднього `ToolSearch` повертає `InputValidationError`. Скіл впроваджує обов'язковий крок `ToolSearch` перед будь-яким викликом MCP. Цей патерн застосовний до будь-якого MCP-сервера з відкладеними інструментами.

```
ToolSearch → authenticate → list_workspaces → create_new_form
           → create_blocks → configure_blocks → update_text → save_form
```

**Патерн ланцюжка блоків**:

Кожен блок повинен посилатися на UUID блоку, що передує йому. Скіл відстежує UUID між викликами для збереження правильного порядку вставки:

```
FORM_TITLE (uuid: "abc")
  → create_blocks([TITLE], insertAfterBlockUuid: "abc") → повертає "def"
  → create_blocks([CHECKBOX × N], insertAfterBlockUuid: "def") → повертає "ghi"
  → create_blocks([PAGE_BREAK], insertAfterBlockUuid: "ghi") → ...
```

**Критичне правило**: Виклик `save_form` є обов'язковим. Без нього форма не існує в Tally, і `list_forms` повертатиме 0 результатів.

**Використання**:

```
/tally-form-builder
Створи форму опитування на тему [topic] з:
- Стор. 1: інтро + питання з чекбоксами та опціями [A, B, C, D]
- Стор. 2: контекстні питання (розмір команди, роль)
- Стор. 3: опціональна контактна інформація (ім'я, email)
Опублікуй як PUBLISHED.
```

```
/tally-form-builder
Редагуй форму [formId]:
- Зміни "2 хв" на "макс. 3 хв" в інтро
- Додай опцію "SMB" до питання про розмір команди
```

**Ключові обмеження (задокументовані у `references/known-issues.md`)**:
- Опції (чекбокс, випадаючий список, множинний вибір) не підтримують HTML — мітки завжди є звичайним текстом
- Опція "Інше" генерує фіксоване мале поле; його неможливо перетворити на textarea через API
- `list_forms` завжди повертає 0, поки не буде викликано `save_form`

**Посилання**:
- Повний скіл: `~/.claude/skills/tally-form-builder/SKILL.md`
- Типи блоків: `~/.claude/skills/tally-form-builder/references/block-types.md`
- Відомі проблеми: `~/.claude/skills/tally-form-builder/references/known-issues.md`
- Шаблон обгортки MCP: `examples/skills/mcp-integration-reference/SKILL.md`

### Пошук через реєстр: ctx7 CLI

Перед тим як занурюватися в конкретні репозиторії, Context7 надає супутній CLI-інструмент (`ctx7`), який автоматизує пошук та встановлення скілів. Замість ручного клонування репозиторіїв, `ctx7 skills suggest` аналізує залежності вашого проекту та рекомендує відповідні скіли з реєстру [context7.com/skills](https://context7.com/skills) — з оцінками довіри для перевірки якості.

**Встановлення**:

```bash
npx ctx7 --help          # Встановлення не потрібне (npx)
npm install -g ctx7       # Глобальне встановлення
```

**Воркфлоу пошуку**:

```bash
# Автовизначення залежностей проекту та пропозиція скілів
npx ctx7 skills suggest

# Пошук за ключовим словом
npx ctx7 skills search terraform

# Встановлення з будь-якого GitHub репозиторію
npx ctx7 skills install antonbabenko/terraform-skill
npx ctx7 skills install owner/repo

# Список / видалення встановлених скілів
npx ctx7 skills list
npx ctx7 skills remove skill-name
```

**Майстер налаштування** (замінює ручне `claude mcp add`):

```bash
# Налаштування Context7 для Claude Code — визначає редактор, обирає режим MCP або CLI+Skills
npx ctx7 setup --claude
```

`ctx7 setup` запускає майстер, який конфігурує Context7 у правильному режимі для вашого редактора. Використовуйте його при першому налаштуванні Context7 замість того, щоб писати `claude mcp add` вручну. Прапор `--claude` націлений саме на Claude Code; `--cursor` та `--universal` доступні для інших редакторів.

**Реєстр проти agentskills.io**: Специфікація [agentskills.io](https://agentskills.io) — це відкритий стандарт, що визначає формат скілів (підтримується 30+ платформами — див. §5.1). Реєстр [context7.com/skills](https://context7.com/skills) — це хостинговий каталог скілів, що відповідають цьому стандарту. Вони доповнюють один одного: agentskills.io визначає формат, context7.com/skills — це місце для пошуку та обміну скілами. Скіли, встановлені через `ctx7`, потрапляють у `~/.claude/skills/` і працюють ідентично до встановлених вручну.

**Генерація скілів** (авторизовано, з лімітами):

```bash
npx ctx7 skills generate    # Кастомний скіл, згенерований ШІ
                             # Free: 6 генерацій/тиждень — Pro: 10/тиждень
```

Генерацію краще залишати для скілів, які не мають аналогів у реєстрі. Для онбордингу команд у масштабі воркфлоу `suggest` + `install` є більш практичним.

**Пошук документації через CLI** (альтернатива MCP):

```bash
# Пошук доступних бібліотек
npx ctx7 library react

# Отримання доксів для конкретної бібліотеки + запит
npx ctx7 docs /facebook/react "useEffect cleanup"
```

Це термінальний еквівалент того, що робить MCP-сервер Context7. Корисно, коли ви хочете знайти щось самостійно без виклику Claude, або в середовищах, де MCP не налаштовано. Користувачам Claude Code, у яких вже активовано MCP-сервер, це не потрібно — Claude обробляє це автоматично.

#### Чому це важливо

Цей скіл демонструє патерн **"скіл-що-створює-скіли"** — мета-підхід, де Claude Code покращує себе через навчання в сесіях. Натхненно академічними роботами про бібліотеки скілів (Voyager, CASCADE, SEAgent, Reflexion).

### Автоматичне покращення скілів: Claude Reflect System

**Репозиторій**: [claude-reflect-system](https://github.com/haddock-development/claude-reflect-system)
**Автор**: Haddock Development | **Статус**: Готово до продакшену (2026)
**Маркетплейс**: [Agent Skills Index](https://agent-skills.md/skills/haddock-development/claude-reflect-system/reflect)

У той час як Claudeception створює нові скіли з виявлених патернів, **Claude Reflect System** автоматично покращує існуючі скіли, аналізуючи фідбек Claude та виявлені виправлення під час сесій.

#### Як це працює

Claude Reflect працює у двох режимах:

**Ручний режим** (`/reflect [skill-name]`):
```bash
/reflect design-patterns  # Аналіз та пропозиція покращень для конкретного скіла
```

**Автоматичний режим** (хук Stop):
1. **Моніторить** тригери хука Stop (кінець сесії, помилка, явна зупинка)
2. **Парсить** транскрипт сесії на наявність фідбеку, пов'язаного зі скілами
3. **Класифікує** тип покращення (виправлення, посилення, новий приклад)
4. **Пропонує** модифікації скіла з рівнем впевненості (HIGH/MED/LOW)
5. **Чекає** на явне рев'ю та схвалення користувача
6. **Бекіпить** оригінальний файл скіла в Git
7. **Застосовує** зміни з валідацією (синтаксис YAML, структура маркдауну)
8. **Робить коміт** з описовим повідомленням

#### Функції безпеки

| Функція | Мета | Реалізація |
|---------|---------|----------------|
| **Рев'ю користувача** | Запобігання небажаним автоматичним змінам | Усі пропозиції потребують схвалення перед застосуванням |
| **Git бекапи** | Можливість відкату невдалих покращень | Авто-коміти перед кожною модифікацією |
| **Валідація синтаксису** | Збереження цілісності файлу скіла | Валідація YAML та маркдауну перед записом |
| **Рівні впевненості** | Пріоритет якісним покращенням | HIGH (чітке виправлення) > MED (ймовірне покращення) > LOW |
| **Механізм блокування** | Запобігання одночасним змінам | Блокування файлів під час аналізу та застосування |

#### Приклад кейсу

**Проблема**: Ви використовуєте скіл `terraform-validation`, який не ловить специфічну помилку конфігурації безпеки. Під час сесії Claude виявляє та виправляє проблему вручну.

**Reflect System виявляє**:
- Claude виправив патерн, не охоплений скілом
- Виправлення було верифіковано (тести пройшли)
- Висока впевненість (чітке покращення)

**Пропозиція**:
```yaml
Skill: terraform-validation
Confidence: HIGH
Change: Додати валідацію шифрування бакетів S3
Diff:
  + - Check bucket encryption: aws_s3_bucket.*.server_side_encryption_configuration
  + - Reject: Шифрування не встановлено або використовується AES256 замість aws:kms
```

**Користувач переглядає** → схвалює → **скіл оновлено** → майбутні сесії автоматично ловитимуть цю проблему.

#### ⚠️ Попередження щодо безпеки

Системи, що самовдосконалюються, несуть специфічні ризики безпеки. Claude Reflect System включає заходи пом'якшення, але користувачі повинні бути пильними:

| Ризик | Опис | Захід пом'якшення | Відповідальність користувача |
|------|-------------|------------|---------------------|
| **Отруєння фідбеку** | Шкідливий ввід маніпулює пропозиціями | Рев'ю користувача, оцінка впевненості | Перевіряйте всі пропозиції HIGH, відхиляйте підозрілі |
| **Отруєння пам'яті** | Шкідливі правки патернів накопичуються | Git бекапи, валідація синтаксису | Періодично аудіюйте історію скіла через Git |
| **Промпт-ін'єкція** | Вбудовані інструкції в транскриптах | Санітизація вводу, ізоляція пропозицій | Ніколи не схвалюйте пропозиції з командами, що виконуються |
| **Роздуття скіла** | Безконтрольне зростання без кураторства | Ручний режим `/reflect [skill]`, регулярне кураторство | Архівуйте або мерджіть надлишкові патерни раз на квартал |
