<constraints>
- Має зберігати REST-ендпоінти під час переходу (6-місячне перекриття)
- Мобільний додаток не можна оновити примусово негайно
</constraints>

<output>
Багатофазний план міграції зі стратегією відкату
</output>
```

Потім використовуйте `/plan` для вивчення в режимі "тільки читання" перед реалізацією.

**XML + Обізнаність про витрати**:

Для великих запитів структуруйте їх за допомогою XML, щоб допомогти Claude зрозуміти обсяг і оцінити використання токенів:

```xml
<instruction>Проаналізуй всі TypeScript файли на наявність невикористаних імпортів</instruction>

<scope>
  директорія src/ (~200 файлів)
</scope>

<output_format>
  Тільки підсумковий звіт (не перелічуй кожен файл)
</output_format>
```

Це допомагає Claude оптимізувати підхід до аналізу та зменшити споживання токенів.

### Бібліотека шаблонів прикладів

Створюйте багаторазові шаблони в `claudedocs/templates/`:

**`claudedocs/templates/code-review.xml`**:

```xml
<instruction>
Перевір наведений нижче код на якість та відповідність найкращим практикам
</instruction>

<context>
[Опишіть призначення компонента та архітектурний контекст]
</context>

<code_example>
[Вставте код сюди]
</code_example>

<focus_areas>
- Вразливості безпеки
- Вузькі місця продуктивності
- Проблеми підтримки коду
- Прогалини в покритті тестами
</focus_areas>

<output>
1. Знайдені проблеми (класифіковані за критичністю)
2. Конкретні рекомендації з прикладами коду
3. Порядок пріоритетності виправлень
</output>
```

**Використання**:

```bash
cat claudedocs/templates/code-review.xml | \
  sed 's/\[Вставте код сюди\]/'"$(cat src/auth.ts)"'/' | \
  claude -p "Оброби цей запит на рев'ю"
```

### Обмеження та зауваження

**Накладні витрати на токени**: XML-теги споживають токени. Для простих запитів природна мова ефективніша.

**Не обов'язково**: Claude чудово розуміє природну мову. Використовуйте XML, коли структура дійсно допомагає.

**Важливість послідовності**: Якщо ви використовуєте XML-теги, будьте послідовними. Змішування стилів у межах однієї сесії може заплутати контекст.

**Крива навчання**: Члени команди повинні розуміти систему тегів. Задокументуйте ваші конвенції в CLAUDE.md.

> **💡 Професійна порада**: Починайте з промптів природною мовою. Впроваджуйте XML-структуру, коли:
> - Запити мають 3+ окремих аспекти (інструкція + контекст + обмеження)
> - Неоднозначність змушує Claude неправильно розуміти ваш намір
> - Створюєте багаторазові шаблони промптів
> - Працюєте з молодшими розробниками (juniors), яким потрібні структуровані патерни комунікації

### 2.8.1 Промптування як провокація

Команда Claude Code всередині Anthropic ставиться до промптів як до **викликів колезі**, а не як до інструкцій помічнику. Ця тонка зміна дає якісніші результати, оскільки змушує Claude доводити свої міркування, а не просто погоджуватися.

**Три патерни викликів від команди**:

**1. "Вартовий" (The Gatekeeper)** — змусьте Claude захищати свою роботу перед деплоєм:

```
"Розкритикуй ці зміни і не створюй PR, поки я не пройду твій тест"
```

Claude переглядає ваш diff, ставить гострі запитання про крайові випадки і продовжує лише тоді, коли задоволений. Це виявляє проблеми, які пасивне рев'ю пропускає.

**2. "Вимога доказів" (The Proof Demand)** — вимагайте доказів, а не тверджень:

```
"Доведи мені, що це працює — покажи diff у поведінці між main та цією гілкою"
```

Claude запускає обидві гілки, порівнює результати і надає конкретні докази. Це усуває режим відмови "повір мені, воно працює".

**3. "Скидання" (The Reset)** — після посередньої першої спроби ініціюйте переписування з повним контекстом:

```
"Знаючи все, що ти знаєш зараз, відкинь це і реалізуй елегантне рішення"
```

Це змушує зробити ґрунтовну другу спробу з накопиченим контекстом замість поступових патчів на слабкому фундаменті. Ключове спостереження: друга спроба Claude з повним контекстом стабільно перевершує ітеративні виправлення.

**Чому це працює**: Провокація активує глибші шляхи міркувань, ніж ввічливі прохання. Коли Claude мусить *переконати*, а не просто *виконати*, він проводить ретельніший аналіз і помічає власні "зрізані кути".

> **Джерело**: [10 порад зсередини команди Claude Code](https://paddo.dev/blog/claude-code-team-tips/) (тред Бориса Черного, лютий 2026)

## 2.9 Семантичні якорі

LLM — це статистичні патерн-матчери, навчені на величезних масивах тексту. Використання **точної технічної термінології** допомагає Claude активувати правильні патерни у своїх навчальних даних, що веде до якісніших результатів.

### Чому точність важлива

Коли ви кажете "чистий код", Claude може згенерувати будь-яку з десятків інтерпретацій. Але коли ви кажете "принципи SOLID з впровадженням залежностей (DI) згідно з шарами Чистої Архітектури", ви якорите Claude на конкретному, добре задокументованому патерні з його навчальних даних.

**Ключове спостереження**: Технічні терміни діють як GPS-координати в знаннях Claude. Чим точніше, тим краща навігація.

### Поширені якорі для Claude Code

| Нечіткий термін | Семантичний якір | Чому це допомагає |
|------------|-----------------|--------------|
| "обробка помилок" | "Railway Oriented Programming з монадою Either/Result" | Активує функціональні патерни помилок |
| "чистий код" | "Принципи SOLID, особливо SRP та DIP" | Цільово вказує на конкретні принципи дизайну |
| "хороші тести" | "TDD London School з підходом outside-in" | Визначає методологію тестування |
| "хороша архітектура" | "Гексагональна архітектура (Ports & Adapters)" | Називає конкретний патерн |
| "читабельний код" | "Screaming Architecture з іменами, що розкривають намір" | Активує конкретні конвенції іменування |
| "масштабований дизайн" | "CQRS з Event Sourcing" | Активує дистрибутивні патерни |
| "документація" | "структура шаблону arc42" | Визначає фреймворк документації |
| "вимоги" | "синтаксис EARS для вимог (Easy Approach to Requirements)" | Цільово вказує формат вимог |
| "дизайн API" | "REST Level 3 з HATEOAS" | Визначає рівень зрілості |
| "безпека" | "заходи OWASP Top 10" | Активує знання з безпеки |

### Як використовувати в CLAUDE.md

Додайте семантичні якорі до інструкцій вашого проекту:

```markdown
# Архітектурні принципи

Дотримуйся цих патернів:
- **Архітектура**: Гексагональна архітектура (Ports & Adapters) з чіткими межами домену
- **Обробка помилок**: Railway Oriented Programming — ніколи не викидай помилки (throw), повертай Result<T, E>
- **Тестування**: TDD London School — використовуй моки для колабораторів, тестуй поведінку, а не реалізацію
- **Документація**: ADR (Architecture Decision Records) для важливих рішень
```

### Поєднання з XML-тегами

Семантичні якорі потужно працюють з XML-структурованими промптами (Розділ 2.8):

```xml
<instruction>
  Рефактори сервіс користувачів згідно з Domain-Driven Design (Evans)
</instruction>

<constraints>
  - Застосуй Гексагональну архітектуру (Ports & Adapters)
  - Використовуй патерн Repository для персистентності
  - Впровадь Railway Oriented Programming для обробки помилок
  - Дотримуйся CQRS для розділення читання/запису
</constraints>

<quality_criteria>
  - Screaming Architecture: структура пакетів розкриває намір
  - Single Responsibility Principle для кожного класу
  - Dependency Inversion: залежність від абстракцій
</quality_criteria>
```

### Семантичні якорі за доменами

**Тестування**:
- TDD London School (mockist) проти Chicago School (classicist)
- Property-Based Testing (у стилі QuickCheck)
- Mutation Testing (PIT, Stryker)
- BDD Gherkin синтаксис (Given/When/Then)

**Архітектура**:
- Гексагональна архітектура (Ports & Adapters)
- Чиста архітектура (Onion layers)
- CQRS + Event Sourcing
- Модель C4 (Context, Container, Component, Code)

**Патерни дизайну**:
- Патерни Банди чотирьох (Gang of Four) (вказуйте: Strategy, Factory, Observer...)
- Тактичні патерни Domain-Driven Design (Aggregate, Repository, Domain Event)
- Функціональні патерни (Monad, Functor, Railway)

**Вимоги**:
- EARS (Easy Approach to Requirements Syntax)
- User Story Mapping (Джефф Паттон)
- Фреймворк Jobs-to-be-Done
- BDD сценарії

> **💡 Професійна порада**: Коли Claude створює занадто загальний код, спробуйте додати конкретніші якорі. "Використовуй чистий код" → "Застосуй каталог рефакторингу Мартіна Фаулера, зокрема Extract Method та Replace Conditional with Polymorphism."

> **Повний каталог**: Див. [examples/semantic-anchors/anchor-catalog.md](../examples/semantic-anchors/anchor-catalog.md) для вичерпного довідника, впорядкованого за доменами.

## 2.10 Потік даних та приватність

> **Важливо**: Все, чим ви ділитеся з Claude Code, надсилається на сервери Anthropic. Розуміння цього потоку даних є критично важливим для захисту конфіденційної інформації.

### Що надсилається в Anthropic

Коли ви використовуєте Claude Code, наступні дані залишають вашу машину:

| Тип даних | Приклад | Рівень ризику |
|-----------|---------|------------|
| Ваші промпти | "Виправ баг з логіном" | Низький |
| Файли, які читає Claude | `.env`, `src/app.ts` | **Високий**, якщо містять секрети |
| Результати запитів MCP | SQL результати з даними користувачів | **Високий**, якщо це продакшн-дані |
| Виводи команд | вивід `env \| grep API` | Середній |
| Повідомлення про помилки | Стек-трейси зі шляхами до файлів | Низький |

### Політики зберігання (Retention)

| Конфігурація | Зберігання | Як увімкнути |
|---------------|-----------|---------------|
| **За замовчуванням** | 5 років | (стандартний стан — навчання увімкнено) |
| **Відмова (Opt-out)** | 30 днів | [claude.ai/settings](https://claude.ai/settings/data-privacy-controls) |
| **Enterprise (ZDR)** | 0 днів | Enterprise-контракт |

**Негайна дія**: [Вимкніть використання даних для навчання](https://claude.ai/settings/data-privacy-controls), щоб скоротити термін зберігання з 5 років до 30 днів.

### Захист конфіденційних даних

**1. Заблокуйте доступ до конфіденційних файлів** у `.claude/settings.json`:

```json
{
  "permissions": {
    "deny": [
      "Read(./.env*)",
      "Edit(./.env*)",
      "Write(./.env*)",
      "Bash(cat .env*)",
      "Bash(head .env*)",
      "Read(./secrets/**)",
      "Read(./**/*.pem)",
      "Read(./**/*.key)",
      "Read(./**/credentials*)"
    ]
  }
}
```

> **Попередження**: `permissions.deny` має відомі обмеження. Див. [Security Hardening Guide](./security/security-hardening.md#known-limitations-of-permissionsdeny) для деталей.

**2. Ніколи не підключайте продакшн-бази даних** до MCP-серверів. Використовуйте dev/staging з анонімізованими даними.

**3. Використовуйте безпекові хуки**, щоб блокувати читання секретних файлів (див. [Розділ 7.4](#74-hooks-automating-workflows)).

> **Повний посібник**: Вичерпну документацію з приватності, включаючи відомі ризики, інциденти в спільноті та корпоративні аспекти, див. у [Data Privacy & Retention Guide](./security/data-privacy.md).

### Головний цикл (The Master Loop)

За своєю суттю Claude Code — це простий цикл `while`:

```
┌─────────────────────────────────────────────────────────────┐
│                 ГОЛОВНИЙ ЦИКЛ (спрощено)                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   Ваш промпт                                                │
│       │                                                     │
│       ▼                                                     │
│   ┌────────────────────────────────────────────────────┐    │
│   │   Claude розмірковує (без класифікатора чи роутера)│    │
│   └───────────────────────┬────────────────────────────┘    │
│                           │                                 │
│          Потрібен інструмент?                               │
│                     ┌─────┴─────┐                           │
│                    ТАК          НІ                          │
│                     │           │                           │
│                     ▼           ▼                           │
│           Виконати інструмент  Текстова відповідь (готово)  │
│                     │                                       │
│                     └────── Передати результат назад Claude │
│                                        │                    │
│                                 (цикл триває)               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**Джерело**: [Anthropic Engineering Blog](https://www.anthropic.com/engineering/claude-code-best-practices)

Тут немає:
- Класифікатора намірів чи роутера завдань
- Пайплайну RAG/ембеддінгів
- Оркестратора DAG
- Розділення на планувальник та виконавець

Сама модель вирішує, коли викликати інструменти, які саме та коли роботу завершено.

### Арсенал інструментів

Claude Code має 8 основних інструментів:

| Інструмент | Призначення |
|------|---------|
| `Bash` | Виконання команд оболонки (універсальний адаптер) |
| `Read` | Читання вмісту файлів (макс. 2000 рядків) |
| `Edit` | Модифікація існуючих файлів (на основі diff) |
| `Write` | Створення/перезапис файлів |
| `Grep` | Пошук у вмісті файлів (на базі ripgrep) |
| `Glob` | Пошук файлів за паттерном |
| `Task` | Створення субагентів (ізольований контекст) |
| `TodoWrite` | Відстеження прогресу (застаріле, див. нижче) |

**Як працює виконання інструментів**: Claude Code може почати виконання інструментів, позначених як безпечні для паралельного запуску (операції тільки для читання, як-от `Read`, `Grep`, `Glob`), поки модель ще генерує відповідь. Це скорочує загальний час очікування. Непаралельні інструменти (запис, bash-команди) чекають завершення генерації та виконуються послідовно. Коли в одній відповіді з'являється кілька інструментів "тільки для читання", вони запускаються паралельно — до 10 одночасно за замовчуванням.

### Система управління завданнями

**Версія**: У Claude Code v2.1.16+ впроваджено нову систему управління завданнями

Claude Code пропонує два підходи до управління завданнями:

| Функція | TodoWrite (Застаріле) | Tasks API (v2.1.16+) |
|---------|-------------------|---------------------|
| **Персистентність** | Тільки пам'ять сесії | Дискове сховище (`~/.claude/tasks/`) |
| **Мультисесійність** | ❌ Втрачається після сесії | ✅ Зберігається між сесіями |
| **Залежності** | ❌ Ручний порядок | ✅ Блокування завдань (A блокує B) |
| **Координація** | Один агент | ✅ Трансляція між багатьма агентами |
| **Відстеження статусу** | pending/in_progress/completed | pending/in_progress/completed/failed |
| **Видимість опису** | ✅ Завжди видимий | ⚠️ Тільки через TaskGet (не в TaskList) |
| **Видимість метаданих** | Н/Д | ❌ Ніколи не видно у виводах |
| **Накладні витрати** | Немає | ⚠️ 1 + N викликів для N повних завдань |
| **Увімкнено** | Завжди доступно | За замовчуванням з v2.1.19 |

#### Tasks API (v2.1.16+)

**Доступні інструменти:**
- `TaskCreate` — Ініціалізація нових завдань з ієрархією та залежностями
- `TaskUpdate` — Зміна статусу, метаданих та залежностей завдання
- `TaskGet` — Отримання деталей конкретного завдання
- `TaskList` — Список усіх завдань у поточному списку
- ~~`TaskOutput`~~ — **Застаріло (v2.1.83+)**. Використовуйте `Read` для `.claude/tasks/<id>/output.log`, щоб отримати вивід завдання напряму.

**Основні можливості:**
- **Персистентне сховище**: Завдання зберігаються в `~/.claude/tasks/<task-list-id>/`
- **Координація між сесіями**: Спільний стан між кількома сесіями Claude
- **Відстеження залежностей**: Завдання можуть блокувати інші (завдання А блокує завдання Б)
- **Життєвий цикл статусу**: pending → in_progress → completed/failed
- **Метадані**: Додавання кастомних даних (пріоритет, оцінки, пов'язані файли тощо)

**Конфігурація:**

```bash
# Увімкнення персистентності завдань між сесіями
export CLAUDE_CODE_TASK_LIST_ID="project-name"
claude

# Приклад: Список завдань для конкретного проекту
export CLAUDE_CODE_TASK_LIST_ID="api-v2-auth-refactor"
claude
```

**⚠️ Важливо**: Використовуйте ID списків завдань, специфічні для репозиторію, щоб уникнути перемішування даних між проектами. Завдання з однаковим ID є спільними для всіх сесій, що використовують цей ID.

**Приклад схеми завдання:**

```json
{
  "id": "task-auth-login",
  "title": "Реалізувати ендпоінт логіну",
  "description": "POST /auth/login з генерацією JWT-токена",
  "status": "in_progress",
  "dependencies": [],
  "metadata": {
    "priority": "high",
    "estimated_duration": "2h",
    "related_files": ["src/auth/login.ts", "src/middleware/auth.ts"]
  }
}
```

**Коли використовувати Tasks API:**
- Проекти, що охоплюють кілька робочих сесій
- Складні ієрархії завдань із залежностями
- Сценарії мультиагентної координації
- Потреба відновити роботу після стиснення контексту (compaction)

**⚠️ Обмеження Tasks API (Критично)**

**Обмеження видимості полів**:

| Інструмент | Видимі поля | Приховані поля |
|------|----------------|---------------|
| `TaskList` | `id`, `subject`, `status`, `owner`, `blockedBy` | `description`, `activeForm`, `metadata` |
| `TaskGet` | Всі поля | — |

**Вплив**:
- **Накладні витрати на виклики**: Перегляд 10 описів завдань = 1 TaskList + 10 TaskGet викликів (11-кратне зростання)
- **Неможливість сканування метаданих**: Неможна фільтрувати/сортувати за кастомними полями (пріоритет, оцінки, теги) без отримання кожного завдання окремо
- **Складність відновлення сесії**: Неможливо швидко переглянути всі замітки до завдань, щоб вирішити, з чого почати

**Приклад вартості**:
```bash
# Неефективно (якщо потрібні описи)
TaskList  # Повертає 10 завдань (без описів)
TaskGet(task-1), TaskGet(task-2), ..., TaskGet(task-10)  # ще 10 викликів

# Разом: 11 API-викликів для огляду 10 завдань
```

**Обхідні патерни**:

1. **Гібридний підхід** (Рекомендовано):
   - Використовуйте Tasks API для **відстеження статусу** та **координації залежностей**
   - Ведіть markdown-файли в репозиторії для **детальних планів реалізації**
   - Приклад: `docs/plans/auth-refactor.md` + Tasks для статусів

2. **Патерн "Тема як резюме" (Subject-as-summary)**:
   - Зберігайте критичну інформацію в полі `subject` (завжди видиме в TaskList)
   - Використовуйте `description` для глибокого контексту (отримуйте за запитом через TaskGet)
   - Приклади тем: `"[P0] Fix login bug (src/auth.ts:45)"` замість `"Fix bug"`

3. **Вибіркове отримання даних**:
   - Використовуйте TaskList для ідентифікації завдань, що потребують уваги (status, blockedBy)
   - Викликайте TaskGet тільки для тих завдань, над якими активно працюєте

**Джерело**: Відгуки практиків спільноти ([Ганг Руй, січень 2026](https://www.linkedin.com/posts/limgangrui_i-explored-the-new-claude-codes-task-system-activity-7420651412881268736-Hpd6))

#### TodoWrite (Застаріле)

**Інструмент**: `TodoWrite` — створює списки завдань, що зберігаються в пам'яті сесії

**Можливості:**
- Просте відстеження завдань у межах однієї сесії
- Статуси: pending/in_progress/completed
- Втрачається при завершенні сесії або стисненні контексту

**Коли використовувати TodoWrite:**
- Проста реалізація в межах однієї сесії
- Швидкі виправлення або експериментальний код
- Версія Claude Code < v2.1.16
- Перевага простоти над персистентністю

**Прапорець міграції** (v2.1.19+):

```bash
# Тимчасове повернення до системи TodoWrite
CLAUDE_CODE_ENABLE_TASKS=false claude

# Використання нового Tasks API (за замовчуванням)
claude
```

#### Найкращі практики

**Дизайн ієрархії завдань:**
```
Проект (parent)
└── Фіча A (child)
    ├── Компонент A1 (leaf task)
    │   ├── Реалізація
    │   └── Тести (залежать від Реалізації)
    └── Компонент A2
```

**Управління залежностями:**
- Завжди визначайте залежності під час створення завдань
- Використовуйте ID завдань (не заголовки) для посилань на залежності
- Перевіряйте залежності за допомогою `TaskGet` перед виконанням

**Переходи статусів:**
- Позначайте `in_progress`, коли починаєте роботу (це запобігає паралельному виконанню)
- Оновлюйте часто для видимості прогресу
- Позначайте `completed` тільки тоді, коли мета повністю досягнута (тести пройдено, перевірено)
- Використовуйте статус `failed` із метаданими помилок для відладки

**Конвенції метаданих:**
```json
{
  "priority": "high|medium|low",
  "estimated_duration": "2h",
  "related_files": ["path/to/file.ts"],
  "related_issue": "https://github.com/org/repo/issues/123",
  "type": "feature|bugfix|refactor|test"
}
```

#### Списки завдань як діагностичний інструмент

**Діагностичний принцип**: Коли список завдань Claude не відповідає вашому наміру, проблема не в Claude — проблема у ваших інструкціях.

Списки завдань діють як **дзеркало** чіткості інструкцій. Якщо ви просите Claude спланувати фічу і результат вас дивує, це розходження є діагностичною інформацією:

```
Ваша інструкція: "Рефактори систему автентифікації"

Список завдань Claude:
- [ ] Прочитати всі файли автентифікації
- [ ] Виявити дублювання коду
- [ ] Винести спільні утиліти
- [ ] Оновити імпорти
- [ ] Запустити тести

Ваша реакція: "Я не це мав на увазі — я хотів перейти з сесій на JWT"

Діагноз: Ваша інструкція була неоднозначною. "Рефакторинг" ≠ "заміна".
```

**Патерни розходжень та що вони розкривають:**

| Тип розходження | Що це означає | Виправлення |
|-----------------|---------------|-----|
| Завдання занадто загальні | Інструкціям бракує специфіки | Додайте ЩО, ДЕ, ЯК, ПЕРЕВІРКА |
| Завдання занадто детальні | Забагато деталей, втрачено загальну картину | Вкажіть мету, а не просто кроки |
| Невірні пріоритети | Відсутній контекст про те, що важливо | Додайте обмеження та пріоритети |
| Пропущені завдання | Не поділилися неявними знаннями | Зробіть припущення явними в промпті |
| Зайві завдання | Claude додумав вимоги, яких ви не мали | Додайте чіткі межі обсягу (scope) |

**Використання розходження завдань як воркфлоу:**

```markdown
## Крок 1: Посів загальної інструкції
Користувач: "Покращи процес оформлення замовлення"

## Крок 2: Перегляд плану Claude (ще не виконувати)
Claude генерує: [список завдань]

## Крок 3: Порівняння з вашою ментальною моделлю
- Пропущено: логіка повторних спроб оплати? → Додати до інструкцій
- Неочікувано: редизайн UI? → Уточнити обсяг (тільки бекенд)
- Невірний порядок: тести в кінці? → Вказати підхід TDD

## Крок 4: Уточнення та перепланування
Користувач: "Насправді мені потрібно ось що: [уточнена інструкція зі специфікою]"
```

**Професійна порада**: Запустіть `TaskList` після початкового планування як **перевірку на адекватність (sanity check)** перед виконанням. Якщо понад 30% завдань вас дивують, ваш промпт потребує доопрацювання. Ітеруйте промпт, а не завдання.

### Управління контекстом

Claude Code працює в межах **вікна контексту 200K токенів** (1M в бета-версії через API — див. [порівняння 200K vs 1M](рядок 1751)):

| Компонент | Орієнтовний розмір |
|-----------|------------------|
| Системний промпт | 5-15K токенів |
| Файли CLAUDE.md | 1-10K токенів |
| Історія розмови | Змінна |
| Результати інструментів | Змінна |
| Резерв для відповіді | 40-45K токенів |

Коли контекст заповнюється (~75% у VS Code, ~95% у CLI), старий вміст автоматично резюмується. Однак **дослідження показують, що це погіршує якість** (падіння продуктивності на 50-70% у складних завданнях). Використовуйте `/compact` проактивно в логічних точках або ініціюйте **передачу сесії (session handoff) при 85%**, щоб зберегти намір замість стиснутої історії. Див. [Session Handoffs](рядок 2140) та [Auto-Compaction Research](../core/architecture.md#auto-compaction).

### Ізоляція субагентів

Інструмент `Task` створює субагентів з:
- Власним свіжим вікном контексту
- Доступом до тих самих інструментів (крім самого Task)
- **Максимальною глибиною 1** (не можуть створювати суб-субагентів)
- У головний контекст повертається тільки їхній підсумковий текст

Це запобігає забрудненню контексту під час дослідницьких завдань.

### TeammateTool (Експериментально)

**Статус**: Частково під прапорцями функцій, триває поступове впровадження.

TeammateTool забезпечує **мультиагентну оркестрацію** з постійним зв'язком між агентами. На відміну від стандартних субагентів, які працюють ізольовано, "колеги" (teammates) можуть координуватися через структуровані повідомлення.

**Основні можливості**:

| Операція | Призначення |
|-----------|---------|
| `spawnTeam` | Створити іменовану команду агентів |
| `discoverTeams` | Список доступних команд |
| `requestJoin` | Запит агента на приєднання до команди |
| `approveJoin` | Лідер команди схвалює запити на вхід |
| Messaging | Комунікація між агентами на базі JSON |

**Бекенди виконання** (автовизначення):
- **In-process**: Асинхронні завдання в тому ж процесі Node.js (найшвидше)
- **tmux**: Персистентні термінальні сесії (виживають при відключенні)
- **iTerm2**: Візуальне розділення вікон (тільки macOS)

**Патерни**:

```
Патерн паралельних спеціалістів:
Лідер створює 3 колег → Кожен рев'ює окремий аспект (безпека, швидкість, архітектура)
→ Колеги працюють одночасно → Звітують лідеру → Лідер синтезує результат

Патерн рою (Swarm):
Лідер створює спільну чергу завдань → Колеги самоорганізуються та беруть завдання
→ Незалежне виконання → Асинхронні оновлення спільного стану
```

**Обмеження**:
- 5-хвилинний таймаут серцебиття (heartbeat) перед видаленням
- Неможливо видалити команду, поки колеги активні
- Прапорці функцій офіційно не задокументовані (знайдені спільнотою)
- Відсутня офіційна підтримка Anthropic для експериментальних функцій

**Коли використовувати**:
- Великі кодобази, що потребують паралельного аналізу (4+ аспекти)
- Тривалі воркфлоу з незалежними підзавданнями
- Рев'ю коду з багатьма специфічними вимогами

**Коли НЕ використовувати**:
- Прості завдання (витрати на оркестрацію не виправдані)
- Послідовні залежності (достатньо стандартних субагентів)
- Критичні для продакшену воркфлоу (експериментальні функції нестабільні)

**Джерела**:
- **Спільнота**: [kieranklaassen - TeammateTool Guide](https://gist.github.com/kieranklaassen/4f2aba89594a4aea4ad64d753984b2ea)
- **Спільнота**: [GitHub Issue #3013 - Parallel Agent Execution](https://github.com/anthropics/claude-code/issues/3013)
- **Спільнота**: [mikekelly/claude-sneakpeek](https://github.com/mikekelly/claude-sneakpeek) — паралельна збірка з увімкненими прапорцями

> ⚠️ **Примітка**: Це експериментальна функція. Можливості можуть змінитися або бути видалені в майбутніх релізах. Завжди перевіряйте поточну поведінку в офіційній документації.

### Антипатерни агентів: Ролі проти контролю контексту

> **"Субагенти призначені не для антропоморфізації ролей, а для контролю контексту"** — Декс Хорті

**Поширена помилка**: Створення агентів так, ніби це людська команда з посадами.

❌ **Невірно** (Антропоморфізація):
```
- Агент Фронтенд (роль: розробник UI)
- Агент Бекенд (роль: API інженер)
- Агент QA (роль: тестувальник)
- Агент Безпека (роль: експерт з безпеки)
```

**Чому це не працює**: Агенти — це не люди з досвідом. Це **інструменти ізоляції контексту** для обчислювальної ефективності.

✅ **Вірно** (Контроль контексту):
```
- Агент для ізольованого аналізу залежностей (область: тільки package.json + lock файли)
- Агент для паралельної обробки файлів (область: пакетні правки без засмічення основного контексту)
- Агент для свіжого аудиту безпеки (область: аналіз безпеки без попередніх припущень)
- Агент для незалежного тестування модулів (область: запуск тестів без втручання в основний воркфлоу)
```

**Ключові відмінності**:

| Антропоморфізація (Невірно) | Контроль контексту (Вірно) |
|----------------------------|-------------------------|
| "Агент-експерт з безпеки" | "Аудит безпеки з ізольованим контекстом" |
| "Агент-фронтенд розробник" | "Аналіз UI-компонентів (область: тільки src/components/)" |
| "Агент для рев'ю коду" | "Рев'ю PR без засмічення основного контексту" |
| Імітує структуру людської команди | Оптимізує обчислювальні ресурси |
| Базується на посадах | Базується на межах обсягу/контексту |

**Коли використовувати агентів** (вагомі причини):
- **Ізоляція контексту**: Запобігання забрудненню основного контексту розмови
- **Паралельна обробка**: Незалежні операції, що можуть виконуватися одночасно
- **Обмеження області (Scope limitation)**: Обмеження аналізу конкретними файлами/директоріями
- **Свіжий погляд**: Аналіз без "багажу" попередніх міркувань
- **Оптимізація ресурсів**: Перенесення важких операцій в окреме вікно контексту

**Коли НЕ використовувати агентів** (погані причини):
- ❌ Створення вигаданої команди з посадами
- ❌ Рольові ігри з різними "експертними" персонами
- ❌ Імітація людської організаційної структури
- ❌ Розділення роботи за дисциплінами (фронт/бек/QA) замість меж контексту

### Спеціалізовані агенти (Scope-Focused Agents)

Крім звичайних субагентів, **оркестрація за межами обсягу** призначає чіткі **межі контексту** різним агентам для аналізу з різних точок зору.

**Патерн**: Замість того, щоб один агент перевіряв усе, створіть **ізольованих** агентів, кожен з яких аналізує окремі аспекти зі свіжим контекстом:

```markdown
Користувач: Перевір новий платіжний сервіс за допомогою спеціалізованого аналізу:

Агент 1 (Безпека): Проаналізуй автентифікацію, валідацію вводу,
  вектори ін'єкцій, роботу з секретами, відповідність PCI DSS.
  Контекст: src/payment/, src/auth/, config/security.yml

Агент 2 (Продуктивність): Проаналізуй SQL-запити, проблему N+1,
  можливості кешування, вузькі місця часу відповіді.
  Контекст: src/payment/repository/, src/database/, логи повільних запитів

Агент 3 (Дизайн API): Проаналізуй повідомлення про помилки, послідовність
  формату відповідей, доступність API, повноту документації.
  Контекст: src/payment/api/, docs/api/, tests/integration/

Синтезуй результати трьох аналізів у єдиний звіт із пріоритетними завданнями.
```

**Реалізація з кастомними агентами**:

```yaml
# .claude/agents/security-audit.md
---
name: security-audit
model: opus
tools: Read, Grep, Glob
---
Проаналізуй код на проблеми безпеки з ізольованим контекстом:
- Вразливості OWASP Top 10
- Недоліки автентифікації/авторизації
- Прогалини у валідації вводу
- Ризики розкриття секретів

Область (Scope): Тільки аналіз безпеки. Звітуй про знахідки з оцінкою
критичності (Критично/Високо/Середньо/Низько) без врахування компромісів
щодо продуктивності чи UX.
```

```yaml
# .claude/agents/perf-audit.md
---
name: perf-audit
model: sonnet
tools: Read, Grep, Glob, Bash
---
Проаналізуй код на вузькі місця продуктивності з ізольованим контекстом:
- Ефективність SQL-запитів (N+1, відсутні індекси)
- Витоки пам'яті та управління ресурсами
- Можливості кешування
- Проблеми алгоритмічної складності

Область (Scope): Тільки аналіз продуктивності. Звітуй про знахідки з
оцінкою впливу (Високий/Середній/Низький) без врахування компромісів
щодо безпеки чи підтримки коду.
```

**Коли використовувати спеціалізованих агентів:**
- Аналіз, що потребує 3+ чітких меж контексту (безпека, швидкість, API)
- Конкуруючі інтереси, що виграють від ізольованої оцінки (швидкість vs безпека vs DX)
- Великі кодобази, де повний контекст завадив би аналізу специфічних аспектів

**Коли НЕ використовувати:**
- Прості рев'ю (один агент з повним контекстом охоплює всі аспекти)
- Обмеження в часі (витрати на синтез перевищують користь)
- Завдання, де області не є по-справжньому незалежними (потрібен спільний контекст)

### Філософія

> "Робіть більше з меншими ресурсами. Розумний вибір архітектури, краща ефективність навчання та фокус на вирішенні проблем можуть конкурувати з масштабом."
> — Даніела Амодей, CEO Anthropic

Claude Code довіряє міркуванням моделі замість побудови складних систем оркестрації. Це означає:
- Менше компонентів = менше сценаріїв збою
- Рішення, керовані моделлю = краща узагальнюваність
- Простий цикл = легка відладка
