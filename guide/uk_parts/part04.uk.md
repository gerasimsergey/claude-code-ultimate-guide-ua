### Патерн передачі сесії (Session Handoff)

Завершуючи сесію або перемикаючи контексти, створюйте **документ передачі** (handoff), щоб зберегти безперервність.

**Мета**: Подолати розрив між сесіями шляхом документування стану, рішень та наступних кроків.

**Шаблон**:

```markdown
# Передача сесії - [Дата] [Час]

## Що було зроблено
- [Завершено ключове завдання 1]
- [Завершено ключове завдання 2]
- [Змінено файли: список]

## Поточний стан
- [Що працює]
- [Що зроблено частково]
- [Відомі проблеми або блокери]

## Прийняті рішення
- [Архітектурний вибір 1: чому]
- [Вибір технології: обґрунтування]
- [Прийняті компроміси]

## Наступні кроки
1. [Найближче наступне завдання]
2. [Залежне завдання]
3. [Подальша валідація]

## Контекст для наступної сесії
- Гілка: [назва-гілки]
- Ключові файли: [список 3-5 найбільш релевантних]
- Залежності: [зовнішні фактори]
```

**Коли створювати документи передачі**:

| Сценарій | Навіщо |
|----------|-----|
| Кінець робочого дня | Безшовне продовження завтра |
| Перед лімітом контексту | Збереження стану перед `/clear` |
| Зміна сфери фокусування | Інше завдання потребує свіжого контексту |
| Очікується переривання | Надзвичайна ситуація або зустріч перериває роботу |
| Складне відлагодження | Документування гіпотез та проведених тестів |

**Місце зберігання**: `claudedocs/handoffs/handoff-YYYY-MM-DD.md`

**Професійна порада**: Попросіть Claude згенерувати передачу:

```
Ви: "Створи документ передачі сесії (handoff) про те, що ми сьогодні зробили"
```

Claude проаналізує git status, історію розмови та згенерує структурований документ.

**Патерн "Тріада передачі" (Handoff Triad Pattern)**: Для команд або багатосесійних воркфлоу протокол із трьох команд додає явну семантику злиття поверх базової передачі:

| Команда | Роль |
|---------|-----|
| `/handoff:create` | Генерує структурований документ із контексту поточної сесії |
| `/handoff:resume` | Завантажує документ передачі, підтверджує розуміння та чекає на схвалення перед стартом |
| `/handoff:update` | Оновлює існуючу передачу за правилами злиття секцій (див. нижче) |

Критичним доповненням у `update` є правила злиття для кожної секції:

| Секція | Правило злиття |
|---------|------------|
| Завдання, обсяг (Scope) | Зберегти або уточнити |
| Файли | Merge — об'єднати оригінальні файли з новими зміненими |
| Відкриття (Discoveries) | Append — додати нові знахідки, ніколи не видаляти попередні |
| Виконана робота | **Append only** — додавати нові записи, ніколи не видаляти історію, включати хеші комітів |
| Статус | Replace — записати поточний стан |
| Наступні кроки | Replace — записати оновлений чек-лист |

Секція "Виконана робота" (Work Done) за принципом append-only створює контрольний слід через усі сесії. Навіть якщо попередня робота була переглянута, зміна з'являється як новий запис, а не перезапис.

Готові шаблони знаходяться в `examples/commands/handoff/` у цьому репозиторії.

> Патерн натхненний [тріадою команд передачі Packmind](https://github.com/packmind/packmind) (Apache 2.0). Див. [Credits](./core/credits.md).

## 2.3 Режим планування (Plan Mode)

Режим планування — це режим Claude Code "дивись, але не чіпай".

### Вхід у режим планування

```
/plan
```

Або запитайте Claude прямо:

```
Ви: Давай сплануємо цю фічу перед впровадженням
```

### Що дозволяє режим планування

- ✅ Читання файлів
- ✅ Пошук у кодобазі
- ✅ Аналіз архітектури
- ✅ Пропонування підходів
- ✅ Запис у файл плану

### Що забороняє режим планування

- ❌ Редагування файлів
- ❌ Запуск команд, що змінюють стан
- ❌ Створення нових файлів
- ❌ Створення комітів

### Коли використовувати режим планування

| Ситуація | Використовувати Plan Mode? |
|-----------|----------------|
| Дослідження незнайомої кодобази | ✅ Так |
| Розслідування багу | ✅ Так |
| Планування нової фічі | ✅ Так |
| Виправлення друкарської помилки | ❌ Ні |
| Швидка правка у відомому файлі | ❌ Ні |

> **Рекомендована частота**: Борис Черний (керівник Claude Code в Anthropic) починає приблизно **80% завдань у режимі планування** — дозволяючи Claude спланувати все перед написанням хоча б одного рядка коду. Після того, як план схвалено, виконання майже завжди виявляється правильним з першої спроби.
> — *Lenny's Newsletter, 19 лютого 2026*

### Вихід із режиму планування

Натисніть `Shift+Tab`, щоб повернутися у звичайний режим (Act Mode). Ви також можете просто написати повідомлення, і Claude запитає: "Готові впроваджувати цей план?"

> **Примітка**: `Shift+Tab` перемикає між Plan Mode та Normal Mode під час сесії. Натисніть `Shift+Tab` двічі у Normal Mode, щоб увійти в Plan Mode; один раз у Plan Mode — щоб повернутися.

### Auto Plan Mode

**Концепція**: Автоматичний запуск режиму планування перед будь-якою ризикованою операцією.

**Конфігураційний файл** (`~/.claude/auto-plan-mode.txt`):
```
Перед виконанням БУДЬ-ЯКОГО інструменту (Read, Write, Edit, Bash, Grep, Glob, WebSearch) ти МАЄШ:
1. СПОЧАТКУ: Використати інструмент exit_plan_mode, щоб представити свій план.
2. ЗАЧЕКАТИ: Явного схвалення користувача перед продовженням.
3. ТІЛЬКИ ПІСЛЯ ЦЬОГО: Виконувати заплановані дії.

Кожен новий запит користувача потребує свіжого плану — попередні схвалення не поширюються.
```

**Запуск з Auto Plan Mode**:

*macOS/Linux:*
```bash
# Прямий запуск
claude --append-system-prompt "Перед виконанням БУДЬ-ЯКОГО інструменту..."

# Через файл (рекомендовано)
claude --append-system-prompt "$(cat ~/.claude/auto-plan-mode.txt)"

# Аліас у .zshrc/.bashrc
alias claude-safe='claude --append-system-prompt "$(cat ~/.claude/auto-plan-mode.txt)"'
```

*Windows (PowerShell):*
```powershell
# Створіть файл конфігурації за шляхом %USERPROFILE%\.claude\auto-plan-mode.txt з тим самим вмістом

# Прямий запуск
claude --append-system-prompt "Перед виконанням БУДЬ-ЯКОГО інструменту..."

# Через файл (додайте в $PROFILE)
function claude-safe {
    $planPrompt = Get-Content "$env:USERPROFILE\.claude\auto-plan-mode.txt" -Raw
    claude --append-system-prompt $planPrompt $args
}
```

**Отриманий воркфлоу**:
```
Користувач: "Додай поле email до моделі User"

Claude (Auto Plan Mode активний):
┌─────────────────────────────────────────────────────────────┐
│ 📋 ЗАПРОПОНОВАНИЙ ПЛАН                                      │
│                                                             │
│ 1. Прочитати schema.prisma, щоб зрозуміти поточну модель    │
│ 2. Додати поле email: String? @unique                       │
│ 3. Згенерувати міграцію Prisma                              │
│ 4. Оновити типи TypeScript                                  │
│ 5. Додати валідацію Zod у роутерах                          │
│                                                             │
│ ⚠️ Вплив: змінено 3 файли, створено 1 міграцію              │
│                                                             │
│ Схвалити цей план? (y/n)                                    │
└─────────────────────────────────────────────────────────────┘

Користувач: "y"

Claude: [Виконує план]
```

**Результат**: на 76% менше токенів з кращими результатами, оскільки план перевіряється перед виконанням.

### Модельні аліаси (Model Aliases)

Claude Code підтримує шість аліасів моделей через `/model` (кожен завжди посилається на останню версію):

| Аліас | На яку модель посилається | Кейс використання |
|-------|-------------|----------|
| `default` | Остання модель для вашого тарифу | Стандартне використання |
| `sonnet` | Claude Sonnet 4.6 | Швидко, економно |
| `opus` | Claude Opus 4.7 | Глибокі міркування |
| `haiku` | Claude Haiku 4.5 | Бюджетно, великі обсяги |
| `sonnet[1m]` | Sonnet з контекстом 1M | Величезні кодобази |
| `opusplan` | Opus (план) + Sonnet (дія) | Гібридний інтелект |

Модель також можна встановити через `claude --model <alias>`, змінну оточення `ANTHROPIC_MODEL` або поле `"model"` у settings.json. Пріоритет: `/model` > прапор `--model` > `ANTHROPIC_MODEL` > settings.json.

**Дати відсічення знань (Knowledge cutoffs)**:

| Модель | Відсічення знань |
|-------|-----------------|
| Claude Opus 4.7 | Ще не опубліковано |
| Claude Sonnet 4.6 | Серпень 2025 |
| Claude Opus 4.6 | Травень 2025 |
| Claude Haiku 4.5 | Лютий 2025 |

Claude Code вставляє дату відсічення знань для активної моделі в системний промпт на початку кожної сесії. Ви можете запитати Claude прямо: "яка твоя дата відсічення знань?", щоб підтвердити, яка дата актуальна для вашої сесії.

### Режим OpusPlan

**Концепція**: Використовувати Opus для планування (кращі міркування) та Sonnet для реалізації (ефективність витрат).

**Чому OpusPlan?**
- **Оптимізація витрат**: токени Opus дорожчі за Sonnet.
- **Найкраще з обох світів**: планування якості Opus + швидкість виконання Sonnet.
- **Економія токенів**: фаза планування зазвичай коротша за фазу впровадження.

**Активація**:
```
/model opusplan
```

Або в `~/.claude/settings.json`:
```json
{
  "model": "opusplan"
}
```

**Як це працює**:
1. У **Plan Mode** (`/plan` або `Shift+Tab` двічі) → використовується **Opus**.
2. У **Act Mode** (звичайне виконання) → використовується **Sonnet**.
3. Автоматичне перемикання залежно від режиму.

**Рекомендований воркфлоу**:
```
1. /model opusplan        → Увімкнути OpusPlan
2. Shift+Tab × 2          → Увійти в Plan Mode (Opus)
3. Описати завдання       → Отримати план якості Opus
4. Shift+Tab              → Вийти в Act Mode (Sonnet)
5. Виконати план          → Sonnet впроваджує ефективно
```

**Альтернативний підхід із субагентами**:

Ви також можете контролювати використання моделей для кожного агента окремо:

```yaml
# .claude/agents/planner.md
---
name: planner
model: opus
tools: Read, Grep, Glob
---
# Агент стратегічного планування
```

```yaml
# .claude/agents/implementer.md
---
name: implementer
model: haiku
tools: Write, Edit, Bash
---
# Агент швидкої реалізації
```

**Примітка для досвідчених користувачів**: OpusPlan особливо цінний для передплатників Pro з обмеженою кількістю токенів Opus. Це дозволяє використовувати потужність Opus для критичного планування, зберігаючи токени для більшої кількості сесій.

**Бюджетний варіант: SonnetPlan (Хак спільноти)**

`opusplan` жорстко закодований на Opus+Sonnet — рідного аліаса `sonnetplan` не існує. Але ви можете перепризначити аліаси `opus` та `sonnet` через змінні оточення, фактично створивши гібрид Sonnet→Haiku:

```bash
# Додайте в ~/.zshrc
sonnetplan() {
    ANTHROPIC_DEFAULT_OPUS_MODEL=claude-sonnet-4-6 \
    ANTHROPIC_DEFAULT_SONNET_MODEL=claude-haiku-4-5-20251001 \
    claude "$@"
}
```

З `sonnetplan` команда `/model opusplan` перенаправить:
- **Plan Mode** → Sonnet 4.6 (через перепризначений аліас `opus`)
- **Act Mode** → Haiku 4.5 (через перепризначений аліас `sonnet`)

> **Застереження**: самозвіт моделі ("яка ти модель?") ненадійний — моделі не завжди знають власну ідентичність. Довіряйте рядку статусу (`Model: Sonnet 4.6` у режимі планування) або перевіряйте через панель білінгу. Питання рідної підтримки відстежується в тикеті GitHub [#9749](https://github.com/anthropics/claude-code/issues/9749).

### "Прогрів двигуна" (Rev the Engine)

**Концепція**: Запуск кількох раундів планування та глибоких роздумів перед виконанням. Наче прогрівання двигуна перед поїздкою.

Стандартний воркфлоу: подумав → спланував → виконав.
"Прогрів": подумав → спланував → подумав глибше → уточнив план → подумав максимально глибоко → фіналізував → виконав.

**Коли використовувати**:
- Критичні архітектурні рішення (незворотні, з високим впливом).
- Складні міграції, що зачіпають 10+ файлів.
- Незнайома область, де перші інстинкти часто помилкові.

**Патерн**:

```markdown
## Раунд 1: Початковий аналіз
Користувач: /plan
Користувач: Проаналізуй поточну систему автентифікації. Які ключові компоненти,
            залежності та потенційні ризики міграції на OAuth2?
Claude: [Початковий аналіз]

## Раунд 2: Глибокий виклик
Користувач: Тепер використай розширене мислення (extended thinking). Кинь виклик власному аналізу:
            - Які припущення ти зробив?
            - Які сценарії відмов ти пропустив?
            - На що звернув би увагу старший інженер з безпеки?
Claude: [Глибший аналіз із самовиправленням]

## Раунд 3: Фінальний план
Користувач: На основі обох раундів напиши остаточний план міграції.
            Включи стратегію відкату та мінімізацію ризиків для кожного кроку.
Claude: [Уточнений план, що враховує обидва раунди]

## Виконання
Користувач: /execute
Користувач: Впроваджуй план з раунду 3.
```

**Чому це працює**: Кожен раунд змушує Claude переглянути припущення. Раунд 2 зазвичай виявляє 30-40% проблем, які раунд 1 пропустив. Раунд 3 синтезує результати в надійніший план.

### Ultraplan

**Статус**: Research preview — потребує Claude Code v2.1.91+ та акаунта Claude Code на вебі.

**Концепція**: Перенесення планування у хмару Anthropic, поки ваш термінал залишається вільним. Claude готує чернетку плану віддалено, використовуючи кілька агентів Opus 4.7+ паралельно; ви переглядаєте його в браузері з інлайновими коментарями, а потім обираєте: виконати в хмарі чи "телепортувати" план назад у термінал.

Це вирішує головну проблему локального Plan Mode: на складних завданнях термінал блокується на хвилини, поки йде планування. Ultraplan працює асинхронно — ви продовжуєте працювати, перевіряючи готовність за посиланням.

**Як це працює**

1. CLI запускає хмарну сесію → у терміналі з'являється живий індикатор статусу.
2. Кілька агентів Opus 4.7+ паралельно досліджують кодобазу (вікна планування до 30 хвилин).
3. У браузері відкривається план із боковою панеллю змісту, інлайновим коментуванням та емодзі-реакціями.
4. Ви ітеруєте над планом — коментуєте конкретні секції, запитуєте правки.
5. Обираєте місце виконання: хмара (створює PR) або термінал (телепортує план назад).

**Активація (3 методи)**

```bash
# 1. Спеціальна команда
/ultraplan migrate the auth service from sessions to JWTs

# 2. Ключове слово у будь-якому промпті
Plan with ultraplan a full refactor of the payments module

# 3. З діалогу схвалення локального плану
# → оберіть "No, refine with Ultraplan on Claude Code on the web"
```

Шляхи через команду та ключове слово спочатку показують діалог підтвердження. Шлях через локальний план пропускає його.

**Індикатори статусу в терміналі**

| Статус | Значення |
|--------|---------|
| `◇ ultraplan` | Claude проводить дослідження та готує чернетку |
| `◇ ultraplan needs your input` | Потрібні уточнення — відкрийте посилання в браузері |
| `◆ ultraplan ready` | План готовий до огляду |

Запустіть `/tasks`, щоб побачити посилання на сесію, активність агентів та дію **Stop ultraplan**.

**Інтерфейс огляду в браузері**

- **Outline sidebar**: навігація між секціями без скролінгу.
- **Inline comments**: виділяйте будь-який фрагмент, залишайте точковий фідбек.
- **Emoji reactions**: сигналізуйте про схвалення або занепокоєння щодо секції без написання коментаря.
- **Цикли правок (Revision cycles)**: просіть Claude врахувати ваші коментарі; він представить оновлену чернетку — ітеруйте стільки разів, скільки потрібно.

**Виконання: два варіанти**

Коли план виглядає правильно, оберіть у браузері:

| Опція | Що відбувається |
|--------|-------------|
| **Approve and start coding** | Хмарна сесія впроваджує план, створює PR; термінал очищується |
| **Approve and teleport back** | План надсилається у ваш термінал із 3 під-варіантами |

Під-варіанти телепортації:
- **Implement here** — вставити план у поточну розмову, почати негайно.
- **Start new session** — нова сесія з планом як контекстом (виводить `claude --resume` для повернення до поточної сесії).
- **Cancel** — зберігає план у файл, виводить шлях до нього.

|-------------|--------|

| Акаунт | Pro, Max, Team або Enterprise (не безкоштовний) |

| Провайдери | Тільки Anthropic API — недоступно на Bedrock, Vertex, Foundry |
| Конфлікт | Несумісно з Remote Control (обидва використовують claude.ai/code) |

**Ultraplan vs. OpusPlan vs. Plan Mode**

| Функція | Plan Mode | OpusPlan | Ultraplan |
|---------|-----------|----------|-----------|
| Виконання | Локально | Локально | Хмарно |
| Термінал заблоковано? | Так | Так | Ні |
| Моделі | Активна модель | Opus (план) + Sonnet (дія) | Opus 4.7 (мультиагентна) |

| Облік токенів | Рахується локально | Рахується локально | Хмарне планування безкоштовне для локальної квоти |

**Коли використовувати Ultraplan**

Найкраще підходить для:
- Складних архітектурних змін, що зачіпають багато файлів (міграції сервісів, великі рефакторинги).
- Завдань, де ви хочете продовжувати працювати, поки йде планування.
- Ситуацій, де зацікавлені сторони (stakeholders) мають перевірити план перед впровадженням.

Пропускайте для:
- Простих, фокусованих змін, де локальний Plan Mode займає менше хвилини.
- Середовищ без інтернету або не на GitHub.
- Сесій з використанням Remote Control.

**Примітка щодо токенів**: Ранні тести показують, що хмарне планування споживає на ~37% менше токенів, ніж еквівалентні локальні плани (82K проти 131K для завдання з міграції на ~55 хв). Токени хмарного планування не враховуються у вашу локальну квоту; рахуються лише токени впровадження.

> **Див. також**: [§9.16 Телепортація сесій](#916-session-teleportation) для детальнішого опису воркфлоу "веб ↔ термінал". Ultraplan використовує ту саму хмарну інфраструктуру з функціями огляду, орієнтованими на планування.

### Ultrareview (v2.1.114+)

Хмарне паралельне мультиагентне рев'ю коду. Де Ultraplan відповідає за планування, Ultrareview — за рев'ю: кілька агентів Opus 4.7 одночасно читають ваші зміни та виявляють баги та архітектурні проблеми, які помітили б лише дуже уважні рев'юери.

**Активація**:

```bash
/ultrareview              # Рев'ю поточної гілки (diff від бази)
/ultrareview <PR#>        # Рев'ю конкретного GitHub PR
```

Ultrareview працює з **diff-файлами, а не з усією кодобазою** — воно перевіряє те, що змінилося на поточній гілці або в конкретному PR. Хмарна сесія запускає паралельних агентів для аналізу diff; результати надходять у браузер і можуть бути опціонально телепортовані назад у термінал.

**Стартова пропозиція**: Передплатники Pro та Max отримують три безкоштовних ultrareviews, щоб спробувати функцію.

**Вимоги**:

| Вимога | Деталі |
|-------------|--------|
| Версія Claude Code | v2.1.114+ |
| Акаунт | Pro або Max |
| Провайдери | Тільки Anthropic API |

**Ultraplan vs. Ultrareview**

| | Ultraplan | Ultrareview |
|---|---|---|
| Мета | Планування перед кодуванням | Рев'ю після кодування |
| Вхідні дані | Промпт з описом завдання | Diff поточної гілки або PR |
| Масштаб | Необмежений | Тільки diff (не вся кодобаза) |
| Результат | Архітектурний план | Звіт про баги та проблеми дизайну |

### Стек механік (Mechanic Stacking)

**Концепція**: Нашарування кількох механізмів Claude Code для максимального інтелекту при прийнятті критичних рішень.

```
Шар 1: Plan Mode          → Безпечне дослідження, без побічних ефектів
Шар 2: Extended Thinking  → Глибокі міркування з токенами мислення
Шар 3: Rev the Engine     → Багаторазове уточнення (раунди)
Шар 4: Split-Role Agents  → Аналіз з різних точок зору
Шар 5: Permutation        → Систематичне тестування варіацій
```

**Вам не потрібні всі шари для кожного завдання.** Підбирайте глибину стека відповідно до впливу рішення:

| Вплив рішення | Глибина стека | Приклад |
|-----------------|-------------|---------|
| Низький (фікс описки) | 0 шарів | Просто зроби це |
| Середній (додати фічу) | 1-2 шари | Plan Mode + Extended Thinking |
| Високий (архітектура) | 3-4 шари | Rev the Engine + Split-Role |
| Критичний (міграція) | 4-5 шарів | Повний стек |

**Антипатерн**: Використання стека для тривіальних рішень. Якщо зміна не є ризикованою і її легко скасувати — просто виконуйте. Надмірне планування так само марнотратне, як і недостатнє.

## 2.4 Відкат (Rewind)

Rewind — це механізм скасування дій у Claude Code.

### Використання Rewind

Доступ через `Esc + Esc` (подвійне натискання Escape) або команду `/rewind`. Це відкриває список чекпоінтів, який можна прокручувати.

### Що робить Rewind

Rewind пропонує чотири різні дії зі списку чекпоінтів:

| Дія | Ефект |
|--------|--------|
| **Restore code and conversation** | Повернути і зміни у файлах, і розмову до обраної точки |
| **Restore conversation** | Залишити поточний код, відкотити тільки розмову |
| **Restore code** | Повернути зміни у файлах, залишити розмову |
| **Summarize from here** | Стиснути розмову від обраної точки вперед (звільняє місце без відкату) |

Ключова відмінність: **Restore** = скасування (повертає стан). **Summarize** = стиснення (звільняє місце без повернення стану). Чекпоінти зберігаються між сесіями (очищення через 30 днів).

### Обмеження

- Працює тільки для змін, зроблених Claude (не для ручних правок).
- Працює в межах поточної сесії.
- Git-коміти НЕ відкочуються автоматично.

### Найкраща практика: Чекпоінт перед ризиком

Перед ризикованою операцією:

```
Ви: Давай закомітимо те, що маємо, перед тим як пробувати цей експериментальний підхід
```

Це створює git-чекпоінт, до якого ви завжди зможете повернутися.

### Драбина відновлення: Три рівні скасування

Коли щось іде не так, у вас є кілька варіантів відновлення. Використовуйте найлегший підхід, який вирішує проблему:

```
┌─────────────────────────────────────────────────────────┐
│               ДРАБИНА ВІДНОВЛЕННЯ                       │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   Рівень 3: Git Restore (радикальний варіант)           │
│   ─────────────────────────────────────                 │
│   • git checkout -- <file>    (скинути незакомічене)    │
│   • git stash                 (зберегти на потім)       │
│   • git reset --hard HEAD~1   (скасувати останній коміт)│
│   • Працює для: ручних правок, багатьох сесій           │
│                                                         │
│   Рівень 2: /rewind (скасування в сесії)                │
│   ─────────────────────────────                         │
│   • Відкочує останні зміни файлів, зроблені Claude      │
│   • Працює тільки в межах поточної сесії                │
│   • Не чіпає git-коміти                                 │
│   • Працює для: поганої генерації коду, хибного шляху   │
│                                                         │
│   Рівень 1: Reject Change (інлайново)                   │
│   ────────────────────────────                          │
│   • Натисніть 'n' під час перегляду diff                │
│   • Зміна ніколи не застосовується                      │
│   • Працює для: виявлення проблем до їх появи          │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**Коли використовувати кожен рівень**:

| Сценарій | Рівень відновлення | Команда |
|----------|----------------|---------|
| Claude запропонував поганий код | Рівень 1 | Натиснути `n` |
| Claude вніс зміни, хочете скасувати | Рівень 2 | `/rewind` |
| Зміни закомічені, потрібен повний відкат | Рівень 3 | `git reset` |
| Експериментальна гілка пішла не туди | Рівень 3 | `git checkout main` |
| Контекст пошкоджений, дивна поведінка | Чистий старт | `/clear` + повтор мети |

**Професійна порада**: Команда `/rewind` показує список змін для скасування. Ви можете вибірково відкотити конкретні файли, а не всі зміни одразу.

### Патерн чекпоінтів: Безпечне експериментування

Для систематичного експериментування використовуйте патерн чекпоінтів для створення безпечних точок відновлення:

```
┌─────────────────────────────────────────────────────────┐
│              ВОРКФЛОУ ЧЕКПОІНТІВ                        │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   1. Створити чекпоінт                                  │
│   ──────────────────                                    │
│   git stash push -u -m "checkpoint-before-refactor"     │
│   (зберігає всі зміни, включаючи невідстежувані файли)  │
│                                                         │
│   2. Експериментувати вільно                            │
│   ──────────────────                                    │
│   Спробуйте ризикований рефакторинг, зміни архітектури  │
│   Якщо спрацювало → комітьте як зазвичай                │
│   Якщо ні → відновіть чекпоінт                          │
│                                                         │
│   3. Відновити чекпоінт                                 │
│   ──────────────────                                    │
│   git stash list              # знайти свій чекпоінт    │
│   git stash apply stash@{0}   # відновити без видалення  │
│   # або                                                 │
│   git stash pop stash@{0}     # відновити та видалити    │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**Автоматичний чекпоінт**: Створіть Stop-хук для автоматичного створення чекпоінта в кінці сесії:

```bash
# .claude/hooks/auto-checkpoint.sh
# Див: examples/hooks/bash/auto-checkpoint.sh

# Автоматично створює git stash при завершенні сесії
# Назва: claude-checkpoint-{branch}-{timestamp}
# Логи у: ~/.claude/logs/checkpoints.log
```

**Типові воркфлоу**:

| Сценарій | Воркфлоу |
|----------|----------|
| Ризикований рефактор | Чекпоінт → Спроба → Коміт або відновлення |
| А/Б тестування підходів | Чекпоінт → Спроба А → Відновлення → Спроба Б → Порівняння |
| Інкрементальна міграція | Чекпоінт → Міграція фрагмента → Тест → Повтор |
| Дослідження прототипу | Чекпоінт → Експеримент → Чисте видалення |

**Переваги над створенням гілок**:
- Швидше, ніж створювати feature-гілки.
- Зберігає незакомічені зміни.
- Легковажний підхід для швидких експериментів.
- Працює відразу для багатьох файлів.

### Рівні зусиль (Effort Levels)

Параметр `effort` (API Opus 4.6) контролює **загальний обчислювальний бюджет** моделі — не лише токени мислення, а й виклики інструментів, багатослівність та глибину аналізу. Low effort = менше викликів інструментів, без передмов. High effort = більше пояснень, детальний аналіз.

**Калібрований градієнт — по одному реальному промпту на рівень:**

- **`low`** — Механічне завдання, дизайн-рішення не потрібні.
  > `"Перейменуй getUserById на findUserById по всьому src/"` — Область пошуку-заміни, мислення не потрібне.

- **`medium`** — Чіткий патерн, визначена область, одна задача.
  > `"Конвертуй fetchUser() у api/users.ts з колбеків на async/await"` — Патерн відомий, область обмежена.

- **`high`** — Дизайн-рішення, крайові випадки, кілька задач.
  > `"Перепроектуй обробку помилок у модулі платежів: додай логіку повторів, відновлення після часткових збоїв та гарантії ідемпотентності"` — Архітектурний вибір, а не просто застосування патерна.

- **`xhigh`** _(Opus 4.7+, v2.1.114+)_ — Надвисокі зусилля між `high` та `max`; значення за замовчуванням для Claude Code (всі плани) з Opus 4.7.
  > `"Відлагоди цей стан гонитви (race condition) у розподіленій черзі завдань із паралельними записами та частковим зчитуванням"` — Більша глибина міркувань, ніж `high`, швидше за `max`.

- **`max`** _(Тільки Opus 4.7+ — повертає помилку на інших моделях)_ — Міжсистемні міркування, незворотні рішення.
  > `"Проаналізуй конвеєр подій мікросервісів на наявність станів гонитви між order-service, inventory-service та notification-service"` — Перевірка гіпотез між сервісами, змагальне мислення.

### Розподіл зусиль на рівні скілів (v2.1.80+)

Скіли можуть оголошувати власний рівень зусиль у фронтматтері. Значення скіла перекриває налаштування сесії на час його виконання, після чого воно повертається до попереднього. Це усуває потребу вручну перемикати `effort` між механічними та аналітичними завданнями.

```yaml
# Механічний скіл — завжди швидкий, не витрачає бюджет міркувань
---
name: release
description: Підняти версію, оновити CHANGELOG, коміт, пуш
effort: low
---

# Аналітичний скіл — завжди глибокий, незалежно від налаштувань сесії
---
name: architecture-review
description: Повний архітектурний аналіз з оцінкою компромісів
effort: high
---
```

**Таблиця рішень для типових типів скілів:**

| Тип скіла | Рекомендовані зусилля | Обґрунтування |
|------------|--------------------|-----------|
| Commit, push, sync | `low` | Послідовні кроки, дизайн-рішення не потрібні |
| Changelog, release notes | `low` | Читає git + форматує, механічна робота |
| Scaffolding, boilerplate | `low` | Створення за шаблоном |
| Рев'ю коду (один PR) | `medium` | Розпізнавання патернів, обмежена область |
| Тріаж ішью, беклог | `medium` | Категоризація + певний аналіз |
| Аудит безпеки | `high` | Моделювання загроз, змагальне мислення |
| Архітектурне рев'ю | `high` | Дизайн-рішення, міркування між компонентами |
| Мультиагентна оркестрація | `high` | Координація + планування |

### Патерни "Модель на агента"

Призначайте моделі агентам залежно від **ролі**, а не важливості:

**Planner** (`examples/agents/planner.md`) — Стратегія, дослідження тільки для читання.

```yaml
---
name: planner
description: Агент стратегічного планування — тільки читання. Використовуй перед впровадженням.
model: opus
tools: Read, Grep, Glob
---
```

**Implementer** (`examples/agents/implementer.md`) — Механічне виконання, обмежена область.

```yaml
---
name: implementer
description: Агент механічного виконання. Область має бути чітко визначена в завданні.
model: haiku
tools: Write, Edit, Bash, Read, Grep, Glob
---
```

> **Примітка**: Haiku — тільки для механічних завдань. Якщо впровадження потребує дизайн-рішень або складної бізнес-логіки, використовуйте Sonnet — вкажіть це у промпті завдання.

**Architecture Reviewer** (`examples/agents/architecture-reviewer.md`) — Критичний огляд дизайну.

```yaml
---
name: architecture-reviewer
description: Рев'ю архітектури та дизайну — тільки читання. Ніколи не змінює код.
model: opus
tools: Read, Grep, Glob
---
```

**Професійна порада**: Додайте нагадування про моделі у ваш CLAUDE.md:
> # Моделі:
> Default: Sonnet. Haiku для механічних завдань. Opus для архітектури та аудитів безпеки.

| Сценарій | Мислення (Thinking) | Причина |

| Boilerplate / генерація тестів| ВИМК | Повторюваний патерн, рішень немає |
| Архітектурна міграція | УВІМК (max) | Незворотні рішення |
| Прямі фактичні питання | ВИМК (low) | Достатньо миттєвої відповіді |
| Рев'ю коду на безпеку | УВІМК (high) | Потрібні змагальні міркування |
