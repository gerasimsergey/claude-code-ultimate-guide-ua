**Найкращі практики Git Worktree:**
1. **Зрозумілі назви**: `git worktree add ../myproject-feature-x feature-x` (замість `temp`).
2. **Додайте в .gitignore**: обов'язково ігноруйте директорії `.worktrees/` або `worktrees/`.
3. **Очищення**: після мерджу видаляйте дерево (`git worktree remove`) та локальну гілку.
4. **Паралельне тестування**: ви можете запустити тести для фічі А в одному дереві, поки кодите фічу Б в іншому.

**Вирішення проблем**:
- "Already checked out": ви не можете відкрити одну гілку в двох деревах одночасно.
- Місце на диску: кожне дерево — це повна копія проекту. Регулярно запускайте `git worktree prune`.

---

### Нативні функції Worktree у Claude Code (v2.1.49+)

Claude Code має вбудовану інтеграцію, що виходить за межі ручного керування:
```bash
claude --worktree  # або -w: створює тимчасове дерево на базі HEAD
```
Дерево створюється автоматично, Claude працює в ньому і видаляє його при виході (якщо не було змін).

**Декларативна ізоляція в агентах (v2.1.50+)**:
Додайте `isolation: "worktree"` у фронтматтер агента. Кожен виклик цього агента автоматично отримуватиме власне ізольоване дерево.

**Хуки життєвого циклу**:
- `WorktreeCreate`: запуск скриптів налаштування (створення гілки БД, `npm install`).
- `WorktreeRemove`: очищення ресурсів після завершення роботи агента.

### Корпоративна безпека (v2.1.83+)

- **`managed-settings.d/`**: дозволяє багатьом командам деплоїти фрагменти політик безпеки без конфліктів мерджу. Файли об'єднуються в алфавітному порядку.
- **`sandbox.failIfUnavailable`**: якщо встановлено `true`, Claude Code видасть помилку замість того, щоб запускатися без пісочниці (compliance-ризик).
- **`CLAUDE_CODE_SUBPROCESS_ENV_SCRUB=1`**: видаляє API-ключі та хмарні секрети зі змінних оточення субпроцесів (Bash, хуки, MCP), щоб вони не потрапили до зовнішніх мережевих запитів.

### Ізоляція баз даних через гілки

Сучасний патерн (2024+): поєднання Git Worktrees із гілками БД для повної ізоляції фіч.
**Як це працює**: Claude автоматично детектує провайдера (Neon, PlanetScale) та пропонує створити гілку БД: `neonctl branches create --name feature-auth`. Це запобігає конфліктам схем при паралельній розробці.

## 9.13 Стратегії оптимізації витрат

### Режим OpusPlan (Рекомендовано)
- **Планування**: Opus для глибокого архітектурного мислення.
- **Виконання**: Sonnet для написання коду.
Це дає стратегічну точність за ціною Sonnet. Активується через `/model opusplan`.

### Економія токенів (Token-Saving)
**Оцінка вартості CLAUDE.md**:
- 50 рядків: ~500-1000 токенів (Ідеально).
- 200 рядків: Верхня межа допустимого.
- 500+ рядків: Потребує розділення.

> **Дослідження (ETH Zürich, лютий 2026)**: написані розробником `CLAUDE.md` покращують успіх на 4%, тоді як згенеровані ШІ — погіршують на 3%, бо ШІ сліпо слідує всім інструкціям, навіть зайвим. **Рекомендація**: включайте лише команди збірки/тестів та специфічні інструменти.

---

### Структурне індексування (CodeXRay)
Замість того, щоб Claude читав файли по черзі (5-10 запитів), побудуйте структурний індекс (SQLite). Це дозволяє:
- Знайти **мертвий код** (функції, які ніхто не викликає).
- Виявити **циклічні залежності**.
- Знайти **Hotspots** (файли з найбільшою кількістю залежностей).

---

### Caveman (Стиснені відповіді ШІ)
Це плагін, який переписує відповіді Claude у стилі "телеграфних фрагментів", видаляючи ввічливість та зайві слова.
- **Економія**: до 65% на вихідних токенах.
- **Команди**: `/caveman` (full), `/caveman lite`.
- **`/caveman-compress`**: стискає ваші файли пам'яті (CLAUDE.md) для економії при старті сесії.

---

### RTK (Rust Token Killer) — Оптимізація виводу команд
RTK фільтрує вивід bash-команд (git, npm, pytest) **перед** тим, як вони потраплять у контекст Claude.
- **Git log**: -92.3% токенів.
- **Vitest run**: -90% токенів.
- **Pnpm list**: -70% токенів.
**Встановлення**: `brew install rtk-ai/tap/rtk`. Інтегрується через хук `PreToolUse`.

---

### Прогресивне дослідження коду (Smart Explore)
Патерн у 3 кроки для економії 90% токенів:
1. **Структура**: отримання сигнатур функцій через `rg` (ripgrep) без читання тіла.
2. **Ціль**: читання лише конкретних рядків (наприклад, 45-90) цільової функції.
3. **Крос-лінки**: пошук місць виклику через grep.
Цей метод валідований на масштабах проектів Aider (40k+ зірок).
