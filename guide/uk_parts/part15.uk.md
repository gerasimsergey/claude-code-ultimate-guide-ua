      "cache_create": 45300
    }

### Повторне вприскування ідентичності після стиснення (Identity Re-injection)

**Проблема**: Коли Claude стискає контекст під час довгої сесії, агенти, налаштовані на певну роль (тімлід, розробник, рев'юер), можуть "забути" свою ідентичність. Стиснутий транскрипт більше не містить оригінальних системних інструкцій, тому наступна відповідь втрачає роль і стає загальною.

Це найбільш помітно в командах агентів з явними префіксами ідентичності. Агент-розробник, який послідовно позначав повідомлення `🔨 DEVELOPER:`, раптово припиняє це робити після стиснення.

**Патерн**: Зберігайте інструкції щодо ідентичності агента у файлі (`.claude/agent-identity.txt`). Після кожного повідомлення користувача хук `UserPromptSubmit` перевіряє, чи містить остання відповідь асистента очікуваний маркер ідентичності. Якщо ні (що трапляється після стиснення) — він впорскує вміст файлу ідентичності як `additionalContext`. Наступна відповідь знову повертається до ролі без втручання людини.

```bash
# .claude/agent-identity.txt
# Інструкції вашого агента — все, що має пережити стиснення

Ви — лід команди фічі. Ви координуєте команду — ви не пишете код
і не робите рев'ю коду.

Додавайте префікс до кожного повідомлення з поточним станом:
  SPAWN / PLANNING / DEVELOPING / REVIEWING / COMMITTING / COMPLETE
```

```bash
# .claude/hooks/identity-reinjection.sh
# Хук UserPromptSubmit — повторно впорскує ідентичність після стиснення

IDENTITY_FILE="${CLAUDE_IDENTITY_FILE:-.claude/agent-identity.txt}"
IDENTITY_MARKER="${CLAUDE_IDENTITY_MARKER:-}"

[[ ! -f "$IDENTITY_FILE" ]] && exit 0

IDENTITY=$(cat "$IDENTITY_FILE")
[[ -z "$IDENTITY" ]] && exit 0

# Маркер за замовчуванням: перший непорожній рядок файлу ідентичності
[[ -z "$IDENTITY_MARKER" ]] && IDENTITY_MARKER=$(grep -m1 -v '^#' "$IDENTITY_FILE" | head -c 40)

TRANSCRIPT_PATH=$(echo "$INPUT" | jq -r '.transcript_path // empty')
[[ -z "$TRANSCRIPT_PATH" || ! -f "$TRANSCRIPT_PATH" ]] && exit 0

LAST_ASSISTANT=$(jq -r '
    [.[] | select(.role == "assistant")] | last | .content |
    if type == "array" then map(select(.type == "text") | .text) | join("") else . end
' "$TRANSCRIPT_PATH" 2>/dev/null)

# Ідентичність на місці: ніяких дій
echo "$LAST_ASSISTANT" | grep -qF "$IDENTITY_MARKER" && exit 0

# Ідентичність відсутня: повторне впорскування
jq -n --arg ctx "[Нагадування про ідентичність]\n\n$IDENTITY" '{"additionalContext": $ctx}'
exit 0
```

**Налаштування** (`settings.json`):

```json
{
  "hooks": {
    "UserPromptSubmit": [{
      "hooks": [{
        "type": "command",
        "command": ".claude/hooks/identity-reinjection.sh"
      }]
    }]
  }
}
```

**Поведінка**:
- Нульові витрати, коли маркер ідентичності присутній (миттєвий вихід при збігу)
- Мовчазна бездіяльність, якщо файл `.claude/agent-identity.txt` відсутній
- Спрацьовує автоматично після стиснення — ручне втручання не потрібне
- Працює як у соло-сесіях, так і в конфігураціях команд агентів

**Кастомізація**: Встановіть `CLAUDE_IDENTITY_MARKER` у вашому оточенні як короткий, характерний рядок із виводу агента (напр. `"LEAD:"`, `"🔨"`). Якщо не встановлено, хук використовує перші 40 символів файлу ідентичності.

> **Повна реалізація**: [`examples/hooks/bash/identity-reinjection.sh`](../../examples/hooks/bash/identity-reinjection.sh)
>
> **Походження**: Патерн взято з [воркфлоу розробки на основі хуків](https://nick-tune.me/blog/2026-02-28-hook-driven-dev-workflows-with-claude-code/) Ніка Тюна (28.02.2026).

## 7.6 Профілі хуків (Hook Profiles)

**Час читання**: 5 хвилин
**Рівень навичок**: Налаштування команди

У міру зростання вашої колекції хуків виникає дилема: деякі розробники хочуть мінімальних накладних витрат (швидкий запуск, жодних блокуючих перевірок), тоді як учасники, орієнтовані на безпеку, або конвеєри CI хочуть суворого дотримання правил. Один `settings.json` не може задовольнити всіх.

**Патерн**: захистіть кожен хук за допомогою змінної оточення, яка визначає бажаний рівень виконання. Три рівні покривають потреби більшості команд:

```
minimal   — тільки критичні хуки безпеки (виявлення секретів, блоки дозволів)
standard  — хуки воркфлоу розробки (форматування, typecheck, лінтинг)
strict    — повний контроль (комплаєнс, стан MCP, ворота якості)
```

### Реалізація

Кожен хук перевіряє `ECC_HOOK_PROFILE` перед виконанням:

```bash
#!/bin/bash
# .claude/hooks/format-on-edit.sh
# Запускається при: standard або strict

REQUIRED_LEVEL="${HOOK_REQUIRED_LEVEL:-standard}"
CURRENT_LEVEL="${ECC_HOOK_PROFILE:-standard}"

# Ієрархія рівнів: minimal < standard < strict
level_value() {
  case "$1" in
    minimal)  echo 1 ;;
    standard) echo 2 ;;
    strict)   echo 3 ;;
    *)        echo 2 ;;
  esac
}

REQUIRED_VAL=$(level_value "$REQUIRED_LEVEL")
CURRENT_VAL=$(level_value "$CURRENT_LEVEL")

if [[ "$CURRENT_VAL" -lt "$REQUIRED_VAL" ]]; then
  exit 0  # Пропустити мовчки
fi

# Далі логіка хука...
```

**Налаштуйте рівень для кожного хука** у `settings.json` через префікс оточення:

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "hooks": [{
          "type": "command",
          "command": "HOOK_REQUIRED_LEVEL=minimal .claude/hooks/secrets-scan.sh"
        }]
      },
      {
        "hooks": [{
          "type": "command",
          "command": "HOOK_REQUIRED_LEVEL=standard .claude/hooks/format-on-edit.sh"
        }]
      },
      {
        "hooks": [{
          "type": "command",
          "command": "HOOK_REQUIRED_LEVEL=strict .claude/hooks/governance-capture.sh"
        }]
      }
    ]
  }
}
```

**Активуйте для сесії** — або експортуйте глобально у вашому профілі оболонки:

```bash
# Сесія дослідження — швидкий запуск, мінімум перевірок
export ECC_HOOK_PROFILE=minimal && claude

# Стандартна сесія розробника (за замовчуванням)
export ECC_HOOK_PROFILE=standard && claude

# Рев'ю безпеки, CI/CD, пре-реліз
export ECC_HOOK_PROFILE=strict && claude
```

Або встановіть для конкретного проекту у `.envrc` (direnv):

```bash
# .envrc — активується автоматично при переході в директорію
export ECC_HOOK_PROFILE=strict
```

### Коли використовувати кожен рівень

| Профіль | Кейс | Активні хуки |
|---------|----------|-------------|
| `minimal` | Дослідження, швидкі прототипи, CI-агенти | Виявлення секретів, блоки дозволів |
| `standard` | Щоденна розробка | + форматування, typecheck, лінтинг |
| `strict` | Рев'ю безпеки, пре-релізи, комплаєнс | + управління, ворота якості, стан MCP |

### Ключові правила дизайну

- **Хуки безпеки** (`minimal`) ніколи не повинні обмежуватися — прописуйте їх без перевірки рівня.
- **За замовчуванням `standard`**, якщо `ECC_HOOK_PROFILE` не встановлено — ніколи не використовуйте `minimal` як запасний варіант.
- **Документуйте в CLAUDE.md**, які хуки працюють на якому рівні, щоб це не було несподіванкою для команди.

> **Подяка**: Патерн обмеження за профілями хуків взято з [Everything Claude Code](https://github.com/affaan-m/everything-claude-code) (Аффан Мустафа, переможець хакатону Anthropic).

## 8.1 Що таке MCP

MCP (Model Context Protocol) — це стандарт для підключення ШІ-моделей до зовнішніх інструментів та джерел даних.

### Чому MCP?

| Без MCP | З MCP |
|-------------|----------|
| Обмеження вбудованими інструментами | Екосистема інструментів, що розширюється |
| Claude вгадує про зовнішні дані | Claude запитує реальні дані |
| Загальне розуміння коду | Глибокий семантичний аналіз |

### Еволюція MCP: Розширення Apps (SEP-1865)

> **🆕 З січня 2026 року**: MCP тепер може надавати інтерактивні інтерфейси (UI) разом із традиційними текстовими відповідями.

#### Проблема розриву контексту

Традиційна взаємодія з ШІ потребує повторюваних промптів для дослідження даних:

**Без MCP Apps**:
```
Ви: "Покажи дані клієнтів"
Claude: "Ось 500 клієнтів [текстовий список]"
Ви: "Відсортуй за доходом"
Claude: "Ось відсортований список [текст]"
Ви: "Відфільтруй за останні 30 днів"
... (кілька циклів промптів)
```

**З MCP Apps**:
```
Ви: "Покажи дані клієнтів"
Claude: [Рендерить інтерактивний дашборд із сортуванням, фільтрами та вибором дат]
Ви: [Сортуєте, фільтруєте безпосередньо в UI — без додаткових промптів]
```

#### Що таке MCP Apps?

MCP Apps дозволяють MCP-серверам надавати **інтерактивні інтерфейси**, які рендеряться безпосередньо у вашій розмові:

- **Дашборди**: Графіки з фільтрацією, деталізацією та експортом.
- **Майстри налаштування**: Форми з залежними полями та валідацією.
- **Переглядачі документів**: PDF-файли з інлайновим підсвічуванням та анотаціями.
- **Монітори в реальному часі**: Метрики, що оновлюються наживо без повторного запуску інструментів.

#### Чому це важливо для користувачів CLI

**Прямий вплив**: **Жодного** — Claude Code CLI не може відображати інтерактивні інтерфейси в терміналі.

**Опосередковані переваги**:
1. **Розуміння екосистеми**: Куди рухається MCP (інтерактивні агентні воркфлоу).
2. **Гібридні воркфлоу**: Claude Desktop для візуального дослідження → Claude Code CLI для автоматизації.
3. **Розробка MCP-серверів**: Якщо ви будуєте кастомні сервери, Apps тепер є опцією.
4. **Контекст інструментів**: Деякі сервери можуть рекламувати можливості UI (видимі в метаданих).

MCP Apps побудовано на **Model Context Protocol** (відкритий стандарт Anthropic):
- **Відкрита специфікація**: [SEP-1865 на GitHub](https://github.com/modelcontextprotocol/ext-apps)
- **Співавтори**: OpenAI, Anthropic, творці MCP-UI.
- **SDK**: `@modelcontextprotocol/ext-apps` (npm)
- **"Build once, deploy everywhere"**: Працює в Claude, VS Code, ChatGPT, Goose.

### Serena (Семантичний аналіз коду)

**Призначення**: Глибоке розуміння коду через семантичний аналіз, індексацію та персистентну пам'ять.

**Чому Serena важлива**: У Claude Code немає вбудованої індексації (на відміну від Cursor). Serena заповнює цю прогалину, індексуючи вашу кодобазу для швидкого та розумного пошуку. Вона також забезпечує **пам'ять сесій** — контекст, що зберігається між розмовами.

**Ключові функції**:
- **Індексація**: Попередня індексація кодобази для ефективного пошуку символів.
- **Пам'ять проекту**: Зберігає контекст у `.serena/memories/` між сесіями.
- **Онбординг**: Автоматично аналізує структуру проекту при першому запуску.

**Інструменти**: `find_symbol`, `get_symbols_overview`, `search_for_pattern`, `find_referencing_symbols`, `replace_symbol_body`, `write_memory`, `read_memory`, `list_memories`.

**Воркфлоу пам'яті сесії**:
```
# Початок сесії
list_memories() → Перегляд існуючого контексту
read_memory("auth_architecture") → Завантаження релевантного контексту

# Під час роботи
write_memory("api_refactor_plan", "...") → Збереження рішень на потім

# Кінець сесії
write_memory("session_summary", "...") → Фіксація прогресу
```

**Налаштування**:
```bash
# Базова індексація (перший запуск)
uvx --from git+https://github.com/oraios/serena serena project index

# Інкрементальна індексація (швидше після першого разу)
uvx --from git+https://github.com/oraios/serena serena project index --incremental

# Паралельна обробка (рекомендовано: 50-75% ядер CPU)
uvx --from git+https://github.com/oraios/serena serena project index --parallel 4
```

> **Важливо**: Команда `serena index-project` застаріла → використовуйте `serena project index`.

### grepai (Рекомендований семантичний пошук)

**Призначення**: Приватний семантичний пошук коду з аналізом графа викликів.

**Чому grepai рекомендовано**: Це **повністю open-source** інструмент, що працює локально через Ollama (жодних проблем із приватністю), і пропонує **аналіз графа викликів** — відстежте, хто викликає певну функцію, і візуалізуйте залежності.

**Ключові функції**:
- **Семантичний пошук**: Пошук коду за описом природною мовою.
- **Граф викликів**: Відстеження callers, callees та повних дерев залежностей.
- **Приватність**: Використання Ollama локально.
- **Фонова індексація**: Демон `grepai watch` підтримує індекс актуальним.

**Приклад**:
```bash
# Семантичний пошук (за змістом, а не точним текстом)
grepai search "user authentication flow"

# Хто викликає цю функцію?
grepai trace callers "createSession"

# Повний граф залежностей
grepai trace graph "createSession" --depth 3
```

**Налаштування**:
```bash
# 1. Встановіть Ollama та модель ембеддінгів
brew install ollama
ollama pull nomic-embed-text

# 2. Встановіть grepai
curl -sSL https://raw.githubusercontent.com/yoanbernabeu/grepai/main/install.sh | sh

# 3. Ініціалізуйте в проекті
grepai init  # Оберіть: ollama, nomic-embed-text

# 4. Запустіть демон індексації
grepai watch &
```

**Патерн "Радіус ураження" (Blast-Radius Pattern):**
Перед модифікацією будь-якої широко використовуваної функції запустіть запит залежностей, щоб перелічити всі зачеплені місця виклику — і тільки потім вирішуйте, чи продовжувати. Це запобігає каскадним поломкам.

### claude-mem (Автоматична пам'ять сесій)

**Призначення**: Автоматична персистентна пам'ять між сесіями Claude Code через стиснуту ШІ-фіксацію інструментів та спостережень.

**Чому claude-mem важливий**: На відміну від ручних інструментів, claude-mem **автоматично фіксує все**, що Claude робить під час сесій, та інтелектуально впорскує релевантний контекст при повторному підключенні. Це вирішує головну проблему: втрату контексту між сесіями.

**Ключові функції**:
- **Автоматична фіксація**: Підключається до подій SessionStart, PostToolUse, Stop, SessionEnd.
- **ШІ-компресія**: Використовує Claude для генерації семантичних підсумків (~10x економія токенів).
- **Прогресивне розкриття**: 3-рівневий пошук (search → timeline → observations) економить ~95% токенів.
- **Гібридний пошук**: Повнотекстовий + векторний (Chroma) + запити природною мовою.
- **Веб-дашборд**: UI на `localhost:37777` для дослідження історії.
- **Приватність**: Теги `<private>` для виключення секретів зі сховища.

**Типи спостережень**:
- `DISCOVERY`: Вивчення коду ("Вивчив модуль auth, знайшов JWT у validateToken()")
- `CHANGE`: Редагування файлів ("Змінив session.middleware.ts: додав логіку оновлення")
- `FEATURE`: Новий функціонал ("Реалізував потік OAuth2")
- `BUGFIX`: Виправлення помилок

**Встановлення**:
```bash
/plugin marketplace add thedotmack/claude-mem
/plugin install claude-mem
# Перезапустіть Claude Code — він активується автоматично
```

**Навички** (`/claude-mem:*`):
- `mem-search`: Пошук в історії ("Як ми вирішили проблему з CORS?")
- `smart-explore`: Економне дослідження кодобази через AST.
- `make-plan`: Створення фазового плану впровадження.
- `timeline-report`: Генерація наративу подорожі проектом.

**Прогресивне розкриття**:
1. **Пошук** (50-100 токенів): "Знайди сесії про автентифікацію" → 5 підсумків.
2. **Таймлайн** (500-1000 токенів): "Покажи таймлайн сесії abc123" → список спостережень.
3. **Деталі** (повний контекст): "Дай деталі спостереження obs_456" → повний результат інструменту.

**Оптимізація витрат — використання Gemini замість Claude**:

```json
// У ~/.claude-mem/settings.json
{
  "provider": "gemini",
  "model": "gemini-2.5-flash-lite",
  "auth_method": "cli"
}
```
