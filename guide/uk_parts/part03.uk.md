| **Керування через** | `/compact`, `/clear` | команда `/memory` (автоматично) | `write_memory()` через Serena MCP |
| **Втрачається при** | Завершенні сесії або `/clear` | Явному видаленні через `/memory` | Явному видаленні з Serena |
| **Потребує** | Нічого | Нічого (v2.1.59+) | [Serena MCP сервер](#82-available-servers) |
| **Кейс використання** | Поточний робочий контекст | Ключові рішення, фрагменти контексту | Архітектурні рішення, патерни |

**Сесійна пам'ять** (короткострокова):
- Усе у вашій поточній розмові
- Файли, що прочитав Claude, запущені команди, прийняті рішення
- Керується через `/compact` (стиснення) та `/clear` (скидання)
- Зникає, коли ви закриваєте Claude Code

**Auto-Memory** *(рідна, v2.1.59+)*:
- Вбудована в Claude Code — не потребує MCP сервера чи налаштування
- Claude автоматично зберігає корисний контекст (рішення, патерни, вподобання) у файли `MEMORY.md`
- Організовано для кожного проекту: `.claude/memory/MEMORY.md` або `~/.claude/projects/<path>/memory/MEMORY.md`
- Керується через `/memory`: перегляд, редагування або видалення збереженого
- Автоматично зберігається між сесіями

**Стійка пам'ять** (довгострокова, Serena MCP):
- Потребує встановленого [Serena MCP сервера](#82-available-servers)
- Явно зберігається за допомогою `write_memory("key", "value")`
- Зберігається між сесіями
- Ідеально для: архітектурних рішень, патернів API, конвенцій кодування

**Патерн: Збереження в кінці сесії**

```
# Перед завершенням продуктивної сесії:
"Збережи наше рішення щодо автентифікації в пам'ять:
- Обрали JWT замість сесій для масштабованості
- Термін дії токена: 15 хв access, 7 днів refresh
- Зберігати refresh-токени в httpOnly cookies"

# Claude викликає: write_memory("auth_decisions", "...")

# Наступна сесія:
"Що ми вирішили щодо автентифікації?"
# Claude викликає: read_memory("auth_decisions")
```

**Коли і що використовувати**:
- **Сесійна пам'ять**: Активне вирішення проблем, відладка, дослідження
- **Auto-memory**: Рішення та контекст, які ви хочете, щоб Claude "перевідкрив" у наступній сесії без ручних зусиль (v2.1.59+)
- **Стійка пам'ять (Serena)**: Структуроване сховище "ключ-значення" для архітектурних рішень у багатьох проектах
- **CLAUDE.md**: Командні конвенції, структура проекту (версіонується через git)

**Auto-compact та PostToolUse захоплення пам'яті — конфлікт, про який варто знати**:

Claude Code автоматично стискає розмову (auto-compact), коли залишок контексту падає нижче фіксованого порогу (приблизно останні 6-7% вікна контексту, або близько 13 тис. токенів до ліміту). На практиці це спрацьовує в діапазоні 90-95% використання, залежно від вікна контексту моделі та зарезервованих вихідних токенів. Перед повним стисненням Claude Code також застосовує **мікро-стиснення** — легший прохід, який вибірково стискає старі результати інструментів (читання файлів, виводи bash, результати пошуку), щоб звільняти місце поступово без підсумовування всієї розмови. Якщо auto-compact не вдається (наприклад, через обмеження швидкості), він повторює спробу до 3 разів поспіль, перш ніж здатися в цій сесії.

Якщо ви використовуєте інструмент захоплення пам'яті на основі хуків (як claude-mem), який зберігає історію сесії через `PostToolUse`, auto-compact може спрацювати і відкинути історію розмови **до того**, як пайплайн збереження встигне її зафіксувати.

Є два способи впоратися з цим:

```json
// Варіант 1: вимкнути auto-compact у налаштуваннях проекту settings.json
// (ви керуєте стисненням вручну через /compact)
{
  "autoCompactEnabled": false
}
```

```bash
# Варіант 2: залишити auto-compact увімкненим, але встановити поріг збереження інструменту
# значно нижче 80% (наприклад, на 60% використання контексту)
# — перевірте конфігурацію кулдаунів/порогів вашого плагіна пам'яті
```

Варіант 1 дає повний контроль, але потребує дисципліни. Варіант 2 безпечніший, якщо ви забуваєте робити компакт вручну. Загальна порада (використовувати `/compact` проактивно на 75%) залишається актуальною — вимкнений auto-compact просто означає, що ви самі відповідаєте за таймінг.

### Патерн свіжого контексту (Fresh Context Pattern / Ralph Loop)

#### Проблема: Гниття контексту (Context Rot)

Дослідження показують, що продуктивність LLM значно погіршується з накопиченням контексту:
- **20-30% розрив у продуктивності** між сфокусованими та "засміченими" промптами ([Chroma, 2025](https://research.trychroma.com/context-rot))
- Деградація починається з ~16K токенів для старих моделей Claude (Chroma, 2025); Anthropic повідомляє про помітну деградацію близько 300-400K токенів у вікні контексту 1M (залежить від завдання, а не є фіксованим порогом)
- Невдалі спроби, трасування помилок та історія ітерацій розмивають увагу

Замість того, щоб керувати контекстом всередині сесії, ви можете **перезапускати нову сесію для кожного завдання**, зберігаючи стан ззовні.

#### Патерн

```bash
# Канонічний "Ralph Loop" (Джеффрі Хантлі)
while :; do cat TASK.md PROGRESS.md | claude -p ; done
```

> **Примітка щодо назви**: "Ralph Loop" використовується в спільноті у двох різних значеннях. Оригінальний патерн Джеффрі Хантлі (вище) стосується ротації контексту — створення нових сесій для уникнення "гниття". Окреме значення, популяризоване Едді Османі та іншими у 2026 році, застосовує той самий термін до *атомарної ітерації завдань* у мультиагентних командах: вибір завдання → реалізація → валідація → коміт → скидання контексту → повтор. Обидва варіанти мають спільну механіку (безстатевий цикл із зовнішнім станом), але відрізняються за масштабом. Якщо термін вживається без уточнення, з'ясуйте, який варіант мається на увазі.

**Стан зберігається через**:
- `TASK.md` — Поточне визначення завдання з критеріями прийняття
- `PROGRESS.md` — Отримані знання, завершені завдання, блокери
- Git коміти — Кожна ітерація комітиться атомарно

**Варіант: tasks/lessons.md**

Легковажна альтернатива для інтерактивних сесій (цикл не потрібен): після кожного виправлення з боку користувача Claude оновлює `tasks/lessons.md` правилом, щоб уникнути тієї самої помилки. Переглядається на початку кожної нової сесії.

```
tasks/
├── todo.md      # Поточний план (пункти для перевірки)
└── lessons.md   # Правила, накопичені з виправлень
```

Різниця від PROGRESS.md: `lessons.md` фіксує *поведінкові правила* ("завжди роби diff перед тим, як позначити виконаним", "ніколи не роби моки без запиту"), а не стан завдання. Воно накопичується з часом — рівень помилок падає в міру зростання набору правил.

| Традиційний підхід | Свіжий контекст |
|-------------|---------------|
| Накопичення в історії чату | Скидання після кожного завдання |
| `/compact` для стиснення | Стан у файлах + git |
| Контекст "перетікає" між завданнями | Кожне завдання отримує повну увагу |

#### Коли використовувати

| Ситуація | Що обрати |
|-----------|-----|
| Контекст 70-90%, залишаєтеся в інтерактиві | `/compact` |
| Контекст 90%+, потрібен чистий старт | `/clear`, потім продовження |
| Довгий автономний запуск на основі завдань | Fresh Context Pattern |
| Виконання вночі або поки вас немає (AFK) | Fresh Context Pattern |

**Добре підходить для**:
- Автономних сесій > 1 години
- Міграцій, великих рефакторингів
- Завдань із чіткими критеріями успіху (тести проходять, білд успішний)

**Погано підходить для**:
- Інтерактивного дослідження
- Проектування без чіткої специфікації
- Завдань із повільними або неоднозначними циклами зворотного зв'язку

**Варіант: Пайплайн "Сесія на кожну область" (Session-per-Concern)**

Замість зациклювання одного й того самого завдання, приділіть окрему нову сесію кожному виміру якості:

1. **Сесія планування** — Архітектура, обсяг, критерії прийняття
2. **Сесія тестування** — Написання unit, інтеграційних та E2E тестів спочатку (TDD)
3. **Сесія реалізації** — Кодування доти, доки всі лінтери та тести не пройдуть
4. **Сесії рев'ю** — Окремі сесії для аудиту безпеки, продуктивності, рев'ю коду
5. **Повторення** — Ітерація з коригуванням обсягу за потреби

Це поєднує Fresh Context (чисті 200K на кожну фазу) з [OpusPlan](#62-opusplan-hybrid-mode) (Opus для сесій рев'ю/стратегії, Sonnet для реалізації). Кожна сесія генерує артефакти прогресу, які живлять наступну.

#### Практична реалізація

**Варіант 1: Ручний цикл**

```bash
# Простий цикл свіжого контексту
for i in {1..10}; do
    echo "=== Ітерація $i ==="
    claude -p "$(cat TASK.md PROGRESS.md)"
    git diff --stat  # Перевірка прогресу
    read -p "Продовжити? (y/n) " -n 1 -r
    [[ ! $REPLY =~ ^[Yy]$ ]] && break
done
```

**Варіант 2: Скрипт** (див. `examples/scripts/fresh-context-loop.sh`)

```bash
./fresh-context-loop.sh 10 TASK.md PROGRESS.md
```

**Варіант 3: Зовнішні оркестратори**

- [AFK CLI](https://github.com/m0nkmaster/afk) — оркестрація з нульовим налаштуванням між різними джерелами завдань

Точність Opus 4.6 падає з 93% до 76% при переході з 256K до 1M; Sonnet 4.5 практично "сиплеться" до 18.5%. **Валідація спільноти**: розробник завантажив ~733K токенів (4 книги про Гаррі Поттера) і Opus 4.6 знайшов 49 з 50 задокументованих заклинань за один промпт ([HN, лютий 2026](https://news.ycombinator.com/item?id=46905735)). Дані MRCR для Sonnet 4.6 ще не опубліковані, але спільнота повідомляє, що модель "важко справляється з виконанням конкретних інструкцій та пошуком точної інформації" при повному заповненні 1M контексту.

**Стратегія 6: Використовуйте кешування промптів для повторюваного контексту (API)**

Якщо ви викликаєте Anthropic API напряму (наприклад, для кастомних агентів або пайплайнів), кешування промптів (prompt caching) знижує витрати до 90% на повторюваних префіксах.

```python
# Позначте стабільні секції через cache_control
response = client.messages.create(
    model="claude-sonnet-4-6-20250514",
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": "<ваш великий системний промпт / контекст кодобази>",
            "cache_control": {"type": "ephemeral"}  # Кешувати цей префікс
        }
    ],
    messages=[{"role": "user", "content": "Fix the bug in auth.ts"}]
)
```

**Економіка кешування промптів**:

| Операція | Мультиплікатор вартості | TTL |
|-----------|-----------------|-----|
| Запис у кеш | 1.25x базової ціни | 5 хвилин (за замовчуванням) |
| Запис у кеш (розширений) | 2x базової ціни | 1 година |
| Читання з кешу (hit) | 0.1x базової ціни | — |
| Зменшення затримки | До 85% для довгих промптів | — |

**Точка беззбитковості**: 2 влучання в кеш при TTL 5 хвилин. Після цього — чиста економія.

**Правила**:
- Макс. **4 точки розриву кешу** (breakpoints) на запит
- Ключ кешу = точний збіг префікса (зміна одного символу = промах повз кеш)
- Ставте точки розриву після великих стабільних секцій: системний промпт, визначення інструментів, контекст кодобази
- Для самого Claude Code: кешування обробляється автоматично CLI-інструментом — це стосується воркфлоу на базі API, які ви будуєте поверх Claude

> Документація: [prompt caching](https://docs.anthropic.com/en/docs/build-with-claude/prompt-caching)

#### Як Claude Code обробляє кешування автоматично

Claude Code керує кешуванням промптів без жодних налаштувань з вашого боку. Розуміння механіки допоможе вам приймати рішення, які тримають рівень влучань у кеш (hit rates) високим, а витрати — низькими.

**Ієрархія префіксів кешу**

Кожен API-виклик, який робить Claude Code, структурує контент у фіксованому порядку: `tools → system → messages`. Зіставлення з кешем завжди починається з початку цього префікса. Стабільний список інструментів + стабільний CLAUDE.md + зростаюча історія повідомлень означає, що перші два шари майже завжди потрапляють у кеш (hits), тоді як тільки нові кроки розмови потребують свіжих обчислень.

**20-блоковий перегляд назад (lookback) — пастка довгих сесій**

Зіставлення з кешем використовує обмежений перегляд назад приблизно на 20 блоків. У довгій сесії з багатьма викликами інструментів та обмінами повідомленнями, блоки з початку розмови випадають із цього вікна і стають промахами повз кеш (misses). Практичний наслідок: дуже довгі сесії поступово втрачають ефективність кешу на рівні повідомлень. Виправлення — `/compact`. Воно стискає історію розмови в один блок підсумку, скидаючи вікно перегляду та відновлюючи високий рівень влучань.

**Мінімальні пороги токенів за моделями**

Блок має відповідати мінімальному розміру, щоб бути придатним для кешування. Блоки, менші за поріг, ніколи не кешуються, незалежно від того, наскільки вони стабільні:

| Сімейство моделей | Мінімум токенів |
|---|---|
| Claude Opus 4.7, Opus 4.6, Opus 4.5, Haiku 4.5 | 4,096 |
| Claude Sonnet 4.6 | 2,048 |
| Claude Sonnet 4.5, Sonnet 4, Sonnet 3.7, Opus 4.1, Opus 4 | 1,024 |
| Claude Haiku 3.5, Haiku 3 | 2,048 |

Короткі файли CLAUDE.md (менше ~1000 токенів) можуть взагалі не кешуватися на моделях Sonnet. Якщо оптимізація витрат важлива, переконайтеся, що ваш системний промпт перетинає поріг для вашої цільової моделі.

**Розмір результату інструменту та економіка кешу**

Результати інструментів потрапляють в історію повідомлень і залишаються там до кінця сесії. Кожен наступний API-виклик перечитує цю історію — за ціною читання з кешу (0.1x), але все одно пропорційно до розміру. Вивід `git status` на 500 токенів коштує 500 × 0.1x для зчитування на кожному наступному кроці. Той самий вивід на 50 токенів (відфільтрований таким інструментом, як RTK) коштує 50 × 0.1x — на 90% менше, що накопичується з кожним кроком у сесії. Компактні виводи інструментів не просто швидше обробляються; вони роблять весь закешований префікс дешевшим у підтримці.

Та сама логіка стосується записів у кеш: менший префікс історії означає дешевші початкові записи (1.25x × менша кількість токенів).

**Моніторинг продуктивності кешу у власних пайплайнах**

При побудові агентів або пайплайнів поверх Anthropic API, об'єкт відповіді `usage` показує метрики кешу напряму:

```python
response = client.messages.create(...)

print(response.usage.cache_creation_input_tokens)  # Токени, записані в кеш цим запитом
print(response.usage.cache_read_input_tokens)       # Токени, прочитані з кешу (hits)
print(response.usage.input_tokens)                  # Не закешовані вхідні токени
```

Розраховуйте свій hit rate як `cache_read / (cache_read + cache_creation)` по всіх запитах. Показник вище 0.8 означає, що структура вашого промпту працює добре. Низькі показники зазвичай означають, що контент у стабільному префіксі змінюється між запитами — перевірте наявність часових міток, випадкових ID або динамічного контенту, вбудованого у ваш системний промпт.

Не існує спеціального інструменту моніторингу конкретно для метрик кешу сесій Claude Code. Відстеження витрат через `ccusage` охоплює загальні витрати, але не розбиває їх на рівень влучань у кеш. Для специфічної видимості кешу в кастомних пайплайнах аналізуйте поля відповіді, наведені вище.

**Практичні правила**

- Тримайте CLAUDE.md стабільним між сесіями — правки одноразово анулюють кеш системи, після чого він "прогрівається" при наступному запиті.
- Запускайте `/compact` до того, як розмова стане дуже довгою, а не після того, як продуктивність впаде.
- Уникайте динамічного контенту в стабільних секціях (дати, випадкові значення, контекст для кожного запиту).
- Більший CLAUDE.md = дорожчий запис у кеш, але також більше збережених токенів на кожне читання — вигідно після ~2 влучань.

**Відомі баги кешу (v2.1.69+)**

Два активних баги мовчки ламають кешування у версії v2.1.69+. Негайно застосуйте ці обхідні шляхи:

- **--resume/--continue** спричиняє повну перебудову кешу (0% hit ratio) при кожному відновленні, оскільки JSONL сесії видаляє відкладені записи інструментів перед записом. Обхідний шлях: уникайте `--resume`, доки це не буде виправлено.
- **Заголовок білінгу за сесію** вставляє унікальний хеш як перший блок системного промпту, що призводить до "холодного" промаху на початку кожної сесії та викликів субагентів. Обхідний шлях: `"CLAUDE_CODE_ATTRIBUTION_HEADER": "false"` у `~/.claude/settings.json`.

Див. [Відомі проблеми → Баги кешу промптів](../core/known-issues.md) та запустіть `/check-cache-bugs` для повного аудиту.

> Документація: [prompt caching](https://docs.anthropic.com/en/docs/build-with-claude/prompt-caching)

#### Вартість vs. Цінність

**Погляд на витрати**: Якщо Claude Code заощаджує вам значний час на заваданні, вартість API зазвичай є мізерною порівняно з вашою погодинною ставкою. Не переоптимізовуйте витрати токенів ціною продуктивності.

**Коли варто оптимізовувати**:
- ✅ Ви маєте обмежений бюджет (студент, хобі-проект)
- ✅ Великий обсяг використання (>4 годин на день)
- ✅ Командне використання (5+ розробників)

**Коли НЕ варто оптимізовувати**:
- ❌ Ваш час дорожчий за витрати на API
- ❌ Ви витрачаєте на оптимізацію більше часу, ніж вона економить
- ❌ Оптимізація шкодить продуктивності (занадто суворі обмеження)

#### Воркфлоу з урахуванням витрат

**Для соло-розробників з обмеженим бюджетом:**

```markdown
1. Починайте з Haiku для дослідження/планування
2. Перемикайтеся на Sonnet для реалізації
3. Використовуйте /compact агресивно (кожні 50-60% контексту)
4. Обмежтеся 1-2 MCP серверами
5. Будьте конкретними в усіх запитах
6. Використовуйте пакетні операції, коли це можливо

Орієнтовна вартість на місяць: $5-$15 за 20-30 годин
```

**Для професійних розробників:**

```markdown
1. Використовуйте Sonnet як стандарт (оптимальний баланс)
2. Використовуйте /compact за потреби (70%+ контексту)
3. Використовуйте повний набір MCP (продуктивність понад усе)
4. Не переоптимізовуйте дрібні запити
5. Використовуйте Opus для критичних архітектурних рішень

Орієнтовна вартість на місяць: $20-$50 за 40-80 годин
```

**Для команд:**

```markdown
1. Спільна інфраструктура MCP (Context7, Serena)
2. Стандартизований CLAUDE.md для уникнення повторних пояснень
3. Бібліотека агентів для уникнення повторного створення патернів
4. Інтеграція CI/CD для автоматизації
5. Відстеження витрат на кожного розробника в консолі Anthropic

Орієнтовна вартість на місяць: $50-$200 за 5-10 розробників
```

#### "Червоні прапорці" (Показники марнотратства)

| Показник | Причина | Вирішення |
|-----------|-------|-----|

#### Тарифні плани та ліміти

> **Примітка**: Плани Anthropic часто змінюються. Завжди перевіряйте актуальні ціни та ліміти на [claude.ai/pricing](https://claude.ai/pricing).

**Як працюють ліміти підписок**

На відміну від використання API (оплата за токени), підписки використовують гібридну модель, яка є свідомо непрозорою:

| Концепція | Опис |
|---------|-------------|
| **5-годинне ковзне вікно** | Основний ліміт; скидається, коли ви надсилаєте наступне повідомлення після спливу 5 годин |
| **Тижневий загальний капітал**| Другорядний ліміт; скидається кожні 7 днів. Обидва діють одночасно |
| **Гібридний підрахунок** | Рекламується як "повідомлення", але фактична ємність базується на токенах і варіюється залежно від складності коду, розміру файлів та контексту |
| **Вагові коефіцієнти моделей**| **Opus споживає в 8-10 разів більше квоти, ніж Sonnet** при однаковому обсязі роботи |

**Приблизні бюджети токенів за планами** (січень 2026, перевірено спільнотою)

| План | 5-годинний бюджет токенів | Промптів Claude Code / 5 год | Годин Sonnet на тиждень | Годин Opus на тиждень | Доступ до Claude Code |
|------|---------------------|------------------------|---------------------|-------------------|-------------------|
| **Free** | 0 | 0 | 0 | 0 | ❌ Немає |
| **Pro** ($20/міс) | ~44,000 токенів | ~10-40 промптів | 40-80 годин | N/A (тільки Sonnet) | ✅ Обмежений |
| **Max 5x** ($100/міс) | ~88,000-220,000 токенів | ~50-200 промптів | 140-280 годин | 15-35 годин | ✅ Повний |

> **Попередження**: Це оцінки, виміряні спільнотою. Anthropic не публікує точні ліміти токенів, і вони зменшувалися без оголошення (зокрема в жовтні 2025 року). Співвідношення Opus/Sonnet 8-10× означає, що користувачі Max 20x отримують лише ~24-40 годин Opus на тиждень, незважаючи на оплату $200 на місяць. "Промптів/5 год" — це грубий практичний переклад бюджету токенів; фактична ємність значно варіюється залежно від складності завдання, розміру контексту та використання субагентів. Місячний ліміт: ~50 активних 5-годинних вікон для всіх планів.

**Чому "години" вводять в оману**

Термін "години Sonnet 4" стосується **витраченого астрономічного часу** під час активної обробки, а не календарних годин. Це неможливо прямо конвертувати в токени, не знаючи:
- Складності коду (більші файли = вищі накладні витрати на токен)
- Використання інструментів (виконання Bash додає ~245 вхідних токенів на виклик; текстовий редактор додає ~700)
- Перечитування контексту та промахів повз кеш

**Патерн досвідченого користувача** (валідовано спільнотою):

```
1. Opus → Створення детального плану (висока якість міркувань)
2. Sonnet/Haiku → Виконання плану (економічно вигідна реалізація)
3. Результат: Найкращі міркування там, де це важливо, менша вартість загалом
```

Це саме те, що режим OpusPlan робить автоматично (див. Розділ 2.3).

**Моніторинг вашого використання**

```bash

```

Anthropic не надає вбудованих метрик використання в реальному часі. Сторонні інструменти, як-от [`ccusage`](https://github.com/ryoppippi/ccusage), допомагають відстежувати споживання токенів між сесіями.

Для перегляду історії використання підписки: Перевірте свою [консоль Anthropic](https://console.anthropic.com/settings/usage) або налаштування Claude.ai.

**Історична довідка**: У жовтні 2025 року користувачі повідомили про значне неоголошене зниження лімітів, що збіглося з релізом Sonnet 4.5. Користувачі Pro, які раніше мали 40-80 годин Sonnet на тиждень, повідомили про досягнення лімітів лише після 6-8 годин. Anthropic підтвердила наявність лімітів, але не пояснила розбіжність.

**Години пік (березень 2026)**: 26 березня 2026 року Anthropic скоригувала те, як споживаються ліміти сесій під час пікового попиту — 5-годинне ковзне вікно вичерпується швидше в **будні з 5:00 до 11:00 за тихоокеанським часом** (13:00–19:00 GMT). Загальний тижневий обсяг той самий, але розподіл інший. Anthropic послалася на обмеження потужності GPU; приблизно 7% користувачів зіткнулися з лімітами, з якими раніше не стикалися. Користувачі Max повідомляли про стрибок використання з 21% до 100% на одному промпті під час піку. Практичний обхідний шлях: переносьте важкі обчислювальні завдання агентів (довгі ланцюжки субагентів, великі рефакторинги) на вечір або вихідні. Позапікове використання "очищується" швидше, дозволяючи розтягнути той самий бюджет на довше.

### "Отруєння" контексту (Контекстне перетікання / Bleeding)

**Визначення**: Коли інформація з одного завдання забруднює інше.

**Патерн 1: Перетікання стилю (Style Bleeding)**
```
Завдання 1: "Створи синю кнопку"
Claude: [Створює синю кнопку]

Завдання 2: "Створи форму"
Claude: [Створює форму... де всі кнопки сині!]
        ↑ "Синє" перетекло в нове завдання

Рішення: Використовуйте явні межі
"---НОВЕ ЗАВДАННЯ---
Створи форму. Використовуй стандартні кольори дизайн-системи."
```

**Патерн 2: Забруднення інструкцій (Instruction Contamination)**
```
Інструкція 1: "Завжди використовуй стрілочні функції"
Інструкція 2: "Дотримуйся конвенцій проекту" (де використовуються звичайні функції)

Claude: [Паралізований, чергує стилі]

Рішення: Уточніть пріоритет
"У разі конфлікту конвенції проекту мають пріоритет над моїми вподобаннями."
```

**Патерн 3: Часова плутанина (Temporal Confusion)**
```
Початок сесії: "auth.ts містить логіку входу"
... 2 години роботи ...
Ви перейменували auth.ts на authentication.ts

Claude: "Я модифікую auth.ts..."
        ↑ Використовує застарілу інформацію

Рішення: Явні оновлення
"Зверни увагу: auth.ts було перейменовано на authentication.ts"
```

**Чек-лист гігієни контексту**:
- [ ] Нові завдання = явні markdown-межі.
- [ ] Структурні зміни = повідомляйте Claude явно.
- [ ] Суперечливі інструкції = уточнюйте пріоритет.
- [ ] Довга сесія (>2 год) = розгляньте `/clear` або нову сесію.
- [ ] Дивна поведінка = перевірте через `/context`.

### Техніка перевірки працездатності (Sanity Check)

Перевірте, чи правильно Claude завантажив вашу конфігурацію.

**Простий метод**:

1. Додайте на початку CLAUDE.md:
```markdown
# Мене звати [Ваше ім'я]
# Проект: [Назва проекту]
# Стек: [Ваш тех. стек]
```

2. Запитайте Claude: "Як мене звати? Над яким проектом я працюю?"

3. Якщо відповідь правильна → Конфігурація завантажена коректно.

**Просунутий метод: Кілька точок перевірки**
```markdown
# === CHECKPOINT 1 === Project: MyApp ===

[... 500 рядків інструкцій ...]

# === CHECKPOINT 2 === Stack: Next.js ===

[... 500 рядків інструкцій ...]

# === CHECKPOINT 3 === Owner: [Name] ===
```

Запитайте "Що таке checkpoint 2?", щоб переконатися, що Claude прочитав документ так далеко.

| Симптом збою | Ймовірна причина | Рішення |
|-----------------|----------------|----------|
| Не знає вашого імені | CLAUDE.md не завантажений | Перевірте розташування файлу |
| Суперечливі відповіді| Друкарська помилка в назві| Має бути `CLAUDE.md` (не `clause.md`) |
| Часткові знання | Контекст вичерпано | `/clear` або нова сесія |
