**Що ви отримуєте**: Claude читає позначені файли та надає семантичний діагноз. Замість простого "висока складність", ви отримуєте: "цей клас обробляє маршрутизацію, кешування, обмеження частоти запитів ТА метрики у 7 137 рядках — винесіть кожну зону відповідальності в окремий модуль".

**Статус**: v0.1 alpha. MIT. Нуль залежностей (Python stdlib + git). Працює на будь-якому репозиторії.

**Джерело**: [chopratejas/vitals](https://github.com/chopratejas/vitals)

#### SE-CoVe — Ланцюжок верифікації (Chain-of-Verification)

**Проблема**: згенерований ШІ код містить тонкі помилки, які проходять рев'ю, оскільки і ШІ, і рев'юер йдуть одним і тим же шляхом міркувань. SE-CoVe розриває це коло, запускаючи незалежний верифікатор, який ніколи не бачить початкового рішення.

**Наукова основа**: адаптація методології Chain-of-Verification від Meta (Dhuliawala et al., ACL 2024 Findings — [arXiv:2309.11495](https://arxiv.org/abs/2309.11495)).

**Як це працює** — 5-етапний конвеєр:

1. **Базовий варіант** — Claude генерує початкове рішення.
2. **Планувальник** — створює верифікаційні питання на основі тверджень у рішенні.
3. **Виконавець** — відповідає на питання, не бачачи базового варіанту (запобігає упередженості підтвердження).
4. **Синтезатор** — порівнює результати, виявляє розбіжності.
5. **Вивід** — видає перевірене рішення.

```bash
# Встановлення (дві окремі команди — обмеження маркетплейсу)
/plugin marketplace add vertti/se-cove-claude-plugin
/plugin install chain-of-verification

# Використання
/chain-of-verification:verify <ваше питання>
/ver<Tab>   # Доступне автодоповнення
```

**Компроміси**: ~2x вартість токенів, менший обсяг виводу. Варто використовувати для критичного до безпеки коду, складного дебагу та архітектурних рішень — не для швидкого прототипування.

**Джерело**: [vertti/se-cove-claude-plugin](https://github.com/vertti/se-cove-claude-plugin) — v1.1.1, MIT

#### Vitals проти SE-CoVe — що обрати

Ці інструменти вирішують різні проблеми на різних етапах циклу розробки:

| | Vitals | SE-CoVe |
|--|--------|---------|
| **Коли** | Обслуговування / щотижневе рев'ю | Генерація під завдання |
| **Проблема** | Накопичений технічний борг | Точність конкретного рішення |
| **Ввід** | Вся історія git | Конкретне питання |
| **Вивід** | Ранжовані проблемні файли + діагноз | Перевірена відповідь |
| **Вартість** | Низька (Python-аналіз + читання файлів) | ~2x стандартна генерація |
| **Найкраще для**| "Який файл скоро зламається?" | "Чи правильне це рішення?" |

**Доповнювальний воркфлоу**: запускайте Vitals щотижня, щоб визначити зони, які потребують уваги, а потім використовуйте SE-CoVe, коли просите Claude провести рефакторинг або виправити ці файли.

#### Легке рев'ю зі зміною ролі

Не кожна зміна потребує 5-етапного конвеєра SE-CoVe. Для повсякденного рев'ю в межах однієї сесії ви можете попросити Claude явно переключитися з ролі автора на роль рев'юера:

```markdown
Ви щойно написали реалізацію вище. Тепер забудьте, що ви її писали.
Зробіть рев'ю як старший інженер, який не є автором цього коду.

Перевірте: відповідність вимогам, межові випадки, обробку помилок, зворотну
сумісність, безпеку, продуктивність. Для кожної знайденої проблеми вкажіть
файл і рядок, поясніть проблему та запропонуйте конкретне виправлення.

Вердикт: APPROVE, REQUEST CHANGES або REJECT.
```

Це працює, тому що явна інструкція "забудьте, що ви це писали" змушує Claude переоцінювати, а не захищати попередні рішення. Це виловлює поверхневі проблеми (відсутність перевірок на null, непослідовна обробка помилок), але оскільки шлях міркувань залишається тим самим, тонкі архітектурні недоліки можуть вижити.

**Коли що використовувати:**

| Підхід | Вартість | Що виловлює | Найкраще для |
|----------|------|---------|----------|
| Зміна ролі (та сама сесія) | 1x | Поверхневі проблеми, неймінг, очевидні баги | Щоденна розробка, швидкі фікси |
| SE-CoVe (плагін) | ~2x | "Сліпі плями" міркувань, логічні помилки | Критичний код, архітектура |
| Крос-модельне рев'ю | 1x-2x | Різні патерни мислення, свіжий погляд | Критичні шляхи, пре-мердж ворота |
| Спеціалізовані агенти | 2-5x | Специфічні для домену проблеми паралельно | Великі PR, багатофакторне рев'ю |

#### Крос-модельне рев'ю (Cross-Model Review)

Коли одна модель перевіряє власний код, вона слідує тим самим патернам міркувань. Використання іншої моделі для рев'ю вносить дійсно незалежний аналіз.

**Патерн**: генеруйте однією моделлю, перевіряйте іншою.

```bash
# Реалізація з Opus (глибокі міркування)
claude --model opus

# Рев'ю дифів з Sonnet (інший шлях мислення, менша вартість)
claude -p "Review the changes in the last commit. Check for logic errors, edge cases, and security. Cite file:line." --model sonnet

# Швидка перевірка з Haiku (швидко, дешево, ловить очевидне)
claude -p "List any bugs or missing error handling in the last commit." --model haiku
```

**З кастомними агентами:**

```yaml
# .claude/agents/cross-model-reviewer.md
---
name: cross-model-reviewer
model: sonnet  # Відрізняється від робочої моделі
tools: Read, Grep, Glob
---
Ви робите рев'ю коду, який не писали. Ваше завдання — знайти проблеми.
Перевірте: логіку, обробку помилок, зворотну сумісність, безпеку, продуктивність.
Для кожної знахідки: severity, file:line, проблема, фікс.
```

**Чому різні моделі ловлять різні баги**: кожна модель має свої когнітивні упередження та "сліпі плями". Баг, який непомітний для однієї моделі, може бути очевидним для іншої. Це той самий принцип, що і в командах розробників-людей.

## 8.6 Безпека MCP

Сервери MCP розширюють можливості Claude Code, але також збільшують поверхню атаки. Перед встановленням будь-якого сервера MCP застосовуйте ту ж прискіпливість, що і до будь-якої сторонньої залежності.

> **Деталі CVE та глибока перевірка**: Інформацію про задокументовані CVE (2025-53109/53110 тощо), "білий список" MCP та процедури реагування на інциденти дивіться в [Security Hardening Guide](./security/security-hardening.md).

### Чек-лист перед встановленням

| Перевірка | Чому |
|-------|-----|
| **Верифікація джерела** | GitHub із зірками, відома організація або офіційний вендор |
| **Аудит коду** | Перевірте вихідний код — уникайте закритих бінарників |
| **Мінімальні дозволи** | Чи потрібен доступ до ФС? Мережі? Навіщо? |
| **Активна підтримка** | Свіжі коміти, реакція на issue |
| **Документація** | Чітке пояснення того, які інструменти він надає |

### Ризики безпеки, які слід розуміти

**Підміна інструментів (Tool Shadowing)**
Шкідливий сервер може оголосити інструменти з поширеними назвами (`Read`, `Write`, `Bash`), які перекривають вбудовані. Claude викликатиме їх, думаючи, що це нативні інструменти.
**Захист**: перевіряйте список інструментів командою `/mcp`. Використовуйте `disallowedTools` у налаштуваннях для блокування підозрілих назв.

**Проблема "заплутаного заступника" (Confused Deputy)**
Сервер із високими привілеями (доступ до БД, ключі) може бути зманіпульований через промпт. Він аутентифікує запит Claude, але не перевіряє повноваження користувача на конкретну дію (наприклад, видалення таблиць).
**Захист**: завжди налаштовуйте сервери MCP з **правами тільки для читання за замовчуванням**.

**Динамічне впорскування можливостей**
Сервер може змінити набір інструментів після проходження початкового аудиту.
**Захист**: фіксуйте версії серверів у конфігурації.

### Безпечні патерни конфігурації

**Налаштування з мінімальними привілеями:**
```json
{
  "mcpServers": {
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": {
        "DATABASE_URL": "postgres://readonly_user:pass@host/db"
      }
    }
  }
}
```

> **Примітка**: `disallowedTools` — це ключ кореневого рівня або CLI-прапорець, а не вкладений у `permissions`. В `settings.json` використовуйте `permissions.deny` для блокування патернів інструментів.

## 9.1 Трійця (The Trinity)

Найпотужніший патерн Claude Code поєднує три техніки:
1. **Plan Mode**: безпечне дослідження без внесення змін.
2. **Extended Thinking**: глибокий аналіз (Opus 4.5/4.6).
3. **Sequential Thinking**: структуровані багатокрокові міркування.

### Extended Thinking (Opus 4.5+) та Adaptive Thinking (Opus 4.6+)

> **⚠️ Зміна парадигми (Opus 4.6, лютий 2026)**: Opus 4.6 замінює мислення на основі бюджету на **Adaptive Thinking**, яке самостійно вирішує, коли використовувати глибокі міркування. Параметр `budget_tokens` **застарів**.

#### Adaptive Thinking (Opus 4.6 та 4.7)
Параметр **`effort`** тепер контролює загальний обчислювальний бюджет — не тільки токени мислення, а й генерацію тексту та виклики інструментів.

**Рівні зусиль (Effort levels):**
- **`max`**: максимальні можливості. Для крос-системних міркувань та незворотних рішень. (Тільки Opus 4.7+).
- **`xhigh`**: надвисокі зусилля. **За замовчуванням у Claude Code з Opus 4.7.**
- **`high`**: складні міркування, агентні завдання. Стандарт для промислових воркфлоу.
- **`medium`**: баланс швидкості, вартості та якості.
- **`low`**: максимальна ефективність. Для класифікації та простих правок.

#### Управління режимом мислення в CLI
1. Команда **/model** зі слайдером зусиль (`low`, `medium`, `high`).
2. Змінна оточення `CLAUDE_CODE_EFFORT_LEVEL`.
3. Поле `effortLevel` у `settings.json`.
4. **Alt+T** (Option+T на Mac) — глобальне ввімкнення/вимкнення мислення.

## 9.3 Інтеграція CI/CD

#### Безголовий режим (Headless Mode)
Запуск Claude Code без інтерактивних промптів: `claude -p "Run tests and report"`.

#### Воркфлоу Unix Piping
Claude Code підтримує операції конвеєрів, що дозволяє потужну інтеграцію з оболонкою:
- `cat file.txt | claude -p 'analyze'`
- `git diff | claude -p 'explain changes'`
- `npm test 2>&1 | claude -p 'suggest fixes'`

**Використання `--output-format`**:
- `text`: людиночитаний вивід (за замовчуванням).
- `json`: структуровані дані для парсингу машиною.
- `stream-json`: потоковий JSON для великих обсягів.

**Приклад у CI/CD**: запуск рев'ю PR через конвеєр, отримання JSON, парсинг через `jq` та додавання коментаря в GitHub через `actions/github-script`.

## 9.4 Інтеграція Git Hooks
Хуки Git дозволяють автоматизувати перевірки перед комітом або пушем. (Продовження в наступній частині).
