```

**Вам не потрібні всі шари для кожного завдання.** Узгоджуйте глибину стека з впливом рішення:

| Вплив рішення | Глибина стека | Приклад |
|-----------------|-------------|---------|
| Низький (фікс описки) | 0 шарів | Просто зробіть це |
| Середній (додавання фічі) | 1-2 шари | Режим планування + Розширене мислення |
| Високий (архітектура) | 3-4 шари | "Розгін двигуна" + Розділення ролей |
| Критичний (міграція) | 4-5 шарів | Повний стек |

**Антипатерн**: Нагромадження шарів для тривіальних рішень. Якщо зміна є зворотною та має низький ризик, просто виконуйте. Надмірне планування так само марнотратне, як і недостатнє.

**Перехресні посилання**:
- Перестановочні фреймворки (Permutation Frameworks): Див. [§9.19](#919-permutation-frameworks)
- Субагенти з розділенням ролей: Див. [Ізоляція субагентів](#sub-agent-isolation)
- Розширене мислення: Див. [§9.1 Трійця](#91-the-trinity)

## 2.4 Rewind (Відмотування)

Rewind — це механізм скасування дій (undo) у Claude Code.

### Використання Rewind

Доступ через `Esc + Esc` (подвійне натискання Escape) або команду `/rewind`. Це відкриває список контрольних точок (checkpoints), який можна прокручувати.

### Що робить Rewind

Rewind пропонує чотири окремі дії зі списку контрольних точок:

| Дія | Ефект |
|--------|--------|
| **Відновити код і розмову** | Повернути зміни у файлах і розмову до обраної точки |
| **Відновити розмову** | Залишити поточний код, відмотати тільки розмову |
| **Відновити код** | Повернути зміни у файлах, залишити розмову |
| **Резюмувати звідси** | Стиснути розмову від обраної точки вперед (звільняє місце без відкату) |

Ключова відмінність: **Відновлення** = скасування (повертає стан). **Резюмування** = стиснення (звільняє місце без відкату). Контрольні точки зберігаються між сесіями (очищення через 30 днів).

### Обмеження

- Працює тільки зі змінами Claude (не з ручними правками)
- Працює в межах поточної сесії
- Коміти Git НЕ відкочуються автоматично

### Найкраща практика: Контрольна точка перед ризиком

Перед ризикованою операцією:

```
Ви: Давай закомітимо те, що маємо, перш ніж пробувати цей експериментальний підхід
```

Це створює контрольну точку в git, до якої ви завжди зможете повернутися.

### Сходи відновлення: Три рівні скасування

Коли щось іде не так, у вас є кілька варіантів відновлення. Використовуйте найлегший підхід, який вирішує вашу проблему:

```
┌─────────────────────────────────────────────────────────┐
│               СХОДИ ВІДНОВЛЕННЯ                         │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   Рівень 3: Git Restore (радикальний варіант)           │
│   ───────────────────────────────────────────           │
│   • git checkout -- <файл>    (відкинути незакомічене)  │
│   • git stash                 (зберегти на потім)       │
│   • git reset --hard HEAD~1   (скасувати останній коміт)│
│   • Працює для: Ручних правок, кількох сесій            │
│                                                         │
│   Рівень 2: /rewind (скасування в сесії)                │
│   ──────────────────────────────────────                │
│   • Повертає нещодавні зміни файлів Claude              │
│   • Працює тільки в межах поточної сесії                │
│   • Не чіпає git-коміти                                 │
│   • Працює для: Поганої генерації коду, невірного напрямку│
│                                                         │
│   Рівень 1: Reject Change (в процесі)                   │
│   ───────────────────────────────────                   │
│   • Натисніть 'n' під час перегляду diff                │
│   • Зміна ніколи не застосовується                      │
│   • Працює для: Виявлення проблем до того, як вони стануть реальними│
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**Коли використовувати кожен рівень**:

| Сценарій | Рівень відновлення | Команда |
|----------|----------------|---------|
| Claude запропонував поганий код | Рівень 1 | Натисніть `n` |
| Claude вніс зміни, хочете скасувати | Рівень 2 | `/rewind` |
| Зміни закомічені, потрібен повний відкат | Рівень 3 | `git reset` |
| Експериментальна гілка пішла не так | Рівень 3 | `git checkout main` |
| Контекст пошкоджений, дивна поведінка | Початок з нуля | `/clear` + повтор мети |

**Професійна порада**: Команда `/rewind` показує список змін для скасування. Ви можете вибірково відкотити конкретні файли, а не всі зміни разом.

### Патерн контрольних точок: Безпечне експериментування

Для систематичного експериментування використовуйте патерн контрольних точок, щоб створювати безпечні точки відновлення:

```
┌─────────────────────────────────────────────────────────┐
│              ВОРКФЛОУ КОНТРОЛЬНИХ ТОЧОК                 │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   1. Створення контрольної точки                        │
│   ──────────────────────────────                        │
│   git stash push -u -m "checkpoint-before-refactor"     │
│   (зберігає всі зміни, включаючи невідстежувані файли)  │
│                                                         │
│   2. Вільне експериментування                           │
│   ───────────────────────────                           │
│   Пробуйте ризикований рефакторинг, архітектурні зміни тощо.│
│   Якщо спрацювало → комітьте як зазвичай                │
│   Якщо не вдалося → відновіть контрольну точку          │
│                                                         │
│   3. Відновлення контрольної точки                      │
│   ────────────────────────────────                      │
│   git stash list              # знайдіть свою точку     │
│   git stash apply stash@{0}   # відновити без видалення │
│   # або                                                 │
│   git stash pop stash@{0}     # відновити та видалити   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**Автоматична контрольна точка**: Створіть Stop-хук для автоматичного створення контрольної точки при завершенні сесії:

```bash
# .claude/hooks/auto-checkpoint.sh
# Див.: examples/hooks/bash/auto-checkpoint.sh

# Автоматично створює git stash при завершенні сесії
# Найменування: claude-checkpoint-{гілка}-{таймстемп}
# Логування в: ~/.claude/logs/checkpoints.log
```

**Типові воркфлоу**:

| Сценарій | Воркфлоу |
|----------|----------|
| Ризикований рефакторинг | Точка → Проба → Коміт або відновлення |
| A/B тестування підходів | Точка → Проба A → Відновлення → Проба B → Порівняння |
| Інкрементальна міграція | Точка → Міграція частини → Тест → Повтор |
| Дослідження прототипу | Точка → Експеримент → Чисте відкидання |

**Переваги перед розгалуженням (branching)**:
- Швидше, ніж створення feature-гілок
- Зберігає незакомічені зміни
- Легковажний підхід для швидких експериментів
- Працює з багатьма файлами одночасно

### Рівні зусиль (Effort Levels)

Параметр `effort` (API Opus 4.6) контролює **загальний обчислювальний бюджет** моделі — не лише токени мислення, а й виклики інструментів, розлогість та глибину аналізу. Низькі зусилля (low) = менше викликів інструментів, відсутність вступу. Високі зусилля (high) = більше пояснень, детальний аналіз.

**Калібрований градієнт — один реальний промпт на рівень:**

- **`low`** — Механічне завдання, дизайн-рішення не потрібні
  > `"Rename getUserById to findUserById across src/"` — Область пошуку-заміни, нуль міркувань.

- **`medium`** — Чіткий патерн, визначена область, одне питання
  > `"Convert fetchUser() in api/users.ts from callbacks to async/await"` — Патерн відомий, область обмежена.

- **`high`** — Дизайн-рішення, крайові випадки, кілька питань
  > `"Redesign error handling in the payment module: add retry logic, partial failure recovery, and idempotency guarantees"` — Архітектурний вибір, а не просто застосування патерна.

- **`xhigh`** _(Opus 4.7+, v2.1.114+)_ — Надвисокі зусилля між `high` та `max`; за замовчуванням для Claude Code (всі плани) з Opus 4.7
  > `"Debug this race condition in the distributed job queue with concurrent writes and partial reads"` — Глибші міркування, ніж у `high`, швидше за `max`.

- **`max`** _(Тільки Opus 4.7+ — повертає помилку на інших моделях)_ — Крос-системні міркування, незворотні рішення
  > `"Analyze the microservices event pipeline for race conditions across order-service, inventory-service, and notification-service"` — Тестування гіпотез у кількох сервісах, змагальне мислення.

### Розподіл зусиль за скілами (v2.1.80+)

Скіли можуть оголошувати власний рівень зусиль у метаданих (frontmatter). Значення скіла перекриває налаштування сесії на час його виконання, а потім повертається до попереднього стану. Це позбавляє необхідності вручну перемикати зусилля між механічними та аналітичними завданнями.

```yaml
# Механічний скіл — завжди швидкий, ніколи не витрачає бюджет на міркування
---
name: release
description: Bump version, update CHANGELOG, commit, push
effort: low
---

# Аналітичний скіл — завжди глибокий, незалежно від налаштувань сесії
---
name: architecture-review
description: Full architectural analysis with trade-off evaluation
effort: high
---
```

**Таблиця рішень для поширених типів скілів:**

| Тип скіла | Рекомендовані зусилля | Обґрунтування |
|------------|--------------------|-----------|
| Commit, push, sync | `low` | Послідовні кроки, жодних дизайн-рішень |
| Changelog, примітки до релізу | `low` | Читає git + форматує, механіка |
| Скеффолдинг, шаблонний код | `low` | Створення екземпляра шаблону |
| Рев'ю коду (один PR) | `medium` | Розпізнавання патернів, обмежена область |
| Тріаж ішью, беклог | `medium` | Категоризація + трохи аналізу |
| Аудит безпеки | `high` | Моделювання загроз, змагальне мислення |
| Архітектурне рев'ю | `high` | Дизайн-рішення, крос-компонентні міркування |
| Мультиагентна оркестрація | `high` | Координація + планування |

> **Модель вартості**: зусилля `low` означають менше викликів інструментів, відсутність вступу, прямий вивід. зусилля `high` означають більше викликів інструментів з поясненнями, детальні резюме, глибше дослідження. Узгоджуйте зусилля там, де аналіз додає цінності — не застосовуйте "зусилля = якість" рівномірно до всього.

## 2.6 Ментальна модель

Розуміння того, як Claude Code "думає", зробить вас ефективнішими.

### Бачення вашого проекту очима Claude

```
┌─────────────────────────────────────────────────────────┐
│                    ВАШ ПРОЕКТ                           │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   ┌─────────────┐    ┌─────────────┐    ┌───────────┐   │
│   │   Файли     │    │ Історія     │    │ Конфіг-   │   │
│   │   (.ts,.py) │    │ Git         │    │ файли     │   │
│   └─────────────┘    └─────────────┘    └───────────┘   │
│          │                  │                  │        │
│          ▼                  ▼                  ▼        │
│   ┌─────────────────────────────────────────────────┐   │
│   │              Розуміння Claude                   │   │
│   │   - Структура файлів та зв'язки                 │   │
│   │   - Патерни коду та конвенції                   │   │
│   │   - Останні зміни (з git)                       │   │
│   │   - Правила проекту (з CLAUDE.md)               │   │
│   └─────────────────────────────────────────────────┘   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### Що Claude знає

1. **Структура файлів**: Claude може навігувати та шукати у ваших файлах.
2. **Вміст коду**: Claude може читати та розуміти код.
3. **Стан Git**: Claude бачить гілки, коміти, зміни.
4. **Правила проекту**: Claude читає CLAUDE.md для дотримання конвенцій.

### Чого Claude не знає

1. **Стан виконання (Runtime)**: Claude не бачить запущених процесів.
2. **Зовнішні сервіси**: Claude не може отримати прямий доступ до ваших баз даних.
3. **Ваші наміри**: Claude потребує чітких інструкцій.
4. **Приховані файли**: Claude за замовчуванням дотримується .gitignore.

> **⚠️ Посилення патернів**: Claude дзеркально відображає патерни, які знаходить. У добре структурованих кодобазах він створює послідовний, ідіоматичний код. У заплутаних кодобазах без чітких абстракцій він продовжує цей хаос. Якщо у вашому коді бракує хороших патернів, вкажіть їх явно в CLAUDE.md або використовуйте семантичні якорі (Розділ 2.9).

### Ви — головний потік (Main Thread)

Уявіть себе планувальником процесора (CPU scheduler). Екземпляри Claude Code — це робочі потоки (worker threads). Ви не пишете код — ви **оркеструєте** роботу.

```
┌─────────────────────────────────────────┐
│          ВИ (Головний потік)            │
│  ┌────────────────────────────────────┐ │
│  │  Обов'язки:                        │ │
│  │  • Визначати завдання та пріоритети│ │
│  │  • Розподіляти бюджет контексту     │ │
│  │  • Переглядати результати          │ │
│  │  • Приймати архітектурні рішення    │ │
│  │  • Обробляти винятки/ескалації     │ │
│  └────────────────────────────────────┘ │
│         │          │          │         │
│    ┌────▼───┐ ┌────▼───┐ ┌────▼───┐    │
│    │Worker 1│ │Worker 2│ │Worker 3│    │
│    │(Claude)│ │(Claude)│ │(Claude)│    │
│    │Фіча    │ │Тести   │ │Рев'ю   │    │
│    └────────┘ └────────┘ └────────┘    │
└─────────────────────────────────────────┘
```

**Висновки**:
- **Не пишіть код**, коли це може зробити Claude. Ваш час призначений для рішень, а не для натискання клавіш.
- **Не займайтеся мікроменеджментом**. Дайте чіткі інструкції, а потім перевірте результат.
- **Перемикайте контекст свідомо**. Як і планувальник, пакетуйте схожі завдання.
- **Ескалюйте на себе**. Коли Claude застряг, втрутьтеся — а потім поверніть роботу йому.

Ця ментальна модель масштабується: один розробник може оркеструвати 2-5 екземплярів Claude на незалежних завданнях (див. [§9.17 Scaling Patterns](#917-scaling-patterns-multi-instance-workflows)).

### Від чат-бота до системи контексту

Найпоширеніша помилка — ставитися до Claude Code як до чат-бота, вводячи випадкові запити і сподіваючись на хороший результат. Те, що відрізняє звичайне використання від професійного воркфлоу, — це зміна мислення:

> **Режим чат-бота**: Ви пишете хороші промпти. **Система контексту**: Ви будуєте структурований контекст, який робить кожен промпт кращим.
>
> *"Досить ставитися до нього як до чат-бота. Дайте йому структурований контекст. CLAUDE.md, хуки, скіли, пам'ять проекту. Це змінює все."*
> — [Робін Лоренц](https://www.linkedin.com/in/robin-lorenz-54055412a/), AI-інженер ([коментар](https://www.linkedin.com/feed/update/urn:li:activity:7426936437746352128?commentUrn=urn%3Ali%3Acomment%3A%28activity%3A7426936437746352128%2C7426941635306987520%29))

Claude Code має чотири шари стійкого контексту, які накопичуються з часом:

| Шар | Що він робить | Розділ | Коли налаштовувати |
|-------|-------------|---------|----------------|
| **CLAUDE.md** | Стійкі правила, конвенції, знання про проект | [§3.1](#31-memory-files-claudemd) | Тиждень 1 |
| **Скіли (Skills)** | Модулі знань для стабільних воркфлоу | [§5](#5-skills) | Тиждень 2 |
| **Хуки (Hooks)** | Автоматичні запобіжники (лінт, безпека, формат) | [§7](#7-hooks) | Тиждень 2-3 |
| **Пам'ять проекту** | Рішення та архітектурний контекст між сесіями | [§3.1](#31-memory-files-claudemd) | Постійно |

Це не незалежні функции. Це шари однієї системи:

- **CLAUDE.md** вчить Claude, *що* потрібно вашому проекту (конвенції, стек, патерни)
- **Скіли** вчать Claude, *як* виконувати конкретні воркфлоу (рев'ю, деплой, тести)
- **Хуки** автоматично встановлюють *запобіжники* (блокування секретів, автоформат, запуск лінтера)
- **Пам'ять** зберігає *рішення* між сесіями (архітектурні вибори, вирішені компроміси)

**До** (режим чат-бота):
> "Використовуй pnpm, а не npm. І пам'ятай, наша конвенція іменування..."
> *(Кожну сесію. Щоразу. Копіювання контексту вручну.)*

**Після** (система контексту):
> CLAUDE.md завантажує конвенції автоматично. Скіли забезпечують стабільні воркфлоу.
> Хуки гарантують якість без ручних зусиль. Пам'ять переносить рішення вперед.

Зміна полягає не в кращому написанні промптів. Вона в побудові системи, де Claude починає кожну сесію, вже знаючи, що вам потрібно.

> **Див. також**: [§9.10 Mindset безперервного вдосконалення](#910-continuous-improvement-mindset) для розвитку цієї системи з часом. Готові обрати правильний механізм? [§2.7 Посібник з вибору конфігурації](#27-configuration-decision-guide) відображає всі сім механізмів у вигляді дерева рішень.

## 2.8 Структуроване промптування з XML-тегами

XML-структуровані промпти забезпечують **семантичну організацію** складних запитів, допомагаючи Claude розрізняти різні аспекти вашого завдання для чіткішого розуміння та кращих результатів.

### Що таке XML-структуровані промпти?

XML-теги виступають як **підписані контейнери**, які явно відокремлюють типи інструкцій, контекст, приклади, обмеження та очікуваний формат виводу.

**Базовий синтаксис**:

```xml
<instruction>
  Ваш основний опис завдання тут
</instruction>

<context>
  Фонова інформація, деталі проекту або релевантний стан
</context>

<code_example>
  Код для довідки або приклади, яких слід дотримуватися
</code_example>

<constraints>
  - Обмеження 1
  - Обмеження 2
  - Вимога 3
</constraints>

<output>
  Очікуваний формат або структура відповіді
</output>
```

### Навіщо використовувати XML-теги?

| Перевага | Опис |
|---------|-------------|
| **Розділення обов'язків** | Різні аспекти завдання чітко розмежовані |
| **Зменшення неоднозначності** | Claude знає, яка інформація для чого служить |
| **Краща обробка контексту** | Допомагає Claude пріоритезувати основні інструкції над фоновими |
| **Послідовне форматування** | Легше створювати шаблони для складних запитів |
| **Багатогранні запити** | Складні завдання з багатьма вимогами залишаються впорядкованими |

### Поширені теги та їх використання

**Основні теги інструкцій**:

```xml
<instruction>Основне завдання</instruction>          <!-- Головна директива -->
<task>Конкретне підзавдання</task>                 <!-- Окремий пункт дії -->
<question>Що мені робити з X?</question>          <!-- Явний запит -->
<goal>Досягти стану Y</goal>                       <!-- Бажаний результат -->
```

**Теги контексту та інформації**:

```xml
<context>Проект використовує Next.js 14</context>            <!-- Фонова інфо -->
<problem>Користувачі скаржаться на повільне завантаження</problem> <!-- Опис проблеми -->
<background>Міграція з Pages Router</background>           <!-- Історичний контекст -->
<state>Зараз у гілці feature-branch</state>                 <!-- Поточна ситуація -->
```

**Теги коду та прикладів**:

```xml
<code_example>
  // Існуючий патерн для наслідування
  const user = await getUser(id);
</code_example>

<current_code>
  // Код, що потребує змін
</current_code>

<expected_output>
  // Який вигляд повинен мати результат
</expected_output>
```

**Теги обмежень та правил**:

```xml
<constraints>
  - Має зберігати зворотну сумісність
  - Жодних критичних змін у публічному API
  - Максимальний час відповіді 100 мс
</constraints>

<requirements>
  - Строгий режим TypeScript
  - 100% покриття тестами
  - Доступність (WCAG 2.1 AA)
</requirements>

<avoid>
  - Не використовуй any для типів
  - Не змінюй схему бази даних
</avoid>
```

### Практичні приклади

**Приклад 1: Рев'ю коду з контекстом**

```xml
<instruction>
Перевір цю мідлвару автентифікації на наявність вразливостей
</instruction>

<context>
Ця мідлвара використовується у фінансовому додатку, що обробляє чутливі дані.
Ми дотримуємося рекомендацій OWASP Top 10 і потребуємо відповідності PCI DSS.
</context>

<code_example>
async function authenticate(req, res, next) {
  const token = req.headers.authorization?.split(' ')[1];
  if (!token) return res.status(401).json({ error: 'No token' });

  const decoded = jwt.verify(token, process.env.JWT_SECRET);
  req.user = decoded;
  next();
}
</code_example>

<constraints>
- Вкажи на будь-які ризики безпеки
- Запропонуй альтернативи, що відповідні PCI DSS
- Врахуй атаки за часом (timing attacks) та витік токенів
</constraints>

<output>
Надай:
1. Список знайдених проблем безпеки
2. Рівень критичності для кожної (Критичний/Високий/Середній/Низький)
3. Конкретні виправлення коду з прикладами
4. Додаткові рекомендації щодо посилення безпеки
</output>
```

**Приклад 2: Впровадження фічі з прикладами**

```xml
<instruction>
Додай систему обмеження частоти запитів (rate limiting) до наших API-ендпоінтів
</instruction>

<context>
Поточний стек: Express.js + Redis
Зараз обмеження відсутні
Спостерігаємо зловживання API з певних IP-адрес
</context>

<requirements>
- 100 запитів на хвилину на IP для авторизованих користувачів
- 20 запитів на хвилину на IP для неавторизованих
- Кастомні ліміти для преміум-користувачів (зберігаються в БД)
- Повертати статус 429 із заголовком Retry-After
</requirements>

<code_example>
// Існуючий патерн мідлвар, який ми використовуємо
app.use(authenticate);
app.use(authorize(['admin', 'user']));
</code_example>

<constraints>
- Не повинно впливати на продуктивність існуючого API
- З'єднання з Redis має використовуватися повторно
- Коректно обробляй збої з'єднання з Redis
</constraints>

<output>
Надай:
1. Реалізацію мідлвари rate limiter
2. Конфігурацію Redis
3. Юніт-тести
4. Документацію для команди
</output>
```

**Приклад 3: Розслідування багу зі станом**

```xml
<task>
Досліди, чому сесії користувачів закінчуються передчасно
</task>

<problem>
Користувачі повідомляють про розлогін через 5-10 хвилин активності,
хоча таймаут сесії налаштований на 24 години.
</problem>

<context>
- Next.js 14 App Router з next-auth
- Сховище сесій у PostgreSQL
- Балансування навантаження на 3 сервери
- Проблема почалася після деплою v2.3.0 минулого тижня
</context>

<state>
Git diff між v2.2.0 (працювала) та v2.3.0 (зламана) показує зміни в:
- middleware.ts (логіка оновлення сесії)
- auth.config.ts (стратегія сесій)
- database.ts (пул з'єднань)
</state>

<constraints>
- Не пропонуй відкочувати деплой
- Проблема в продакшені, потрібне швидке рішення
- Необхідно зберегти безпеку сесій
</constraints>

<output>
Надай:
1. Гіпотезу про першопричину
2. Файли для перевірки (в порядку пріоритетності)
3. Команди для відладки, які слід запустити
4. Потенційні виправлення з аналізом компромісів
</output>
```

### Просунуті патерни

**Вкладені теги для складної ієрархії**:

```xml
<task>
Рефакторинг системи автентифікації
  <subtask priority="high">
    Оновити модель користувача
    <constraints>
      - Зберегти існуючі ID користувачів
      - Додати міграцію для верифікації email
    </constraints>
  </subtask>

  <subtask priority="medium">
    Впровадити OAuth провайдерів
    <requirements>
      - Google та GitHub OAuth
      - Повторно використати існуючу логіку сесій
    </requirements>
  </subtask>
</task>
```

**Кілька прикладів з мітками**:

```xml
<code_example label="поточна_реалізація">
  // Старий підхід з callback hell
  getUser(id, (user) => {
    getOrders(user.id, (orders) => {
      res.json({ user, orders });
    });
  });
</code_example>

<code_example label="бажаний_патерн">
  // Новий патерн async/await
  const user = await getUser(id);
  const orders = await getOrders(user.id);
  res.json({ user, orders });
</code_example>
```

**Умовні інструкції**:

```xml
<instruction>
Оптимізуй продуктивність запитів до бази даних
</instruction>

<context>
Запит зараз займає 2.5 секунди для 10 000 записів
</context>

<constraints>
  <if condition="PostgreSQL">
    - Використовуй EXPLAIN ANALYZE
    - Розглянь матеріалізовані представлення
  </if>

  <if condition="MySQL">
    - Використовуй EXPLAIN з аналізом плану запиту
    - Розглянь кеш запитів
  </if>
</constraints>
```

### Коли використовувати XML-структуровані промпти

| Сценарій | Рекомендовано? | Чому |
|----------|--------------|-----|
| Прості запити в один рядок | ❌ Ні | Витрати часу перевищують користь |
| Багатокрокове впровадження фічі | ✅ Так | Відокремлює цілі, обмеження, приклади |
| Дослідження багу з контекстом | ✅ Так | Розрізняє симптоми та середовище |
| Рев'ю коду за специфічними критеріями | ✅ Так | Чіткий поділ коду, контексту, вимог |
| Планування архітектури | ✅ Так | Організовує цілі, обмеження, компроміси |
| Швидкий фікс описки | ❌ Ні | Зайва складність |

### Найкращі практики

**Що варто робити (Do's)**:
- ✅ Використовуйте описові назви тегів, що пояснюють суть
- ✅ Дотримуйтеся однакових тегів для схожих запитів
- ✅ Поєднуйте з CLAUDE.md для проектних конвенцій щодо тегів
- ✅ Логічно вкладайте теги при відображенні ієрархії
- ✅ Використовуйте теги, щоб розділити "що", "чому" та "як"

**Чого не варто робити (Don'ts)**:
- ❌ Не структуруйте занадто сильно прості запити (створює шум)
- ❌ Не змішуйте призначення тегів (наприклад, обмеження всередині прикладів коду)
- ❌ Не використовуйте загальні теги (`<tag>`, `<content>`) без чіткого значення
- ❌ Не робіть занадто глибоку вкладеність (>3 рівнів важко читати)

### Інтеграція з CLAUDE.md
