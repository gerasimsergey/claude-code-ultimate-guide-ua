## 7.1 Система подій

Хуки — це скрипти, які запускаються автоматично при виникненні певних подій.

### Типи подій

**Життєвий цикл (події рівня сесії)**:

| Подія | Коли спрацьовує | Чи може блокувати? | Кейс |
|-------|---------------|------------|----------|
| `SessionStart` | Сесія починається або відновлюється | Ні | Ініціалізація, завантаження контексту розробки |
| `Setup` | Фаза налаштування середовища на старті | Ні | Встановлення інструментів, валідація передумов |
| `SessionEnd` | Сесія завершується | Ні | Очищення, логування |

**Дії агента (конвеєр виконання інструментів)**:

| Подія | Коли спрацьовує | Чи може блокувати? | Кейс |
|-------|---------------|------------|----------|
| `Stop` | Claude завершує відповідь | Так | Пост-відповідні дії, продовження циклів |
| `StopFailure` | Хід завершується помилкою API (ліміти, автентифікація) | Ні | Сповіщення про вичерпання квот, обсервабіліті |
| `PreToolUse` | Перед виконанням виклику інструменту | Так | Валідація безпеки, модифікація вводу |
| `PostToolUse` | Після успішного завершення інструменту | Ні | Форматування, логування |
| `PostToolUseFailure` | Після невдалого виклику інструменту | Ні | Логування помилок, дії для відновлення |

**Дозволи (процес схвалення)**:

| Подія | Коли спрацьовує | Чи може блокувати? | Кейс |
|-------|---------------|------------|----------|
| `PermissionRequest` | З'являється діалог запиту дозволу | Так | Кастомна логіка схвалення |
| `PermissionDenied` | Відмова у наданні дозволу | Ні | Аудит заборонених операцій, алерти |

**Стиснення (управління контекстом)**:

| Подія | Коли спрацьовує | Чи може блокувати? | Кейс |
|-------|---------------|------------|----------|
| `PreCompact` | Перед стисненням контексту | Ні | Збереження стану перед стисненням |
| `PostCompact` | Після завершення стиснення контексту | Ні | Відновлення стану, логування стиснення |

**Мультиагентність (оркестрація)**:

| Подія | Коли спрацьовує | Чи може блокувати? | Кейс |
|-------|---------------|------------|----------|
| `SubagentStart` | Створено субагента | Ні | Ініціалізація субагента |
| `SubagentStop` | Субагент завершує роботу | Так | Очищення після субагента |
| `TeammateIdle` | Член команди агентів збирається перейти в очікування | Так | Координація команди, ворота якості |
| `TaskCreated` | Завдання створено через TaskCreate | Ні | Моніторинг завдань, аудит логів |
| `TaskCompleted` | Завдання позначається як завершене | Так | Перевірка критеріїв завершення |

**Конфігурація (налаштування та інструкції)**:

| Подія | Коли спрацьовує | Чи може блокувати? | Кейс |
|-------|---------------|------------|----------|
| `ConfigChange` | Конфіг змінюється під час сесії | Так (крім policy) | Корпоративний аудит, блокування несанкціонованих змін |
| `InstructionsLoaded` | CLAUDE.md або файл інструкцій завантажено | Ні | Аудит активних файлів інструкцій |

**Файлова система (зміни в робочому просторі)**:

| Подія | Коли спрацьовує | Чи може блокувати? | Кейс |
|-------|---------------|------------|----------|
| `CwdChanged` | Робоча директорія змінюється під час сесії | Ні | Перезавантаження direnv, зміна тулчейну |
| `FileChanged` | Файл змінено під час сесії | Ні | Перезавантаження конфігу, тригери вочерів |
| `WorktreeCreate` | Створення worktree | Так (вихід != 0) | Кастомне налаштування VCS |
| `WorktreeRemove` | Видалення worktree | Ні | Очищення стану VCS |

**Взаємодія з користувачем (промпти та сповіщення)**:

| Подія | Коли спрацьовує | Чи може блокувати? | Кейс |
|-------|---------------|------------|----------|
| `UserPromptSubmit` | Користувач надсилає промпт, до обробки Claude | Так | Збагачення контексту, валідація промпту |
| `Notification` | Claude надсилає сповіщення | Ні | Звукові алерти, кастомні сповіщення |
| `Elicitation` | Claude запитує інформацію у користувача (headless) | Так | Перехоплення та попередня відповідь в автоматизації |
| `ElicitationResult` | Отримано відповідь на запит інформації | Ні | Аудит відповідей користувача на питання Claude |

> **`Stop` та `SubagentStop` — поле `last_assistant_message` (v2.1.47+)**: Ці події тепер включають поле `last_assistant_message` у вхідному JSON, що дає прямий доступ до фінальної відповіді Claude без парсингу файлів транскрипту. Корисно для конвеєрів оркестрації.
>
> ```bash
> # У вашому скрипті хука Stop
> LAST_MSG=$(cat | jq -r '.last_assistant_message // ""')
> echo "$LAST_MSG" >> ~/.claude/logs/session-outputs.log
> ```

### Модель виконання хуків (v2.1.0+)

Claude Code підтримує дві моделі виконання хуків:

#### Синхронна (за замовчуванням)

- Claude **блокується**, поки хук не завершиться
- Код виходу та stdout доступні негайно для фідбеку
- **Кейс**: Критична валідація (безпека, перевірка типів, блокуючі операції)
- **Конфігурація**: Пропустіть `async` або встановіть `async: false`

#### Асинхронна (опціонально)

- Claude **продовжує роботу негайно**, хук виконується у фоні
- Код виходу/stdout НЕ доступні Claude (немає петлі фідбеку)
- **Кейс**: Некритичні операції (логування, сповіщення, форматування, метрики)
- **Конфігурація**: Додайте `async: true` у визначення хука

#### Приклад конфігурації

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": ".claude/hooks/auto-format.sh",
            "timeout": 10000,
            "async": true  // ← Неблокуюче виконання
          },
          {
            "type": "command",
            "command": ".claude/hooks/typecheck.sh"
            // Синхронно за замовчуванням - блокує до завершення
          }
        ]
      }
    ]
  }
}
```

#### Матриця прийняття рішень

| Призначення хука | Режим виконання | Причина |
|--------------|---------------|--------|
| Форматування коду (Prettier) | **Async** | Косметична зміна, фідбек не потрібен |
| Лінтинг з автофіксом | **Async** | Некритичні покращення |
| Перевірка типів (tsc) | **Sync** | Помилки мають блокувати для ітерації |
| Валідація безпеки | **Sync** | Має блокувати небезпечні операції |
| Логування/метрики | **Async** | Чистий побічний ефект |
| Сповіщення (Slack) | **Async** | Сповіщення користувача, не блокує |
| Виконання тестів | **Sync** | Результати впливають на наступну дію |
| Вприскування Git-контексту | **Sync** | Збагачує промпт перед обробкою |

#### Вплив на продуктивність

**Приклад сесії (10 редагувань файлів):**
- **Sync хуки**: `auto-format.sh` (500мс) × 10 = 5с блокування
- **Async хуки**: `auto-format.sh` у фоні = 0с блокування
- **Виграш**: ~5-10с за типову сесію розробки

#### Обмеження асинхронних хуків

⚠️ Асинхронні хуки не можуть:
- Блокувати Claude при помилках (код виходу 2 ігнорується)
- Надавати фідбек у реальному часі через stdout або `systemMessage`
- Гарантувати порядок виконання з іншими хуками
- Повертати `additionalContext` для використання Claude

Використовуйте async тільки тоді, коли завершення хука справді не залежить від воркфлоу Claude.

### Скрипти оболонки проти AI-агентів: Що і коли використовувати

Не все потребує ШІ. Обирайте правильний інструмент:

| Тип завдання | Найкращий інструмент | Чому | Приклад |
|-----------|-----------|-----|---------|
| **Детерміноване** | Bash-скрипт | Швидко, передбачувано, 0 токенів | Створити гілку, отримати коментарі PR |
| **На основі патернів** | Bash + regex | Надійно для відомих патернів | Пошук секретів, валідація формату |
| **Потребує інтерпретації** | ШІ-агент | Потрібне судження | Рев'ю коду, архітектурні рішення |
| **Залежне від контексту** | ШІ-агент | Потрібне розуміння | "Чи відповідає це вимогам?" |

**Золоте правило**: Якщо ви можете написати regex або просту умову — використовуйте bash-скрипт. Якщо завдання потребує "розуміння" або "судження" — використовуйте агента.

## 7.2 Створення хуків

### Реєстрація хуків (settings.json)

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash|Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": ".claude/hooks/security-check.sh",
            "timeout": 5000
          }
        ]
      }
    ]
  }
}
```

### Поля конфігурації

| Поле | Опис |
|-------|-------------|
| `matcher` | Регулярний вираз для фільтрації запуску хука (назва інструменту тощо) |
| `if` | Фільтр за правилами дозволів (напр., `Bash(git *)`) — v2.1.85+ |
| `type` | Тип хука: `"command"`, `"http"`, `"prompt"` або `"agent"` |
| `command` | Команда оболонки (для типу `command`) |
| `prompt` | Текст промпту для оцінки LLM. Використовуйте `$ARGUMENTS` для JSON вводу |
| `timeout` | Макс. час виконання в секундах (за замовчуванням: 600с команда) |
| `model` | Модель для оцінки (для `prompt`/`agent`). За замовчуванням швидка модель |
| `async` | Якщо `true`, працює у фоні (тільки для `command`) |
| `statusMessage` | Кастомне повідомлення спінера під час роботи хука |
| `once` | Якщо `true`, запускається один раз за сесію (тільки для скілів) |

### Хуки рівня сесії (Session-Scoped Hooks)

Хуки не обов'язково зберігати в `settings.json`. Claude Code підтримує ефемерні **хуки рівня сесії**, які реєструються під час роботи і діють лише до кінця поточної сесії.

Це механізм, який скіли використовують внутрішньо: при виклику скіл може зареєструвати один або кілька хуків для цього виклику. Після завершення сесії ці хуки зникають.

**Коли використовувати**:
- Скіли, яким потрібні зворотні виклики (callbacks) тільки поки вони активні
- Тимчасова автоматизація (напр., "аудіюй кожен файл, який я редагую, тільки в цій сесії")
- Конвеєри CI, що вприскують хуки через API програмно

**Типи хуків:**

- **`command`**: Запускає команду оболонки. Отримує JSON на stdin, повертає JSON на stdout.
- **`http`** *(v2.1.63+)*: Відправляє POST-запит JSON на URL і читає JSON-відповідь. Корисно для CI/CD вебхуків.
- **`prompt`**: Надсилає промпт + ввід хука моделі Claude для оцінки в один хід.
- **`agent`**: Створює субагента з доступом до інструментів для багатокрокової верифікації.

**Приклад HTTP-хука** (v2.1.63+):

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "http",
            "url": "https://ci.example.com/webhook/claude-hook",
            "allowedEnvVars": ["CI_TOKEN"]
          }
        ]
      }
    ]
  }
}
```

### Умовні хуки з `if` (v2.1.85+)

Поле `if` фільтрує запуск хука, використовуючи синтаксис правил дозволів. Це дозволяє уникнути створення зайвих підпроцесів на кожну подію.

```json
// ПІСЛЯ: хук спрацьовує тільки коли Bash виконує команду git
{
  "event": "PostToolUse",
  "if": "Bash(git *)",
  "command": "./scripts/log-git-usage.sh"
}
```

| Патерн | Коли спрацьовує |
|---------|-----------|
| `Bash(git *)` | Будь-який виклик Bash, що починається з `git` |
| `Edit` | Будь-який виклик інструменту Edit |
| `Write(/tmp/*)` | Запис у шляхи всередині `/tmp/` |

### Ввід хука (JSON на stdin)

Хуки отримують JSON зі спільними полями плюс специфічні для події поля:

```json
{
  "session_id": "abc123",
  "cwd": "/project",
  "hook_event_name": "PreToolUse",
  "tool_name": "Bash",
  "tool_input": {
    "command": "git status"
  }
}
```

### Вивід хука

Хуки передають результати через коди виходу та опціональний JSON на stdout. Claude Code обробляє JSON тільки при коді виходу 0.

**Універсальні поля JSON**:
- `continue`: Якщо `false`, Claude повністю зупиняє обробку.
- `stopReason`: Повідомлення користувачу, якщо `continue: false`.
- `systemMessage`: Попередження, що показується користувачу.

**Управління рішеннями (PreToolUse)**:
Використовує `hookSpecificOutput` з `permissionDecision` (allow/deny/ask/defer). Пріоритет при конфліктах: `deny` > `defer` > `ask` > `allow` (v2.1.89+).

**Приклад блокування PreToolUse**:
```json
{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "deny",
    "permissionDecisionReason": "Руйнівна команда заблокована хуком"
  }
}
```

**Вприскування контексту PreToolUse**:
```json
{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "additionalContext": "Поточна гілка git: feature/auth. 3 незакомічені файли."
  }
}
```

**PreToolUse рішення `defer` (v2.1.89+ — тільки для headless)**:
При виборі `defer` Claude зупиняється з причиною `tool_deferred`. Зовнішній процес може зібрати дозвіл людини і продовжити сесію через `--resume`.

### Коди виходу

| Код | Значення | Результат |
|------|---------|--------|
| `0` | Успіх | Дозволити операцію, парсити stdout як JSON |
| `2` | Блокуюча помилка | Заборонити операцію, передати stderr Claude |
| Інші | Неблокуюча помилка | Stderr показується у фоновому режимі, робота триває |

### Патерн "Silent Success"

Ключовий принцип чистого контексту: **хуки мають мовчати при успіху і бути багатослівними тільки при помилці.**

```bash
#!/bin/bash
# .claude/hooks/build-check.sh — патерн Silent Success
# При успіху: повна тиша (нічого не потрапляє в контекст)
# При помилці: вивід помилок + код виходу 2

OUTPUT=$(bun run build 2>&1)
EXIT_CODE=$?

if [[ $EXIT_CODE -eq 0 ]]; then
    exit 0  # Тиша
fi

# Помилка: надіслати агенту для виправлення
echo "$OUTPUT" >&2
exit 2
```

Ця асиметрія запобігає накопиченню успішних логів білду чи тестів як "контекстного шуму".

## 7.3 Шаблони хуків

### Шаблон 1: PreToolUse (Блокувальник безпеки)

```bash
#!/bin/bash
# .claude/hooks/security-blocker.sh
# Блокує небезпечні команди

INPUT=$(cat)
TOOL_NAME=$(echo "$INPUT" | jq -r '.tool_name')
COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command // ""')

DANGEROUS_PATTERNS=(
    "rm -rf /"
    "rm -rf ~"
    "rm -rf *"
    "sudo rm"
    "git push --force origin main"
    "npm publish"
)

for pattern in "${DANGEROUS_PATTERNS[@]}"; do
    if [[ "$COMMAND" == *"$pattern"* ]]; then
        echo "BLOCKED: Виявлено небезпечну команду: $pattern" >&2
        exit 2
    fi
done

exit 0
```

### Шаблон 2: PostToolUse (Автоформатувальник)

```bash
#!/bin/bash
# .claude/hooks/auto-format.sh
# Автоформатування коду після редагування

INPUT=$(cat)
TOOL_NAME=$(echo "$INPUT" | jq -r '.tool_name')

if [[ "$TOOL_NAME" != "Edit" && "$TOOL_NAME" != "Write" ]]; then
    exit 0
fi

FILE_PATH=$(echo "$INPUT" | jq -r '.tool_input.file_path // ""')

if [[ -z "$FILE_PATH" ]]; then
    exit 0
fi

if [[ "$FILE_PATH" =~ \.(ts|tsx|js|jsx|json|md|css|scss)$ ]]; then
    npx prettier --write "$FILE_PATH" 2>/dev/null
fi

exit 0
```

### Шаблон 3: UserPromptSubmit (Збагачувач контексту)

```bash
#!/bin/bash
# .claude/hooks/git-context.sh
# Додає контекст git до кожного промпту

BRANCH=$(git branch --show-current 2>/dev/null || echo "не репозиторій")
LAST_COMMIT=$(git log -1 --format='%h %s' 2>/dev/null || echo "немає комітів")

cat << EOF
{
  "hookSpecificOutput": {
    "additionalContext": "[Git] Гілка: $BRANCH | Останній: $LAST_COMMIT"
  }
}
EOF

exit 0
```

### Нативний інструмент PowerShell (Windows, v2.1.84+ preview)

На Windows Claude Code може використовувати PowerShell як повноцінний інструмент поруч із Bash. Це дозволяє використовувати скрипти `.ps1` та нативні команди Windows без потреби в WSL чи Git Bash.

Увімкніть у `~/.claude/settings.json`:
```json
{
  "tools": {
    "powershell": { "enabled": true }
  }
}
```

### Шаблони хуків для Windows

#### Шаблон W1: Перевірка безпеки PreToolUse (PowerShell)

Створіть `.claude/hooks/security-check.ps1`:

```powershell
# security-check.ps1
# Блокує небезпечні команди

$inputJson = [Console]::In.ReadToEnd() | ConvertFrom-Json
$command = $inputJson.tool_input.command

$dangerousPatterns = @(
    "rm -rf /",
    "Remove-Item -Recurse -Force C:\",
    "git push --force origin main",
    "npm publish"
)

foreach ($pattern in $dangerousPatterns) {
    if ($command -like "*$pattern*") {
        Write-Error "BLOCKED: Виявлено небезпечну команду: $pattern"
        exit 2
    }
}

exit 0
```

# auto-format.ps1
# Автоформатування після редагування

}
