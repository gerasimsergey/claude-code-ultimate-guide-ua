**Поведінка агента**:
- Працюючи над сервісом → читає перехресні посилання.
- Знаходить `conflict-resolver.ts` → використовує його замість повторної реалізації.
- Знає, що треба перевірити ADR для розуміння бізнес-логіки.

**Патерн: ланцюжки "Див. також"**:
Додавайте в JSDoc посилання на суміжні файли. Наприклад, у `calendar-sync-service.ts` додайте посилання на клієнти Google та Outlook. Це дозволяє агенту зрозуміти повний потік даних від сервісу до конкретної інтеграції.

#### Самодокументовані команди (--help)
Ваші CLI-інструменти мають пояснювати себе самі. Додавайте блок `--help` з прикладами використання та посиланнями на документацію. Claude зможе прочитати цей вивід і самостійно зрозуміти, як запускати скрипти синхронізації або міграції.

#### Вбудована технічна документація
Замість окремих Вікі-сторінок, тримайте документацію поруч із кодом.
Структура:
- `README.md` (як використовувати).
- `RATE_LIMITS.md` (ліміти сторонніх API).
- `TROUBLESHOOTING.md` (каталог помилок та рішень).
Агент знайде ці файли природним шляхом через файлову систему і зможе виправити баг без галюцинацій.

### 9.18.10 Інструкції з використання (Doc Blocks)

Надавайте чіткі приклади використання через тег `@example`.
- **❌ Погано**: `function validateEmail(email: string): boolean`. Агент мусить вгадувати, чи це лише перевірка формату, чи і унікальності в БД.
- **✅ Добре**: Детальний JSDoc із прикладами для правильних імейлів, тимчасових доменів та існуючих адрес. Агент тепер точно знає, що функція повертає Promise і викидає специфічні помилки (`ValidationError`, `DuplicateEmailError`).

**API з розумними параметрами за замовчуванням (Sensible Defaults)**:
Проектуйте API так, щоб вони працювали з мінімальною конфігурацією. Це дозволяє агенту почати роботу негайно, не витрачаючи час на дослідження всіх можливих опцій ретраїв або таймаутів.

### 9.18.11 Чек-лист Agent-Friendly кодобази

Оцініть свій проект за 33-бальною шкалою:
1. **Доменні знання** (ADR, CLAUDE.md).
2. **Discoverability** (повні назви, теги, README).
3. **Ефективність токенів** (файли до 500 рядків, відсутність шуму).
4. **Тестування** (TDD, E2E, 80% coverage).
5. **Конвенції** (стандартні патерни, mainstream tech).
6. **Guardrails** (хуки, CI-валідація).

## 9.19 Permutation Frameworks (Фреймворки перестановок)

Замість того, щоб обирати один підхід і сподіватися на краще, ви можете систематично генерувати та оцінювати варіанти архітектури.
**Приклад**: REST vs GraphQL vs tRPC.
1. Визначаєте спільні обмеження (база даних, логіка).
2. Генеруєте варіанти `CLAUDE.md` для кожного підходу.
3. Реалізуєте прототипи в паралельних гілках Git.
4. Просите Claude-рев'юера порівняти їх за складністю, покриттям тестами та зручністю для команди.

## 9.20 Agent Teams (Мультиагентна координація)

**Статус**: ⚠️ Експериментально (v2.1.32+, потрібен Opus 4.6+).
**Agent Teams** дозволяють декільком інстансам Claude працювати паралельно, координуючись автономно. Один сеанс виступає як **Team Lead**, розбиваючи завдання на підзавдання для **Teammates**.
- **Координація**: автоматична, на базі Git (task locking, конфлікт-менеджмент).
- **Найкраще для**: складного рев'ю (безпека + API + фронтенд), масштабного рефакторингу (47+ файлів) або глибокого аналізу всієї кодобази.
- **Попередження**: це в 3 рази дорожче за токенами, тому використовуйте лише для дійсно складних завдань.
