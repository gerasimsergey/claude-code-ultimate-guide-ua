| Конфігурації VCS та інструментів | `.gitconfig`, `.mcp.json`, `.claude.json` |

Конкретні правила `allow` (наприклад, `Bash(npm publish:*)`), визначені в `settings.json` або CLAUDE.md, також зберігаються в режимі `bypassPermissions` — вони продовжують застосовуватися як додаткові фільтри поверх будь-якого режиму дозволів. Це дозволяє створювати точні обмеження (наприклад, "завжди запитувати перед публікацією в npm"), які діють незалежно від того, як запущено сесію.

### Втома від дозволів (Permission Fatigue — антипатерн)

Типова пастка: ви занурені в завдання, запити постійно з'являються, і ви починаєте схвалювати їх, не читаючи. Це **втома від дозволів** — і вона повністю нівелює сенс системи дозволів.

Виправлення полягає в тому, щоб заздалегідь обрати правильний режим, а не клацати по запитах один за одним:

| Ситуація | Правильний режим | Чому |
|-----------|-----------|-----|
| Дослідницька робота, незнайома кодобаза | Plan mode | Неможливо випадково щось змінити |
| Надійні локальні правки, без shell-операцій | `acceptEdits` | Мовчки схвалює правки, але все ще контролює команди |
| Довгі агентні завдання, план Max | Auto mode | Claude оцінює дії; менше переривань з меншим ризиком, ніж bypass |
| Автоматизований пайплайн, ізольоване середовище | `bypassPermissions` | Жодних запитів — але безпечно тільки в ізоляції |
| Потрібне авто-схвалення одного інструменту | `permissions.allow` у CLAUDE.md | Гранулярно, а не "все або нічого" |
| Нова сесія за замовчуванням | Default mode | Явний огляд кожної дії |

Режим, якого слід уникати: використання `--dangerously-skip-permissions` на робочій машині, де є SSH-ключі, API-токени або доступ до продакшену. Система дозволів має цінність лише тоді, коли ви дійсно читаєте те, що схвалюєте — або налаштовуєте режим, що відповідає вашому реальному рівню довіри.

### Посібник з міграції: GitHub Copilot → Claude Code

#### Що Copilot робить добре

- **Інлайнові пропозиції** — Швидке автодоповнення під час друку
- **Звичний воркфлоу** — Працює всередині вашого редактора
- **Низький рівень тертя** — Відсутність перемикання контексту
- **Режим агента** — Редагування багатьох файлів, команди терміналу, автономна ітерація (доступно в VS Code + JetBrains)
- **Безкоштовний план** — 2K доповнень + 50 преміум-запитів за $0/місяць
- **Вибір моделі** — Можливість обирати Claude, Codex, GPT моделі з лютого 2026 року

#### Що Claude Code робить краще

- **Термінальний воркфлоу** — Немає залежності від IDE; працює через SSH, у CI/CD, у будь-якому терміналі
- **Система стійкого контексту** — CLAUDE.md + скіли + хуки накопичуються з часом; кастомні інструкції Copilot новіші та менш гранулярні
- **Оркестрація агентів** — Команди агентів, субагенти, паралельне виконання з детермінованою координацією багатьох файлів
- **Модель оплати за використання** — Немає квот на преміум-запити; режим агента Copilot обмежений щомісячними лімітами (300/міс на Pro, 1500/міс на Pro+)
- **Headless/CI режим** — Запуск у пайплайнах, автоматизація, неінтерактивні контексти
- **Глибока кастомізація** — Кастомні slash-команди, хуки подій, модулі скілів, композиція MCP серверів

#### Гібридний підхід (Рекомендовано)

**Використовуйте Copilot для:**
- Швидкого автодоповнення під час друку
- Генерації шаблонного коду (boilerplate)
- Простих доповнень функцій
- Нескладних багатофайлових завдань всередині IDE (режим агента)
- Швидких запитань у чаті про видимий код

**Використовуйте Claude Code для:**
- Реалізації фіч, що охоплюють кілька репозиторіїв або перетинають архітектурні межі
- Систематичного відлагодження, що потребує глибокого аналізу кодобази
- CI/CD автоматизації та headless виконання
- Рев'ю коду та рефакторингу у великих масштабах
- Розуміння незнайомих кодобаз
- Написання тестів для цілих модулів

**Приклад воркфлоу**:

```bash
# Ранок: Планування фічі з Claude Code

# День: Відлагодження з Claude Code
claude
Ви: "Логін не працює на мобільних пристроях, але працює на десктопі. Відлагоди це."

# Кінець дня: Рев'ю з Claude Code
claude
Ви: "Перевір мої сьогоднішні зміни. Пошукай проблеми з безпекою."
```

#### Що Cursor робить добре

- **Інлайнове редагування** — Прямі модифікації коду в редакторі
- **GUI інтерфейс** — Звичний досвід VS Code
- **Чат + автодоповнення** — Обидва режими в одному інструменті

- **Просунута кастомізація** — Агенти, скіли, хуки, команди

#### Коли варто переходити

**Залишайтеся на Cursor, якщо:**
- Ви рішуче віддаєте перевагу GUI перед CLI
- Ви хочете універсальний досвід IDE "все в одному"
- Вам подобається GUI-центричний воркфлоу з інтегрованим режимом агента
- Вам не потрібна глибока кастомізація

**Переходьте на Claude Code, якщо:**
- Вам комфортно працювати в терміналі
- Ви хочете глибшої кастомізації (агенти, хуки)
- Ви працюєте зі складними проектами з багатьма репозиторіями
- Ви хочете інтегрувати AI в CI/CD
- Ви хочете пряму тарифікацію через API без пулів кредитів

### Типові проблеми при міграції

**Проблема 1: "Мені бракує інлайнових пропозицій"**

- **Рішення**: Продовжуйте використовувати Copilot/Cursor для автодоповнення, а Claude Code — для складних завдань
- **Альтернатива**: Просіть Claude генерувати фрагменти коду, які ви можете вставити

**Проблема 2: "Перемикання контексту дратує"**

- **Рішення**: Використовуйте розділений термінал (редактор зліва, Claude Code справа)
- **Порада**: Налаштуйте гарячу клавішу для перемикання фокуса на термінал

**Проблема 3: "Я не знаю, який інструмент коли використовувати"**

- **Емпіричне правило**:
  - **<5 рядків коду** → Використовуйте Copilot/автодоповнення
  - **5-50 рядків, один файл** → Будь-який інструмент підійде
  - **>50 рядків або кілька файлів** → Використовуйте Claude Code

**Проблема 4: "Claude Code повільніший за автодоповнення"**

- **Реальність**: Claude Code вирішує інші завдання
- **Не порівнюйте**: Автодоповнення vs. повне виконання завдання
- **Оптимізуйте**: Використовуйте конкретні запити, добре керуйте контекстом

**Проблема 5: "Витрати непередбачувані"**

- **Рішення**: Відстежуйте витрати в консолі Anthropic
- **Бюджет**: Встановіть ментальний бюджет на сесію ($0.10-$0.50)
- **Оптимізуйте**: Використовуйте `/compact`, будьте конкретними в запитах

## 1.7 Калібрування довіри: Коли і скільки перевіряти

Код, згенерований AI, потребує **пропорційної перевірки** залежно від рівня ризику. Сліпе прийняття всього результату або параноїдальна перевірка кожного рядка однаково марнують час. Цей розділ допоможе вам відкалібрувати вашу довіру.

### Проблема: Борг за перевірку (Verification Debt)

Дослідження послідовно показують, що код AI має вищий рівень дефектів, ніж код, написаний людиною:

| Метрика | AI vs Людина | Джерело |
|--------|-------------|--------|
| Логічні помилки | в 1.75 рази більше | [ACM study, 2025](https://dl.acm.org/doi/10.1145/3716848) |
| Вразливості безпеки | 45% містять вразливості | [Veracode GenAI Report, 2025](https://veracode.com/blog/genai-code-security-report) |
| XSS вразливості | в 2.74 рази більше | [CodeRabbit study, 2025](https://coderabbit.ai/blog/state-of-ai-vs-human-code-generation-report) |
| Збільшення розміру PR | +18% | [Jellyfish, 2025](https://jellyfish.co) |
| Інциденти на PR | +24% | [Cortex.io, 2026](https://cortex.io) |
| Рівень невдач при змінах | +30% | [Cortex.io, 2026](https://cortex.io) |

**Ключовий інсайт**: AI створює код швидше, але перевірка стає вузьким місцем. Питання не в тому, "чи працює це?", а в тому, "як я дізнаюся, що це працює?".

> **Нюанс щодо зручності обслуговування**: 2-фазне сліпе рандомізоване дослідження (Borg et al., 2025, n=151 професійних розробників) не виявило суттєвої різниці в часі, необхідному для подальшої розробки коду, згенерованого AI, порівняно з людським. Вищезгадані рівні дефектів реальні — але вони не перетворюються систематично на вищий тягар обслуговування для наступного розробника. Ризик більш вузько локалізований, ніж зазвичай вважається. ([arXiv:2507.00788](https://arxiv.org/abs/2507.00788))

### Спектр перевірки

Не весь код потребує однакової ретельності. Узгоджуйте зусилля з перевірки з ризиком:

| Тип коду | Рівень перевірки | Інвестиція часу | Техніки |
|-----------|-------------------|-----------------|------------|
| **Boilerplate** (конфіги, імпорти) | Легкий перегляд | 10-30 сек | Швидкий погляд, довіра структурі |
| **Utility functions** (форматери, хелпери) | Швидкий тест | 1-2 хв | Один тест "щасливого шляху" |
| **Бізнес-логіка** | Глибоке рев'ю + тести | 5-15 хв | Порядковий аналіз, крайові випадки |
| **Критично для безпеки** (auth, crypto, валідація) | Максимальний + інструменти | 15-30 хв | Статичний аналіз, фаззинг, peer review |
| **Зовнішні інтеграції** (API, бази даних) | Інтеграційні тести | 10-20 хв | Тест на моках + реальний ендпоінт |

### Перевірка: соло vs команда

**Стратегія соло-розробника:**

Без колег для рев'ю, компенсуйте це наступним:

1. **Високе покриття тестами (>70%)**: Ваша сітка безпеки
2. **Vibe Review**: Проміжний шар між "прийняти наосліп" та "перевіряти кожен рядок":
   - Прочитайте повідомлення коміту / підсумок
   - Перегляньте diff на предмет несподіваних змін у файлах
   - Запустіть тести
   - Швидка перевірка працездатності в додатку
   - Відправляйте (ship), якщо все зелене
3. **Інструменти статичного аналізу**: ESLint, SonarQube, Semgrep зловлять те, що ви пропустите
4. **Тайм-боксинг**: Не витрачайте 30 хвилин на перевірку 10-рядкового хелпера

```
Соло воркфлоу:
Генерація → Vibe Review → Тести пройдено? → Ship
                 ↓
           Тести впали? → Глибоке рев'ю → Fix
```

**Командна стратегія:**

З кількома розробниками:

1. **Перший етап рев'ю від AI**: Нехай Claude або Copilot перевірять першими (ловить 70-80% проблем)
2. **Обов'язкове людське підтвердження**: Рев'ю AI ≠ схвалення
3. **Експерти для критичних шляхів**: Код безпеки → рев'юер, навчений безпеці
4. **Ротація рев'юерів**: Запобігайте утворенню сліпих зон

```
Командний воркфлоу:
Генерація → AI Рев'ю → Людське Рев'ю → Merge
               ↓              ↓
        Позначити хиби   Фінальне схвалення
```

### Чек-лист "Доведи, що це працює"

Перед відправкою згенерованого AI коду, перевірте:

**Функціональна коректність:**
- [ ] "Щасливий шлях" працює (ручний або авто-тест)
- [ ] Крайові випадки оброблені (null, empty, межові значення)
- [ ] Стани помилок обробляються коректно (немає мовчазних збоїв)

**Базова безпека:**
- [ ] Валідація введення присутня (ніколи не довіряйте вводу користувача)
- [ ] Відсутність захардкоджених секретів (пошукайте `password`, `secret`, `key`)
- [ ] Перевірки auth/authz збережені (немає обходу існуючого захисту)

**Цілісність інтеграції:**
- [ ] Існуючі тести все ще проходять
- [ ] У diff немає несподіваних змін файлів
- [ ] Додані залежності обґрунтовані та перевірені

**Якість коду:**
- [ ] Відповідає конвенціям проекту (іменування, структура)
- [ ] Немає очевидних проблем із продуктивністю (N+1, витоки пам'яті)
- [ ] Коментарі пояснюють "чому", а не "що"

### Антипатерни, яких слід уникати

| Антипатерн | Проблема | Кращий підхід |
|--------------|---------|-----------------|
| **"Воно компілюється, відправляй"** | Синтаксис ≠ коректність | Запустіть принаймні один тест |
| **"AI написав, значить безпечно"** | AI оптимізує під правдоподібність, а не безпеку | Завжди перевіряйте критичний код вручну |
| **"Тести проходять, готово"** | Тести можуть не охоплювати нову зміну | Перевірте покриття змінених рядків |
| **"Так само, як минулого разу"** | Контекст змінюється, AI може видати інший код | Кожна генерація незалежна |
| **"Промпт писав Senior"** | Досвід не гарантує якість виводу | Перевіряйте результат, а не вхідні дані |
| **"Це просто шаблонний код"** | Навіть у шаблонах можуть ховатися помилки | Принаймні пробіжіться очима на предмет сюрпризів |

### Калібрування з часом

Ваша стратегія перевірки має еволюціонувати:

1. **Почніть обережно**: Перевіряйте все, поки ви новачок у Claude Code
2. **Відстежуйте патерни помилок**: Де баги прослизають найчастіше?
3. **Посильте контроль критичних шляхів**: Збільште увагу до зон, де вже траплялися інциденти
4. **Послабте контроль зон низького ризику**: Більше довіряйте AI у стабільних, перевірених типах коду
5. **Періодичні аудити**: Час від часу вибірково перевіряйте "довірений" код

**Ментальна модель**: Сприймайте AI як здібного молодшого розробника (junior). Ви б не деплоїли його код без рев'ю, але ви також не переписуєте все, що він створив.

## 1.8 Вісім помилок початківців (та як їх уникнути)

Типові пастки, що сповільнюють нових користувачів Claude Code:

### 1. ❌ Пропуск планування

**Помилка**: Перехід одразу до "виправи цей баг" без пояснення контексту.

**Виправлення**: Використовуйте формат ЩО/ДЕ/ЯК/ПЕРЕВІРКА:
```
ЩО: Виправити помилку таймауту логіну
ДЕ: src/auth/session.ts
ЯК: Збільшити термін дії токена з 1 год до 24 год
ПЕРЕВІРКА: Логін зберігається після оновлення браузера
```

### 2. ❌ Ігнорування лімітів контексту

**Помилка**: Робота доти, доки контекст не досягне 95% і якість відповідей не впаде.

**Виправлення**: Слідкуйте за `Ctx(u):` у рядку статусу. Використовуйте `/compact` на 70%, `/clear` на 90%.

### 3. ❌ Використання розпливчастих промптів

**Помилка**: "Зроби цей код кращим" або "Перевір на баги"

**Виправлення**: Будьте конкретними: "Рефактори `calculateTotal()`, щоб він обробляв нульові ціни без помилок"

### 4. ❌ Сліпе прийняття змін

**Помилка**: Натискання "y" без читання diff.

**Виправлення**: Завжди переглядайте diff. Використовуйте "n", щоб відхилити, а потім поясніть, що не так.

### 5. ❌ Відсутність безпеки версійного контролю

**Помилка**: Внесення великих змін без комітів.

**Виправлення**: Робіть коміт перед великими змінами. Використовуйте гілки фіч. Claude може допомогти: `/commit`

### 6. ❌ Занадто широкі дозволи

**Помилка**: Налаштування `Bash(*)` або `--dangerously-skip-permissions`

**Виправлення**: Починайте з обмежень, розширюйте за потреби. Використовуйте дозволені списки: `Bash(npm test)`, `Bash(git *)`

### 7. ❌ Змішування непов'язаних завдань

**Помилка**: "Виправи баг автентифікації ТА рефактори базу даних ТА додай нові тести"

**Виправлення**: Одне сфокусоване завдання на сесію. Використовуйте `/clear` між різними завданнями.

**Як оцінити розмір завдання для Claude Code:**

| Сигнал | Занадто велике | Правильний розмір | Занадто мале |
|--------|---------|------------|-----------|
| Опис | Використовує "ТА" між різними поведінками | Один вертикальний зріз, одна поведінка | Зміна одного рядка, яку швидше зробити вручну |
| Сесія | Вичерпує контекст або втрачає фокус | Завершується за одну сесію | Займає 30 секунд |
| Рев'ю | Рев'юер не може осягнути весь diff | Diff можна перевірити за один прохід | Не варте рев'ю |
| Відкат | Скасування ламає інші речі | `git revert` чисто скасовує все | N/A |

**Евристика розділення**: якщо опис вашого завдання потребує сполучника "та" між двома різними поведінками для користувача — розділіть його. "Користувачі можуть скидати паролі" — це одне завдання. "Користувачі можуть скидати паролі ТА адміни можуть примусово завершувати сесії" — це два.

> **Глибоке занурення**: [Spec-First Workflow — Task Granularity](./workflows/spec-first.md#task-granularity-sizing-work-for-agents) охоплює патерн вертикального зрізу, чек-лист якості PRD та конкретні приклади "до/після".

### 8. ❌ Ставлення до Claude Code як до чат-бота

**Помилка**: Набір випадкових інструкцій кожну сесію. Повторення конвенцій проекту, архітектури, ручне нагадування про перевірки якості.

**Виправлення**: Будуйте структурований контекст, який накопичується:
- **CLAUDE.md**: Ваші конвенції, стек та патерни — завантажуються автоматично кожну сесію
- **Skills**: Багаторазові воркфлоу (`/review`, `/deploy`) для стабільного виконання
- **Hooks**: Автоматичні запобіжники (лінт, безпека, форматування) — нуль ручних зусиль

Почніть з CLAUDE.md на першому тижні. Див. [§2.6 Ментальна модель](#from-chatbot-to-context-system) для повної схеми.

### Швидка самоперевірка

## 2.2 Управління контекстом

🔴 **Це найважливіша концепція в Claude Code.**

### 📌 Швидка довідка з управління контекстом

**Зони**:
- 🟢 0-50%: Працюйте вільно
- 🟡 50-75%: Будьте вибірковими
- 🔴 75-90%: `/compact` зараз
- ⚫ 90%+: Необхідно `/clear`

**Коли контекст високий**:
1. `/compact` (зберігає контекст, звільняє місце)
2. `/clear` (чистий старт, втрачається історія)

**Запобігання**: Завантажуйте лише необхідні файли, регулярно компактіть, часто робіть коміти.

---

### Що таке контекст?

Контекст — це "робоча пам'ять" Claude для вашої розмови. Вона включає:
- Усі повідомлення в розмові
- Файли, які прочитав Claude
- Виводи команд
- Результати роботи інструментів

### Бюджет контексту

Claude має вікно контексту в **200 000 токенів**. Уявіть це як оперативну пам'ять (RAM) — коли вона заповнюється, все сповільнюється або ламається.

### Читання рядка статусу

Рядок статусу показує використання контексту:

```
Claude Code │ Ctx(u): 45% │ Cost: $0.23 │ Session: 1h 23m
```

| Метрика | Значення |
|--------|---------|
| `Ctx(u): 45%` | Ви використали 45% контексту |
| `Cost: $0.23` | Вартість API на даний момент |
| `Session: 1h 23m` | Час, що минув |

### Налаштування кастомного рядка статусу

Стандартний рядок статусу можна покращити, додавши детальнішу інформацію: гілку git, назву моделі та зміни у файлах.

**Варіант 1: [ccstatusline](https://github.com/sirmalloc/ccstatusline) (рекомендовано)**

Додайте в `~/.claude/settings.json`:

```json
{
  "statusLine": {
    "type": "command",
    "command": "npx -y ccstatusline@latest",
    "padding": 0
  }
}
```

Це відображатиме: `Model: Sonnet 4.6 | Ctx: 0 | ⎇ main | (+0,-0) | Cost: $0.27 | Session: 0m | Ctx(u): 0.0%`

**Варіант 2: Власний скрипт**

Створіть свій скрипт, який:
1. Читає дані JSON зі stdin (модель, контекст, вартість, інфо git)
2. Виводить один відформатований рядок у stdout
3. Підтримує ANSI кольори для стилізації

```json
{
  "statusLine": {
    "type": "command",
    "command": "/path/to/your/statusline-script.sh",
    "padding": 0
  }
}
```

Використовуйте команду `/statusline` у Claude Code, щоб автоматично згенерувати стартовий скрипт.

**Доступні поля JSON (stdin)**:

| Поле | Тип | Опис |
|-------|------|-------------|
| `model` | string | Назва поточної моделі |
| `context` | object | `used`, `total`, `percentage` |
| `cost_usd` | number | Вартість сесії |
| `git` | object | Гілка, кількість staged/unstaged файлів |
| `rate_limits` | object | Використання на Claude.ai (v2.1.80+) |

**Об'єкт `rate_limits`** (v2.1.80+) — відображає використання токенів Claude.ai прямо в рядку статусу без відкриття дашборду:

```json
{
  "rate_limits": {
    "5h":  { "used_percentage": 42, "resets_at": "2026-03-20T15:30:00Z" },
    "7d":  { "used_percentage": 18, "resets_at": "2026-03-23T00:00:00Z" }
  }
}
```

Приклад використання у скрипті рядка статусу:

```bash
#!/usr/bin/env bash
input=$(cat)
pct_5h=$(echo "$input" | jq -r '.rate_limits["5h"].used_percentage // "?"')
echo "RL: ${pct_5h}%"
```

### Зони контексту

| Зона | Використання | Дія |
|------|-------|--------|
| 🟢 Зелена | 0-50% | Працюйте вільно |
| 🟡 Жовта | 50-75% | Почніть бути вибірковими |
| 🔴 Червона | 75-90% | Використовуйте `/compact` або `/clear` |
| ⚫ Критична| 90%+ | Обов'язково очистіть, інакше будуть помилки |

### Стратегії відновлення контексту

Коли контекст стає високим:

**Варіант 1: Compact** (`/compact`)
- Підсумовує розмову
- Зберігає ключовий контекст
- Зменшує використання приблизно на 50%

> **Коли `/compact` йде не так**: Стиснення запускається тоді, коли модель має найбільше накопиченого контексту, тобто вона перебуває в точці найбільшої відволікання. Якщо модель не може передбачити, куди рухається робота (наприклад, авто-стиснення спрацювало посеред відлагодження, а ваше наступне повідомлення — "тепер виправ те попередження в bar.ts"), вона може видалити релевантну для майбутнього інформацію з підсумку. Пом'якшіть це, компактячи проактивно та з контекстом: `/compact focus on the auth refactor, drop the test debugging` спрямовує підсумок на те, що важливо далі. (Джерело: внутрішні настанови Anthropic)

**Варіант 2: Clear** (`/clear`)
- Починає з чистого аркуша
- Втрачає весь контекст
- Використовуйте при зміні тем

> **"Одне завдання, один чат"** — змішування непов'язаних тем між кроками знижує точність моделі приблизно на 39%. Контекст накопичує шум ("гниття контексту"), що спотворює судження, навіть якщо загальна кількість токенів залишається низькою. Використовуйте `/clear` агресивно між різними завданнями, а не лише тоді, коли смужка контексту стає червоною.

**Варіант 3: Summarize from here** (v2.1.32+)
- Використовуйте `/rewind` (або `Esc + Esc`), щоб відкрити список чекпоінтів
- Оберіть чекпоінт і виберіть "Summarize from here"
- Claude підсумує все, що було після цього моменту, зберігаючи попередній контекст недоторканим
- Звільняє місце, зберігаючи критично важливий контекст
- Точніше, ніж повний `/compact`

**Варіант 4: Цільовий підхід**
- Будьте конкретними в запитах
- Уникайте "прочитай весь файл"
- Використовуйте посилання на символи: "прочитай функцію `calculateTotal`"

### Тріаж контексту: Що залишити, а що видалити

При наближенні до червоної зони (75%+), одного лише `/compact` може бути недостатньо. Вам потрібно активно вирішувати, яку інформацію зберегти перед стисненням.

**Пріоритет: Залишити**

| Залишити | Чому |
|------|-----|
| Вміст CLAUDE.md | Основні інструкції мають зберігатися |
| Файли, що активно редагуються | Контекст поточної роботи |
| Тести для поточного компонента | Контекст валідації |
| Прийняті критичні рішення | Архітектурні вибори |
| Повідомлення про помилки | Контекст проблеми |

**Пріоритет: Видалити**

| Видалити | Чому |
|----------|-----|
| Файли, що прочитані, але більше не релевантні | Разові перегляди |
| Виводи відлагодження вирішених проблем | Історичне сміття |
| Довга історія розмови | Буде підсумована /compact |
| Файли завершених завдань | Більше не потрібні |
| Великі конфіг-файли | Можна перечитати за потреби |

**Чек-лист перед стисненням (Pre-Compact)**:

1. **Задокументуйте критичні рішення** у CLAUDE.md або нотатках сесії
2. **Закомітьте зміни** в git (створює точку відновлення)
3. **Чітко вкажіть поточне завдання** ("Ми впроваджуємо X")
4. **Запустіть `/compact`**, щоб підсумувати та звільнити місце

**Порада**: Якщо ви знаєте, що вам знадобиться конкретна інформація після стиснення, скажіть про це Claude прямо: "Перед тим, як ми стиснемо контекст, пам'ятай, що ми вирішили використовувати Стратегію А для автентифікації через X." Claude включить це в підсумок.

### Сесійна vs Стійка пам'ять

Claude Code має три різні системи пам'яті. Розуміння різниці є ключовим для ефективної довгострокової роботи:

| Аспект | Сесійна пам'ять | Auto-Memory (рідна) | Стійка пам'ять (Serena) |
|--------|----------------|----------------------|---------------------------|
| **Охоплення** | Тільки поточна розмова | Між сесіями, для проекту | Між усіма сесіями |
