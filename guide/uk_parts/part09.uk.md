5. **CI**: Додайте щоденне завдання GitHub Actions, яке перегенерує всі вихідні дані та запускає `git diff --exit-code`, щоб виявити розбіжності.

### Коли НЕ варто це використовувати

Цей патерн має реальні накладні витрати. Будьте чесними щодо того, чи потрібен він вам:

| Ситуація | Рекомендація |
|-----------|----------------|
| Розробник-одинак | Не варте зусиль. Одного CLAUDE.md достатньо. |
| Команда з 2-3 осіб, однакові інструменти | На межі. Використовуйте правила пріоритетності CLAUDE.md (Розділ 3.4). |
| Команда 5+, кілька інструментів | Цей патерн окупається. |
| Інструкції, що швидко змінюються | Висока вартість обслуговування. Спочатку стабілізуйте правила, потім модулюйте. |
| Короткострокові проекти (<3 місяців) | Занадто складно. Використовуйте спільний CLAUDE.md. |

Точка беззбитковості — приблизно **3+ розробники з 2+ різними інструментами ШІ**. Нижче цього порогу накладні витрати на управління файлами перевищують вигоду.

> Повний покроковий воркфлоу реалізації див. у [Командні інструкції ШІ](workflows/team-ai-instructions.md).

### Політика розкриття коду ШІ (Командне управління)

Коли кілька розробників використовують Claude Code над однією кодобазою, прихована генерація ШІ створює тиху проблему якості: код мержиться без розуміння кимось того, що він робить або чому.

**Патерн** (від команд у продакшені): зробити генерацію ШІ видимою, не блокуючи її.

**Поріг розкриття**: якщо Claude генерує більше ніж ~10 рядків поспіль, автор заявляє про це в PR.

**Доповнення до шаблону PR**:

```markdown
## Участь ШІ

**Що зробив ШІ**: [список змінених файлів або розділів]
**Що зробив я**: [перевірив, адаптував, протестував, зрозумів]
**Рев'ю**: [так / ні — поясніть, якщо ні]
```

**Чому це працює**:
- Змушує автора справді прочитати та зрозуміти згенерований код перед мерджем
- Робить рев'ю коду ефективнішим (рев'юери знають, що саме ретельно перевіряти)
- Запобігає накопиченню технічного боргу через "vibe coding"
- Створює документальний слід для архітектурних рішень

**Градуйоване впровадження** — підберіть під рівень зрілості вашої команди:

| Рівень розробника | Вимога до розкриття |
|-----------------|----------------------|
| Junior / онбординг | Обов'язково — кожен блок, згенерований ШІ |
| Intermediate | Рекомендовано — нетривіальні фічі |
| Senior | Опціонально — на власний розсуд |

**Чим це НЕ є**:
- Не забороною на генерацію ШІ
- Не вправою з підрахунку рядків
- Не механізмом пошуку винних

> **Антипатерн**: Пропуск розкриття заради швидкості. Прихована вартість полягає в тому, що рев'юери схвалюють код, який ніхто не розуміє, що протягом місяців перетворюється на непрозорі для всієї команди ділянки кодобази.

### 3 принципи Бориса Черного для команд ШІ

> Ці принципи Борис Черний (керівник Claude Code в Anthropic) поширює серед кожного нового члена команди.
> — *Lenny's Newsletter, 19 лютого 2026 року*

**1. Навмисно недофінансовуйте проекти**

Мати одного чудового інженера над великою проблемою — замість цілої команди — змушує до глибокої утилізації ШІ. Це обмеження прискорює доставку коду, а не уповільнює її. Вузьке місце зміщується з кількості людей на якість промптів та воркфлоу.

**2. Спочатку дайте інженерам необмежену кількість токенів**

Не оптимізуйте витрати токенів на ранніх етапах. Дайте інженерам свободу для максимальних експериментів. Божевільні, інноваційні патерни з'являються лише тоді, коли ніхто не стежить за лічильником. Оптимізуйте витрати *після того*, як успішна ідея довела свою цінність і потребує масштабування.

**3. Заохочуйте людей діяти швидше**

Перший інстинкт при роботі з інструментами ШІ — обережність: перевірка кожного виводу, сумніви щодо кожної пропозиції. Кращий інстинкт: деплой, валідація, ітерація. Claude Code створений для високошвидкісних циклів, а не для довгих роздумів.

> **Коли застосовувати**: Команди від 2 осіб, що використовують Claude Code професійно. Розробникам-одинакам варто зосередитися на перших двох принципах (недофінансування = сприймайте себе як команду з однієї людини з важелем ШІ; необмежені токени = не цензуруйте свої експерименти).

### Йдемо далі: Розповсюдження стандартів у масштабах організації

Збірка модулів на основі профілів вирішує проблему узгодженості для кожного розробника. Однак це все одно потребує від вашої команди ручного обслуговування модулів та запуску збирача. При наявності 50+ розробників у 30+ репозиторіях навіть це стає перешкодою.

Інструменти на кшталт [Packmind](ecosystem/third-party-tools.md#packmind) розвивають цей принцип далі: визначте стандарти один раз у центральному плейбуку, і вони будуть автоматично розповсюджені як файли `CLAUDE.md`, слеш-команди та скіли — між репозиторіями та різними інструментами ШІ (Claude Code, Cursor, Copilot, Windsurf). Плейбук також може вбирати знання з коментарів до рев'ю PR, обговорень у Slack та звітів про інциденти, щоб підтримувати стандарти актуальними без ручного втручання.

> **Коли варто розглянути це**: Команди від 10+ розробників, 5+ репозиторіїв, використання більше одного кодуючого агента ШІ.

## 4.1 Що таке агенти

Агенти — це спеціалізовані підпроцеси, яким Claude може делегувати завдання.

### Навіщо використовувати агентів?

| Без агентів | З агентами |
|----------------|-------------|
| Один Claude робить усе | Спеціалізовані експерти для кожної галузі |
| Контекст захаращується | Кожен агент має фокусований контекст |
| Загальні відповіді | Специфічна для домену експертиза |
| Ручний вибір інструментів | Попередньо налаштований доступ до інструментів |

### Агент проти прямого промпту

```
Прямий промпт:
Ви: Перевір цей код на проблеми безпеки, зосередившись на OWASP Top 10,
     перевіряючи SQL-ін'єкції, XSS, CSRF та вразливості автентифікації...

З агентом:
Ви: Використовуй агента security-reviewer для аудиту цього коду
```

Агент інкапсулює всю цю експертизу.

### Вбудовані проти кастомних агентів

| Тип | Джерело | Приклад |
|------|--------|---------|
| Вбудовані | Claude Code за замовчуванням | Explore, Plan |
| Кастомні | Ваша папка `.claude/agents/` | Backend architect, Code reviewer |

## 4.2 Створення кастомних агентів

Агенти — це markdown-файли в `.claude/agents/` з YAML frontmatter.

### Структура файлу агента

```markdown
---
name: agent-name
description: Чіткий тригер активації (50-100 символів)
model: sonnet
tools: Read, Write, Edit, Bash, Grep, Glob
---

[Інструкції для агента у форматі Markdown]
```

### Поля Frontmatter

Усі офіційні поля, що підтримуються Claude Code ([джерело](https://code.claude.com/docs/en/sub-agents)):

| Поле | Обов'язково | Опис |
|-------|----------|-------------|
| `name` | ✅ | Ідентифікатор у kebab-case |
| `description` | ✅ | Коли активувати цього агента (використовуйте "PROACTIVELY" для автовиклику) |
| `model` | ❌ | `sonnet` (за замовчуванням), `opus`, `haiku` або `inherit` |
| `tools` | ❌ | Дозволені інструменти (через кому). Підтримує синтаксис `Task(agent_type)` для обмеження субагентів, яких можна створювати |
| `disallowedTools` | ❌ | Інструменти для заборони, видаляються з успадкованого або вказаного списку |
| `permissionMode` | ❌ | `default`, `acceptEdits`, `dontAsk`, `bypassPermissions` або `plan` |
| `maxTurns` | ❌ | Максимальна кількість кроків агента перед зупинкою субагента |
| `skills` | ❌ | Скіли для попереднього завантаження в контекст агента при запуску (вприскується повний зміст, а не просто доступ) |
| `mcpServers` | ❌ | Сервери MCP для цього субагента — назви серверів або вбудовані конфіги |
| `hooks` | ❌ | Хуки життєвого циклу для цього субагента (`PreToolUse`, `PostToolUse`, `Stop`) |
| `memory` | ❌ | Область персистентної пам'яті: `user`, `project` або `local` |
| `background` | ❌ | `true`, щоб завжди запускати як фонове завдання (за замовчуванням: `false`) |
| `isolation` | ❌ | `worktree`, щоб запускати в тимчасовій git worktree (автоочищення, якщо немає змін) |
| `color` | ❌ | Колір виводу в CLI для візуального розрізнення (наприклад, `green`, `magenta`) |

**Області пам'яті** — обирайте залежно від того, наскільки широко мають застосовуватися знання:

| Область | Сховище | Коли використовувати |
|-------|---------|----------|
| `user` | `~/.claude/agent-memory/<name>/` | Навчання між проектами |
| `project` | `.claude/agent-memory/<name>/` | Специфічно для проекту, можна ділитися через git |
| `local` | `.claude/agent-memory-local/<name>/` | Специфічно для проекту, не комітиться |

> Повне охоплення пам'яті агентів — ліміт ін'єкції 200 рядків, структура MEMORY.md, посібник із вибору області — у [§4.5 Пам'ять агентів](#45-agent-memory).

### Вибір моделі

| Модель | Найкраще для | Швидкість | Вартість |
|-------|----------|-------|------|
| `haiku` | Швидких завдань, простих змін | Швидко | Низька |
| `sonnet` | Більшості завдань (за замовчуванням) | Збалансовано | Середня |
| `opus` | Складних міркувань, архітектури | Повільно | Висока |

## 4.4 Найкращі практики

### Що варто і чого не варто робити

| ✅ Варто | ❌ Не варто |
|-------|----------|
| Робіть агентів спеціалістами | Не створюйте агентів-універсалів |
| Визначайте чіткі тригери | Не використовуйте розпливчасті описи |
| Додавайте конкретні приклади | Не залишайте активацію неоднозначною |
| Обмежуйте доступ до інструментів | Не давайте всі інструменти всім агентам |
| Компонуйте через скіли | Не дублюйте експертизу |

### Спеціалізація проти універсалізації

**Добре**: Агент для кожної конкретної задачі
```
backend-architect    → дизайн API, база даних, продуктивність
security-reviewer    → OWASP, автентифікація, шифрування
test-engineer        → стратегія тестування, покриття, TDD
```

**Погано**: Один агент для всього
```
full-stack-expert    → робить усе (посередньо)
```

### Явні тригери активації

**Гарний опис**:
```yaml
description: Використовуй при проектуванні API, рев'ю схем баз даних або оптимізації продуктивності бекенду
```

**Поганий опис**:
```yaml
description: Речі для бекенду
```

### Композиція скілів

Замість дублювання знань:

```yaml
# security-reviewer.md
skills:
  - security-guardian  # успадковує знання OWASP
```

### Чек-лист валідації агента

Перед впровадженням кастомного агента перевірте його за цими критеріями:

**Ефективність** (Чи він працює?)
- [ ] Протестовано на 3+ реальних кейсах вашого проекту
- [ ] Вивід стабільно відповідає очікуваному формату
- [ ] Гідно обробляє крайові випадки (порожній ввід, помилки, таймаути)
- [ ] Коректно інтегрується в існуючі воркфлоу

**Економічність** (Чи він виправданий за ціною?)
- [ ] <5000 токенів на типове виконання
- [ ] <30 секунд для стандартних завдань
- [ ] Не дублює роботу інших агентів/скілів
- [ ] Виправдовує своє існування порівняно з нативними можливостями Claude

**Безпека** (Чи він безпечний?)
- [ ] Інструменти обмежені мінімально необхідними
- [ ] Без доступу до Bash, якщо це не обов'язково
- [ ] Доступ до файлів обмежений релевантними директоріями
- [ ] У визначенні агента немає облікових даних чи секретів

**Підтримка** (Чи він довговічний?)
- [ ] Чітке, описове ім'я та опис
- [ ] Задокументовано явні тригери активації
- [ ] Приклади показують типові патерни використання
- [ ] Зазначено сумісність версій, якщо є залежність від фреймворку

> 💡 **Правило трьох**: Якщо агент не економить значну кількість часу щонайменше на 3 повторюваних завданнях, це, ймовірно, надмірна інженерія. Починайте зі скілів, переходьте до агентів тільки тоді, коли цього вимагає складність.

> **Автоматизований аудит**: Запустіть `/audit-agents-skills` для комплексного аудиту якості всіх агентів, скілів та команд. Кожен файл оцінюється за 16 критеріями з ваговими балами (32 бали для агентів/скілів, 20 для команд). Див. `examples/skills/audit-agents-skills/` для повної методології оцінювання.

### Фотові (background) субагенти

Субагенти можуть працювати у фоновому режимі, не блокуючи основну сесію. Це корисно для завдань типу "запустив і забув", як-от виконання тестів, лінтинг або сповіщення.

| Режим | Поведінка | Коли використовувати |
|------|----------|----------|
| Стандартний | Батьківський процес чекає на вивід агента | Потрібен результат перед продовженням |

## 4.5 Пам'ять агентів

Впроваджена в **Claude Code v2.1.33** (лютий 2026), поле `memory` у frontmatter дає субагентам персистентні знання на основі маркдауну, які зберігаються між сесіями. До цього кожен виклик агента починався з чистого аркуша, незалежно від попередніх запусків.

### Чому пам'ять агентів важлива

Без пам'яті агент-рев'юер, який виявив, що ваша команда надає перевагу патернам раннього повернення (early-return) замість вкладених `if`, не має можливості перенести це спостереження далі. Наступний виклик почнеться "з нуля". Пам'ять агентів виправляє це: агент записує свої висновки в структурований файл, і майбутні виклики продовжують з того місця, де зупинився останній.

Це відрізняється від інших систем пам'яті в Claude Code. Кожна з них служить своїй меті:

| Система | Ким пишеться | Ким читається | Область | Зберігається |
|--------|------------|---------|-------|----------|
| **CLAUDE.md** | Вами (вручну) | Main Claude + усі агенти | Проектна або глобальна | Відстежується Git |
| **Авто-пам'ять** | Main Claude (авто) | Тільки Main Claude | На проект на користувача | Ігнорується Git |
| **Пам'ять агента** | Самим агентом | Тільки цим специфічним агентом | Налаштовується | Залежить від області |

Агент читає як `CLAUDE.md` (спільний контекст проекту), так і власну пам'ять (накопичені знання конкретного агента). Ці два шари доповнюють один одного.

### Області пам'яті (Memory Scopes)

Обирайте область залежно від того, де ці знання будуть корисними:

| Область | Місце зберігання | Контроль версій | Найкраще для |
|-------|-----------------|-------------------|----------|
| `user` | `~/.claude/agent-memory/<agent-name>/` | Ні | Навчання між проектами — рев'юер коду, що накопичує знання про патерни в кожному репозиторії |
| `project` | `.claude/agent-memory/<agent-name>/` | Так (комітиться) | Знання специфічні для проекту, якими має ділитися вся команда — напр., конвенції API, виявлені агентом-скаффолдером |
| `local` | `.claude/agent-memory-local/<agent-name>/` | Ні (gitignore) | Особисті знання специфічні для проекту, які не варто комітити |

Ці області дзеркально відображають ієрархію налаштувань (`~/.claude/settings.json` → `.claude/settings.json` → `.claude/settings.local.json`), роблячи ментальну модель узгодженою для всієї системи.

Активуйте пам'ять, додавши один рядок у frontmatter агента:

```yaml
---
name: code-reviewer
description: Рев'ю коду на якість, безпеку та послідовність
tools: Read, Grep, Glob
memory: user
---
```

### Як працює вприскування 200 рядків

Коли агент запускається, Claude Code зчитує перші 200 рядків файлу `MEMORY.md` у директорії пам'яті агента і вприскує їх безпосередньо в системний промпт агента. Це відбувається автоматично — явний виклик інструменту не потрібен.

```
~/.claude/agent-memory/code-reviewer/
├── MEMORY.md                   ← перші 200 рядків вприскуються при старті
├── react-patterns.md           ← тематичний файл, завантажується за запитом
└── security-checklist.md       ← тематичний файл, завантажується за запитом
```

Як тільки `MEMORY.md` перевищує 200 рядків, агент повинен перенести детальний вміст у тематичні файли, залишивши `MEMORY.md` як лаконічний індекс із посиланнями. Агент керує цим сам — інструменти `Read`, `Write` та `Edit` автоматично доступні будь-якому агенту з встановленим параметром `memory`.

**Практичний наслідок**: структуруйте `MEMORY.md` як розумний підсумок, а не як журнал, куди тільки дописують дані. Найважливіші записи — зверху, тематичні файли — для глибини.

### Структура MEMORY.md

Добре структурований файл пам'яті агента робить вприснутий вміст миттєво корисним:

```markdown
# пам'ять code-reviewer
Останнє оновлення: 2026-03-10

## Конвенції проекту (підтверджено)
- Раннє повернення замість вкладених умов (підтверджено у 12 рев'ю)
- `zod` для всієї валідації на межах API — ніколи не використовувати `joi` чи чисті перевірки типів
- Middleware автентифікації має застосовуватися перед будь-якою логікою контролера

## Повторювані проблеми
- Відсутній `await` у асинхронних викликах БД у `/src/services/` (помічено 4 рази цього місяця)
- Приведення до `any` у скриптах міграції допускається як відоме виключення

## Патерни, за якими варто стежити
- Нові контриб'ютори схильні пропускати обгортку error boundary у деревах React

## Тематичні файли
- [react-patterns.md](react-patterns.md) — структура компонентів, використання хуків, правила мемоїзації
- [security-checklist.md](security-checklist.md) — нотатки OWASP Top 10 за категоріями
```

### Промптування агентів використовувати свою пам'ять

Пам'ять корисна лише тоді, коли агент послідовно читає та пише в неї. Явне промптування в тілі агента має велике значення:

```yaml
---
name: api-developer
description: Реалізація ендпоінтів API згідно з конвенціями команди
tools: Read, Write, Edit, Bash
memory: project
---

Перед початком будь-якого завдання перевір свою пам'ять на наявність відповідних
конвенцій та минулих рішень. Після завершення завдання онови свою пам'ять новими
патернами, архітектурними рішеннями або повторюваними проблемами, які ти помітив.
Тримай MEMORY.md у межах 200 рядків — перенось детальні нотатки у тематичні файли.
```

Цей патерн — скіли для статичних знань при старті, пам'ять для динамічних накопичених знань — дає агентам найкраще з обох світів. Скіли вприскують кураторський довідковий матеріал при першому запуску; пам'ять несе далі те, що агент виявляє самостійно.

### Вибір правильної області

| Ситуація | Рекомендована область |
|-----------|------------------|
| Загальний рев'юер коду, що використовується в багатьох проектах | `user` — знання накопичуються глобально |
| Агент для створення каркасу API, що вивчає конвенції ендпоінтів вашої команди | `project` — комітьте пам'ять, щоб колеги могли нею скористатися |
| Особистий агент для рефакторингу з вашими вподобаннями щодо стилю | `local` — залишається тільки на вашій машині |
| Агент для клієнтського проекту, який ви не хочете змішувати з особистими знаннями | `local` — ізольований, не комітиться |

# Code Reviewer

## Визначення обсягу

Виконуйте комплексне рев'ю коду з ізольованим контекстом, зосереджуючись на:
- Якості коду та зручності підтримки
- Найкращих практиках безпеки (OWASP Top 10)
- Оптимізації продуктивності
- Аналізі покриття тестами

Область: Тільки аналіз рев'ю коду. Надавайте висновки без впровадження виправлень.

## Тригери активації

Використовуйте цього агента, коли:
- Завершуєте фічу перед PR (потрібен "свіжий погляд" на код)
- Рев'юєте чужий код (ізольований контекст рев'ю)
- Проводите аудит безпеки критичного коду (фокус на безпеці)
- Аналізуєте вузькі місця продуктивності (фокус на продуктивності)

## Методологія

1. **Зрозуміти контекст**: Прочитати код і зрозуміти його призначення
2. **Перевірити якість**: Оцінити читабельність, підтримку, принципи DRY
3. **Сканування безпеки**: Шукати вразливості OWASP Top 10
4. **Рев'ю продуктивності**: Виявити потенційні вузькі місця
5. **Надати фідбек**: Структурований звіт із рівнями критичності

## Формат виводу

### Звіт з рев'ю коду

**Підсумок**: [огляд 1-2 реченнями]

**Критичні проблеми** (Must Fix):
- [Проблема з посиланням файл:рядок]

**Попередження** (Should Fix):
- [Проблема з посиланням файл:рядок]

**Пропозиції** (Nice to Have):
- [Можливість для покращення]

**Позитивні моменти**:
- [Що було зроблено добре]
```

## Визначення обсягу

1. **Відтворити**: Підтвердити наявність проблеми
2. **Ізолювати**: Звузити до найменшого випадку відтворення
3. **Проаналізувати**: Прочитати код, перевірити логи, простежити виконання
4. **Сформулювати гіпотезу**: Висунути теорії про причину
5. **Протестувати**: Перевірити гіпотезу мінімальними змінами
6. **Виправити**: Реалізувати рішення
7. **Верифікувати**: Підтвердити, що виправлення працює і не ламає інше

## Формат виводу

### Звіт про налагодження

**Проблема**: [Опис]
**Першопричина**: [Що насправді не так]
**Докази**: [Як ви про це дізналися]
**Виправлення**: [Що змінити]
**Верифікація**: [Як підтвердити, що це працює]
```

## 4.7 Просунуті патерни агентів

### Tool SEO — Оптимізація описів агентів

Поле `description` визначає, коли Claude автоматично активує вашого агента. Оптимізуйте його як SEO:

```yaml
# ❌ Поганий опис
description: Рев'юїть код

# ✅ Хороший опис (Tool SEO)
description: |
  Security code reviewer - використовуй PROACTIVELY при:
  - Рев'ю коду автентифікації/авторизації
  - Аналізі ендпоінтів API
  - Перевірці валідації вводу
  - Аудиті обробки даних
  Тригери: security, auth, vulnerability, OWASP, injection
```

**Техніки Tool SEO**:
1. **"використовуй PROACTIVELY"**: Заохочує автоматичну активацію
2. **Явні тригери**: Ключові слова, що активують агента
3. **Перелічені контексти**: Коли агент є релевантним
4. **Короткі нікнейми**: `sec-1`, `perf-a`, `doc-gen`

### Класифікація "ваги" агентів

| Категорія | Токени | Час ініціалізації | Оптимальне використання |
|----------|--------|-----------|-------------|
| **Легкий** | <3K | <1s | Часті завдання, воркери |
| **Середній** | 10-15K | 2-3s | Аналіз, рев'ю |
| **Важкий** | 25K+ | 5-10s | Архітектура, повні аудити |

**Золоте правило**: Легкий агент, використаний 100 разів > Важкий агент, використаний 10 разів

### Метод 7 паралельних завдань (7-Parallel-Task)

Запускайте 7 субагентів з фокусом на певну область паралельно для повної реалізації фіч:

└─────────────────────────────────────────────────────────────┘

```
Проаналізуй цей PR з ізольованими областями:
1. Архітектурна область: патерни дизайну, принципи SOLID, модульність
2. Область безпеки: вразливості, ризики ін'єкцій, недоліки auth/authz
3. Область продуктивності: запити до БД, алгоритмічна складність, кешування
4. Область підтримки: ясність коду, документація, конвенції іменування
5. Область тестування: покриття тестами, крайові випадки, тестованість

Контекст: src/**, tests/**, тільки файли, змінені у PR
```

**Промпт для рев'ю UX** (фокус на області):
```
Оціни цей інтерфейс з ізольованими областями:
1. Візуальний дизайн: послідовність із дизайн-системою, відступи, типографіка
2. Юзабіліті: виявляємість, шлях користувача, когнітивне навантаження
3. Ефективність: гарячі клавіші, функції для досвідчених користувачів
4. Доступність: відповідність WCAG 2.1 AA, скрінрідери, навігація клавіатурою
5. Адаптивність: мобільні брейкпоінти, touch-цілі, обробка в'юпорту

Контекст: src/components/**, styles/**, тільки файли, пов'язані з UI
```

**Приклад із продакшену: Мультиагентне рев'ю коду** (Пет Каллен, січень 2026):

Спеціалізовані агенти для комплексного рев'ю PR:

1. **Область послідовності**: дубльована логіка, порушення патернів, відповідність DRY (контекст: повний diff PR)
2. **Область SOLID**: порушення SRP, вкладені умови (>3 рівнів), цикломатична складність >10 (контекст: змінені класи/функції)
3. **Область захисного коду**: "тихі" catch, проковтнуті виключення, приховані фолбеки (контекст: код обробки помилок)

**Ключові патерни** (поза загальним Split Role):

- **Передпольотна перевірка**: `git log --oneline -10 | grep "Co-Authored-By: Claude"` для виявлення попередніх проходів та уникнення повторень
- **Анти-галюцинація**: використовуйте `Grep`/`Glob` для верифікації патернів перед тим, як їх рекомендувати (правило частоти: >10 = встановлено, <3 = не встановлено)
- **Узгодження**: пріоритезуйте існуючі патерни проекту над ідеальними, пропускайте пропозиції з задокументованим обґрунтуванням
- **Класифікація критичності**: 🔴 Must Fix (блокери) / 🟡 Should Fix (покращення) / 🟢 Can Skip (бажано)
- **Цикл збіжності**: рев'ю → виправлення → повторне рев'ю → повтор (макс. 3 ітерації), поки не залишаться тільки опціональні покращення

**Запобіжники в продакшені**:
- Читання повного контексту файлу (не тільки рядків diff)
- Умовне завантаження контексту залежно від змісту diff (запити до БД → перевірка індексів, маршрути API → перевірка middleware автентифікації)
- Список файлів-виключень (package.json, міграції, .env)
- Ворота якості (Quality gates): валідація `tsc && lint` перед кожною ітерацією

### Агенти іменованих перспектив (Named Perspective Agents)

Посібник перелічує "рольові персони експертів" як погану причину для використання агентів (див. §3.x, Коли НЕ використовувати агентів). Агенти іменованих перспектив — це інший патерн, і їх не слід плутати з цим.

**Відмінність**:

| Патерн | Що це таке | Проблема |
|---------|-----------|---------|
| Рольова гра (антипатерн) | "Ти — старший бекенд-розробник з 10 роками досвіду" | Загальна роль, нічого не додає до хорошого промпту |
| Іменована перспектива | "Зроби рев'ю з перспективи DHH" | Кодує специфічний, впізнаваний набір інженерних поглядів |

Агент іменованої перспективи використовує відоме інженерне ім'я як стислий промпт. Назва агента "DHH" об'єднує наступне без зайвих слів: "товсті" моделі, "тонкі" контролери, конвенції REST над конфігурацією, скептицизм щодо передчасної абстракції, прагматизм Rails. Ім'я — це шлях до чіткого стилю думок, а не костюм.
