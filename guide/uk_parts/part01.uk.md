## TL;DR - 5-хвилинне резюме

Якщо у вас є лише 5 хвилин, ось що вам потрібно знати:

### Основні команди
```bash
claude                    # Запустити Claude Code
/help                     # Показати всі команди
/powerup                  # Інтерактивні уроки: CLAUDE.md, /rewind, пам'ять, режими зусиль
/status                   # Перевірити використання контексту
/compact                  # Стиснути контекст, коли >70%
/clear                    # Почати з чистого аркуша
/plan                     # Безпечний режим тільки для читання
Ctrl+C                    # Скасувати операцію
```

### Воркфлоу
```
Опис → Claude аналізує → Огляд Diff → Прийняти/Відхилити → Перевірка
```

### Управління контекстом (Критично!)
| % Контексту | Дія |
|-----------|--------|
| 0-50% | Працюйте вільно |
| 50-70% | Будьте вибірковими |
| 70-90% | `/compact` зараз |
| 90%+ | Необхідно `/clear` |

*Ці пороги базуються на моєму досвіді. Ваш оптимальний воркфлоу може відрізнятися залежно від складності завдання та стилю роботи.*

### Ієрархія пам'яті
```
~/.claude/CLAUDE.md       → Глобальна (всі проекти)
/project/CLAUDE.md        → Проектна (комітиться)
/project/.claude/         → Особиста (не комітиться)
```

### Потужні функції
| Функція | Що вона робить |
|---------|--------------|
| **Agents** | Спеціалізовані AI-персони для конкретних завдань |
| **Skills** | Багаторазові модулі знань |
| **Hooks** | Скрипти автоматизації, що запускаються подіями |
| **MCP Servers** | Зовнішні інструменти (Serena, Context7, Playwright...) |
| **Plugins** | Пакети розширень, створені спільнотою |

### Золоті правила
1. **Завжди переглядайте diff** перед прийняттям змін
2. **Використовуйте `/compact`**, перш ніж контекст стане критичним
3. **Будьте конкретними** у ваших запитах (ЩО, ДЕ, ЯК, ПЕРЕВІРКА)
4. **Починайте з режиму планування (Plan Mode)** для складних/ризикованих завдань
5. **Створюйте CLAUDE.md** для кожного проекту

### Дерево швидких рішень
```
Просте завдання → Просто запитайте Claude
Складне завдання → Використовуйте TodoWrite для планування
Ризикована зміна → Спочатку увійдіть у режим планування
Повторюване завдання → Створіть агента або команду
Контекст заповнений → /compact або /clear
```

**Тепер прочитайте Розділ 1 для повного швидкого старту або перейдіть до будь-якого потрібного розділу.**

### Оновлення Claude Code

Тримайте Claude Code в актуальному стані для отримання нових функцій, виправлень багів та покращень моделей:

```bash
# Перевірка наявності оновлень
claude update

# Альтернатива: Оновлення через npm
npm update -g @anthropic-ai/claude-code

# Перевірка версії
claude --version

# Перевірка стану системи після оновлення
claude doctor
```

**Доступні команди обслуговування:**

| Команда | Призначення | Коли використовувати |
|---------|-------------|----------------------|
| `claude update` | Перевірити та встановити оновлення | Щотижня або при виникненні проблем |
| `claude doctor` | Перевірити здоров'я авто-оновлювача | Після змін у системі або якщо оновлення не вдається |
| `claude --version` | Показати поточну версію | Перед повідомленням про баги |
| `claude auth login` | Автентифікація через командний рядок | CI/CD, devcontainers, скриптовані налаштування |
| `claude auth status` | Перевірити поточний стан автентифікації | Переконатися, який акаунт/метод активний |
| `claude auth logout` | Очистити збережені облікові дані | Спільні машини, очищення безпеки |

**Рекомендації щодо частоти оновлення:**
- **Щотижня**: Перевіряйте оновлення під час звичайної розробки
- **Перед великою роботою**: Переконайтеся, що маєте останні функції та виправлення
- **Після змін у системі**: Запустіть `claude doctor`, щоб перевірити здоров'я
- **При несподіваній поведінці**: Спочатку оновіть, а потім шукайте причину

### Desktop App: Claude Code без терміналу

Claude Code доступний у двох формах: CLI (на якому зосереджений цей посібник) та **вкладка Code** у додатку Claude Desktop. Той самий механізм під капотом, графічний інтерфейс замість терміналу. Доступно на macOS та Windows — інсталяція Node.js не потрібна.

**Що додає десктопний додаток до стандартного Claude Code:**

| Функція | Деталі |
|---------|---------|
| Візуальний огляд diff | Переглядайте зміни у файлах з коментарями перед прийняттям |
| Живий перегляд додатка | Claude запускає ваш dev-сервер, відкриває вбудований браузер, автоматично перевіряє зміни |
| Моніторинг GitHub PR | Авто-виправлення помилок CI, авто-мердж після проходження перевірок |
| Паралельні сесії | Кілька сесій у бічній панелі, кожна з автоматичною ізоляцією Git worktree |
| Конектори | GitHub, Slack, Linear, Notion — GUI налаштування, без ручної конфігурації MCP |
| Вкладення файлів | Прикріплюйте зображення та PDF безпосередньо до промптів |
| Дистанційні сесії | Запускайте довгі завдання в хмарі Anthropic, продовжуйте після закриття додатка |
| SSH сесії | Підключайтеся до віддалених машин, хмарних VM, dev-контейнерів |

**Коли обирати Desktop vs CLI:**

| Використовуйте Desktop, коли... | Використовуйте CLI, коли... |
|--------------------|-----------------|
| Вам потрібен візуальний огляд diff | Вам потрібні скрипти або автоматизація (`--print`, piping) |
| Ви проводите онбординг колег | Ви використовуєте сторонніх провайдерів (Bedrock, Vertex, Foundry) |
| Вам потрібне управління сесіями в бічній панелі | Вам потрібен режим дозволів `dontAsk` |
| Ви робите live-демо або спільний огляд | Вам потрібні команди агентів / мультиагентна координація |
| Вам потрібні вкладення (зображення, PDF) | Ви на Linux (Desktop тільки для macOS + Windows) |

**Чого НЕМАЄ в Desktop** (тільки в CLI): сторонні API провайдери, прапори скриптування (`--print`, `--output-format`), `--allowedTools`/`--disallowedTools`, команди агентів, `--verbose`, Linux.

**Спільна конфігурація**: Desktop та CLI читають ті самі файли — CLAUDE.md, MCP сервери (через `~/.claude.json` или `.mcp.json`), хуки, скіли та налаштування. Ваші налаштування CLI переносяться автоматично.

> **Порада для міграції**: запустіть `/desktop` у терміналі, щоб перенести активну сесію CLI у додаток Desktop. Тільки на macOS та Windows.

> **Примітка щодо MCP серверів**: MCP сервери, налаштовані в `claude_desktop_config.json` (вкладка Chat), відокремлені від Claude Code. Щоб використовувати MCP сервери у вкладці Code, налаштуйте їх у `~/.claude.json` або у вашому проектному `.mcp.json`. Дивіться [Розділ 8.1 — MCP](#81-what-is-mcp).

> **Повна довідка**: [code.claude.com/docs/en/desktop](https://code.claude.com/docs/en/desktop)

#### Shell-команди з `!`

Виконуйте команди миттєво, не просячи Claude зробити це за вас:

```bash
# Швидка перевірка статусу
!git status
!npm run test
!docker ps

# Перегляд логів
!tail -f logs/app.log
!cat package.json

# Швидкий пошук
!grep -r "TODO" src/
!find . -name "*.test.ts"
```

**Коли використовувати `!` vs запитувати Claude**:

| Використовуйте `!` для... | Запитуйте Claude про... |
|----------------|-------------------|
| Швидких перевірок статусу (`!git status`) | Git-операцій, що потребують рішень |
| Команд перегляду (`!cat`, `!ls`) | Аналізу та розуміння файлів |
| Вже відомих команд | Побудови складних команд |
| Швидкої ітерації в терміналі | Команд, у яких ви не впевнені |

**Приклад воркфлоу**:
```
Ви: !git status
Вивід: Показує 5 змінених файлів

Ви: Створи коміт із цими змінами, дотримуючись conventional commits
Claude: [Аналізує файли, пропонує повідомлення коміту]
```

#### Посилання на файли через `@`

Вказуйте конкретні файли у ваших промптах для цільових операцій:

```bash
# Один файл
Перевір @src/auth/login.tsx на предмет проблем із безпекою

# Кілька файлів
Рефактори @src/utils/validation.ts та @src/utils/helpers.ts, щоб прибрати дублювання

# З масками (у деяких контекстах)
Проаналізуй всі файли тестів @src/**/*.test.ts

# Відносні шляхи працюють
Перевір @./CLAUDE.md на відповідність конвенціям проекту
```

**Чому варто використовувати `@`**:
- **Точність**: Вказуйте точні файли замість того, щоб дозволяти Claude шукати
- **Швидкість**: Пропуск фази пошуку файлів
- **Контекст**: Сигналізує Claude прочитати ці файли за запитом через інструменти
- **Ясність**: Робить ваші наміри чіткими

**Приклад**:
```
# Без @
Ви: Виправи баг в автентифікації
Claude: Який файл містить логіку автентифікації? [Витрачає час на пошук]

# З @
Ви: Виправи баг в автентифікації у @src/auth/middleware.ts
Claude: [Читає файл за запитом і пропонує виправлення]
```

#### Робота із зображеннями та скріншотами

Claude Code підтримує **пряме введення зображень** для візуального аналізу, реалізації мокапів та відгуків щодо дизайну.

**Як використовувати зображення**:

1. **Вставити безпосередньо в термінал** (macOS/Linux/Windows із сучасним терміналом):
   - Скопіюйте скріншот або зображення в буфер обміну (`Cmd+Shift+4` на macOS, `Win+Shift+S` на Windows)
   - У сесії Claude Code вставте за допомогою `Cmd+V` / `Ctrl+V`
   - Claude отримує зображення і може його аналізувати

2. **Drag and drop** (деякі термінали):
   - Перетягніть файл зображення у вікно терміналу
   - Claude завантажує та обробляє зображення

3. **Посилання через шлях**:
   ```bash
   Проаналізуй цей мокап: /path/to/design.png
   ```

**Типові кейси**:

```bash
# Реалізація UI за мокапом

Реалізуй цей екран логіну на React з Tailwind CSS

# Відлагодження візуальних проблем
Ви: [Вставка скріншота зламаної верстки]
Кнопка зміщена. Виправ CSS.

# Аналіз діаграм
Ви: [Вставка архітектурної діаграми]
Поясни архітектуру цієї системи та ідентифікуй потенційні вузькі місця

# Код з маркерної дошки
Ви: [Вставка фото алгоритму з дошки]
Перетвори цей алгоритм на код Python

# Аудит доступності
Ви: [Вставка скріншота UI]
Перевір цей інтерфейс на відповідність вимогам WCAG 2.1
```

**Підтримувані формати**: PNG, JPG, JPEG, WebP, GIF (статичні)

**Кращі практики**:
- **Високий контраст**: Переконайтеся, що текст/діаграми чітко видно
- **Обрізайте зайве**: Видаляйте непотрібні елементи UI для фокусування аналізу
- **Анотуйте при необхідності**: Обводьте/підсвічуйте конкретні зони, на яких Claude має зосередитися
- **Комбінуйте з текстом**: "Зосередься на секції хедера" дає додатковий контекст

**Приклад воркфлоу**:
```
Ви: [Вставка скріншота повідомлення про помилку з консолі браузера]
Ця помилка з'являється, коли користувачі натискають кнопку відправки. Відлагоди її.

```

**Обмеження**:
- Зображення споживають значну кількість токенів контексту (еквівалентно ~1000-2000 словам тексту)
- Використовуйте `/status`, щоб стежити за використанням контексту після вставки зображень

- Деякі термінали можуть не підтримувати вставку зображень з буфера обміну (альтернатива: збережіть і вкажіть шлях до файлу)

> **💡 Порада**: Робіть скріншоти повідомлень про помилки, мокапів дизайну та документації замість того, щоб описувати їх текстом. Візуальне введення часто швидше та точніше за письмові описи.

##### Інтеграція Figma MCP

Figma надає **офіційний MCP сервер** (анонсовано у 2025), який дає Claude прямий доступ до ваших файлів дизайну, радикально зменшуючи споживання токенів порівняно зі скріншотами.

**Варіанти налаштування**:

```bash
# Віддалений MCP (всі плани Figma, будь-яка машина)
claude mcp add --transport http figma https://mcp.figma.com/mcp

# Десктопний MCP (потребує додатка Figma Desktop з Dev Mode)
claude mcp add --transport http figma-desktop http://127.0.0.1:3845/mcp
```

**Доступні інструменти через Figma MCP**:

| Інструмент | Призначення | Токени |
|------|---------|--------|
| `get_design_context` | Витягує структуру React+Tailwind з фреймів | Низько |
| `get_variable_defs` | Отримує дизайн-токени (кольори, відступи, типографіка) | Дуже низько |
| `get_code_connect_map` | Пов’язує компоненти Figma з вашою кодобазою | Низько |
| `get_screenshot` | Робить візуальний скріншот фрейму | Високо |
| `get_metadata` | Повертає властивості вузлів, ID, позиції | Дуже низько |

**Чому варто використовувати Figma MCP замість скріншотів?**
- **У 3-10 разів менше токенів**: Структуровані дані замість аналізу зображень
- **Прямий доступ до токенів**: Кольори та відступи витягуються, а не інтерпретуються
- **Мапінг компонентів**: Code Connect пов'язує Figma з реальними файлами коду
- **Ітеративний воркфлоу**: Невеликі зміни не потребують нових скріншотів

**Рекомендований воркфлоу**:
```
1. get_metadata          → Розуміння загальної структури
2. get_design_context    → Отримання ієрархії компонентів для конкретних фреймів
3. get_variable_defs     → Витяг дизайн-токенів один раз на проект
4. get_screenshot        → Тільки коли потрібне візуальне підтвердження
```

**Приклад сесії**:
```bash
Ви: Реалізуй хедер дашборду з Figma
Claude: [Викликає get_design_context для фрейму хедера]
→ Повертає: структуру React з класами Tailwind, точні відступи
Claude: [Викликає get_variable_defs]
→ Повертає: --color-primary: #3B82F6, --spacing-md: 16px
Claude: [Реалізує компонент, що точно відповідає Figma]
```

**Передумови**:
- Акаунт Figma (Free tier працює для remote MCP)

##### Оптимізація зображень для Claude Vision

Розуміння того, як Claude обробляє зображення, допомагає оптимізувати швидкість та точність.

**Рекомендації щодо роздільної здатності**:

| Діапазон | Ефект |
|-------|--------|
| **< 200px** | Втрата точності, текст нечитабельний |
| **200-1000px** | Оптимально для більшості вайрфреймів |
| **1000-1568px** | Найкращий баланс якості та токенів |
| **1568-8000px** | Автоматичне зменшення (марна трата часу на завантаження) |
| **> 8000px** | Відхиляється API |

**Розрахунок токенів**: `(ширина × висота) / 750 ≈ спожиті токени`

| Розмір зображення | Приблизна кількість токенів |
|------------|-------------------|
| 200×200 | ~54 токени |
| 500×500 | ~334 токени |
| 1000×1000 | ~1,334 токени |
| 1568×1568 | ~3,279 токенів |

**Рекомендації щодо форматів**:

| Формат | Коли використовувати |
|--------|----------|
| **PNG** | Вайрфрейми, діаграми, текст, чіткі лінії |
| **WebP** | Загальні скріншоти, гарне стиснення |
| **JPEG** | Тільки фото — артефакти стиснення шкодять розпізнаванню ліній |
| **GIF** | Уникайте (тільки статика, низька якість) |

**Чек-лист оптимізації**:
- [ ] Обрізайте лише релевантну область
- [ ] Зменшуйте до 1000-1200px, якщо зображення більше
- [ ] Використовуйте PNG для вайрфреймів/діаграм
- [ ] Перевіряйте `/status` після вставки
- [ ] Розгляньте текст, якщо контекст >70%

> **💡 Порада щодо токенів**: Вайрфрейм 1000×1000 споживає ~1,334 токени. Та сама інформація у вигляді структурованого тексту (через Figma MCP) може зайняти 200-400 токенів. Використовуйте скріншоти для візуального контексту, а структуровані дані — для реалізації.

#### Продовження та відновлення сесій

Claude Code дозволяє вам **продовжувати попередні розмови** між термінальними сесіями, зберігаючи повний контекст та історію.

**Два способи відновлення**:

1. **Продовжити останню сесію** (`--continue` або `-c`):
   ```bash
   # Автоматично відновлює вашу останню розмову
   claude --continue
   # Коротка форма
   claude -c
   ```

2. **Відновити конкретну сесію** (`--resume <id>` або `-r <id>`):
   ```bash
   # Відновити сесію за конкретним ID
   claude --resume abc123def
   # Коротка форма
   claude -r abc123def
   ```

3. **Прив'язка до GitHub PR** (`--from-pr <number>`, v2.1.49+):
   ```bash
   # Почати сесію, прив'язану до конкретного PR
   claude --from-pr 123

   # Сесії, створені через gh pr create під час роботи в Claude,
   # автоматично прив'язуються до цього PR — використовуйте --from-pr для відновлення
   gh pr create --title "Add auth" --body "..."
   # Пізніше:
   claude --from-pr 123  # Відновлює контекст сесії для цього PR
   ```

   Корисно для продовження роботи над фічею саме там, де ви зупинилися відносно конкретного PR — не потрібно запам'ятовувати ID сесій.

**Пошук ID сесій**:

```bash
# Вбудовано: інтерактивний вибір сесії
claude --resume

# Вбудовано: список через Serena MCP (якщо налаштовано)
claude mcp call serena list_sessions

# Рекомендовано: швидкий пошук готовими командами
# Див. examples/scripts/session-search.sh (bash, нуль залежностей, список за 15мс)
# Див. examples/scripts/cc-sessions.py (Python, інкрементальний індекс, фільтр гілок)
cs                    # Список 10 останніх сесій
cs "authentication"   # Повнотекстовий пошук по всіх сесіях

# Сесії також відображаються при виході
Ви: /exit
Session ID: abc123def (збережено для відновлення)
```

> **Інструменти пошуку сесій**: Для швидкого пошуку див. [session-search.sh](../examples/scripts/session-search.sh) (легкий bash) та [cc-sessions.py](../examples/scripts/cc-sessions.py) (Python, просунуті функції: пошук, фільтр гілок та `discover` для автоматичного аналізу патернів — [GitHub](https://github.com/FlorianBruniaux/cc-sessions)). Також: [Observability Guide](./ops/observability.md#session-search--resume).

**Типові кейси**:

| Сценарій | Команда | Навіщо |
|----------|---------|-----|
| Робота перервана | `claude -c` | Повернутися рівно туди, де зупинилися |
| Фіча на кілька днів | `claude -r abc123` | Продовжити складне завдання через кілька днів |
| Після перерви/мітингу | `claude -c` | Відновити без втрати контексту |
| Паралельні проекти | `claude -r <id>` | Перемикання між контекстами різних проектів |
| Доопрацювання після рев'ю | `claude -r <id>` | Виправлення зауважень в оригінальному контексті |

**Приклад воркфлоу**:

```bash
# День 1: Початок реалізації автентифікації
cd ~/project
claude
Ви: Реалізуй JWT автентифікацію з refresh-токенами
Claude: [Аналіз та початкова реалізація]
Ви: /exit
Session ID: auth-feature-xyz (використано 27% контексту)

# День 2: Продовження роботи
cd ~/project
claude --continue
Claude: Відновлюю сесію auth-feature-xyz...
Ви: Додай rate limiting для ендпоінтів автентифікації
Claude: [Продовжує з повним контекстом роботи першого дня]
```

**Кращі практики**:

- **Правильний вихід**: Завжди виходьте через `/exit` або `Ctrl+D` (не вбивайте процес), щоб сесія збереглася.
- **Описові фінальні повідомлення**: Завершуйте сесії контекстом ("Готово до тестування"), щоб згадати стан при відновленні.
- **Проактивне управління контекстом**: Стежте через `/status` та використовуйте перевірені пороги:
  - **< 70%**: Оптимально — повна потужність міркувань.
  - **75%**: Гарний час для ручного `/compact` — до того, як якість почне падати.
  - **85%**: Зона авто-стиснення — Claude Code автоматично стисне історію, коли залишиться ~6-7% вікна. Рекомендовано передати справу вручну до цього моменту ([research-backed](../core/architecture.md#auto-compaction)).
  - **95%**: Критична межа — сильна деградація якості, негайно скидайте сесію.
- **Найменування сесій**: Використовуйте `/rename` для надання описових назв — критично при паралельній роботі (див. [Патерн Auto-Rename](#session-auto-rename) нижче).

**Відновлення vs Новий старт**:

| Відновлюйте, коли... | Починайте заново, коли... |
|-------------------|---------------------|
| Продовжуєте конкретну фічу/завдання | Переходите до непов'язаної роботи |
| Будуєте на основі попередніх рішень | Попередня сесія пішла не туди |
| Контекст ще релевантний (< 75%) | Контекст роздутий (> 85%) |
| У процесі багатоступенева реалізація | Швидкі поодинокі питання |

**Обмеження**:

- Сесії зберігаються локально (не синхронізуються між машинами)
- Дуже старі сесії можуть видалятися (залежить від лімітів сховища)
- Пошкоджені сесії неможливо відновити (починайте заново з `/clear`)
- Неможливо відновити сесію, розпочату з іншою моделлю або конфігурацією MCP

**Збереження контексту**:

При відновленні Claude зберігає:
- ✅ Повну історію розмови
- ✅ Файли, що раніше читалися/редагувалися
- ✅ CLAUDE.md та налаштування проекту
- ✅ Стан MCP серверів (якщо використовується Serena)
- ✅ Знання про незакомічені зміни в коді

**Поєднання з MCP Serena**:

Для просунутого управління сесіями з пам’яттю проекту та відстеженням символів:

```bash
# Ініціалізація пам'яті Serena для проекту
claude mcp call serena initialize_session

# Робота з повною персистентністю
Ви: Реалізуй автентифікацію користувачів
Claude: [Працює, Serena відстежує символи та контекст]

# Вихід та повернення пізніше з повною пам'яттю
claude -c
Claude: [Відновлює з персистентним розумінням проекту від Serena]
```

> **💡 Порада**: Використовуйте `claude -c` як стандартний спосіб запуску Claude Code в активних проектах. Це гарантує, що ви не втратите контекст, поки самі не захочете почати з чистого аркуша через `claude` (без прапорів).

> **Джерело**: [DeepTo Claude Code Guide - Context Resume Functions](https://cc.deeptoai.com/docs/en/best-practices/claude-code-comprehensive-guide)

### Виявлення патернів сесій (cc-sessions discover) {#session-pattern-discovery}

Ваша історія сесій — це джерело даних. Кожного разу, коли ви просите Claude зробити щось схоже у різних сесіях, це сигнал: винесіть це у скіл, команду або правило CLAUDE.md і перестаньте платити "податок на контекст" при кожному запиті.

`cc-sessions discover` автоматизує цей аналіз. Він читає вашу історію сесій, знаходить повторювані патерни в повідомленнях користувача та каже вам, що варто винести.

**Встановлення**:

```bash
curl -sL https://raw.githubusercontent.com/FlorianBruniaux/cc-sessions/main/cc-sessions \
  -o ~/.local/bin/cc-sessions && chmod +x ~/.local/bin/cc-sessions
```

**Два режими**:

| Режим | Як працює | Вартість | Швидкість |
|------|-----|------|-------|
| N-gram (default) | Токенізує повідомлення, будує індекс частоти фраз (3-6 слів) | Безкоштовно, локально | ~3с для 12 проектів |
| `--llm` | Дедуплікує повідомлення, відправляє пачку в `claude --print` | Використовує вашу підписку | ~15с |

```bash
# Режим N-gram: всі проекти, останні 90 днів
cc-sessions --all discover

# Нижчий поріг, вужче вікно
cc-sessions --all discover --since 60d --min-count 2 --top 15

# Семантичний аналіз через claude --print
cc-sessions --all discover --llm

# Вивід у JSON для скриптів
cc-sessions --all discover --json | jq '.[] | select(.category == "skill")'
```

**Приклад виводу**:

```

  📋  ПРАВИЛО CLAUDE.md
  ────────────────────────────────────────────────────────────
  пиши тести перед реалізацією
    234 сесії (28%) · 891 входження · score 0.416
    → 3a72f1c4-...

  🧩  СКІЛ
  ────────────────────────────────────────────────────────────
  security review потоку автентифікації
    71 сесія (8%) · 203 входження · score 0.084
    → 9f1c3a22-...

  ⚡  КОМАНДА
  ────────────────────────────────────────────────────────────
  generate prisma migration rollback script
    18 сесій (2%) · 44 входження · score 0.021
    → 44aab71c-...
```

**Правило 20%, вбудоване в скоринг**: патерни вище 20% сесій стають пропозиціями для `правил CLAUDE.md` (завантажуються завжди), 5-20% стають `скілами` (завантажуються за запитом), нижче 5% — `командами` (явний виклик). Бонус за крос-проектність (1.5×) пріоритезує патерни, що повторюються у різних кодобазах — їх варто виносити навіть при нижчій частоті.

Дивіться також: [§5.1 Розуміння Скілів](#51-understanding-skills) для розрізнення між правилами CLAUDE.md, скілами та командами, та [правило 20%](#the-20-rule) для фреймворку прийняття рішень.

### Авто-перейменування сесій

При роботі в кількох паралельних сесіях Claude Code (розділені термінали, вкладки WebStorm, паралельні потоки завдань) вибір через `/resume` показує сесії за часом або обрізаним першим промптом — неможливо розрізнити їх з першого погляду.

Два взаємодоповнюючі підходи вирішують це. Використовуйте один або обидва разом.

#### Підхід А: Інструкція поведінки у CLAUDE.md (під час сесії)

Інструкція в `~/.claude/CLAUDE.md` змушує Claude викликати `/rename` автоматично після 2-3 обмінів повідомленнями. Не потребує додаткових інструментів, працює в усіх IDE та терміналах.

```markdown
# Session Naming (auto-rename)

## Очікувана поведінка

1. **Раннє перейменування**: Як тільки основна тема сесії стане зрозумілою (після 2-3 повідомлень),
   виконай `/rename` з короткою, описовою назвою (макс 50 символів)
2. **Оновлення в кінці сесії**: Якщо фокус значно змістився, запропонуй нову назву перед закриттям

## Формат назви

`[дія] [об'єкт]` — приклади:
- "fix whitepaper PDF build"
- "add auth middleware + tests"
- "refactor hook system"
- "update CC releases v2.2.0"

## Правила

- Макс 50 символів, без префікса "Session:", без дати
- Дієслово першим (fix, add, refactor, update, research, debug...)
- Багатотемність: тільки домінуючий предмет, не вичерпний список
- НЕ запитуй підтвердження для раннього перейменування (просто роби це)
```

Це добре працює під час активних сесій, але залежить від того, чи послідовно Claude дотримується інструкції.

#### Підхід Б: Хук SessionEnd (автоматичний, на базі AI)

Хук `SessionEnd` читає файл JSONL сесії безпосередньо з `~/.claude/projects/`, витягує перші кілька повідомлень користувача як контекст і викликає `claude -p --model claude-haiku-4-5-20251001` для генерації описової назви з 4-6 слів. Якщо Haiku недоступна, використовується очищена версія першого повідомлення.

Хук оновлює як `sessions-index.jsonl` (для кастомних браузерів сесій), так і поле slug у файлі JSONL (для нативної сумісності з `/resume`).

```json
// .claude/settings.json
{
  "hooks": {
    "SessionEnd": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "~/.claude/hooks/auto-rename-session.sh"
          }
        ]
      }
    ]
  }
}
```

Вимоги: `claude` CLI у PATH, `python3` для парсингу JSON. Встановіть `SESSION_AUTORENAME=0`, щоб вимкнути для конкретної сесії.

Після завершення сесії список `/resume` покаже `"fix auth middleware"` замість `"2026-03-04T14:23..."`.

#### Використання обох підходів разом

Вони обробляють різні моменти життєвого циклу сесії. Підхід А перейменовує рано, щоб сесію можна було ідентифікувати, поки вона ще запущена. Підхід Б перейменовує в кінці з назвою, що відображає повний обсяг сесії, потенційно перезаписуючи проміжну назву більш точною.

**Обмеження (обох підходів)**: Назви вкладок терміналу у WebStorm та iTerm2 не змінюються. JetBrains фільтрує ANSI-послідовності. Перейменовується сесія Claude, а не вкладка ОС.

> Див. повний шаблон: [examples/claude-md/session-naming.md](../examples/claude-md/session-naming.md)
> Див. шаблон хука: [examples/hooks/bash/auto-rename-session.sh](../examples/hooks/bash/auto-rename-session.sh)

## 1.4 Режими дозволів

Claude Code має п’ять режимів дозволів, які контролюють ступінь автономності Claude:

### Default Mode (За замовчуванням)

Claude запитує дозвіл перед:
- Редагуванням файлів
- Виконанням команд
- Створенням комітів

Це найбезпечніший режим для навчання.

### Auto-accept Mode (`acceptEdits`)

```
Ви: Увімкни auto-accept до кінця цієї сесії
```

Claude автоматично підтверджує редагування файлів, але все ще запитує про shell-команди. Використовуйте, коли довіряєте правкам і хочете швидкості.

⚠️ **Попередження**: Використовуйте auto-accept лише для чітко визначених, зворотних операцій.

### Plan Mode

```
/plan
```

Claude може лише читати та аналізувати, модифікації заборонені. Ідеально для:
- Розуміння незнайомого коду
- Дослідження архітектурних варіантів
- Безпечного вивчення перед змінами

Вийдіть за допомогою `/execute`, коли будете готові вносити зміни.

### Don't Ask Mode (`dontAsk`)

Автоматично відхиляє інструменти, якщо вони не схвалені заздалегідь через `/permissions` або правила `permissions.allow`. Claude ніколи не перериває роботу запитами на дозвіл: якщо інструмент не дозволений явно, він мовчки відхиляється.

Використовуйте для суворих воркфлоу, де ви хочете повного контролю над тим, які інструменти запускаються, без інтерактивного підтвердження.

### Auto Mode (Для підписників Max, v2.1.114+)

Режим Auto дозволяє Claude приймати рішення щодо дозволів від вашого імені під час тривалих завдань. Замість того, щоб зупинятися кожного разу, коли ризикована дія потребує схвалення, Claude застосовує власне судження — а ви переглядаєте результат, а не схвалюєте кожен крок.

```
# Увімкнення через settings.json
{ "permissionMode": "auto" }
```

На відміну від `bypassPermissions` (який схвалює наосліп), режим Auto використовує класифікатор для оцінки кожної дії. Хуки `PermissionDenied` спрацьовують, коли класифікатор щось блокує, даючи вам знати, що було відхилено. Призначено для довгих завдань з меншою кількістю переривань та меншим ризиком, ніж пропуск усіх дозволів.

**Вимоги**: Підписка на план Max. Доступно з версії v2.1.114.

### Bypass Permissions Mode (`bypassPermissions`)

Автоматично схвалює все, включаючи shell-команди. Запити на дозвіл відсутні взагалі.

⚠️ **Попередження**: Використовуйте лише в ізольованих CI/CD середовищах. Потребує `--dangerously-skip-permissions` для увімкнення з CLI. Ніколи не використовуйте на робочих системах або з недовіреним кодом.

**Інваріант безпеки — деякі шляхи ЗАВЖДИ запитують дозвіл, навіть у режимі `bypassPermissions`**:

Певні операції запису вважаються занадто чутливими для автоматичного схвалення за будь-якої конфігурації. Claude Code завжди запитує підтвердження перед зміною:

| Захищена ціль | Приклади |
|-----------------|---------|
| Директорія `.git/` | git hooks, refs, config всередині репозиторію |
| Директорія `.claude/` | агенти, скіли, хуки, налаштування — крім `.claude/worktrees/` |
| Файли конфігурації Shell | `.bashrc`, `.zshrc`, `.bash_profile`, `.profile` |
