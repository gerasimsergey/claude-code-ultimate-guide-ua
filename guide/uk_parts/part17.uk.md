**Ключові відмінності від Kairn/doobidoo**:
- **Єдиний Rust-бінарник**: ніякого Python, pip або віртуальних оточень — `brew install icm` і готово.
- **Подвійна архітектура в одному інструменті**: Memories (епізодична пам'ять із розпадом) + Memoirs (постійний типізований граф) — Kairn покриває шар графа, doobidoo — семантичний шар, ICM — обидва.
- **Автовилучення**: тришарова автоматична фіксація (хуки патернів, перед стисненням, початок сесії) без явних викликів `store_memory`.
- **Автодедуплікація**: блокує записи, схожість яких із наявним контентом перевищує 85%.

| Функція | doobidoo | Kairn | ICM |

| Мова | Python | Python | Rust (єдиний бінарник) |
| Встановлення | pip | pip | Homebrew / curl |

### Стек пам'яті MCP: Патерни взаємодоповнюваності

> **⚠️ Експериментально** - Ці патерни поєднують кілька MCP-серверів. Протестуйте у своєму воркфлоу перед використанням.

**4-шаровий стек знань**:

```
┌─────────────────────────────────────────────────────┐
│                    ШАР ЗНАНЬ                        │
├─────────────────────────────────────────────────────┤
│  doobidoo     │ Рішення, ADR, бізнес-контекст       │
│  (семантика)  │ "Чому ми це зробили?"               │
├───────────────┼─────────────────────────────────────┤
│  Serena       │ Символи, структура, пам'ять K-V     │
│  (індекс коду)│ "Де визначено X?"                   │
├───────────────┼─────────────────────────────────────┤
│  grepai       │ Семантичний пошук + граф викликів   │
│  (пошук коду) │ "Знайди код, що робить X"           │
├───────────────┼─────────────────────────────────────┤
│  Context7     │ Офіційна документація бібліотек     │
│  (доки)       │ "Як використовувати бібліотеку X?"  │
└─────────────────────────────────────────────────────┘
```

**Матриця порівняння**:

| Можливість | Serena | grepai | doobidoo | Kairn | ICM |
|------------|--------|--------|----------|-------|-----|
| Пам'ять між сесіями | Key-value | Ні | Семантична | Граф знань | Епізодична + граф |
| Пам'ять між IDE | Ні | Ні | Так | Так | Так (14 клієнтів) |
| Синхронізація пристроїв | Ні | Ні | Так (Cloudflare) | Ні | Ні |
| Граф знань | Ні | Граф викликів | Граф рішень | Типізовані зв'язки | Типізовані зв'язки |
| Нечіткий пошук | Ні | Код | Пам'ять | Текст + семантика | Гібрид BM25 + вектор |
| Теги/категорії | Ні | Ні | Так | Так | Так (теми) |
| Розпад пам'яті | Ні | Ні | Ні | Так (біологічний) | Так (налаштовуваний) |
| Автовилучення | Ні | Ні | Ні | Ні | Так (3 шари) |
| Runtime | — | — | Python | Python | Rust (бінарник) |

**Приклади воркфлоу**:

```bash
# Сценарій 1: Розуміння фічі
retrieve_memory("який статус модуля оплати?")   # doobidoo → бізнес-контекст
grepai search "payment processing"               # grepai → пошук коду
find_symbol("PaymentProcessor")                  # Serena → точне місце

# Сценарій 2: Онбординг (Сесія 1 → Сесія N)
# Сесія 1 (старший розробник)
store_memory("Архітектура: гексагональна з портами/адаптерами", tags=["onboarding"])
store_memory("DANGER: ніколи не чіпайте legacy/payment.ts без рев'ю", tags=["onboarding", "danger"])

# Сесія N (новий розробник)
retrieve_memory("архітектура проекту?")
retrieve_memory("небезпечні зони?")

# Сценарій 3: ADR (Записи архітектурних рішень)
store_memory("""
ADR-001: FastAPI vs Flask
- Рішення: FastAPI
- Причина: нативна асинхронність, авто OpenAPI
""", tags=["adr", "api"])

# Сценарій 4: Спільна робота в різних IDE
# В Claude Code (термінал)
store_memory("Рефакторю AuthService, не чіпайте його", tags=["wip"])
# В Cursor (інше вікно)
retrieve_memory("що зараз у роботі?") → бачить попередження
```

**Коли який інструмент обрати**:
- **Serena `read_memory()`**: "Я знаю точний ключ". Швидкий пошук.
- **doobidoo `retrieve_memory()`**: "Я пам'ятаю тему, а не ключ". Семантичний пошук.
- **doobidoo**: "Треба спільна пам'ять для різних IDE або пристроїв".
- **Serena `find_symbol()`**: "Де цей символ у коді?". Індексація коду.
- **grepai `search()`**: "Знайди код за наміром". Семантичний пошук по коду.
- **Kairn**: "Довготривала пам'ять проекту з автоочищенням застарілих даних".

### Git MCP Server (Офіційний від Anthropic)

**Призначення**: Програмний доступ до Git через 12 структурованих інструментів для комітів, дифів, логів та управління гілками.

**Чому Git MCP краще за Bash `git`**: Інструмент Bash повертає сирий вивід термінала, який потребує парсингу та споживає більше токенів. Git MCP повертає структуровані дані, які Claude може використовувати безпосередньо, з вбудованими фільтрами (дата, автор, гілка) та економними дифами через параметр `context_lines`.

> **⚠️ Статус**: Рання розробка — API може змінюватися. Підходить для локальних воркфлоу.

**Інструменти (12)**: `git_status`, `git_diff_unstaged`, `git_diff_staged`, `git_diff`, `git_commit`, `git_add`, `git_reset`, `git_log`, `git_create_branch`, `git_checkout`, `git_show`, `git_branch`.

**Налаштування**:
```bash
# Встановлення не потрібне — uvx завантажить його при першому запуску
uvx mcp-server-git --repository /path/to/repo
```

**Конфігурація Claude Code** (`~/.claude.json`):
```json
{
  "mcpServers": {
    "git": {
      "command": "uvx",
      "args": ["mcp-server-git", "--repository", "/absolute/path/to/repo"]
    }
  }
}
```

### GitHub MCP Server (Офіційний від GitHub)

**Призначення**: Повний доступ до платформи GitHub — Issues, Pull Requests, Projects, пошук по коду та GitHub Enterprise.

**Git MCP vs GitHub MCP**:
- **Git MCP**: Локальні операції (коміти, дифи, гілки).
- **GitHub MCP**: Хмарна платформа (Issues, PR, рев'ю, проекти).

Вони доповнюють один одного: Git MCP для локальної роботи, GitHub MCP для колаборації.

**Два режими налаштування**:
1. **Remote** (`api.githubcopilot.com`): Потрібна підписка на GitHub Copilot.
2. **Self-hosted**: Потрібен лише GitHub PAT (Personal Access Token). Ніякого Copilot не треба.

**Налаштування Remote (Copilot)**:
> **⚠️ Відома проблема**: `claude mcp add` намагається зареєструватися через OAuth, що не підтримується Copilot. Треба додати токен вручну.

Крок 1 — Додайте сервер:
```bash
claude mcp add --transport http github https://api.githubcopilot.com/mcp/
```

Крок 2 — Отримайте токен: `gh auth token`.

Крок 3 — Відредагуйте `~/.claude.json`, додавши заголовок `Authorization`:
```json
"github": {
  "type": "http",
  "url": "https://api.githubcopilot.com/mcp/",
  "headers": {
    "Authorization": "Bearer gho_xxxxxxxxxxxx"
  }
}
```

**Воркфлоу**: "Виведи всі відкриті PR, призначені мені", "Підготуй чернетку коментаря для PR #456", "Знайди використання застарілої функції по всій організації".

## 8.3 Конфігурація

### Місцезнаходження
```
~/.claude.json          # Глобальна конфігурація (поле "mcpServers")
.mcp.json               # Конфігурація проекту (комітиться в Git)
```

### Динамічні заголовки (v2.1.85+)
Якщо один скрипт `headersHelper` обслуговує кілька серверів, ви можете повертати різні токени залежно від `CLAUDE_CODE_MCP_SERVER_NAME`.

### Управління великими наборами серверів
Велика кількість увімкнених серверів погіршує точність вибору інструментів.
**Патерн**: мінімальний глобальний конфіг (2-3 сервери) + специфічні сервери в `.mcp.json` проекту.

#### MCP Tool Search — ліниве завантаження
У версії v4 впроваджено **MCP Tool Search**: схеми інструментів завантажуються лише тоді, коли вони потрібні. Це економить до **85% токенів** на старті та підвищує точність вибору інструментів на 25 пунктів.

### 8.3.1 Управління секретами MCP

**Проблема**: Зберігання API-ключів у відкритому тексті в `mcp.json` — це ризик безпеки.
**Рішення**: Використовуйте системні зв'язки ключів (OS Keychain), змінні оточення або секретні сховища.

#### Підхід 1: Системна зв'язка ключів (Рекомендовано)
Найкращий вибір для розробників на macOS/Linux. Секрети зашифровані на диску.

**Налаштування macOS**:
```bash
# Зберегти секрет
security add-generic-password -a "claude-mcp" -s "github-token" -w "ghp_your_token"
```

**Використання в MCP**:
```json
"github": {
  "command": "bash",
  "args": ["-c", "GITHUB_TOKEN=$(security find-generic-password -s 'github-token' -w) npx @github/mcp-server"]
}
```

**Linux (libsecret)**: використовуйте `secret-tool store` та `secret-tool lookup`.
**Windows**: використовуйте `cmdkey` для управління обліковими даними.
