### Фонові завдання для Fullstack розробки

**Проблема**: Fullstack розробка часто потребує тривалих процесів (dev-сервери, ворчери), які блокують основну сесію Claude.

**Рішення**: Використовуйте **`Ctrl+B`**, щоб перевести завдання у фоновий режим і підтримувати швидкі цикли фідбеку.

#### Коли використовувати фоновий режим

| Сценарій | Команда для фону | Навіщо |
|----------|-------------------|-----|
| **Dev сервер** | `pnpm dev` → `Ctrl+B` | Тримає сервер живим під час правок фронтенду |
| **Test watcher** | `pnpm test --watch` → `Ctrl+B` | Моніторинг тестів під час написання коду |
| **Build watcher** | `pnpm build --watch` → `Ctrl+B` | Виявлення помилок збірки без блокування сесії |
| **Міграція БД** | `pnpm migrate` → `Ctrl+B` | Тривала міграція, поки ви працюєте над іншим |

**Порада**: Перевіряйте статус фонових завдань командою **/tasks**.

#### Обмеження фонових завдань
- **Немає команди foreground**: неможливо повернути завдання на передній план (поки що).
- **Втрата контексту**: тривалі завдання можуть втратити актуальність.
- **Відсутність стрімінгу виводу**: вивід не видно, поки ви не перевірите `/tasks`.
- **Залежність від сесії**: завдання вмирають при виході з Claude.

### Claude in Chrome: Візуальний цикл зворотного зв'язку

Жоден текстовий тест не скаже Claude, чи виглядає UI правильно. Без браузера Claude лише припускає результат.
**Claude in Chrome** закриває цю прірву. Це розширення Chrome, яке дає Claude Code прямий контроль над вашим браузером: навігація, кліки, читання консолі, скріншоти та валідація візуального результату.

**Активація**:
```bash
claude --chrome          # запуск із підтримкою Chrome
/chrome                  # перевірка статусу з'єднання
```

> **Цитата Бориса Черного (творця Claude Code)**: "Якщо Claude не бачить результату, він не може його покращити."

## 9.6 Todo як дзеркало інструкцій

TodoWrite — це не просто трекінг, це механізм інструкцій.
**Принцип дзеркала**: те, що ви пишете в todo, стає вказівкою для Claude.
- ❌ **Розмите Todo**: "Виправити баг" → Розмите виконання.
- ✅ **Конкретне Todo**: "Виправити null pointer у getUserById, якщо користувача не знайдено — повертати null замість помилки" → Точне виконання.

## 9.7 Стилі виводу (Output Styles)

Ви можете керувати тим, як Claude відповідає, через `/config` → "Preferred output style" або `settings.json`.

| Стиль | Дія Claude | Для кого |
|-------|-----------------|----------|
| **Default** | Виконує завдання лаконічно | Досвідчені девелопери, швидкість |
| **Explanatory** | Додає блоки "Insights", пояснює дизайн-рішення | Архітектурне рев'ю, онбординг |
| **Learning** | Зупиняється на кроках, додає `TODO(human)` | Junior-розробники, парне програмування |

**Кастомні стилі**: з грудня 2025 ви можете створювати власні стилі у `.claude/styles/` (файли Markdown).

## 9.8 Vibe Coding та скелетні проекти

**Vibe Coding** — це швидке прототипування через природну розмову, дослідження ідей перед реалізацією.
**Антипатерн: Перевантаження контексту**: dumping гігантських файлів у надії, що Claude розбереться.
**Рішення: Фазова стратегія контексту**:
1. **Дослідження**: режим `/plan` (тільки читання, написання плану).
2. **Реалізація**: звичайний режим (виконання плану).
3. **Новий старт**: передача естафети (Session handoff), коли контекст >75%.

#### Боротьба з деградацією коду ("Slop")
Швидкий ШІ-кодинг накопичує "шлак" (slop) — застарілі абстракції, сміттєвий код.
**Desloppify** — інструмент спільноти (лютий 2026), що встановлюється як навичка Claude Code для циклу: scan → find issue → fix → repeat.

#### Скелетні проекти (Skeleton Projects)
Це мінімальні робочі шаблони, що задають патерни до повної реалізації.
**Принцип**: він повинен запускатися (`pnpm dev`) з першого дня і містити хоча б один повний вертикальний зріз (full stack).

## 9.10 Мислення безперервного вдосконалення

Мета — не просто кодити з ШІ, а **покращувати воркфлоу**.
Після кожного ручного втручання запитуйте: *"Як покращити процес, щоб уникнути цієї ручної правки наступного разу?"*
1. Чи може це зловити лінтер? (Додайте правило ESLint).
2. Чи можна це додати в конвенції? (Додайте в **CLAUDE.md**).
3. Чи потрібен новий приклад імпорту? (Оновіть документацію).

> **Мислення ШІ-нативної розробки**: "Я покращую воркфлоу та контекст, щоб ШІ писав кращий код."
