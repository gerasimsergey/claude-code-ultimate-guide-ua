**✅ Видаляйте шум, зберігайте цінність**:
Не пишіть коментарі типу "Імпорт React" або "Поле імені". Це марна трата токенів.
- **❌ Погано**: `// Повертає користувача за ID`.
- **✅ Добре**: `// Повертає користувача. Поле age потрібне для майбутньої перевірки віку (див. ADR-012)`.
**Економія**: Скорочення токенів на ~47% без втрати критичної інформації.

**Прапорці Verbose для налагодження**:
Використовуйте змінну `DEBUG=true`, щоб умовно включати детальний вивід. Claude зможе бачити трасування виконання лише тоді, коли це потрібно для вирішення багів, не засмічуючи контекст у звичайному режимі.

### 9.18.6 Тестування для автономності

**Проблема**: Агенти слідують тестам надійніше, ніж документації.
**Рішення**: Тести — це і є специфікація. Людина має писати їх вручну, а не перекладати це на ШІ.

**Чому TDD критичний для агентів?**
- **Люди**: можуть додумати контекст із нечітких вимог.
- **Агенти**: реалізують РІВНО ТЕ, що вказано в тестах. Немає тесту — немає фічі.
**Порада**: Ніколи не просіть агента самого писати тести для свого коду. Це призведе до "циклічної валідації", де тест просто підтверджує помилки реалізації.

**Воркфлоу TDD для агентів**:
1. Ви (людина) пишете тест, що ПАДАЄ.
2. Даєте тест агенту з інструкцією: "Реалізуй функцію, щоб пройти цей тест".
3. Агент реалізує код.
4. Верифікація запуском тесту.

**Автоматизація браузера (Playwright)**:
Для UI-фіч давайте агенту E2E-тести. Це дає візуальний доказ коректності та ловить помилки доступності (A11y) або CSS, які агент може пропустити в юніт-тестах.

---

### 9.18.7 Конвенції та патерни

**Перевага "нудних технологій" (Boring Tech)**:
Агенти краще працюють із React та Express, бо мають гігабайти тренувальних даних за цими технологіями. Використання екзотичних або самописних фреймворків без детальної документації гарантує галюцинації.

**ADR (Architecture Decision Records)**:
Документуйте архітектурні рішення. Наприклад, **ADR-011** пояснює 3-шарову архітектуру: Контролери → Сервіси → Репозиторії. Прочитавши це, агент знатиме, що не можна викликати базу даних напряму з контролера.

---

### 9.18.8 Guardrails (Захисні бар'єри)

**Хуки як валідатори антипатернів**:
Використовуйте хук `PreToolUse`, щоб блокувати порушення шарів архітектури. Якщо агент намагається додати виклик репозиторію в контролер, хук видасть помилку: "❌ Порушення шарів! Див. ADR-011".

**Філософія "Tainted Code" (Забруднений код)**:
Ставтеся до будь-якого коду від ШІ як до потенційно небезпечного, поки він не пройде:
1. Лінтер (синтаксис).
2. Type Checker (типи).
3. Тести (логіка).
4. CI checks (покриття, TODO-мітки).
5. **Human review** (бізнес-контекст та безпека).

**Чек-лист для рев'ю PR від ШІ**:
- [ ] **Намір**: чи вирішує код реальну проблему, чи просто проходить тести?
- [ ] **Крайні випадки**: чи оброблені null, негативні значення, пусті масиви?
- [ ] **Безпека**: чи немає вразливостей до ін'єкцій або обходу авторизації?
- [ ] **Підтримка**: чи зрозумілий код для майбутніх розробників-людей?

### 9.18.9 Перехресні посилання (Serendipity)

Додавайте в JSDoc посилання на суміжні модулі. Наприклад, у `event-service.ts` додайте:
`@related calendar-sync-service, conflict-resolver`. Це підштовхне агента дослідити ці файли, якщо завдання стосується синхронізації, і запобіжить дублюванню коду.
