### Команда /btw

`/btw` дозволяє поставити швидке побічне питання, поки Claude працює, не перериваючи основний потік. Введіть `/btw що повертає ця функція?` і отримайте миттєву відповідь у спливаючому вікні — основне завдання продовжуватиметься без зупинки.

**Як це працює**: Claude створює тимчасового ефемерного агента БЕЗ доступу до інструментів. Він не може читати файли, запускати команди або виконувати дії. Він відповідає один раз, базуючись виключно на поточному контексті розмови, після чого оверлей закривається. Цей обмін репліками ніколи не потрапляє в основну історію розмови.

**Ключові обмеження:**
- Тільки читання — немає доступу до файлів або шеллу
- Одинарна відповідь — без подальшого діалогу в оверлеї
- Тільки контекст — відповіді на основі того, що вже в розмові, а не на диску
- "Повна обізнаність про контекст" означає контекст розмови, а не файли проекту

**Коли використовувати:**
- Швидке уточнення посеред завдання ("btw який порт за замовчуванням для Postgres?")
- Перевірка термінології без зупинки роботи
- Швидка перевірка того, що Claude щойно згадав

**Синтаксис**: Почніть повідомлення з `btw` (малими літерами, слеш не обов'язковий), а потім ваше питання. Claude Code виявляє префікс `btw` і переспрямовує його на ефемерного агента.

> Примітка: Ця функція (`btw-side-question`) була введена близько версії v2.0.73 і стабілізована до v2.1.23.

### Розгалуження сесій (Session Forking)

Розгалуження створює нову незалежну сесію, яка починається з певної точки існуючої історії. Використовуйте це, коли доходите до точки прийняття рішення і хочете вивчити два напрямки, не починаючи з нуля.

**Два способи розгалуження:**

```bash
# Зсередини активної сесії
/branch

# З CLI при відновленні
claude --resume <session-id> --fork-session
```

Команда `/branch` була додана у v2.1.77, замінивши `/fork` (яка залишилася як аліас).

**Коли розгалужувати замість перезапуску:**
- Ви перебуваєте в робочому стані і хочете спробувати ризикований рефакторинг, не втрачаючи його
- Ви хочете спробувати два різні підходи до однієї проблеми паралельно
- Ви знайшли хорошу точку посеред сесії і хочете відгалузитися для тестування гіпотези

**Після розгалуження**: обидві гілки стають незалежними — зміни в одній не впливають на іншу. Продовжити будь-яку з них можна пізніше через `claude --resume` та інтерактивний вибір сесії.

**Порада**: запустіть `/rename` перед розгалуженням, щоб ви могли розрізнити дві гілки у списку вибору.

### /recap: Контекст сесії при поверненні

`/recap` надає підсумок контексту, коли ви повертаєтеся до сесії після перерви. Claude автоматично виявляє вашу відсутність і генерує короткий переказ того, над чим велася робота, останні вжиті дії та що планується далі. Це робить повернення до тривалої сесії значно менш дезорієнтуючим, особливо після нічної перерви або стиснення контексту.

**Поведінка**: Recap спрацьовує автоматично при повторному вході в сесію. Він не спрацьовує в кінці сесії; тригером є саме *повернення* до тієї, що була неактивною.

**Варіанти конфігурації:**

| Метод | Ефект |
|--------|--------|
| `/config` пошук "recap" | Увімкнути/вимкнити функцію в інтерфейсі |
| `CLAUDE_CODE_ENABLE_AWAY_SUMMARY=1` | Примусово увімкнути (корисно, якщо телеметрія вимкнена) |
| `CLAUDE_CODE_ENABLE_AWAY_SUMMARY=0` | Повністю вимкнути |

Функція працює навіть з вимкненою телеметрією (Bedrock, Vertex, Foundry, `DISABLE_TELEMETRY`). Ви також можете перемикати її через `/config` без дотику до змінних оточення.

**Історія версій**: Введено у v2.1.108. Розширено на середовища з вимкненою телеметрією у v2.1.110. Регресія, що викликала автоматичне спрацьовування, поки користувач ще писав повідомлення, була виправлена у v2.1.113.

### Команда /insights

`/insights` аналізує історію вашого використання Claude Code для генерації комплексного звіту, що виявляє патерни, точки тертя (friction points) та можливості для оптимізації.

#### Що вона аналізує

Команда обробляє дані ваших сесій для виявлення:

- **Області проектів**: Автоматично групує вашу роботу за тематичними областями (наприклад, "Frontend Development", "CLI Tooling", "Documentation") з підрахунком сесій
- **Стиль взаємодії**: Визначає ваші патерни воркфлоу (керований планом, дослідницький, ітеративний, контролюючий)
- **Успішні патерни**: Підсвічує те, що працює добре (координація між багатьма файлами, підходи до дебагу, вибір інструментів)
- **Категорії тертя**: Вказує на повторювані проблеми (баг в коді, неправильні директорії, втрата контексту, нерозуміння запитів)
- **Використання інструментів**: Відстежує, якими інструментами ви користуєтеся найчастіше (Bash, Read, Edit, Grep тощо), та виявляє можливості для оптимізації
- **Поведінка "мульти-клодингу"**: Виявляє патерни паралельних сесій (одночасний запуск кількох екземплярів Claude)
- **Часові патерни**: Визначає ваші найбільш продуктивні часові вікна та розподіл часу відповіді

#### Що вона створює

Запуск `/insights` генерує інтерактивний HTML-звіт за шляхом `~/.claude/usage-data/report.html`, що містить:

**Короткий огляд (At a Glance)**:
- Що працює: 2-3 речення про успішні патерни
- Що заважає: 2-3 речення про основні точки тертя
- Швидкі перемоги: 1-2 дієві поради (час налаштування < 5 хвилин)
- Амбітні воркфлоу: 1-2 просунуті патерни для майбутнього вивчення

**Детальні розділи**:
1. **Над чим ви працюєте**: 3-5 автовизначених проектних областей з описами
2. **Як ви використовуєте Claude Code**: Наративний аналіз вашого стилю взаємодії + резюме ключових патернів
3. **Вражаючі речі, які ви зробили**: 3 "великі перемоги" — складні воркфлоу, виявлені системою (наприклад, рев'ю кількома агентами, кастомні шари автоматизації)
4. **Де щось йде не так**: 3 категорії тертя з прикладами та стратегіями пом'якшення
5. **Функції CC, які варто спробувати**:
   - 6+ доповнень до CLAUDE.md (відформатовані, готові до копіювання)
   - 3 функції з кодом налаштування (Custom Skills, Hooks, Task Agents)
6. **Нові способи використання Claude Code**: 3 патерни використання з промптами для копіювання
7. **На горизонті**: 3 амбітні воркфлоу з детальними промптами для впровадження (300+ токенів кожен)
8. **Веселе завершення**: Анекдот з ваших сесій (наприклад, пам'ятне втручання користувача)

**Інтерактивні елементи**:
- Кнопки копіювання для всіх фрагментів коду та промптів
- Чекбокси для доповнень у CLAUDE.md (масове копіювання)
- Графіки та візуалізації (використання інструментів, типи тертя, результати, розподіл за часом доби)
- Навігаційний зміст із якірними посиланнями
- Адаптивний дизайн (працює на мобільних пристроях)

#### Як її використовувати

**Базове використання**:
```bash
/insights
```

Команда виконується мовчки і займає ~10-30 секунд залежно від кількості сесій. Ви побачите:
```
1281 сесій · 10,442 повідомлення · 3445г · 1160 комітів
2025-12-15 до 2026-02-06

## At a Glance
[4 розділи резюме...]

Report URL: file:///Users/you/.claude/usage-data/report.html
```

**Відкриття звіту**:
- CLI: `open ~/.claude/usage-data/report.html` (macOS)
- Звіт є автономним HTML-файлом (без зовнішніх залежностей)

**Коли запускати**:
- **Після великих проектів**: Визначити, що спрацювало, а що покращити наступного разу
- **Щомісяця**: Відстежувати еволюцію ваших патернів воркфлоу
- **Коли відчуваєте застій**: Отримати поради на основі даних щодо точок тертя
- **Перед оптимізацією CLAUDE.md**: Подивитися, які патерни варто кодифікувати

#### Типові інсайти

Звіт може виявити такі патерни:

**Категорії тертя**:
- "Buggy Code Requiring Multiple Fix Rounds" (22 випадки) → Пропонує цикли build-check-fix після кожної правки
- "Wrong Directory Before Starting Work" (12 випадків) → Рекомендує явне підтвердження робочої директорії в CLAUDE.md
- "Context Loss" → Позначає сесії, де розмова відхилилася від початкової мети

**Успішні патерни**:
- "Plan-Driven Execution at Scale" → Виявляє користувачів, які надають нумеровані плани та досягають 80%+ успішності
- "Multi-Agent Review and Challenge Loops" → Ідентифікує просунутих користувачів, які створюють субагентів для критичного рев'ю

**Пропозиції для CLAUDE.md** (приклад):
```markdown
## Директорії проекту
Завжди підтверджуй правильну робочу директорію перед початком роботи:
- Frontend: /path/to/web-app
- Backend: /path/to/api
Ніколи не припускай, у якому проекті працювати — запитуй, якщо є двозначність.
```

#### Технічні деталі

- **Рушій аналізу**: Використовує Claude Haiku (швидкий, економний)
- **Ліміт сесій**: Аналізує до 50 останніх сесій за один запуск
- **Бюджет токенів**: Макс. 8192 токени на один прохід аналізу
- **Місце зберігання даних**: `~/.claude/usage-data/` (сесії зберігаються як JSONL)
- **Приватність**: Весь аналіз виконується локально; дані не надсилаються зовнішнім сервісам, окрім стандартного використання API Claude Code

#### Як працює /insights (Архітектурний огляд)

Конвеєр аналізу обробляє дані сесій через 7 етапів:

1. **Фільтрація сесій**: Завантаження з `~/.claude/projects/`, виключення суброзмов агентів, сесій з <2 повідомленнями користувача або тривалістю <1 хвилини
2. **Підсумовування транскриптів**: Поділ сесій, що перевищують 30 000 символів, на сегменти по 25 000 символів
3. **Витяг аспектів (Facets Extraction)**: Використання Claude Haiku для класифікації сесій за структурованими категоріями
4. **Агрегований аналіз**: Виявлення патернів між сесіями та повторюваних воркфлоу
5. **Резюме для керівника (Executive Summary)**: Генерація синтезу "At a Glance" за чотирма вимірами
6. **Генерація звіту**: Рендеринг інтерактивного HTML з візуалізаціями
7. **Кешування аспектів**: Збереження класифікацій у `~/.claude/usage-data/facets/<session-id>.json` для швидких наступних запусків

**Система класифікації аспектів (Facets)**:

Система категоризує сесії за такими вимірами:
**Цілі (13 типів)**: Debug/Investigate, Implement Feature, Fix Bug, Write Script/Tool, Refactor Code, Configure System, Create PR/Commit, Analyze Data, Understand Codebase, Write Tests, Write Docs, Deploy/Infra, Cache Warmup.
**Типи тертя (12 категорій)**: Нерозуміння запитів, Неправильний підхід, Баг у коді, Відмова користувача від дій, Блокування Claude, Рання зупинка користувачем, Неправильні локації файлів, Зайве ускладнення, Повільність/багатослівність, Помилки інструментів, Нечіткі запити, Зовнішні проблеми.
**Рівні задоволеності (6)**: Frustrated → Dissatisfied → Likely Satisfied → Satisfied → Happy → Unsure.
**Результати (4 стани)**: Not Achieved → Partially Achieved → Mostly Achieved → Fully Achieved.
**Категорії успіху (7)**: Швидкий точний пошук, Правильні правки коду, Гарні пояснення, Проактивна допомога, Зміни в багатьох файлах, Гарний дебаг, Нічого.
**Типи сесій (5)**: Одне завдання, Багато завдань, Ітеративне шліфування, Дослідження, Швидке питання.

Розуміння цих категорій допомагає інтерпретувати ваш звіт:
- Багато тертя "Баг у коді" → Розгляньте впровадження pre-commit хуків
- Низька задоволеність цілями "Implement Feature" → Покращуйте специфічність фази планування

**Оптимізація продуктивності**: Система кешування гарантує, що наступні запуски аналізуватимуть лише нові сесії, що робить регулярні щомісячні запуски швидкими навіть при великій історії.

> **Джерело**: Деталі архітектури з [Zolkos Technical Deep Dive](https://www.zolkos.com/2026/02/04/deep-dive-how-claude-codes-insights-command-works.html) (2026-02-04)

#### Обмеження

- **Потребує історії**: Потрібно хоча б ~10 сесій для значущих патернів
- **Упередженість щодо свіжості**: Фокусується на останніх 50 сесіях
- **Оціночна задоволеність**: Бали задоволеності виводяться моделлю, а не є явними оцінками користувача
- **Немає агрегації між проектами**: Кожен проект аналізується незалежно

| Інструмент | Область | Вивід | Кейс |
|------|-------|--------|----------|
| `/insights` | Поведінка в сесіях, тертя, патерни | Інтерактивний HTML-звіт | Оптимізація воркфлоу |
| `/status` | Тільки поточна сесія | Текстове резюме (контекст, витрати) | Моніторинг у реальному часі |
| `ccboard` | Економіка, аналіз вартості | TUI/Web дашборд | Відстеження бюджету |
| Git history | Тільки зміни коду | Лог комітів | Показники доставки |

> **Порада**: Запускайте `/insights` щомісяця, `/status` під час кожної сесії, а `ccboard` щотижня для повної видимості.

### Команда /simplify

Додана у v2.1.63, `/simplify` — це вбудована slash-команда, яка переглядає ваш нещодавно змінений код на предмет зайвого ускладнення та надлишкових абстракцій, а потім виправляє знайдені проблеми.

#### Коли використовувати

Запускайте її після завершення фічі, перед створенням pull request:

```bash
# Переглянути все змінене з останнього коміту
/simplify

# Зосередитися на конкретній проблемі
/simplify зосередься на обробці помилок
/simplify перевір наявність зайвих залежностей
/simplify подивись на патерни запитів до БД
```

#### Що вона робить

`/simplify` аналізує змінений код на предмет:
- **Повторного використання** — дубльована логіка, яку можна винести
- **Якості** — патерни, що погіршують читабельність або підтримку
- **Ефективності** — алгоритмічні та структурні покращення

Вона працює на рівні архітектури та структури, а не на рівні форматувальника чи лінтера. `/simplify` доповнює такі інструменти, як ESLint або Prettier, а не замінює їх.

#### Позиціонування

| Інструмент | Рівень | Що виправляє |
|------|-------|-------|
| Prettier | Форматування | Стиль, відступи |
| ESLint | Правила синтаксису | Прості патерни, невикористані змінні |
| `/simplify` | Архітектура | Зайві абстракції, дублювання, дизайн |

> **Примітка**: `/simplify` є вбудованою командою, а не кастомним скілом. Доступна з версії v2.1.63+.

### Команда /batch

Додана у v2.1.63, `/batch` оркеструє масштабні зміни в кодобазі, розподіляючи роботу між 5–30 паралельними агентами в ізольованих git worktrees, кожен з яких відкриває власний pull request.

#### Як це працює

1. **Дослідження та план** — аналіз кодобази та поділ зміни на незалежні одиниці
2. **Паралельне виконання** — одночасний запуск 5–30 ізольованих агентів у git worktree
3. **PR для кожного агента** — кожен агент завершує свою частину та відкриває PR

#### Використання

```bash
/batch migrate from react to vue
/batch замініть усі використання lodash нативними еквівалентами
/batch додайте анотації типів до всіх файлів JavaScript
```

#### Коли використовувати

`/batch` — це нативний еквівалент патерну мультиагентної роботи в паралельних worktree (див. §15). Використовуйте її для великих, повторюваних змін на рівні файлів, які можна розділити на незалежні блоки: міграції, рефакторинг, масове типізування, заміна залежностей.

### Заплановані завдання: Три методи

Claude Code надає три окремі механізми для запуску повторюваних завдань. Вони відрізняються місцем виконання, способом тригера та необхідністю увімкненого локального комп'ютера.

#### Порівняльна таблиця

| | Routines (Рутини) | Десктопні завдання | `/loop` |
|--|--|--|--|
| Виконується на | Хмарі Anthropic | Локальній машині | Локальній машині |
| Машина має бути увімкнена | Ні | Так | Так |
| Сесія має бути відкрита | Ні | Ні | Так |
| Зберігається між перезапусками | Так | Так | Ні |
| Доступ до лок. файлів | Ні (клонування репо) | Так | Так |
| Типи тригерів | Розклад / API / GitHub | Тільки розклад | Тільки в сесії |
| MCP-сервери | Конфігуруються для завдання | Конфіги + конектори | Успадковуються з сесії |
| Запити дозволів | Немає (автономно) | Конфігуруються | Успадковуються з сесії |
| Мін. інтервал | 1 година (за розкладом) | 1 хвилина | 1 хвилина |
| Денний ліміт запусків | 5–25 на день | Безлімітно | Обмежено сесією |

#### Рутини (Cloud Automation)

Рутини працюють на інфраструктурі Anthropic — ваш комп'ютер може бути повністю вимкненим. Кожен запуск клонує свіжу копію вашого GitHub-репозиторію. Три типи тригерів можна комбінувати в одній рутині.

> **Research preview**: поведінка, ліміти та API можуть змінюватися.

**Доступ**: плани Pro, Max, Team та Enterprise.

**Денні ліміти запусків**:
| План | Запусків/день |
|------|----------|
| Pro | 5 |
| Max | 15 |
| Team / Enterprise | 25 |

Додаткові запуски доступні при увімкненій тарифікації понад ліміт.

**Створення рутини**:
- `claude.ai/code/routines` — веб-інтерфейс
- Десктопний додаток — **New task** → **New remote task**
- `/schedule` у CLI (тільки для тригера за розкладом; API та GitHub тригери потребують веб-інтерфейсу)

**Як працює кожен запуск**: Anthropic клонує ваш репозиторій, запускає сесію Claude із налаштованим середовищем та MCP-конекторами, виконує завдання, а потім пушить коміти у гілку з префіксом `claude/` за замовчуванням.

**Ключові обмеження**:
- Немає доступу до локальних файлів (тільки ті, що в репозиторії GitHub)
- Мінімальний інтервал — 1 година для тригера за розкладом
- Підтримує MCP-конектори: Slack, Linear, Google Drive та інші, налаштовані для кожної рутини
- Запуски відображаються як повноцінні сесії, які можна перевірити, продовжити або створити з них PR

##### Тригер за розкладом (Schedule Trigger)

Працює за розкладом cron. Чотири пресети (щогодини / щодня / по буднях / щотижня), плюс кастомні вирази через `/schedule update` у CLI.

```bash
/schedule "щопонеділка о 9 ранку відкривай PR із підсумком замерджених PR за минулий тиждень"
/schedule "щоночі о 2 ранку бери найпріоритетніший баг із Linear та відкривай чернетку PR із фіксом"
/schedule "щоп'ятниці скануй замерджені PR на предмет розбіжностей у документації та відкривай PR з оновленнями"
```

##### API-тригер

Кожна рутина отримує виділений HTTP-ендпоінт. Відправляйте POST-запит до нього з будь-якої зовнішньої системи — інструментів сповіщення, конвеєрів деплою, CI-скриптів — і Claude відкриє нову автономну сесію.

```bash
curl -X POST https://api.anthropic.com/v1/claude_code/routines/trig_01.../fire \
  -H "Authorization: Bearer sk-ant-oat01-xxxxx" \
  -H "anthropic-beta: experimental-cc-routine-2026-04-01" \
  -H "anthropic-version: 2023-06-01" \
  -H "Content-Type: application/json" \
  -d '{"text": "Спрацював алерт Sentry SEN-4521 у продакшені. Стек-трейс додано."}'
```

Опціональне поле `text` передає контекст, специфічний для запуску (тіло алерту, ID деплою, фрагмент логу), у промпт рутини. Відповідь повертає `session_url` для спостереження за запуском у прямому ефірі.

**Налаштування**: додайте API-тригер на сторінці редагування рутини у веб-інтерфейсі, натисніть **Generate token** (показується один раз — збережіть його негайно), скопіюйте URL ендпоінту. Токени створюються для кожної рутини окремо; їх можна ротувати або відкликати на тій самій панелі.

##### Тригер подій GitHub (GitHub Event Trigger)

Автоматично запускає нову сесію при виникненні відповідних подій у репозиторії GitHub. Потребує встановлення Claude GitHub App для цільового репозиторію (окремо від `/web-setup`).

**17 підтримуваних типів подій**: pull request, push, issues, releases, check run, check suite, workflow run, workflow job, workflow dispatch, repository dispatch, pull request review, PR review comment, issue comment, discussion, discussion comment, commit comment, merge queue entry.

**PR-фільтри**: можна звузити за автором, заголовком, тілом, базовою/цільовою гілкою, лейблами, станом чернетки, станом мерджу або походженням з форку. Усі умови мають збігатися.

```
# Приклади комбінацій фільтрів
PR відкритий з форку                          → рутина рев'ю безпеки
PR з лейблом "needs-backport", замерджений    → рутина backport-to-next
Будь-який замерджений PR, що змінює /sdk/python/ → рутина авто-портування в Go SDK
PR відкритий, не є чернеткою                   → рутина чек-листа командного рев'ю
```

**Важливо**: кожна відповідна подія відкриває власну незалежну сесію. Два відкриті PR = дві сесії. Повторного використання сесій між подіями немає.

**Офіційна документація**: `https://code.claude.com/docs/en/routines`

##### Пошук кейсів для вашого проекту

Хороший кандидат для Рутини має три властивості: він щоразу виконує одну й ту саму логіку (або реагує на чітко визначену подію), результат є конкретним (відкритий PR, надіслане повідомлення, оновлений файл), і під час виконання не потрібна участь людини.

П'ять напрямків для аудиту будь-якого проекту:

| Напрямок | Питання для аудиту |
|-------|-----------------|
| Планове обслуговування | Що ви робите вручну за розкладом і іноді забуваєте? Аудит залежностей, тріаж застарілих PR, звіти про мертвий код. |
| Реакції на події | Що має відбуватися при кожному відкритті або мерджі PR, але не відбувається, бо ніхто до цього не доходить? Чек-листи рев'ю, оновлення чейнджлогу. |
| Реагування на алерти | Коли спрацьовує моніторинг, що насамперед робить розробник? Чи може цей крок виконуватися автоматично до того, як людина подивиться? |
| Синхронізація систем | Що розходяться через ручну синхронізацію? Два SDK, сайт документації та API, GitHub issues та Linear. |
| Автоматизація релізів | Що ви запускаєте вручну перед або після деплою? Smoke-тести, нотатки до релізу, сповіщення стейкхолдерів. |

Використовуйте команду `/routines-discover` для аналізу будь-якої кодобази — вона читає репозиторій, виявляє конкретних кандидатів за цими п'ятьма напрямками та ранжує їх за співвідношенням цінності до зусиль.

```bash
/routines-discover
```

#### Десктопні заплановані завдання

Десктопні завдання працюють на вашій локальній машині через додаток Claude Code Desktop. Ваша машина має бути увімкнена, але активна термінальна сесія не потрібна.

На відміну від хмарних завдань, десктопні мають повний доступ до локальних файлів та вашої існуючої конфігурації MCP. Мінімальний інтервал — 1 хвилина.

**Створення завдання**: відкрийте додаток Desktop, перейдіть на сторінку **Schedule**, натисніть **New task**. Ви також можете створити віддалене (хмарне) завдання з цієї ж сторінки.

**Офіційна документація**: `https://code.claude.com/docs/en/desktop-scheduled-tasks`

#### DIY: System Cron + `claude --print`

Для повного контролю без десктопного додатка підключіть системний cron безпосередньо з прапором headless:

```bash
# crontab -e
0 8 * * 1-5 bash -c 'source /home/user/.env && cd /your/repo && claude --print "підсумуй зміни в git з учора" >> /var/log/claude-daily.log 2>&1'
```

Цей підхід працює повністю офлайн без інфраструктури Anthropic і не має мінімального інтервалу. Три речі, які потрібно зробити правильно: використовуйте повний шлях до `claude` (перевірте через `which claude`), завантажуйте `ANTHROPIC_API_KEY` із файлу, а не хардкодьте його, і переспрямовуйте stdout та stderr у лог-файл, щоб мати запис кожного запуску.

#### Команда /loop

`/loop [interval] [prompt]` запускає промпт або slash-команду з заданим інтервалом у межах поточної сесії. Вона зупиняється, коли ви натискаєте `Ctrl+C` або надсилаєте будь-яке нове повідомлення.

```bash
/loop 5m перевір деплой
/loop 30m /slack-feedback
/loop 1h /pr-pruner
```

**Як це працює**: Claude виконує промпт, чекає вказаний час, виконує знову і так по колу. Кожне виконання позначається часовою міткою в транскрипті. Ви можете посилатися на slash-команду (наприклад, `/loop 30m /review-pr`) або писати довільний промпт.

**Кейси від Бориса Черного (творця Claude Code):**
| Цикл | Що робить |
|------|-------------|
| `/loop 5m /babysit` | Автоматично обробляє рев'ю коду, ребейз, пушить PR вперед |
| `/loop 30m /slack-feedback` | Публікує PR для фідбеку команди кожні 30 хв |
| `/loop 1h /pr-pruner` | Очищує застарілі PR за розкладом |

**Обмеження**: Тільки в межах сесії. Макс. 3 дні роботи, мін. інтервал 1 хвилина, макс. 50 завдань на сесію.

> `/loop` додано у v2.1.71. Маркери часу в транскриптах циклів додані у v2.1.86. Хмарні та десктопні заплановані завдання запущені 9 березня 2026 року.

## 6.2 Створення кастомних команд

Команди — це маркдаун-файли, що визначають певний процес.

### Місце зберігання файлів команд

```
.claude/commands/
├── tech/           # Воркфлоу розробки
│   ├── commit.md
│   └── pr.md
├── product/        # Продуктові воркфлоу
│   └── problem-framer.md
└── support/        # Воркфлоу підтримки
    └── ticket-analyzer.md
```

### Іменування команд

| Файл | Виклик |
|------|------------|
| `commit.md` у `tech/` | `/tech:commit` |
| `pr.md` у `tech/` | `/tech:pr` |
| `problem-framer.md` у `product/` | `/product:problem-framer` |

### Інтерполяція змінних

Команди можуть приймати аргументи:

```markdown
# Моя команда

Ви отримали такі аргументи: $ARGUMENTS[0] $ARGUMENTS[1] $ARGUMENTS[2]
(Або використовуйте скорочено: $0 $1 $2)

Обробіть їх належним чином.
```

Використання:
```
/tech:deploy production
```
`$ARGUMENTS[0]` (або `$0`) стане `production`.

> **⚠️ Зміна синтаксису (v2.1.19)**: Синтаксис аргументів змінився з крапки (`$ARGUMENTS.0`) на квадратні дужки (`$ARGUMENTS[0]`). Якщо у вас є існуючі команди зі старим синтаксисом, оновіть їх.

**Поєднуйте `$ARGUMENTS` з `argument-hint`**, щоб користувачі бачили доступні опції у списку вибору команд. Підказка відображається як текст-плейсхолдер при введенні команди:

```yaml
---
description: Деплой у цільове середовище
argument-hint: "<env> [--skip-tests] [--dry-run]"
---
Деплой у середовище $ARGUMENTS[0].
```

Коли користувач вводить `/deploy`, меню покаже: `/deploy <env> [--skip-tests] [--dry-run]`

### Шаблон рецепта: Чек-листи валідації контексту

Стандартний шаблон вище добре підходить для робочих команд. Для команд, які є процедурно ризикованими (деплой, міграція даних, незворотні операції), додайте розділ "Context Validation Checkpoints" перед основними кроками:

```markdown
## Чек-листи валідації контексту

Перед виконанням будь-якого кроку переконайтеся, що всі ці твердження є істинними.
Якщо будь-яка перевірка не проходить, зупиніться і поясніть причину.

* [ ] Цільова гілка існує та синхронізована з main
* [ ] Немає незакомічених змін у відповідних файлах
* [ ] Необхідний конфігураційний файл існує за шляхом X
* [ ] Доступні облікові дані або дозволи
```

Чек-лист змушує Claude явно перевіряти передумови, замість того щоб виявляти помилки посеред виконання. Провал перевірки дає чітку помилку з причиною, яку можна виправити; помилка посеред кроку створює частковий стан, який важче відновити.

## 6.4 Приклади команд

### Приклад 1: Команда Commit

```markdown
# Коміт поточних змін

## Мета
Створити добре відформатований git-коміт згідно з Conventional Commits.

## Процес

1. **Перевірка статусу**
   Запустити `git status`, щоб побачити всі зміни.

2. **Аналіз змін**
   Запустити `git diff`, щоб зрозуміти, що змінилося.

3. **Перегляд історії**
   Запустити `git log -5 --oneline`, щоб побачити стиль останніх комітів.

4. **Чернетка повідомлення**
   Створити повідомлення коміту згідно з:
   - `feat`: Нова фіча
   - `fix`: Виправлення багу
   - `refactor`: Реструктуризація коду
   - `docs`: Документація
   - `test`: Зміни в тестах
   - `chore`: Обслуговування

5. **Індексація та коміт**
   ```bash
   git add [відповідні файли]
   git commit -m "[type](scope): опис"
   ```

6. **Верифікація**
   Запустити `git status`, щоб підтвердити успішність коміту.

## Аргументи
Якщо $ARGUMENTS[0] надано:
- Використовуй як підказку для повідомлення коміту: "$ARGUMENTS[0]"

## Формат виводу
Commit: [hash] [message]
Files: [number] changed
```

### Приклад 2: Команда PR

```markdown
# Створити Pull Request

## Мета
Створити добре задокументований pull request на GitHub.

## Процес

1. **Перевірка стану гілки**
   - `git status` - Перевірити чистоту робочої директорії
   - `git branch` - Підтвердити перебування на гілці фічі
   - `git log main..HEAD` - Переглянути всі коміти

2. **Аналіз змін**
   - `git diff main...HEAD` - Побачити всі зміни порівняно з main
```
