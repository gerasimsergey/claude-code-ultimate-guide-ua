Вчіться на поширених помилках, щоб уникнути розчарування та максимізувати продуктивність.

### Пастки безпеки

**❌ Не робіть:**
- Використання `--dangerously-skip-permissions` на промислових системах.
- Хардкод секретів у командах, конфігах або CLAUDE.md.
- Надання занадто широких дозволів, як-от `Bash(*)`, без обмежень.
- Коміт `.claude/settings.local.json` (містить API-ключі) у Git.

**✅ Робіть:**
- Зберігайте секрети у змінних оточення або захищених сховищах.
- Починайте з мінімальних дозволів і розширюйте їх поступово.
- Регулярно перевіряйте активні дозволи командою `claude config list`.
- Використовуйте хуки **PreToolUse** для виявлення випадкового розкриття секретів.

**Приклад хука безпеки:**
```bash
#!/bin/bash
# .claude/hooks/PreToolUse.sh - Блокування секретів у комітах
# Перевіряє git diff --cached на наявність патернів password/token/api_key
# Повертає exit 2 для блокування операції.
```

### Пастки продуктивності

**❌ Не робіть:**
- Завантаження всього монорепозиторію, коли потрібен лише один пакет.
- Ігнорування очищення сесій — старі сесії накопичуються і сповільнюють роботу.
- Підтримання контексту на рівні 90%+ тривалий час.

**✅ Робіть:**
- Керуйте режимом мислення: вимикайте його (Alt+T) для простих правок.
- Використовуйте **/compact** проактивно, коли контекст досягає 70%.
- Моніторьте витрати через **/status** та налаштуйте `cleanupPeriodDays`.

### Пастки воркфлоу та "Парадокс артефакту"

**❌ Не робіть:**
- Пропуск контексту проекту (`CLAUDE.md`) — це веде до повторюваних помилок.
- Прийняття змін наосліп без перевірки дифів.

> **⚠️ Парадокс артефакту — Anthropic AI Fluency Index (лютий 2026)**
> Дослідження Anthropic показує: коли Claude видає "відшліфований" результат (код, який компілюється), **користувачі стають на 5.2% менш критичними**. Ми схильні довіряти тому, що виглядає завершеним.
> **Контрзаходи**: запускайте тести *перед* прийняттям коду; явно запитуйте: "Які межові випадки ти не врахував?"; використовуйте хуки валідації виводу.

**Ефективний формат промпту: WHAT/WHERE/HOW/VERIFY**
- **WHAT**: Що зробити (додати валідацію).
- **WHERE**: Де (шляхи до файлів).
- **HOW**: Як (використовувати Zod, показати помилки inline).
- **VERIFY**: Як перевірити (критерії успіху).

**✅ Робіть:**

## 9.12 Найкращі практики Git та воркфлоу

### Стандарти атрибуції ШІ
Для прозорості використовуйте трейлер `Co-Authored-By: Claude <noreply@anthropic.com>` або стандарт LLVM `Assisted-by:`. Детальніше у [AI Traceability Guide](./ops/ai-traceability.md).

### Воркфлоу Git Amend
Використовуйте `git commit --amend` тільки для **неопублікованих (unpushed)** комітів. Якщо коміт вже в репозиторії — створюйте новий `fix:` коміт.

### Rewind проти Revert
- **`/rewind`**: скасовує останні зміни Claude у поточній сесії (локально, без комітів).
- **`git revert`**: створює новий коміт, що скасовує попередній (безпечно для історії).

---

### Git Worktrees для паралельної розробки

**Проблема**: робота над фічею А, терміново потрібен хотфікс → `git stash`, перемикання гілок, втрата концентрації.
**Рішення**: `git worktree` дозволяє мати кілька робочих директорій з одного репо одночасно.

**Команди життєвого циклу**:
- **/git-worktree**: створення з автоматичними перевірками типів та тестів у фоні.
- **/git-worktree-status**: перевірка результатів фонової верифікації.
- **/git-worktree-remove**: безпечне видалення після мерджу.

> **💡 Порада — Симлінки node_modules**: `/git-worktree` за замовчуванням створює символічні посилання на `node_modules` з головного дерева, заощаджуючи ~30с часу та багато місця на диску.

**Порада команди Claude Code**: Використовуйте аліаси з однієї літери (`za`, `zb`, `zc`) для миттєвого стрибка між робочими деревами фіч. Створіть окреме дерево `zlog` виключно для аналізу логів та запитів до БД, щоб не засмічувати робочі гілки.
