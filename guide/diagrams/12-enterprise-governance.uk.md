# Корпоративне управління

Патерни рівня організації для команд, що впроваджують Claude Code масштабно — рівні використання, воркфлоу схвалення MCP та конфігурації обмежень.

---

### Рівні ризику управління — що і коли контролювати

Не все потребує жорсткого контролю. Ця схема допоможе обрати рівень контролю залежно від ризику — від особистого воркфлоу до регульованих середовищ.

```mermaid
flowchart TD
    A([Що ви регулюєте?]) --> B{Контекст<br/>використання?}

    B --> P["Особистий воркфлоу<br/>Локальний код<br/>Один розробник"]
    B --> T["Командна кодобаза<br/>Спільний репо<br/>5–20 розробників"]
    B --> PR["Продакшн-система<br/>Реальні дані клієнтів<br/>Будь-яка команда"]
    B --> REG["Регульоване середовище<br/>HIPAA, SOC2, фінанси<br/>Юридичні зобов'язання"]

    P --> TIER1(["Рівень 1: Стартовий<br/>Правила CLAUDE.md<br/>+ хуки блокування дій<br/>Налашт. за 10 хв"])
    T --> TIER2(["Рівень 2: Стандарт<br/>Спільний settings.json + реєстр MCP<br/>+ PR-гейти + аудит-лог<br/>Налашт. за ~2 год"])
    PR --> TIER3(["Рівень 3: Суворий<br/>Повний deny-лист дозволів<br/>+ воркфлоу схвалення<br/>+ аудит сесій"])
    REG --> TIER4(["Рівень 4: Регульований<br/>Все вищезгадане<br/>+ аудит відповідності<br/>+ контролі SOC2/ISO27001"])

    style A fill:#F5E6D3,color:#333
    style TIER1 fill:#7BC47F,color:#333
    style TIER4 fill:#E85D5D,color:#fff

    click A href "../security/enterprise-governance.uk.md#1-локальне-vs-спільне-розподіл-управління" "Що ви регулюєте?"
```

<details>
<summary>ASCII версія</summary>

```
Контекст використання?
├─ Особистий воркфлоу       → Рівень 1: Стартовий (CLAUDE.md + хуки)
├─ Командна кодобаза        → Рівень 2: Стандарт (settings.json + реєстр MCP)
├─ Продакшн-система         → Рівень 3: Суворий (deny-лист + схвалення + аудит)
└─ Регульоване середовище   → Рівень 4: Регульований (все + аудит відповідності)

Ви МОЖЕТЕ контролювати: settings.json в репо, CLAUDE.md, хуки, CI/CD, реєстр MCP.
Ви НЕ МОЖЕТЕ контролювати: особисті ~/.claude, особисті API-ключі.
```

</details>

### Воркфлоу управління MCP

Корпоративне управління MCP — це 5-кроковий пайплайн, який гарантує безпеку серверів та контроль версій.

```mermaid
sequenceDiagram
    participant DEV as Розробник
    participant TL as Тімлід + Безпека
    participant REG as Реєстр MCP<br/>.claude/mcp-registry.yaml
    participant REPO as Спільний конфіг-репо<br/>settings.json

    DEV->>TL: Запит на MCP сервер<br/>Назва, URL, кейс, об'єм даних

    TL->>TL: 5-хв аудит безпеки<br/>Зірки >50? Свіжі комміти?<br/>CVE? Небезпечні прапорці?
    TL->>TL: Класифікація ризику: НИЗЬКИЙ / СЕРЕДНІЙ / ВИСОКИЙ

    alt НИЗЬКИЙ ризик
        TL->>REG: Схвалено — додати в реєстр<br/>Закріпити версію, термін 6 міс.
    else СЕРЕДНІЙ/ВИСОКИЙ ризик
        TL->>TL: 2-тижневий тест у пісочниці<br/>+ підпис команди безпеки
        TL->>REG: Схвалено з обмеженнями<br/>Термін 3 міс.
    else ВИСОКИЙ ризик (неприйнятно)
        TL->>DEV: Відмовлено — причина в реєстрі
    end

    REG->>REPO: Деплой через settings.json<br/>Без локальних перезаписів
    Note over REPO: Фіксована версія, на всю команду

    REPO->>TL: Моніторинг кожні 30 днів<br/>Патчі: автосхвалення<br/>Мінорні+: ручне рев'ю<br/>Квартально: повний аудит
```

### Класифікація даних та правила доступу Claude Code

Класифікація даних визначає, що Claude Code дозволено читати. Це найважливіший аспект управління.

```mermaid
flowchart LR
    subgraph PUBLIC["🟢 ПУБЛІЧНІ"]
        PU1["Open source код<br/>Публічна документація<br/>Контент блогів"]
        PU2(["Дозволено — без обмежень"])
    end

    subgraph INTERNAL["🔵 ВНУТРІШНІ"]
        IN1["Внутрішні інструменти<br/>Нечутливий код<br/>Командна документація"]
        IN2(["Дозволено — стандартний конфіг"])
    end

    subgraph CONFIDENTIAL["🟠 КОНФІДЕНЦІЙНІ"]
        CO1["Бізнес-таємниці<br/>Інтелектуальна власність<br/>Архів архітектури"]
        CO2(["Тільки Enterprise план<br/>Zero Data Retention"])
    end

    subgraph RESTRICTED["🔴 СУВОРО ОБМЕЖЕНІ"]
        RE1["Персональні дані (PII)<br/>Платіжні дані / Медичні дані<br/>Паролі та API-ключі"]
        RE2(["НІКОЛИ в контекст ШІ<br/>Блок через permissions.deny<br/>Без винятків"])
    end

    PU1 --> PU2
    IN1 --> IN2
    CO1 --> CO2
    RE1 --> RE2

    style PU2 fill:#7BC47F,color:#333
    style IN2 fill:#6DB3F2,color:#fff
    style CO2 fill:#E87E2F,color:#fff
    style RE2 fill:#E85D5D,color:#fff

    click PU1 href "../security/enterprise-governance.uk.md#2-хартія-використання-ші" "Класифікація даних"
```

<details>
<summary>ASCII версія</summary>

```
ПУБЛІЧНІ       → Дозволено, без обмежень
ВНУТРІШНІ      → Дозволено, стандартний конфіг
КОНФІДЕНЦІЙНІ  → Тільки Enterprise (Zero Data Retention)
СУВОРО ОБМЕЖЕНІ → НІКОЛИ в контекст ШІ (PII, PCI, ключі)
                 Блокувати через: permissions.deny Read(.env, *.key, secrets/**)

Залізне правило: СУВОРО ОБМЕЖЕНІ дані ніколи не потрапляють у вікно контексту.
Ні в промпти, ні у файли, ні як приклади.
```

</details>
