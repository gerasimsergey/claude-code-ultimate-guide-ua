# Правила безпеки на продакшені

> **Аудиторія**: Команди, що розгортають Claude Code у продакшн-середовищах.
> **Для тих, хто навчається самостійно**: Див. замість цього [Початок роботи](#початок-роботи).

---

## Коротко (TL;DR)

**6 правил для професійних команд, що не підлягають обговоренню**:

1. ✅ **Стабільність портів**: Ніколи не змінюйте порти бекенду/фронтенду.
2. ✅ **Безпека бази даних**: Завжди робіть бекап перед руйнівними операціями.
3. ✅ **Повнота реалізації**: Ніколи не випускайте "напівготові" фічі.
4. ✅ **Блокування інфраструктури**: Зміни Docker/env потребують окремого дозволу.
5. ✅ **Безпека залежностей**: Жодних нових бібліотек без схвалення.
6. ✅ **Дотримання патернів**: Дотримуйтесь конвенцій існуючої кодової бази.

---

## Правило 1: Стабільність портів

### Проблема
Зміна портів ламає локальні середовища розробки, конфігурації Docker Compose та налаштування проксі-серверів (nginx/Vercel).
**Реальний інцидент**: Порт бекенду змінили з 3000 на 8080 під час рефакторингу. Уся команда втратила день на переналаштування локальних середовищ, а стейджинг "ліг", бо проксі все ще звертався до 3000.

### Правило
**Ніколи не змінюйте порти бекенду/фронтенду без явного дозволу команди.**

### Реалізація (CLAUDE.md)
```markdown
## Конфігурація портів
**КРИТИЧНО**: Порти заблоковані для координації команди.
- Frontend (Vite): 5173
- Backend (Express): 3000
- Database: 5432
```

## Правило 2: Безпека бази даних

### Проблема
Випадкові видалення в продакшені = втрата даних.
**Реальний інцидент**: Команда `DELETE FROM users WHERE id = 123` перетворилася на `DELETE FROM users`, бо розробник забув частину `WHERE`.

### Правило
**Завжди робіть бекап перед руйнівними операціями (DROP, DELETE, TRUNCATE, ALTER...DROP).**

### Реалізація (Хук PreToolUse)
Створіть скрипт, який перехоплює команди Bash і блокує їх, якщо вони містять небезпечні SQL-ключові слова без попередньої команди бекапу.

---

## Правило 3: Повнота реалізації фіч

### Проблема
Claude іноді "лінується", коли контекст вичерпується: додає `// TODO` замість логіки або створює "заглушки" (mocks).

### Правило
**Ніколи не випускайте недороблені фічі. Якщо почали — завершіть до робочого стану.**

### Стандарти в CLAUDE.md:
- Жодних `TODO` для основної функціональності.
- Жодних заглушок типу `throw new Error("Not implemented")`.
- Повна обробка помилок (try/catch) для кожного асинхронного виклику.

---

## Правило 4: Блокування інфраструктури

### Правило
**Модифікація конфігурацій інфраструктури вимагає явного дозволу.**

**Файли під захистом**:
- `docker-compose.yml`, `Dockerfile`.
- `.env.example` (шаблони).
- `terraform/`, `kubernetes/`, `.github/workflows/`.
- Схеми баз даних (`prisma/schema.prisma`).

---

## Правило 5: Безпека залежностей

### Проблема
Додавання нових бібліотек збільшує розмір бандла, створює ризики вразливостей та ліцензійних конфліктів.

### Правило
**Жодних нових залежностей без схвалення.**
Claude повинен спочатку перевірити, чи не вирішують проблему вже встановлені пакети (наприклад, використати `axios`, якщо він уже є, замість додавання `fetch-node`).

---

## Правило 6: Дотримання патернів

### Правило
**Дотримуйтесь існуючих конвенцій кодової бази.**
Якщо проект використовує функціональні компоненти React — не пишіть класові. Якщо використовується Zustand — не додавайте Redux.

---

## Правило 7: Парадокс верифікації

### Проблема
Коли ШІ успішний у 99% випадків, людська пильність притупляється. Люди починають схвалювати PR "не дивлячись", що призводить до пропуску критичних помилок в 1% випадків.

### Правило
**Будуйте автоматизовані системи безпеки замість того, щоб покладатися лише на людську увагу.**
Коли надійність ШІ перевищує 95%, змістіть акцент з ручного рев'ю на автоматичні "запобіжники" (guardrails):
- Тести типу (TypeScript).
- Лінтери.
- Unit та E2E тести.
- Автоматичні сканери безпеки.

---

## Безпека автономних циклів

Для сесій, що працюють автономно (unattended) довше кількох хвилин, впроваджуйте механізм **heartbeat** (серцебиття).
- **Хук PostToolUse**: записує мітку часу після кожної дії.
- **Watchdog (сторожовий пес)**: окремий процес, який вбиває всю групу процесів Claude, якщо мітка часу не оновлювалася довше встановленого ліміту (наприклад, 30 секунд).

Це запобігає "зацикленню" ШІ та безглуздому спалюванню бюджету API.
