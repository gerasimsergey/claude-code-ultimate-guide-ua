# Посібник зі зміцнення безпеки

> **Впевненість**: Рівень 2 — На основі розкриттів CVE, досліджень безпеки (2024-2026) та валідації спільнотою.
>
> **Сфера**: Актуальні загрози (атаки, ін'єкції, CVE). Питання конфіденційності даних див. у [data-privacy.md](./data-privacy.md).

---

## TL;DR - Матриця рішень

| Ваша ситуація | Негайні дії | Час |
|----------------|------------------|------|
| **Соло-розробник, публічні репозиторії** | Встановіть хук сканування виводу | 5 хв |
| **Команда, чутлива кодова база** | + Веттинг MCP + хуки проти ін'єкцій | 30 хв |
| **Корпорація, продакшн** | + ZDR + верифікація цілісності | 2 години |

**Прямо зараз**: Перевірте ваші MCP-сервери за [Безпечним списком](#безпечний-список-mcp-перевірено-спільнотою) нижче.

> **НІКОЛИ**: Не схвалюйте MCP з невідомих джерел без фіксації версії.
> **НІКОЛИ**: Не запускайте MCP для баз даних на продакшені без облікових даних тільки для читання (read-only).

## Частина 1: Запобігання (До початку роботи)

### 1.1 Воркфлоу перевірки (веттингу) MCP

Сервери Model Context Protocol (MCP) розширюють можливості Claude Code, але створюють значну поверхню атаки.

#### Атака: MCP Rug Pull (Різке відкликання довіри)

1. Зловмисник публікує корисний MCP (напр., "форматувальник коду").
2. Користувач додає його в конфіг і схвалює один раз.
3. MCP працює нормально 2 тижні, завойовуючи довіру.
4. Зловмисник випускає шкідливе оновлення (повторне схвалення не потрібне!).
5. MCP викрадає ключі SSH, `.env`, облікові дані.

**ЗАХИСТ**: Фіксація версій + перевірка хешу + моніторинг.

#### Підсумок CVE (2025-2026)

| CVE | Рівень | Вплив | Захист |
|-----|----------|--------|------------|
| **CVE-2025-54135** | Високий | RCE в Cursor через ін'єкцію в mcp.json | Хук моніторингу цілісності |
| **CVE-2026-24052** | Високий | SSRF через обхід валідації доменів | Оновіть до v1.0.111+ |
| **ADVISORY-CC-2026-001** | Високий | Втеча з пісочниці (Sandbox bypass) | **Оновіть до v2.1.34+ негайно** |
| **CVE-2026-33032** (MCPwn) | **Критичний** | Захоплення nginx через MCP без авторизації | **Оновіть nginx-ui до >= v2.3.4** |

### 1.2 Ризики ланцюжка поставок скілів

Сторонні скіли (Agent Skills), встановлені через `npx add-skill`, несуть ризики, схожі на npm-пакети. Дослідження Snyk (лютий 2026) виявило:
- **36.82%** скілів мають вразливості в безпеці.
- **13.4%** становлять критичний ризик (малварь, викрадення секретів).

**Захист**:
- Скануйте перед встановленням за допомогою `mcp-scan`.
- Перевіряйте `SKILL.md` на наявність доступу до `Bash`.
- Фіксуйте версії через хеші комітів.

---

### 1.3 Обмеження permissions.deny

Налаштування `permissions.deny` у `.claude/settings.json` є офіційним методом блокування доступу до файлів.

**Рекомендована конфігурація**:
Блокуйте **всі** вектори доступу, а не лише `Read`:

```json
{
  "permissions": {
    "deny": [
      "Read(./.env*)",
      "Edit(./.env*)",
      "Write(./.env*)",
      "Bash(cat .env*)",
      "Read(./secrets/**)",
      "Read(./**/*.pem)"
    ]
  }
}
```

## Частина 2: Детекція (Під час роботи)

### 2.1 Виявлення промпт-ін'єкцій

ШІ-помічники вразливі до непрямих ін'єкцій через контекст коду. Зловмисники вбудовують інструкції у файли, які Claude читає автоматично.

**Методи обходу**:
- **Символи нульової ширини**: Інструкції, невидимі для людини (`U+200B`).
- **RTL override**: Зміна напрямку тексту (`U+202E`) для приховування команд.
- **Гомогліфи**: Використання кириличної "а" замість латинської "a".
- **Вкладені команди**: `$(evil_command)` всередині легітимних аргументів.

---

### 2.2 Налаштування стека хуків

Рекомендована конфігурація безпеки для `~/.claude/settings.json`:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": ["~/.claude/hooks/dangerous-actions-blocker.sh"]
      },
      {
        "matcher": "Edit|Write",
        "hooks": [
          "~/.claude/hooks/prompt-injection-detector.sh",
          "~/.claude/hooks/unicode-injection-scanner.sh"
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Bash",
        "hooks": ["~/.claude/hooks/output-secrets-scanner.sh"]
      }
    ]
  }
}
```

## Частина 3: Реагування (Коли щось пішло не так)

### 3.1 Витік секретів

1. **Негайно відкличте ключі** (AWS, GitHub, Stripe тощо).
2. **Перевірте масштаб витоку**: чи потрапив секрет у віддалений репозиторій (`git log origin/main..HEAD`).
3. **Перепишіть історію Git** (за потреби) за допомогою `git filter-repo`.
4. **Ротуйте ВСІ пов'язані облікові дані**.

---

### 3.2 Аварійний вимикач (Kill Switch) ШІ

| Рівень | Механізм | Коли використовувати |
|-------|----------------------|-------------|
| **1. Обмеження прав** | `permissions.deny`, хук блокування дій | Підозріла поведінка |
| **2. Контроль швидкості** | Хук `velocity-governor.sh` | Агент діє хаотично |
| **3. Повна зупинка** | `Ctrl+C`, вимкнення конфігу, видалення | Підтверджена компрометація |

## Частина 4: Безпека дистанційного керування (Remote Control)

Функція **Remote Control** (лютий 2026) дозволяє керувати локальною сесією Claude Code через браузер або телефон.

**Модель загроз**:
- **Витік URL сесії**: URL є фактичним паролем до вашого терміналу. Не діліться ним у Slack чи скріншотах.
- **RCE через віддалені команди**: Якщо зловмисник отримає URL, він зможе виконувати команди від вашого імені.
- **Корпоративні ризики**: Трафік проходить через реле Anthropic.

**Найкращі практики**:
- Активуйте лише за потреби, не вмикайте автозапуск.
- Закривайте сесію (`Ctrl+C`) відразу після завершення роботи.
- Не використовуйте на машинах з доступом до критичних секретів продакшену.
