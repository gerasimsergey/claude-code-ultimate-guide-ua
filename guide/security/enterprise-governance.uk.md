# Корпоративне управління ШІ для Claude Code

> **Аудиторія**: Техліди, менеджери з розробки, офіцери безпеки, які впроваджують Claude Code у командах.
>
> **Сфера**: Управління на рівні організації (політики, воркфлоу схвалення, рівні доступу, комплаєнс). Питання безпеки окремих розробників див. у [security-hardening.md](./security-hardening.md).

---

## Коротко (TL;DR)

**Прогалина в управлінні**: Документація з безпеки Claude Code зазвичай описує, що має робити окремий розробник. Вона не каже, що робити, коли інструментом користується вся організація (наприклад, 50 розробників) з різними профілями ризику та без єдиної політики.

**Що ви отримаєте в цьому посібнику**:
- Матрицю ризиків для локального та спільного використання.
- Шаблон Хартії (Статуту) використання ШІ.
- Воркфлоу управління MCP-серверами та реєстр у форматі YAML.
- 4 готові рівні обмежень (Guardrail Tiers), які можна скопіювати в `settings.json`.
- Поради щодо аудиту для SOC2 та ISO27001.

---

## 1. Локальне vs Спільне: Розподіл управління

Найбільша помилка в корпоративному управлінні ШІ — застосування однакових правил до всього. Локальне використання розробником та спільна робота над репозиторієм мають фундаментально різні ризики.

### 1.1 Матриця ризиків

| Вимір | Локальне використання | Спільне використання |
|-----------|-------------|--------------|
| **Дані** | Власні файли розробника | Дані клієнтів, спільний код, секрети |
| **Зона впливу** | Одна машина | Весь репозиторій, CI/CD, продакшн |
| **Відповідальність** | Індивідуальна | Команда / Організація |
| **Комплаєнс** | Зазвичай відсутній | SOC2, ISO27001, HIPAA |

## 2. Хартія використання ШІ

Хартія відповідає на головне запитання: "Що нам дозволено робити з Claude Code у цій компанії?".

### Шаблон хартії (Коротко):
- **Схвалені інструменти**: Claude Code (Pro/Team/Enterprise).
- **Класифікація даних**:
    - **PUBLIC**: Дозволено без обмежень.
    - **INTERNAL**: Дозволено зі стандартним конфігом.
    - **CONFIDENTIAL**: Дозволено лише для Enterprise плану.
    - **RESTRICTED** (PII, дані карток): ЗАБОРОНЕНО.
- **Заборонені кейси**: Автономний деплой без людини, обробка даних пацієнтів без рев'ю.

## 3. Управління MCP-серверами

Індивідуальна перевірка MCP важлива, але організація повинна мати централізований реєстр схвалених інструментів.

**Воркфлоу схвалення**:
1. Розробник подає запит на новий MCP.
2. Техлід проводить 5-хвилинний аудит безпеки.
3. MCP вноситься до реєстру `.claude/mcp-registry.yaml`.
4. Конфігурація розгортається через спільний `settings.json` у репозиторії.

---

## 4. Рівні обмежень (Guardrail Tiers)

Ми підготували 4 конфігурації залежно від ваших потреб:

### Рівень 1: Стартовий (Starter)
Для команд до 5 осіб, внутрішні проекти.
- Блокування `.env` та ключів.
- Базовий хук блокування небезпечних дій.

### Рівень 2: Стандартний (Standard)
Для команд 5-20 осіб, робота з кодом, близьким до продакшену.
- Блокування інфраструктурних файлів (Docker, Terraform).
- Детектор промпт-ін'єкцій.
- Сканер секретів у виводі.

### Рівень 3: Суворий (Strict)
Для команд 20+, критичні системи, дані клієнтів.
- Повне блокування команд встановлення пакетів (`npm install`).
- Блокування схем баз даних.
- Обов'язкове логування сесій.

### Рівень 4: Регульований (Regulated)
Для медицини, фінансів, HIPAA/SOC2/PCI.
- Повна заборона читання будь-яких файлів, що можуть містити PII.
- Обов'язкові "шлюзи схвалення" (approval gates) перед комітами.
- Криптографічне підтвердження авторства коду.

## 5. Впровадження в масштабі

**Головний принцип**: Конфігурація управління живе в репозиторії, а не на машинах розробників.
- Створіть окремий репозиторій `your-org-config`.
- Використовуйте скрипти онбордингу для автоматичного налаштування Claude Code при приєднанні до команди.
- Впроваджуйте перевірки в CI/CD, щоб блокувати PR з небезпечними змінами в конфігах безпеки.

## 6. Аудит та комплаєнс
