# Ізоляція в пісочниці для кодувальних агентів

> **Впевненість**: Рівень 2 — Офіційна документація Docker + верифіковані дані вендорів.
> **Час читання**: ~10 хвилин.

---

## Коротко (TL;DR)

| Рішення | Ізоляція | Локально/Хмара | Найкраще для |
|----------|-----------|-------------|----------|
| **Пісочниці Docker** | microVM (гіпервізор) | Локально | Макс. безпека, потрібен Docker-in-Docker |
| **Нативна пісочниця CC** | Процес (Seatbelt/bubblewrap) | Локально | Легка щоденна розробка, довірений код |
| **Fly.io Sprites** | Firecracker microVM | Хмара | API-керовані воркфлоу агентів |
| **E2B** | Firecracker microVM | Хмара | Мульти-фреймворк ШІ-додатки |
| **Vercel Sandboxes** | Firecracker microVM | Хмара | Екосистема Next.js / Vercel |

## 1. Проблема: Безпечна автономія

Система дозволів Claude Code захищає вас, але створює дилему:
- **`--dangerously-skip-permissions`** прибирає всі обмеження — Claude може виконати `rm -rf` або `DROP TABLE` без запиту. На основній хост-машині це небезпечно.
- **Втома від дозволів** — підтвердження кожної дії уповільнює автономну роботу.

**Рішення**: Ізолюйте середовище виконання. Нехай агент вільно працює всередині пісочниці, де "зона впливу" обмежена. Пісочниця стає межею безпеки, а не система дозволів.

---

## 2. Пісочниці Docker (Docker Sandboxes)

> **Вимоги**: Docker Desktop 4.58+ (macOS або Windows).

Пісочниці Docker запускають агентів у ізоляції на базі microVM. Кожна пісочниця має власний приватний демон Docker та файлову систему. Вони **не відображаються** у `docker ps`, оскільки це віртуальні машини, а не контейнери.

### Швидкий старт
```bash
# Створити та запустити пісочницю з вашим проектом
docker sandbox run claude ~/my-project

# Запустити в автономному режимі (безпечно всередині пісочниці)
docker sandbox run claude ~/my-project -- --dangerously-skip-permissions
```

**Ключові властивості**:
- **Повна ізоляція**: Агент не має доступу до хост-демона Docker або файлів поза робочою директорією.
- **Приватний Docker**: Кожна пісочниця може збирати та запускати власні контейнери.

---

## 3. Нативна пісочниця Claude Code

Claude Code v2.1.0+ включає вбудовану пісочницю, що використовує примітиви ОС для ізоляції на рівні процесів. Docker не потрібен.

**Примітиви ОС**:
- **macOS**: Використовує **Seatbelt** (ядерна фільтрація системних викликів). Працює "з коробки".
- **Linux/WSL2**: Використовує **bubblewrap** (namespaces + seccomp). Потребує встановлення: `sudo apt-get install bubblewrap socat`.

### Порівняння рівнів ізоляції

| Аспект | Нативна пісочниця | Пісочниці Docker |
|--------|---------------|------------------|
| **Рівень ізоляції** | Процес | microVM (гіпервізор) |
| **Ядро** | Спільне з хостом | Окреме ядро для кожної VM |
| **Накладні витрати** | Мінімальні (~1-3% CPU) | Помірні (~5-10% CPU) |
| **Docker-in-Docker** | ❌ Не підтримується | ✅ Приватний демон Docker |

## 4. Патерн: Docker Sandbox + DSP

Рекомендований підхід для локальної автономної розробки:
1. **Створіть пісочницю**: `docker sandbox create my-feature ~/my-project`.
2. **Налаштуйте мережу**: дозвольте лише `api.anthropic.com` та реєстри пакетів.
3. **Запустіть Claude з DSP**: `docker sandbox run my-feature -- --dangerously-skip-permissions "Рефакторинг модуля..."`.
4. **Перевірте зміни на хості**: робоча директорія синхронізується автоматично.

---

## 5. Антипатерни

- **DSP без пісочниці**: Агент отримує необмежений доступ до вашої реальної машини.
- **Монтування всієї файлової системи**: Це нівелює сенс ізоляції. Монтуйте лише папку проекту.
- **Пропуск `git diff`**: Ізоляція захищає хост, але не гарантує якість коду. Завжди перевіряйте зміни перед комітом.
